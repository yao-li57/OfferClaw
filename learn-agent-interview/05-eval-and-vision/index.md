# 评测与前沿洞见

## Q：Agent 系统应该用哪些指标来评估？

> 来源：AI 产品工程师 + 研究员联合面试

**新手答**："看模型的准确率和响应速度就行。"

**高手答**：

1. **任务级指标（最重要）**：Task Success Rate（任务完成率，用户目标是否实现）是核心指标，比任何模型层面的指标更直接反映业务价值。按任务类型分层统计（简单/复杂），整体成功率会掩盖子类问题。

2. **效率指标**：Turn Efficiency（完成任务需要的平均轮数）、Token Efficiency（每任务消耗的 token 数）。效率指标反映 Agent 的"聪明程度"——聪明的 Agent 用更少的交互、更少的资源完成任务。

3. **可靠性指标**：Error Rate（报错率）、Fallback Rate（触发降级的频率）、Tool Call Success Rate（工具调用成功率）。可靠性问题会严重损害用户信任，比功能缺失更危险。

4. **用户体验指标**：TTFT（Time-to-First-Token，用户感知到系统"开始响应"的时间）、Session Abandonment Rate（用户中途放弃的比例）。这些指标捕捉到模型指标看不到的用户满意度信号。

5. **成本指标**：Cost per Task（每任务平均成本）、Cost per Session。AI 产品的单位成本直接决定商业可行性，要和成功率一起看（高成功率+高成本 vs 低成功率+低成本的权衡）。

**差距在哪**：新手用模型 Benchmark 评估 Agent，高手用任务完成率+效率+可靠性+成本的多维框架，因为 Benchmark 反映模型能力，但不反映系统在真实场景的业务价值。

## Q：LLM 评测有哪些核心难点？

> 来源：AI 研究工程师面试

**新手答**："准确率不好算，因为答案有很多种表达方式。"

**高手答**：

1. **主观性问题**：大量 LLM 任务（写作质量、回答有用性）没有客观正确答案，评测结果受评估者主观影响大。标注者间一致性（Inter-annotator Agreement）往往比想象中低，<0.7 的 kappa 系数很常见。

2. **输出多样性**：同一含义的不同表达方式都是正确的，传统 BLEU/ROUGE 等指标严重低估模型能力（因为它们是 n-gram 匹配，与参考答案不同就扣分）。

3. **幻觉的难测性**：事实性幻觉很难自动检测，需要对每个声明做知识库验证，规模化成本极高。简单的"LLM 评判是否正确"本身也可能出错。

4. **Benchmark 污染**：模型训练数据可能包含 Benchmark 的测试集，导致 Benchmark 高分不代表真实能力提升（Goodhart's Law：当一个指标变成目标，它就不再是一个好指标）。

5. **分布偏移**：Benchmark 题目固定，但真实用户的问题分布持续变化。高 Benchmark 分数不等于好的生产体验，需要持续收集生产数据做评测，而不只依赖 Benchmark。

**差距在哪**：新手认为 LLM 评测难只是因为"答案多样"，高手知道这是一个系统性难题——主观性、污染、分布偏移，每个维度都需要专门的工程策略来缓解。

## Q：什么是 Goodhart's Law？它如何影响 AI 产品的评测设计？

> 来源：AI 产品设计与工程面试

**新手答**："没听说过。"

**高手答**：

1. **Goodhart's Law 原文**："When a measure becomes a target, it ceases to be a good measure."（当一个度量变成目标，它就不再是一个好度量。）

2. **AI 场景的体现**：针对 MMLU 优化的模型，MMLU 分数高但真实场景能力未必好；针对用户满意度评分优化的 RLHF，模型学会了"讨好用户"（verbose、假装自信），而非真正有用。

3. **对评测设计的影响**：不要把评测指标作为优化的唯一目标，要维护一个"元指标"——真实业务价值的代理指标（如用户续费率、任务真实完成率），定期检验评测指标和业务价值是否还对齐。

4. **防污染策略**：评测集不能公开，不能参与模型训练，要定期更新，防止被针对性优化。类似 holdout test set 的原则。

5. **多指标防对冲**：单一指标容易被游戏（Gaming），用互相制约的多指标组合评估（如同时考察准确率+幻觉率+拒绝率+成本），让优化一个指标的同时不能过度牺牲其他指标。

**差距在哪**：新手建立评测就是为了"跑出好成绩"，高手知道评测体系本身需要被设计为"不容易被游戏"，否则评测好但产品差是 AI 行业的常见陷阱。

## Q：如何在生产环境下设计 LLM 功能的 A/B 测试？

> 来源：AI 产品工程师面试

**新手答**："分一半流量用新模型，看哪个好。"

**高手答**：

1. **实验单元选择**：按 user_id 而非 request_id 分组，确保同一用户在整个实验周期内只看到一个版本，避免同一用户先看到 A 再看到 B 产生学习效应污染实验。

2. **统计显著性**：LLM 评测指标方差很大（不同任务的得分差异巨大），需要足够大的样本量才能达到统计显著性。建议运行 2 周以上，使用 power analysis 预计算所需样本量，避免早停（underpowered）实验。

3. **护栏指标（Guardrail Metrics）**：除了主要实验指标（任务成功率），要设置护栏指标——如果实验组的错误率、幻觉率、拒绝率超过阈值，自动停止实验，防止在追求提升主指标时破坏基础质量。

4. **多指标同时追踪**：LLM 改动经常是"得此失彼"——回答更详细了但 token 成本上升了，用户满意度提高了但幻觉率也上升了。A/B 测试报告必须展示所有关键指标的变化，而不是只展示改善了的指标。

5. **用户反馈信号**：除了自动指标，要收集隐式用户行为信号（如：用户在 AI 回答后是否继续追问同一问题，追问可能意味着回答不满意）作为辅助指标，捕捉自动指标无法覆盖的用户体验维度。

**差距在哪**：新手把 A/B 测试当简单的"分流然后看数字"，高手知道实验单元选择、样本量计算、护栏指标、多指标权衡是保证实验结果可信的工程基础。

## Q：LLM-as-judge 如何设计？有哪些主要偏差？

> 来源：AI 评测工程师面试

**新手答**："让 GPT-4 打分，分高说明好。"

**高手答**：

1. **LLM-as-judge 的优势**：比人工评估便宜 100 倍以上，可以按维度给出结构化评分（准确性 / 相关性 / 流畅性），评分速度快，适合大规模评测。

2. **Position Bias（位置偏差）**：当让 LLM 比较两个答案时，模型倾向于给位置靠前（第一个）的答案更高分。解决方案：对每对比较做两次（A/B 和 B/A），取平均，消除位置偏见。

3. **Length Bias（长度偏差）**：LLM 裁判倾向于给更长的答案打高分，即使长答案并不更好（只是更 verbose）。需要在评分 Prompt 中明确指出"长度不是优点，准确性和简洁性才是"。

4. **Self-Preference Bias**：GPT-4 倾向于给 GPT-4 的输出打更高分，Claude 倾向于给 Claude 的输出打更高分。评估模型要尽量选择中立的，或者用多个不同模型的平均分。

5. **评分 Prompt 的设计**：参考 Rubric（评分标准）要具体——不要只说"评估回答质量（1-10分）"，而是给出每个分段的具体标准（9-10分：完整、无错误、有具体示例；7-8分：基本正确但缺乏深度；...）。

**差距在哪**：新手把 LLM-as-judge 当免费的高质量评估，高手知道它有系统性偏差，设计时要用位置随机化、长度控制、多模型平均、具体 Rubric 来减少偏差。

## Q：如何对 Agent 的执行 trace 做离线分析？

> 来源：AI 质量工程师面试

**新手答**："看一下日志，找问题在哪。"

**高手答**：

1. **Trace 结构化存储**：每条 trace 包含：session_id、完整 message history、每轮的 tool calls + results、最终输出、token 消耗、耗时。以 JSON 格式存入数据库，支持按维度查询。

2. **失败案例挖掘**：定期查询失败的 trace（task_success=false 或用户给了负反馈），手动分析失败原因分布——是工具选择错误、LLM 理解偏差、还是任务本身不可完成？这是持续改进的信息来源。

3. **Trace Replay（轨迹回放）**：能够复现某条 trace——用相同的输入和 context 重跑 Agent，验证问题是否可复现（区分偶发性错误和系统性 bug），以及修复后是否解决了问题。

4. **行为漂移检测**：定期对比本周和上周的 Trace 分布——工具调用分布是否变化？平均 token 消耗是否上升？某类任务的成功率是否下降？自动检测模型或 Prompt 变化带来的行为漂移。

5. **高价值 case 提取**：从 trace 中提取高质量成功案例，作为 few-shot 示例加入 Prompt 或 fine-tuning 数据集；提取典型失败案例，构建评测集的 hard cases。

**差距在哪**：新手的 trace 分析是"出事了翻日志"，高手把 trace 作为持续改进的数据飞轮——定期挖掘、系统分析、反哺 Prompt 和模型训练，让每次失败都变成系统提升的素材。

## Q：当前 LLM 在 Agent 场景的核心能力局限是什么？

> 来源：AI 前沿洞见面试（考察技术判断力）

**新手答**："有时候会编造事实，不够准确。"

**高手答**：

1. **长程规划（Long-horizon Planning）**：当任务需要 20+ 步骤时，LLM 难以维持一致的计划，中途容易偏离原始目标或丢失早期约束。这不是 context 长度问题，而是 LLM 本质上是"下一个 token 预测"而非"目标导向规划"。

2. **精确数学和逻辑推理**：即使是最强的模型，在复杂数学（>5 步推导）和严格逻辑（多重嵌套条件）上的可靠性仍然不足，且错误经常发生在看似简单的步骤上。Code interpreter 是当前的工程 workaround。

3. **工作记忆的局限**：LLM 在一次推理过程中需要同时追踪多个变量时会出错（如在一段推理中维护 10 个约束条件）。更长的 context 不能完全解决这个问题——信息在 context 里存在，但模型不一定能"同时使用"所有信息。

4. **自我校正的局限**：让 LLM 检查自己输出中的错误，效果有限且不稳定。模型更容易确认而非纠正自己的错误（self-confirmation bias），Critic Agent 用另一个 LLM 比让同一 LLM 自我批评效果好。

5. **工具选择准确率**：当工具数量超过 20 个时，LLM 的工具选择准确率显著下降，容易选错工具或遗忘某些工具的存在。这是实际 Agent 系统工具数量受限的根本原因。

**差距在哪**：新手用"幻觉"一词概括所有 LLM 问题，高手能清晰区分规划、推理、记忆、自我校正、工具选择各维度的局限，这是做出正确工程架构决策的前提。

## Q：如何为 RAG 系统建立评测体系？

> 来源：RAG 工程师面试

**新手答**："看回答准不准确就行了。"

**高手答**：

1. **四个核心维度**（RAGAS 框架）：**Faithfulness**（答案是否忠实于检索内容，无幻觉）、**Answer Relevance**（答案是否回答了问题）、**Context Recall**（检索是否找回了必要信息）、**Context Precision**（检索到的内容中有多少是相关的）。

2. **分层评测**：检索层指标（Recall@K、Precision@K、MRR，评估检索本身的质量）和生成层指标（Faithfulness、Answer Correctness，评估 LLM 利用检索结果的质量）分开评测，才能定位是检索问题还是生成问题。

3. **Golden Set 的构建**：人工标注 100-500 条（query, relevant_documents, expected_answer）三元组，作为离线评测集。关键是 coverage——要覆盖高频查询类型和 hard case（如多跳查询、需要推断的问题）。

4. **在线评测信号**：用户给了负反馈（踩 / 反驳 / 继续追问）的对话是潜在的质量问题信号，定期采样分析，作为离线评测集的补充。

5. **评测频率**：知识库更新后、检索策略变更后、生成模型升级后，都要重跑评测集，验证没有质量回归。不能只在上线前测，而不在变更后测。

**差距在哪**：新手把 RAG 评测等同于 QA 评测（答案对不对），高手知道 RAG 要分检索和生成两层分别评测，才能定位问题、针对性改进。

## Q：Long Context 和 RAG 如何选型？

> 来源：AI 系统架构设计面试

**新手答**："文档短用 Long Context，文档多用 RAG。"

**高手答**：

1. **核心区别**：Long Context 把所有文档塞进 context（暴力注入），RAG 先检索相关片段再注入（精准注入）。前者无遗漏但成本极高，后者低成本但有召回失败风险。

2. **选 Long Context 的场景**：文档总量 < 100k tokens，且需要 LLM 在全文中做跨段推理（如理解文档整体结构、寻找多个段落之间的关联）。文档量少、推理要求跨全文时 Long Context 优势明显。

3. **选 RAG 的场景**：知识库大（>100k tokens）、成本敏感、查询可以通过局部片段回答（大多数问答场景）。RAG 适合"用户问了特定问题，找相关片段回答"的模式。

4. **Lost in the Middle 的权衡**：Long Context 虽然把全文塞进去了，但 LLM 对中间部分的注意力较低，相关信息在文档中间时反而可能被 RAG 的精准检索表现更好。

5. **混合方案**：对于复杂场景，可以先用 RAG 粗筛（召回 10-20 个 chunk），再把这些 chunk 全部注入 context（总量通常 < 20k tokens），兼顾检索精度和推理完整性。

**差距在哪**：新手用文档数量做判断标准，高手知道选型要综合考虑推理需求（需不需要全文上下文）、成本约束（token 预算）、和检索质量（能不能可靠地找到相关片段）三个维度。

## Q：如何量化 Prompt 改动的效果？

> 来源：AI 工程效能面试

**新手答**："改了之后测几条，感觉好了就行。"

**高手答**：

1. **改动前定指标**：在改 Prompt 之前，明确"这次改动要提升什么指标"（如 JSON 解析成功率、用户满意度分）。没有预设指标就无法评估是否成功。

2. **Golden Set 基准测试**：维护一套有 ground truth 的测试集，每次 Prompt 改动都跑完整测试集，和上次版本做数值对比，而不是凭感觉抽测。

3. **Regression Test（回归测试）**：关注提升的同时，检测是否在其他维度退化——Prompt 改动带来"副作用"是非常常见的（在简单 case 上改善但在 edge case 上退化）。分层统计不可省略。

4. **成本/质量 Pareto**：记录改动前后的平均 token 消耗，有时候 5% 的质量提升伴随 50% 的成本上升，这个权衡需要明确决策。

5. **统计显著性**：测试集要足够大（通常 100+ 条），效果改善需要超过噪声水平（不能只比较平均分，要做假设检验或至少看置信区间）。

**差距在哪**：新手凭感觉评估 Prompt 改动，高手把它当软件功能改动来对待——有测试集、有基准、有回归检测、有成本分析，让每次迭代都可测量、可追溯。

## Q：GPT-4o / Claude / Gemini 从工程角度有什么核心差异？

> 来源：多模型 AI 平台工程师面试

**新手答**："能力差不多，看定价选就行了。"

**高手答**：

1. **Tool Calling 实现差异**：OpenAI 的 function call 参数是字符串 JSON（需要解析），Anthropic 的 tool_use 直接返回 object（无需解析）；并行 tool call 的格式也不同，需要 Provider 抽象层统一。

2. **流式输出格式差异**：三家的 SSE event 格式完全不同，tool call 在流式中的拼接方式各异（OpenAI 在多个 delta 中分散 tool_call 字段，Anthropic 有独立的 content_block 类型），统一处理需要 Provider 适配层。

3. **context window 与成本**：Claude 3.5 Sonnet context 200k tokens，GPT-4o 128k，Gemini 1.5 Pro 1M tokens，但大 context 的成本差异显著。生产中要评估 actual 使用的 context 大小，而非只看最大值。

4. **Prompt 敏感性差异**：Claude 对 System Prompt 的遵循更严格，在角色扮演场景表现更好；GPT-4 在代码生成和工具调用上更稳定；Gemini 在多语言场景上更强。选模型要根据具体任务类型。

5. **Speed/Cost Profile**：GPT-4o mini 和 Claude Haiku 是高速低成本选项（适合实时交互），GPT-4o 和 Claude Sonnet 是平衡选项，o1 和 Claude Opus 是高质量慢速选项（适合深度推理任务）。

**差距在哪**：新手看 Benchmark 选模型，高手看工程特性（Tool Calling 格式、流式结构、成本曲线、任务适配性），这些工程细节决定了集成的复杂度和生产稳定性。

## Q：什么是 Evals 驱动开发？如何在实践中落地？

> 来源：AI 工程实践面试

**新手答**："测试驱动开发的 AI 版本？"

**高手答**：

1. **核心理念**：在改动 Prompt / 模型 / RAG 策略之前，先建立评测基准，改动后立即运行评测，用数据而非主观感受决策是否上线。把 eval 放在迭代循环的最内层。

2. **Eval Set 的构建**：从生产数据中挑选代表性 case（正常 case + 边界 case + 已知失败 case），标注 ground truth 或评估维度，形成 golden dataset。这是 Evals 驱动的基础设施。

3. **CI/CD 集成**：把 eval 加入 CI pipeline，每次 Prompt 变更自动触发评测，如果关键指标低于阈值则拒绝合并，防止"感觉好了"但实际指标回归的改动上线。

4. **eval 的层级**：Unit eval（单个 Prompt 功能的快速测试）→ Integration eval（完整对话链路的测试）→ A/B eval（生产流量的真实效果）。三层 eval 覆盖不同粒度的质量保障。

5. **Eval 本身的维护**：eval 集会随业务发展过时，需要定期更新（增加新场景 case，删除过时 case），并且 eval 的 ground truth 本身也需要版本管理。

**差距在哪**：新手凭人工感觉迭代，高手用数据驱动迭代——Evals 是 LLM 工程的"测试套件"，没有它就像盲目重构代码，无法保证改动不引入回归。

## Q：推理模型（o1/DeepSeek-R1 类）在 Agent 场景有哪些工程影响？

> 来源：前沿 AI 工程师面试

**新手答**："推理模型更聪明，用它代替原来的模型就好了。"

**高手答**：

1. **显著的延迟代价**：推理模型在输出可见 token 之前有内部思考过程（几十秒），TTFT 极高，用户体验差。适合后台异步任务，不适合实时交互场景。

2. **Thinking Token 的成本**：推理过程的 thinking tokens 也计入成本（且通常是计费的），一个复杂问题的总 token 消耗可能是普通模型的 10-20 倍。要在质量收益和成本之间做 ROI 计算。

3. **工具调用的稳定性**：推理模型的 tool calling 行为有时不如非推理模型稳定（可能在 thinking 过程中过度思考而导致 tool call 参数异常），需要在生产中充分测试。

4. **适用任务类型**：推理模型在需要多步推理的任务（数学、代码 debug、战略规划）上有显著提升，在简单 QA 和格式化输出任务上的优势不明显。不要对所有任务都换推理模型。

5. **Hybrid 策略**：生产中可以使用"轻量路由"——先用普通模型处理，当检测到任务复杂度超过阈值（如需要多步推理）时，切换到推理模型。这比全量使用推理模型成本低很多。

**差距在哪**：新手看到推理模型更强就想全面替换，高手知道推理模型在延迟、成本、适用场景上都有明确局限，要用 Hybrid 策略在合适的任务上使用合适的模型。

## Q：如何保持对 AI 领域最新进展的跟踪？

> 来源：AI 工程师面试（考察学习方式和视野）

**新手答**："关注几个 AI 博主，看微信推文。"

**高手答**：

1. **信源分层**：一手信源（ArXiv 每日更新、官方 Blog 如 OpenAI/Anthropic/Google Research）→ 二手信源（Hugging Face Papers、AI Digest）→ 社区讨论（Twitter/X AI 研究社区，LessWrong）。信源要分层，不能只看聚合内容。

2. **论文的实用阅读法**：对于 arXiv 新论文，先看 Abstract + Conclusion + Figure（5 分钟），判断是否值得深读；值得深读的才读方法论和实验结果。每周精读 2-3 篇 > 粗读 20 篇。

3. **工程实现跟进**：学术论文到工程落地有巨大 gap。关注 Hugging Face 的模型发布（有可直接使用的实现）、GitHub 上高 star 的新 repo、工程博客（如 Cloudflare、Stripe 等的 AI 工程实践）。

4. **项目驱动学习**：把新技术用在真实项目中（如这个项目的知识库、向量检索、Agent 架构）才能真正理解它的工程价值和局限，而不只是停留在概念理解。

5. **建立知识体系**：用个人 Wiki 或笔记系统记录关键洞见，建立概念之间的联系（而不是孤立的技术点），这让新知识更容易被类比理解，也更容易在面试中清晰表达。

**差距在哪**：新手依赖二手信息（推文、博主解读），高手建立主动的信息获取体系——直接读一手源、分层筛选、工程验证，把新知识转化为系统性理解而非碎片化追热点。

## Q：你认为当前 LLM 应用中最被高估和最被低估的方向各是什么？

> 来源：高级 AI 工程师/架构师面试（考察技术判断力和独立思考）

**新手答**："LLM 能做很多事，我觉得都挺有价值的。"

**高手答**：（以下是参考思路，面试时应表达自己的真实判断）

1. **被高估的方向（示例）**：全自动 Agentic 工作流（No Human in the Loop）——目前 LLM 的可靠性和推理能力还不足以支撑长链路无监督自动化，在高风险场景（代码部署、财务决策）中全自动是危险的。短期内 Human-in-the-Loop 是必须的，不是可选的。

2. **被低估的方向（示例）**：受限领域的微调小模型——很多企业场景不需要 GPT-4 级别的通用能力，一个 7B 参数的微调模型在特定领域可以达到 GPT-4 90% 的效果，成本低 100 倍，延迟低 10 倍，私有化部署更安全。

3. **回答框架**：给出具体判断 + 技术依据（为什么高估/低估）+ 你在实际项目中的观察/验证，而不是泛泛而谈。面试官要看的是你的技术判断力和独立思考，而不是"正确答案"。

4. **避免套话**：不要说"AI 会取代程序员"或"AI 只是工具"这类无意义的极端。面试官希望看到你对具体技术方向的细粒度判断，有时间窗口感（1-2 年 vs 5 年），有工程落地视角。

5. **展示信息来源**：引用你读过的论文、亲自验证过的实验、或在项目中亲身经历的案例，让判断有根基。

**差距在哪**：新手给不出有具体依据的技术判断，高手能清晰表达自己的技术视角——既有明确的观点（不回避），又有支撑判断的证据（不武断），这正是高级工程师和研究员的核心素质。

## Q：如何检测生产中的模型行为退化？

> 来源：AI 平台 SRE 面试

**新手答**："出了问题用户会反馈，再查日志就行了。"

**高手答**：

1. **基准 eval 定期自动运行**：每天（或每次部署后）对 golden set 自动运行评测，对比昨日分数，超过阈值的下降立即告警。不要等用户投诉才发现质量问题。

2. **用户行为信号监控**：追踪"连续追问同一问题"（暗示回答没解决问题）、"用户打了负反馈"、"对话在 3 轮内放弃"的比率变化。这些行为信号往往比自动指标更早反映质量变化。

3. **输出分布监控**：分析 LLM 输出的统计特征——平均长度、特定词汇/短语的使用频率、拒绝率变化。当这些分布发生突变时，通常意味着模型行为有变化（如 Provider 更新了底层模型版本）。

4. **错误模式聚类**：把一段时间内的失败 case 做聚类（用 embedding 聚类相似错误），发现新出现的错误类型。新错误类型的出现往往是退化的信号，比监控总体错误率更早发现方向性问题。

5. **Canary 对比**：维护一个 "shadow" 评测流量（占真实流量的 1%），实时对比 shadow 输出和 golden answers 的相似度。持续下降说明生产模型行为在偏离期望。

**差距在哪**：新手的监控是被动的（等用户报告），高手的监控是主动且多层的——自动 eval + 行为信号 + 分布监控 + 错误聚类，在问题影响大量用户前就发现和响应退化。

## Q：开源模型 vs 闭源 API 的工程权衡是什么？

> 来源：AI 技术选型架构面试

**新手答**："闭源更好用，开源更便宜，主要看预算。"

**高手答**：

1. **能力边界**：2024 年以来，开源模型（Llama 3.1 70B、Qwen 2.5、DeepSeek）已经接近 GPT-4 Turbo 的能力水平，差距主要在复杂推理和长文本理解。对于大多数生产任务，能力差距不是主要决策因素。

2. **数据隐私与合规**：涉及用户敏感数据（医疗、法律、金融）的场景，不能把数据发送到第三方 API，必须自托管开源模型。这是 Hard Constraint，不是选择题。

3. **运维成本**：闭源 API 的工程成本极低（调用即用，无需维护基础设施）；自托管开源模型需要 GPU 服务器、模型管理、推理优化（量化/批处理/KV Cache）、版本升级，这些都是真实的工程成本。

4. **延迟和定制化**：自托管开源模型可以针对 hardware 做 inference 优化（vLLM + 量化），P50 延迟可以低于闭源 API；同时可以做 fine-tuning 深度定制，而闭源 API 的定制化选项有限。

5. **决策框架**：数据隐私要求强 or 高流量需要最低 token 成本 → 开源自托管；快速迭代、团队小、工程资源有限 → 闭源 API；高质量需求 + 预算充足 → 高端闭源；领域专精 + 愿意投入 fine-tuning → 开源微调。

**差距在哪**：新手用"便宜贵"二分法，高手用数据隐私、运维能力、延迟需求、定制化需求、团队规模多维度判断，选型是系统工程决策而非简单的成本比较。

## Q：如何设计 Evals 来评估 Agent 的规划能力？

> 来源：Agent 能力评测工程面试

**新手答**："看 Agent 最终有没有完成任务就行了。"

**高手答**：

1. **子目标完成率**：把任务分解为若干子目标（milestone），评测 Agent 在每个子目标的完成率，而不只是最终结果。这能区分"整体失败但中间做得不错"和"随机成功"。

2. **计划质量评估**：在 Agent 开始执行前，让它描述计划，评测计划的合理性（是否包含必要步骤、顺序是否正确、是否考虑了 edge case）。计划质量往往能预测任务完成率。

3. **轨迹效率评估**：比较 Agent 的实际执行路径和最优路径（Oracle）的差异——多用了几次工具、走了哪些弯路。效率低的 Agent 即使最终成功，也说明规划能力有提升空间。

4. **鲁棒性测试**：在任务中途注入干扰（错误工具结果、资源不可用），测试 Agent 能否调整计划、优雅恢复，而不是卡住或放弃。规划能力强的 Agent 应该能够重规划（Re-planning）。

5. **对比基准**：和 chain-of-thought baseline（只用 LLM 不用工具规划）以及 human planning（人工规划最优路径）做对比，量化 Agent 规划的实际价值。

**差距在哪**：新手用最终成功率代替规划能力评估，高手知道规划能力需要专门的评测维度——子目标跟踪、计划质量、执行效率、鲁棒性，每个维度都反映不同方面的规划能力。
