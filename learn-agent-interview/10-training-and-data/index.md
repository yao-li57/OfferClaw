# 训练与数据工程

## Q：什么时候应该 Fine-tuning 而不是 Prompt Engineering？

> 来源：AI 技术选型面试

**新手答**："模型效果不好就去 Fine-tuning。"

**高手答**：

1. **先穷尽 Prompt Engineering**：Fine-tuning 成本高、周期长、需要标注数据。Prompt Engineering 通常能解决 70-80% 的效果问题，且迭代速度快、成本低。先确认 Prompt Engineering 已经无法进一步提升，再考虑 Fine-tuning。

2. **适合 Fine-tuning 的场景**：①需要改变模型的输出风格/格式（让模型始终以特定 JSON 格式输出）；②特定领域专业知识的内化（减少 RAG 检索开销）；③减少 Prompt 中的重复示例（把 100 条 few-shot 内化成模型本身）；④推理速度和成本（小模型微调后可以替代大模型）。

3. **不适合 Fine-tuning 的场景**：知识动态更新（微调模型的知识截止于训练数据）；需要引用来源（微调不能追溯答案来源）；希望模型知道最新信息（Fine-tuning 不是实时的）。

4. **数据量要求**：SFT 通常需要 1000-10000 条高质量数据对才有明显效果，数量不足的 Fine-tuning 比高质量 Prompt + RAG 效果差。

5. **成本比较**：OpenAI Fine-tuning 的调用成本高于基础模型（约 1.5-2 倍），需要计算 Fine-tuning 在成本和质量上是否都优于 Prompt Engineering + 更大模型。

**差距在哪**：新手把 Fine-tuning 当解决模型不够好的万能方案，高手知道它的适用条件、数据需求和成本代价，会先确认 Prompt 优化空间是否已经用尽，再投入 Fine-tuning。

## Q：SFT（监督微调）需要什么质量的数据？多少条够用？

> 来源：LLM 微调工程面试

**新手答**："数据越多越好，质量差不多就行。"

**高手答**：

1. **质量远比数量重要**：Stanford Alpaca 用 52k 条数据 Fine-tuning LLaMA，而后来的研究（LIMA）显示 1000 条精心挑选的数据可以达到甚至超过 52k 条随机数据的效果（Less is More for Alignment）。

2. **数据质量标准**：①Task Diversity（覆盖目标任务的各种形态）；②Response Quality（答案要有深度，不是泛泛而谈）；③Format Consistency（输出格式要一致，模型才能学会格式）；④No Noise（过滤错误、矛盾、低质答案）。

3. **数量经验参考**：新任务入门 100-500 条可以验证效果；生产级 SFT 通常需要 1000-10000 条；超过 10000 条之后边际收益递减，重点转向质量提升而非数量增加。

4. **Task Diversity 的重要性**：1000 条多样化数据 > 10000 条同质数据。过于集中的数据会导致模型在其他子任务上退化（Catastrophic Forgetting 的数据层面体现）。

5. **数据来源**：生产系统中收集的真实用户交互（经人工筛选高质量部分）是最有价值的数据；其次是专家标注；最后才是 LLM 生成 + 人工验证（合成数据）。

**差距在哪**：新手认为数量是关键，高手知道数据质量和多样性比数量更重要，LIMA 的"1000 条足矣"是对"更多数据 = 更好"直觉的有力反驳，是面试中展示前沿认知的好素材。

## Q：RLHF 的完整流程是什么？核心挑战在哪里？

> 来源：LLM 对齐工程研究面试

**新手答**："让模型根据人类反馈学习，效果更好。"

**高手答**：

1. **三阶段流程**：①SFT（Supervised Fine-tuning）：用高质量 instruction-response 对做初步微调；②RM（Reward Model）训练：对同一 prompt 的多个响应做人类偏好排名（A > B），训练奖励模型预测哪个响应更好；③PPO（Proximal Policy Optimization）：用奖励模型的打分信号对语言模型做强化学习，最大化奖励分数。

2. **核心挑战一：奖励 Hacking**：模型学会了最大化 Reward Model 的分数，而不是真正的人类偏好。例如学会产生"听起来好但内容空洞"的回答（Reward Model 打高分，但实际没用），或极端地输出过长的响应（Reward Model 对长度有偏好）。

3. **核心挑战二：数据成本**：高质量偏好数据的收集极昂贵——需要专业标注员对成千上万个输出对做排名，成本是 SFT 数据的 5-10 倍以上。

4. **核心挑战三：KL Divergence 控制**：PPO 训练时用 KL penalty 防止模型偏离 SFT 初始版本太远，但 KL 权重的调参非常敏感——太大导致学不到新行为，太小导致过度偏移崩溃。

5. **工程复杂度**：RLHF 需要维护多个模型（SFT 模型、Reward 模型、PPO Actor、Reference 模型），训练过程复杂、不稳定，工程团队的实现和维护成本极高。

**差距在哪**：新手认为 RLHF 是"加了人类反馈就更好"，高手知道奖励 Hacking、数据成本、训练不稳定是三个核心工程挑战，也是 DPO 诞生的背景。

## Q：DPO（Direct Preference Optimization）和 RLHF 有什么区别？

> 来源：LLM 对齐工程面试

**新手答**："DPO 比 RLHF 简单，效果差不多。"

**高手答**：

1. **核心差异**：RLHF 需要先训练独立的 Reward Model，再用 RL 算法（PPO）优化；DPO 直接从偏好数据（chosen vs rejected 对）通过数学推导，把偏好学习变成一个分类问题，绕过了显式的 Reward Model 和 RL 训练。

2. **工程优势**：DPO 训练稳定（无 PPO 的 KL 不稳定问题）、只需要标准的监督学习框架（无需 RL 基础设施）、不需要在训练时保持多个模型副本。实现难度和计算成本都低于 RLHF。

3. **数据格式**：DPO 的数据是三元组 `(prompt, chosen_response, rejected_response)`，比 RLHF 的排名数据更容易收集（只需要二选一，而不是多个排名）。

4. **性能对比**：研究表明 DPO 在大多数对齐任务上与 RLHF（PPO）性能相当，在某些任务上略差（尤其是需要在线数据的任务）。但实现复杂度和成本的优势让 DPO 成为了工程实践中的主流选择。

5. **DPO 的变体**：IPO、KTO、SimPO 等 DPO 的改进变体进一步降低了偏好数据的质量要求（KTO 甚至不需要对比对，只需要标注好/不好）。

**差距在哪**：新手知道 DPO 比 RLHF 简单但说不出为什么，高手能清晰解释 DPO 绕过 Reward Model 的数学直觉、工程优势和性能对比，展示对 LLM 对齐领域的深入理解。

## Q：LoRA 的数学原理是什么？适用场景是什么？

> 来源：LLM 高效微调工程面试

**新手答**："LoRA 是低秩分解，只训练部分参数，省显存。"

**高手答**：

1. **数学核心**：原始权重矩阵 W（d×d），Full Fine-tuning 要更新 W，ΔW 有 d² 个参数；LoRA 假设 ΔW 是低秩的，用两个小矩阵 A（d×r）和 B（r×d）的乘积来近似（BA≈ΔW），其中 r << d。训练时只训练 A 和 B，参数量从 d² 降到 2dr。

2. **rank（r）的选择**：r 越大，拟合能力越强但参数量越多；r 越小，正则化更强但可能欠拟合。经验值：简单任务 r=4-8；复杂任务 r=16-64；通常先从 r=8 开始，根据评测结果调整。

3. **适用层的选择**：原论文建议只对 Attention 层（Q/K/V 矩阵）应用 LoRA，实践中有时对 FFN 层也应用效果更好。要通过实验决定，而不是盲目全应用。

4. **alpha 参数**：LoRA 的缩放因子 `lora_alpha / r`，控制 LoRA 更新幅度相对于原始权重的比例。通常设 alpha = 2*r（如 r=8, alpha=16），实践中这个比例是经验起点。

5. **QLoRA（量化 LoRA）**：在 LoRA 基础上，先将基础模型量化为 4-bit（NF4 量化），大幅减少显存需求，使得在消费级 GPU（24GB）上可以微调 70B 参数模型。代价是少量精度损失。

**差距在哪**：新手知道 LoRA 省显存但说不清原理，高手能解释低秩假设的数学直觉、rank 选择的经验法则、alpha 的作用，以及 QLoRA 如何在 LoRA 基础上进一步降低资源要求。

## Q：训练数据应该如何清洗和去重？

> 来源：数据工程 + LLM 训练面试

**新手答**："删掉明显的错误数据，其余保留就行了。"

**高手答**：

1. **去重的重要性**：重复数据会让模型过度记忆特定样本（memorization），导致过拟合；还会浪费训练计算资源（相同数据反复学习）。LLM 训练数据中重复率可能高达 20-30%，去重是必要步骤。

2. **精确去重**：基于 MD5/SHA 哈希，完全相同的样本只保留一个。最快、最准，适用于明显重复（爬虫多次抓取同一页面）。

3. **模糊去重（MinHash + LSH）**：用 MinHash 生成文档签名，用 LSH 找到相似签名的候选对，再计算 Jaccard 相似度。能发现"略有修改的重复"（如改了标点、改了数字但内容相同），比精确去重质量更高。

4. **质量过滤**：过短样本（< 50 tokens，通常是无意义片段）、过长样本（超过 context limit，需要截断）、高重复率样本（同一句话重复多次，LLM 生成的常见质量问题）、有害内容（需要内容安全过滤）。

5. **类别平衡**：如果训练数据中某类任务过多（如 70% 都是代码生成），模型可能对其他任务退化。要分析数据分布，按类别上采样少数类或下采样多数类，保持 Task Diversity。

**差距在哪**：新手的数据清洗是"删明显错误"，高手知道数据质量工程包含去重（精确+模糊）、质量过滤、类别平衡三个维度，且每个维度都有具体的工程实现方法。

## Q：合成数据（Synthetic Data）的生成方法有哪些？风险是什么？

> 来源：LLM 数据工程面试

**新手答**："让 GPT-4 生成训练数据，便宜又快。"

**高手答**：

1. **Self-Instruct**：从少量人类写的 seed instruction 出发，用 LLM 自动生成新的 instruction-response 对。生成后需要质量过滤（去除重复、低质、有害内容），斯坦福 Alpaca 就是用这种方法生成了 52k 数据。

2. **Evol-Instruct**：对已有指令做"进化"——让 LLM 把简单指令改写成更复杂/更具体/更多约束的版本，生成难度递增的数据集，提高模型处理复杂任务的能力（WizardLM 使用这种方法）。

3. **Back-translation**：对于翻译或改写任务，先生成目标语言文本，再让模型"翻译回"源语言，生成伪平行语料。质量低于真实平行数据，但成本极低。

4. **核心风险一：Model Collapse**：用 LLM 生成的数据训练 LLM，如果没有足够的真实数据混合，模型的方差会逐渐下降（回答越来越单一），多样性退化，这就是 Model Collapse 现象。

5. **核心风险二：错误传播**：生成 LLM 的错误会被下游模型"学习"，特别是事实性错误。合成数据必须经过人工或程序验证，尤其是事实类任务，不能直接使用生成数据。

**差距在哪**：新手把合成数据当廉价的完美替代品，高手知道 Model Collapse 和错误传播是合成数据的两个核心风险，需要真实数据混合、人工验证、多样性监控来防范。

## Q：模型量化（INT4/INT8/FP16）对质量有什么影响？

> 来源：LLM 部署工程师面试

**新手答**："量化了精度下降，不量化精度高，看资源决定。"

**高手答**：

1. **FP16/BF16**：FP32 参数压缩到 16 位，显存减少一半，速度提升 1.5-2 倍，精度损失极小（通常 < 0.5%）。现代 LLM 训练和推理的标准配置，基本无质量损失，是 "safe default"。

2. **INT8 量化**：权重量化到 8 位，显存减少到 FP16 的一半，推理速度提升（整数运算快于浮点），精度损失约 1-3%。大多数任务可以接受，适合部署在显存有限的服务器。

3. **INT4/NF4 量化（GPTQ/AWQ）**：显存减少到 FP16 的 1/4，可以在 RTX 3090（24GB）部署 70B 模型，但精度损失 3-5%，在复杂推理任务上明显。适合资源极度受限的场景，不适合高精度要求的应用。

4. **Activation 量化 vs Weight-only 量化**：Weight-only 量化（如 GPTQ）只量化权重，激活保持高精度，精度损失更小；同时量化权重和激活（如 INT8 的 LLM.int8()）可以减少计算开销，但精度损失更大。

5. **任务敏感度不同**：代码生成和数学推理对量化更敏感（精度损失更显著），自然语言生成和分类任务对量化不敏感。选择量化级别要在目标任务上测试，而不是依赖通用 Benchmark。

**差距在哪**：新手把量化当"精度 vs 资源"的二元选择，高手知道不同量化方法（FP16/INT8/INT4）有不同的精度-速度-资源曲线，且任务敏感度不同，要基于目标任务的评测数据做选择。

## Q：知识蒸馏在 Agent 场景有什么应用价值？

> 来源：AI 系统优化工程面试

**新手答**："知识蒸馏是让小模型学大模型，效果接近大模型。"

**高手答**：

1. **蒸馏原理**：用大模型（Teacher）的软标签（soft logits，不只是 one-hot label）训练小模型（Student）。Teacher 的软标签包含词汇间的关联概率，比人工标注的硬标签信息更丰富，让 Student 在更少数据下学到更多。

2. **Agent 场景的价值**：用 GPT-4 或 Claude 作为 Teacher，生成高质量的 Reasoning Traces（详细的思考过程 + 最终答案），把这些 Traces 作为训练数据蒸馏小模型（7B-13B）。小模型学会了大模型的推理方式，在特定任务上接近大模型效果，但推理成本低 10-100 倍。

3. **Speculative Execution 架构**：用小模型（Draft Model，如 13B）快速生成候选 token，用大模型（Verify Model，如 70B）并行验证/修正。整体吞吐量提升 2-4 倍，质量由大模型保证，速度由小模型提供。

4. **Task-specific 蒸馏**：不需要训练一个"通用"的 Student，只需要在目标任务上蒸馏。面试诊断 Agent 场景，用 GPT-4 生成 10000 条高质量诊断数据，蒸馏一个专门做面试诊断的 7B 模型，成本和质量都优于直接调 GPT-4 API。

5. **评估蒸馏质量**：不要只看 Perplexity，在目标任务的 eval 集上对比 Teacher 和 Student 的任务指标（Task Success Rate、F1 等），确认 Student 在业务指标上达到了可接受的水平。

**差距在哪**：新手把蒸馏当压缩技术（小模型学大模型），高手知道它在 Agent 场景中是成本优化的核心策略——通过 Teacher 生成高质量 Reasoning Traces，把大模型的推理能力以低成本转移到小模型。

## Q：持续学习和灾难性遗忘如何解决？

> 来源：长期 AI 系统维护工程面试

**新手答**："定期重新训练模型就行了。"

**高手答**：

1. **灾难性遗忘的本质**：神经网络用梯度更新共享参数，在新任务上训练时，旧任务的关键参数被覆盖，旧能力快速下降。这是 LLM 持续学习的核心挑战。

2. **EWC（Elastic Weight Consolidation）**：计算每个参数对旧任务的重要性，在新任务训练时，对重要参数的更新施加额外的正则化惩罚，防止这些参数被大幅修改。代价是计算和存储重要性权重的开销。

3. **Replay 方法**：在新任务训练时，混入一定比例的旧任务数据（Experience Replay），让模型同时在旧任务和新任务上训练。简单有效，缺点是需要保留旧数据（隐私和存储成本）。

4. **Parameter Isolation（参数隔离）**：为不同任务分配独立的参数子集（如不同任务使用不同的 LoRA 适配器），切换任务时加载对应的 LoRA，核心参数不被不同任务覆盖。这是目前工程上最实用的方案。

5. **在线学习的现实考量**：LLM 的在线持续学习在生产中极难实现（不稳定、难控制），更实用的方案是"定期批量微调"（每季度或每月用新数据做一次完整的 Fine-tuning）+ RAG 覆盖实时知识更新需求。

**差距在哪**：新手认为持续学习是"周期性重训"，高手知道灾难性遗忘是真实的工程问题，EWC/Replay/参数隔离是不同代价下的解决方案，而在大多数生产场景中 RAG + 定期批量微调是最可行的架构。

## Q：SFT 训练过程中的关键监控指标是什么？

> 来源：LLM 训练工程师面试

**新手答**："看 loss 就行了，loss 下降说明在学。"

**高手答**：

1. **Training Loss 和 Validation Loss**：Training Loss 持续下降但 Validation Loss 开始上升 → 过拟合信号，需要 Early Stopping 或增加 Dropout/Weight Decay。Validation Loss 不下降 → 学习率可能过小或数据质量差。

2. **Eval 任务指标（比 Loss 更重要）**：在目标任务的评测集上的实际表现指标（JSON 解析成功率、任务完成率），这些指标才直接反映业务价值。Loss 下降了但任务指标没改善，说明 Eval 数据分布有偏差。

3. **Per-token Loss 分析**：对不同类型 token（instruction tokens vs response tokens）分别统计 Loss。通常只在 response token 上计算 Loss（instruction 被 mask 掉），如果 instruction 的 Loss 也在影响训练，可能是数据 format 有问题。

4. **梯度范数（Gradient Norm）**：梯度范数突然爆炸 → 学习率过大或数据异常，需要梯度裁剪（Gradient Clipping）；梯度范数长期接近 0 → 梯度消失，模型几乎没在学习。

5. **训练速度（Tokens per second）**：MFU（Model FLOPs Utilization），衡量 GPU 计算资源的利用效率。MFU 持续低（< 30%）说明数据加载或通信是瓶颈，而不是计算，需要优化 DataLoader 或通信策略。

**差距在哪**：新手把 Loss 当唯一训练指标，高手知道需要同时监控 Train/Val Loss 分离、任务指标、梯度范数、训练效率，这四类指标覆盖了 Overfitting、数据质量、训练稳定性、资源效率四个维度。

## Q：如何从生产对话数据构建 SFT 训练集？

> 来源：数据飞轮工程面试

**新手答**："把用户的对话导出来，处理一下就能用。"

**高手答**：

1. **正样本选取**：不是所有生产数据都值得训练，需要筛选高质量样本：用户给了正反馈（如点赞）、任务被成功完成（task_success=true）、专家人工审核标注为高质量的对话。

2. **格式转换**：生产对话的 JSON 结构（session messages 数组）需要转换为 SFT 格式——通常是 `{"instruction": "...", "response": "..."}` 或 `{"messages": [{"role": "user", ...}, {"role": "assistant", ...}]}`（OpenAI SFT 格式）。

3. **敏感信息脱敏**：生产数据包含用户真实信息（姓名、邮件、工作单位），在使用前必须做 PII 脱敏处理。一旦带有真实 PII 的数据进入训练集，无法完全"训练掉"这些信息（模型可能记忆并在推理时复现）。

4. **质量平衡**：生产数据的分布往往不均衡（某类任务极多），要按任务类型做采样平衡，防止过度训练某类任务导致其他任务退化。

5. **数据飞轮设计**：生产数据 → SFT 训练 → 更好的模型 → 更多成功的对话 → 更多高质量训练数据，这个正向循环是数据飞轮（Data Flywheel）的核心。需要设计好反馈信号的采集（什么是成功的对话）才能让飞轮转动。

**差距在哪**：新手认为"生产数据导出来就是训练数据"，高手知道生产数据到训练数据需要质量筛选、格式转换、PII 脱敏、分布平衡四道工序，且设计好飞轮是长期数据积累的关键。

## Q：如何评估 Fine-tuning 是否成功？

> 来源：LLM 工程效果评估面试

**新手答**："看 loss 降了多少，越低越好。"

**高手答**：

1. **任务指标优先**：在目标任务的 eval 集上对比微调前后的任务成功率、格式正确率、关键指标。这些指标直接反映业务价值，比 Loss 更有意义。

2. **与 Baseline 比较**：Fine-tuning 的正确比较对象是"更好的 Prompt + 更大的基础模型"，而不是"Fine-tuning vs 什么都不做"。如果用 GPT-4 + 精心设计的 Prompt 能达到相同效果，Fine-tuning 可能不值得。

3. **通用能力退化检测**：微调通常会在提升特定任务的同时，造成通用能力的轻微退化。要在通用 Benchmark（如 MMLU、HellaSwag）上测试，确认退化在可接受范围内。

4. **对齐测试**：测试微调后模型是否仍然遵循安全指南，是否出现了意外的行为变化（如开始拒绝某些合理请求，或不再拒绝某些有害请求）。

5. **成本比较**：同样的任务质量，Fine-tuning 后的小模型和未 Fine-tuning 的大模型，成本相差多少？如果 Fine-tuning 的年化成本节省不抵数据标注和训练费用，可能不划算。

**差距在哪**：新手用 Loss 评估微调效果，高手用多维度评估——任务指标、与 Baseline 比较、通用能力退化、安全对齐、成本效益，微调是否"成功"需要这些维度共同判断。

## Q：什么是 Instruction Tuning？数据设计有什么要求？

> 来源：LLM 微调工程面试

**新手答**："就是用指令格式的数据微调，让模型听指令。"

**高手答**：

1. **Instruction Tuning 的作用**：预训练模型只会"续写文本"，无法理解"执行指令"。Instruction Tuning 用 instruction-response 格式的数据微调，让模型学会将自然语言指令映射为相应的行为，是现代 Chat 模型的基础。

2. **指令多样性要求**：训练数据的指令覆盖范围要广——分类、生成、摘要、翻译、代码、推理、对话……单一类型的指令会导致模型只擅长该类型，在其他类型上退化。FLAN 用 1800+ 种任务格式大幅提升了 Instruction Following 能力。

3. **回答质量比格式更重要**：模型学的是 (instruction, response) 的映射关系，如果 response 质量差（内容浅、有错误、格式不一致），模型会学到差的输出模式，不是学到"如何遵循指令"。

4. **Negative Instruction 覆盖**：训练数据中要包含拒绝有害请求的示例（instruction = 有害请求，response = 礼貌拒绝 + 解释原因），否则模型不知道什么时候应该拒绝，容易被越狱。

5. **Format Token 的一致性**：如果使用特殊格式 Token（`[INST]...` `[/INST]` 或 `<|user|>...` `<|assistant|>`），训练数据中的格式必须完全一致，否则模型在推理时对格式的响应不稳定。

**差距在哪**：新手把 Instruction Tuning 当"用 QA 对话数据微调"，高手知道指令多样性、响应质量、Negative Instruction 覆盖、格式一致性是数据设计的核心要素，每一个影响微调后模型的实际表现。

## Q：训练基础设施如何选择？云 GPU vs 本地 vs 托管服务？

> 来源：LLM 训练工程运维面试

**新手答**："小实验用自己的机器，大训练用云。"

**高手答**：

1. **本地 GPU**：RTX 4090/3090，适合 LoRA/QLoRA 微调 7B-13B 模型的小实验，低延迟，可以 24/7 占用。缺点：规模有限，多卡训练需要额外配置，长时间占用大型 GPU 浪费（实验不连续）。

2. **云 GPU（按需）**：AWS p3/p4、GCP A100、Azure NDv4，适合需要大规模 GPU（> 8 张 A100）的完整 SFT 训练。按小时计费，弹性好，适合不连续的训练任务。成本随时长快速增长，建议用 Spot/Preemptible 实例节省 60-70% 成本。

3. **托管微调服务**：OpenAI Fine-tuning API、Together.ai、Modal、RunPod，零运维，按 token 计费，适合不想管基础设施的团队。缺点：模型不在自己服务器（数据隐私）、定制化能力有限。

4. **成本估算**：7B 模型 LoRA 微调 1k 条数据，单张 A100（40GB）约 30-60 分钟，云端成本 $3-5；70B 模型 Full SFT 10k 条数据，8 × A100 约需 8-16 小时，成本 $200-400。这些数字帮助在不同方案之间做决策。

5. **实验 vs 生产的分层**：探索实验（验证数据质量、超参数搜索）→ 本地小 GPU 或 Spot 实例；正式训练（确认方案后的完整运行）→ 按需云 GPU；超大规模（> 100B 参数）→ 考虑 TPU/专用训练集群。

**差距在哪**：新手用"大小"判断训练在哪里跑，高手基于数据量、模型规模、数据隐私要求、团队运维能力做分层决策，并能给出具体的成本估算。

## Q：混合精度训练（BF16/FP16）有什么工程注意事项？

> 来源：深度学习训练工程师面试

**新手答**："用 FP16 省显存，没什么特别注意的。"

**高手答**：

1. **FP16 vs BF16**：FP16 有更高的数值精度但范围更小（易溢出，需要 Loss Scaling）；BF16 范围和 FP32 相同（不易溢出），精度略低。现代 GPU（Ampere 以上）推荐 BF16，稳定性更好，无需 Loss Scaling 的额外开销。

2. **Loss Scaling（FP16 专用）**：FP16 的梯度容易下溢（变成 0）或上溢（变成 inf），需要把 Loss 乘以一个大数再计算梯度，最后再除回来。PyTorch 的 GradScaler 自动处理，但需要在代码中显式启用。

3. **Master Weights 的 FP32**：混合精度训练中，虽然前向传播和梯度计算用 FP16/BF16，但参数的主副本（Master Weights）保持 FP32 精度，确保权重更新的精度。这是"混合"的含义——计算用低精度，存储和更新用高精度。

4. **Adam Optimizer 状态的精度**：Adam 的 first moment 和 second moment（优化器状态）通常保持 FP32，即使模型参数用 FP16，这会占用大量显存（优化器状态是模型参数的 2-3 倍）。8-bit Adam 可以把优化器状态量化为 8-bit，节省 75% 的优化器状态显存。

5. **数值溢出的检测**：训练中定期检查梯度是否有 NaN/Inf（PyTorch 的 `torch.isnan(loss).any()` 检查），发现后调整 Learning Rate 或检查数据是否有异常值。出现 NaN 就 skip 那个 batch 是工程常见做法。

**差距在哪**：新手认为"用 FP16 开启就行了"，高手知道 BF16 vs FP16 的稳定性差异、Loss Scaling 的必要性、Master Weights 的精度原因、8-bit Adam 的优化，这些是真正工程化训练的基础知识。

## Q：如何设计 Fine-tuning 的 Checkpoint 策略？

> 来源：LLM 训练可靠性工程面试

**新手答**："每个 Epoch 保存一次就行了。"

**高手答**：

1. **保存频率的权衡**：Checkpoint 保存频率高（如每 100 步）→ 故障恢复损失少，但 I/O 开销大，占用大量存储；频率低（如每 Epoch）→ 故障可能导致大量重训，但 I/O 开销小。通常按"可接受的最大重训时间"来决定频率。

2. **Best Model Tracking**：不只是按步数保存，还要追踪 Validation Loss / 任务指标最好的模型，单独保留为 "best_model" Checkpoint。训练结束后用 best_model 而不是最后一步的模型。

3. **滚动保存**：保留最近 K 个 Checkpoint（如最近 5 个），自动删除更早的，避免存储无限增长。通常 K=3-5 已经足够做故障回滚。

4. **断点续训（Resuming）**：Checkpoint 不只是模型权重，还要保存：优化器状态（Adam 的 first/second moment）、学习率调度器状态（step 计数）、随机数种子（DataLoader 的位置）。缺少任何一项，续训结果可能和未中断的训练不同。

5. **多节点训练的 Checkpoint 同步**：多 GPU/多节点训练时，需要协调所有节点同步保存 Checkpoint，防止只有部分 rank 保存导致 Checkpoint 不完整。DDP 训练通常只让 rank 0 保存，其他 rank 等待。

**差距在哪**：新手的 Checkpoint 是"周期性备份"，高手的 Checkpoint 是"完整可恢复的训练状态"——包含优化器状态、调度器状态、随机数状态，支持精确续训，并有 Best Model 追踪和滚动清理策略。

## Q：PEFT（参数高效微调）有哪些主要方法？各适合什么场景？

> 来源：高效 LLM 微调工程面试

**新手答**："LoRA 就是 PEFT，省显存。"

**高手答**：

1. **LoRA（Low-Rank Adaptation）**：在 Transformer 的注意力矩阵上注入低秩更新，只训练 A、B 两个小矩阵。最主流，灵活，支持多任务（不同任务的 LoRA 权重可以合并或分离），推理时可以 merge 到基础权重，无额外推理开销。

2. **Prefix Tuning**：在每一层的 Key 和 Value 矩阵前面添加可训练的"前缀向量"，基础模型参数完全冻结。训练参数量极少，但效果通常比 LoRA 差，适合资源极度受限的场景。

3. **Prompt Tuning（Soft Prompts）**：在输入端添加可训练的 embedding 向量（"soft tokens"），只训练这些 embedding。最省参数（只有几百到几千个），但效果比 LoRA 差，主要用于研究。

4. **Adapter（Bottleneck Adapters）**：在 Transformer 的每个 FFN 层后插入小的 MLP（Adapter）模块，只训练 Adapter 参数。灵活，不同任务可以有不同的 Adapter，但推理时有额外的计算开销（LoRA merge 后没有推理开销，Adapter 有）。

5. **选型建议**：需要合并多个任务 → LoRA（可以叠加多个 LoRA 权重）；极端低资源 → Prefix Tuning 或 Prompt Tuning；需要动态切换任务而不想重新加载模型 → Adapter。绝大多数场景 LoRA 是默认选择。

**差距在哪**：新手知道 LoRA 是 PEFT 方法之一，高手知道 PEFT 的方法谱系（LoRA/Prefix/Prompt/Adapter），每种方法的技术原理、参数效率、推理开销不同，选型要根据场景需求来决定。

## Q：数据标注质量如何控制？

> 来源：AI 数据工程团队面试

**新手答**："让标注员认真标就行，质量不行就返工。"

**高手答**：

1. **标注指南（Annotation Guide）**：详细的标注规范文档，包含：每个标签的定义、正/负例对比、边界 case 的处理原则。标注指南越详细，标注员间的一致性越高。

2. **Inter-annotator Agreement（标注一致性）**：对同一批样本让 2-3 个标注员独立标注，计算 Cohen's Kappa 或 Fleiss Kappa 系数。Kappa > 0.7 表示质量较好；< 0.6 说明指南不够清晰，需要澄清边界 case 并重新培训。

3. **仲裁机制**：对于标注员之间有分歧的样本（不一致的标注），设立仲裁流程（高级标注员或领域专家做最终决定），确保每个样本有确定的 ground truth。

4. **质量抽查（Quality Sampling）**：周期性从每个标注员的输出中随机抽取 5-10% 样本，由专家重新审核，计算该标注员的个人准确率。低于阈值（如 90%）的标注员需要重新培训或返工。

5. **激励对齐**：按照质量而非数量付费（或设置质量奖金），防止标注员为了完成数量而牺牲质量。同时避免设置过于宽松的质量标准让标注员"刷"高通过率。

**差距在哪**：新手把标注质量控制当"管理问题"（认真就行），高手知道需要清晰的指南、量化的一致性指标、仲裁机制、周期性抽查四个系统性机制，把"标注质量"变成可测量和可改善的工程指标。

## Q：数据版本管理和血缘追踪如何实现？

> 来源：MLOps / 数据工程面试

**新手答**："数据存在 S3，文件名加时间戳就算版本了。"

**高手答**：

1. **数据版本化工具**：DVC（Data Version Control）是最主流的工具，类似 Git 但针对大文件——将大数据文件存储在远程存储（S3/GCS），用 Git 跟踪 .dvc 指针文件（记录数据的 hash 和位置），实现数据版本与代码版本的绑定。

2. **版本绑定的价值**：能够精确复现某次训练——从 Git history 找到训练代码版本，再用 .dvc 文件拉取同一版本的训练数据，完整重现实验环境。这是"可复现性"的基础。

3. **血缘追踪（Data Lineage）**：记录每个数据集的来源——从哪个原始数据来、经过了哪些处理步骤（清洗、过滤、增强）、由哪个脚本生成。血缘信息让"这批数据有问题，影响了哪些模型"的排查变得可能。

4. **Feature Store 的实践**：对于频繁复用的特征（如 Embedding 向量、用户标签），使用 Feature Store（如 Feast、Tecton）集中管理，确保训练和推理使用完全相同的特征版本，消除 Training-Serving Skew。

5. **实验数据的不变性**：一旦训练完成，该实验对应的数据集版本不应被修改（只追加，不修改），确保实验结果可以被后续重现验证。"数据不可变性"是数据工程的重要原则。

**差距在哪**：新手用文件名时间戳"管理"版本，高手用 DVC 等工具把数据版本管理提升到 Git 同等水平——可 diff、可回滚、可绑定到代码版本、可追踪血缘，让机器学习实验真正可复现。

## Q：如何在有限预算内最大化微调效果？

> 来源：AI 工程成本优化面试

**新手答**："节省一些标注成本，用更便宜的硬件。"

**高手答**：

1. **数据质量 > 数据数量**：预算有限时，优先把钱花在 100-500 条高质量标注数据上（专家标注），而不是 10000 条低质量数据（众包标注）。LIMA 的 1000 条高质量数据经验是最有力的支撑。

2. **QLoRA 最大化显存效率**：用 4-bit 量化基础模型 + LoRA 微调，可以在单张消费级 GPU（RTX 4090，24GB）上微调 70B 参数模型，成本比 A100 实例低 10 倍以上。

3. **超参数搜索的资源分配**：不要在完整数据集上搜索超参数，先在 10-20% 数据的子集上做快速搜索，找到最优学习率、LoRA rank、alpha 等参数，再用全数据做正式训练，减少搜索阶段的计算成本。

4. **Spot/Preemptible 实例**：云端训练用 Spot 实例成本比按需实例低 60-70%。代价是实例可能被抢占，需要配合 Checkpoint 策略（频繁保存），被抢占后自动重新申请实例续训。

5. **小模型替代大模型**：不要默认选最大的基础模型（13B > 7B > 3B）。在数据有限的场景，小模型 Fine-tuning 后的任务成功率可能和大模型 Zero-shot 相当，而推理成本低 2-4 倍。先用 7B 的 LoRA 做 baseline，再评估是否值得升级到 13B。

**差距在哪**：新手在有限预算下的直觉是"花尽量少的钱"（用最便宜的服务），高手知道最优的资源分配是"把钱花在刀刃上"——高质量数据 > 数量，QLoRA 降低硬件要求，超参数快速搜索减少计算浪费，Spot 实例节省云成本。

## Q：RLHF 的替代方案有哪些？DPO、GRPO、DAPT 分别适合什么场景？

> 来源：LLM 对齐工程前沿技术面试

**新手答**："DPO 比 RLHF 简单，现在大家都用 DPO 了。"

**高手答**：

1. **DPO（Direct Preference Optimization）**：把偏好学习转化为分类损失，直接在 (prompt, chosen, rejected) 三元组上训练，无需独立的 Reward Model 和 RL 训练过程。工程实现简单、训练稳定，是目前 Fine-tuning 场景的主流方案。适合：有明确 chosen/rejected 对的对齐任务，团队没有 RL 基础设施。

2. **GRPO（Group Relative Policy Optimization）**：DeepSeek-R1 采用的方法，对同一 prompt 生成多个回答（Group），用组内相对排名作为奖励信号，避免了 PPO 需要 Reference Model 的问题。核心优势是可以在线生成训练数据（不需要预先收集偏好对），特别适合推理任务（数学、代码）——有 Ground Truth 可以自动评分，不需要人工标注偏好。适合：有明确可验证答案的任务（代码可执行、数学题有标准答案）。

3. **DAPT（Domain-Adaptive Pre-Training）**：在目标领域的大量无标注数据上继续预训练，让模型内化领域语言模式，而不是通过偏好学习调整行为。不涉及 RL，是最低成本的"领域适配"方式。适合：目标领域有大量文本但标注成本极高（法律、医学），且任务是理解/生成领域文本而非遵循指令。

4. **方案选型矩阵**：有人工标注偏好对 + 需要行为对齐 → DPO；有可验证 Ground Truth + 推理增强 → GRPO；只有无标注领域数据 + 提升领域理解 → DAPT；需要精细化奖励建模 + 团队有 RL 能力 → RLHF（PPO）。

5. **工程成本排序**：DAPT（继续预训练，只需无标注数据）< DPO（需要偏好对，无 RL 基础设施）< GRPO（需要在线生成和评分基础设施）< RLHF（需要 Reward Model + RL 训练栈）。

**差距在哪**：新手知道 DPO 比 RLHF 简单，高手能区分 DPO/GRPO/DAPT 的技术路线差异，知道 GRPO 在可验证任务上的独特优势（DeepSeek-R1 的路径），以及 DAPT 适合无标注领域适配的场景，能根据数据形态和任务特点选择正确方案。

## Q：LLM 训练数据如何做隐私保护？差分隐私和 PII 脱敏怎么落地？

> 来源：AI 合规与数据安全工程面试

**新手答**："把数据里的名字和电话删掉就行了。"

**高手答**：

1. **PII 脱敏的范围**：PII（Personally Identifiable Information）不只是姓名、电话，还包括邮件地址、身份证号、IP 地址、家庭住址、医疗记录等。脱敏方式分两类：①假名化（替换为占位符，如将"张三"替换为"[PERSON]"）；②匿名化（删除或替换为随机假数据，如用随机生成的邮件替换真实邮件）。假名化可逆，匿名化不可逆，训练数据通常选后者。

2. **自动化 PII 识别**：人工逐条检查不现实，需要用 NER 模型（命名实体识别）自动识别 PII 实体。开源工具：Microsoft Presidio（支持多语言、多类型 PII 检测 + 自动脱敏），规则库 + ML 模型结合，准确率通常在 85-95%，需要人工抽检补漏。

3. **差分隐私（Differential Privacy, DP）**：在梯度更新阶段注入随机噪声（Gaussian Noise），数学上保证攻击者无法从模型参数中还原单条训练数据。核心参数 ε（epsilon）控制隐私预算，ε 越小噪声越大、隐私越强但模型效用损失越大。通常 ε=1~8 是实践中的权衡区间。实现：Google 的 DP-SGD（差分隐私随机梯度下降），PyTorch 有 Opacus 库支持。

4. **DP 的工程代价**：DP 训练需要对每个样本的梯度单独计算和裁剪（Per-sample Gradient Clipping），计算开销是普通训练的 3-5 倍，且为了保证隐私预算通常需要更多轮次训练才能收敛。生产中只在监管要求严格（医疗、金融）时才强制使用。

5. **合成数据作为隐私方案**：用差分隐私生成合成数据替代真实数据——在真实数据上用 DP 训练生成器，再用生成的合成数据训练下游模型，合成数据本身不含真实 PII。这是医疗 AI 领域常用的数据共享方案，规避了直接共享真实数据的合规风险。

**差距在哪**：新手把隐私保护等同于"删敏感字段"，高手知道需要系统化的 PII 识别工具链、差分隐私的数学保证及其工程代价，以及在不同合规要求下（GDPR、HIPAA）选择合适的保护级别。

## Q：如何构建用于 Fine-tuning 的 Preference Dataset？chosen/rejected 对如何收集和控制质量？

> 来源：LLM 对齐数据工程面试

**新手答**："让人选哪个回答更好就行了。"

**高手答**：

1. **数据结构**：Preference Dataset 的基本单元是三元组 `(prompt, chosen_response, rejected_response)`。chosen 是被偏好的回答，rejected 是较差的回答。两者之间的"差距"越明显，训练信号越强；差距过小（near-tie）的样本会引入噪声，应该过滤。

2. **收集方式一：A/B 人工标注**：对同一 prompt 生成 2-4 个候选回答，让标注员做两两比较（A vs B），汇总比较结果形成全序排名，然后从排名中采样 (chosen, rejected) 对。关键：两两比较比直接打分更可靠（人对"哪个更好"的判断比给绝对分数更一致）。

3. **收集方式二：生产日志挖掘**：从线上系统的用户行为中提取偏好信号——用户采纳/编辑了哪个回答（positive signal），用户放弃对话或投诉（negative signal）。这是成本最低的偏好数据来源，但信号噪声大，需要较强的过滤逻辑。

4. **质量控制关键点**：①过滤 near-tie（两个回答质量差不多，标注员随机选，引入噪声）；②过滤有害 chosen（被选中的回答虽然"更好"但绝对质量仍然差）；③多标注员投票（2-3 人标注同一样本，只保留一致性高的样本）；④覆盖多样性（prompt 分布要覆盖模型会遇到的各种任务，不能集中在某一类）。

5. **自动化构建（AI Feedback）**：用更强的模型（GPT-4、Claude Opus）做"AI 裁判"，自动对候选回答进行排名，替代人工比较。成本低 10-100 倍，但 AI 裁判有系统性偏好（偏长、偏格式化），需要定期用人工数据校验 AI 裁判的可靠性。

**差距在哪**：新手认为偏好数据只是"让人选哪个更好"，高手知道需要控制 near-tie 噪声、多标注员一致性、prompt 分布多样性，并能在人工标注和 AI Feedback 之间根据预算和质量要求做权衡。

## Q：数据飞轮（Data Flywheel）如何系统设计？如何用生产流量持续改善模型质量？

> 来源：AI 产品增长与数据工程架构面试

**新手答**："把用户数据存起来，定期重训模型。"

**高手答**：

1. **飞轮的核心逻辑**：更好的模型 → 更好的用户体验 → 更多用户使用 → 更多高质量交互数据 → 再次改善模型。飞轮能转动的前提是：每一步都有明确的正向放大机制，其中"数据质量筛选"是不能省略的环节——低质量数据不会提升模型，会让飞轮空转甚至倒转。

2. **反馈信号的设计**：需要在产品层设计可量化的反馈信号，常见方式：①显式反馈（点赞/踩）——信号明确但响应率低（< 5%）；②隐式行为信号（用户是否采纳推荐、对话是否完成、是否触发了重试）——覆盖率高但信号噪声大；③专家审核（周期性人工抽审）——质量最高但成本最高。三种信号需要组合使用。

3. **数据管道的关键节点**：原始对话日志 → PII 脱敏 → 自动质量评分（长度、格式、是否完成任务）→ 人工抽审 → 数据标注（need_label 的样本进入标注队列）→ SFT/DPO 数据集构建 → 触发再训练。每个节点需要监控吞吐量和质量指标，防止某一节点成为瓶颈。

4. **触发再训练的策略**：不应该是"定期重训"（时间驱动），而应该是"数据驱动"——当新增高质量样本达到阈值（如 500 条新 DPO 对），或模型在 eval 集上的指标下降超过阈值，才触发新一轮训练。减少无效的训练计算浪费。

5. **飞轮的冷启动问题**：新产品没有用户数据时，飞轮无法启动。解决方案：①先用合成数据或公开数据集做初始模型；②用灰度发布（少量用户先用）积累真实数据；③用竞品的公开评测数据做"借力"——飞轮的冷启动需要人工投入，而非等待有机增长。

**差距在哪**：新手把飞轮理解为"存数据 + 定期重训"，高手知道飞轮的核心是信号设计（显式 + 隐式 + 专家审核）、数据管道各节点的质量保证、数据驱动而非时间驱动的再训练策略，以及冷启动问题的系统性解决。

## Q：预训练、微调、RAG 三种知识注入方案怎么选？成本和适用场景如何对比？

> 来源：AI 产品技术选型面试

**新手答**："知识更新用 RAG，任务优化用微调，基础能力靠预训练。"

**高手答**：

1. **预训练（Continual Pre-training / DAPT）**：在领域语料上继续预训练，让模型内化领域语言模式。成本最高（需要大量数据 + 长时间训练），适合：领域语言差异极大（医学、法律文书），且有充足的无标注领域语料（> 1B tokens）。效果是让模型"懂这个领域的语言"，而非遵循特定指令。

2. **SFT 微调**：用 (instruction, response) 对让模型学会特定任务的行为模式。成本中等（需要 1k-10k 条高质量标注数据），适合：任务行为高度可预测（格式固定、规则明确）、需要低延迟推理（小模型微调后替代大模型）、Prompt 无法稳定解决的输出格式问题。效果是让模型"会这个任务"。

3. **RAG（检索增强生成）**：推理时动态检索知识库注入上下文。无需训练，知识更新实时（改知识库即生效），适合：知识频繁更新（新闻、产品文档）、需要引用来源、知识量超出模型上下文窗口。成本是每次推理的检索延迟（50-200ms）和向量库维护成本。

4. **三维对比表**：

   | 维度 | 预训练 | 微调 | RAG |
   |------|--------|------|-----|
   | 知识更新速度 | 极慢（需重训）| 慢（需重训）| 实时 |
   | 推理成本 | 无额外成本 | 无额外成本 | +检索延迟 |
   | 初始投入 | 极高 | 中等 | 低 |
   | 知识边界 | 参数化记忆 | 参数化记忆 | 可引用来源 |
   | 适合知识量 | 海量无标注 | 有限有标注 | 无上限 |

5. **组合使用的最佳实践**：大多数生产场景是 RAG + SFT 的组合——RAG 覆盖动态知识和来源引用，SFT 解决固定格式输出和指令遵循问题。只有在特定领域语言理解严重不足时，才在 RAG + SFT 基础上叠加 DAPT。三层叠加成本最高，需要明确的业务 ROI 支撑。

**差距在哪**：新手对三种方案有粗略认知但无法量化对比，高手能从知识更新速度、推理成本、初始投入、知识边界四个维度给出结构化对比，并知道生产中 RAG + SFT 组合是最常见的起点，DAPT 是成本最高的最后手段。

## Q：Scaling Laws 是什么？对训练数据规模决策有什么指导意义？

> 来源：LLM 预训练与基础研究面试

**新手答**："模型越大效果越好，数据越多越好。"

**高手答**：

1. **Scaling Laws 的核心发现（Kaplan et al., 2020）**：模型性能（loss）与模型参数量 N、训练 token 数 D、计算量 C 之间存在幂律关系。固定计算预算 C 时，最优配置是同时扩大模型规模和数据量，而非只扩大其中一个。

2. **Chinchilla Scaling Laws（Hoffmann et al., 2022）**：DeepMind 的研究发现原始 Kaplan scaling law 严重低估了数据量的重要性。最优配置下，训练 token 数应该大约等于模型参数量的 20 倍（N tokens ≈ 20 × N params）。例如 70B 模型应该用约 1.4T tokens 训练，而不是 GPT-3 时代的 300B tokens。Llama 2 就是基于 Chinchilla 定律设计训练数据规模的。

3. **对数据规模决策的指导**：给定计算预算，先确定可以负担的 FLOPs（C = 6ND），再根据 Chinchilla 比例（N:D ≈ 1:20）分配参数量和数据量。如果数据量不够，扩大模型规模没有意义——数据是瓶颈。

4. **Scaling Law 的局限性**：Scaling Law 预测的是 Next-Token Prediction Loss，不能直接预测下游任务表现（Loss 下降不总是等于任务准确率提升）。Emergent Abilities（涌现能力）——如 Chain-of-Thought——在 Scaling Law 预测的 Loss 曲线上无法被预见，它们在特定规模阈值后突然出现。

5. **数据质量对 Scaling 的影响**：Scaling Law 假设数据质量一致，实际训练中低质量数据的边际收益远低于高质量数据。FineWeb、DCLM 等预训练数据集的研究表明，同等 token 数量下，高质量过滤后的数据（FineWeb-Edu）的 Scaling 效率可以比未过滤数据高 30-50%。

**差距在哪**：新手停留在"大模型 + 多数据"的直觉，高手了解 Chinchilla Scaling Law 给出的参数量与数据量的最优比例（20:1），知道在固定计算预算下如何分配模型规模和数据规模，以及数据质量对 Scaling 效率的放大作用。

## Q：Constitutional AI 和 RLAIF 是什么？如何用 AI 反馈替代人工标注？

> 来源：LLM 自对齐与规模化对齐工程面试

**新手答**："就是让 AI 自己检查自己，减少人工。"

**高手答**：

1. **Constitutional AI（CAI）的核心思想（Anthropic, 2022）**：不依赖人工标注偏好数据，而是给模型一套"原则（Constitution）"（如：不得助长伤害、应该诚实、应该有帮助），让模型先自行生成回答，再用这套原则对自己的回答进行批判和修订，最后把修订过程转化为训练数据。分两阶段：①SL-CAI（监督阶段，用 AI 自我修订的数据做 SFT）；②RL-CAI（用 AI 生成的偏好数据做强化学习，即 RLAIF）。

2. **RLAIF（Reinforcement Learning from AI Feedback）**：用强大的 LLM（如 GPT-4 或 Claude Opus）充当 Reward Model，对候选回答自动打分，替代人工偏好标注。核心优势：成本低（API 调用 vs 人工标注），可以快速生成海量偏好数据。核心风险：AI 裁判的系统偏见（偏好更长的回答、偏好格式化内容）会被训练放大。

3. **AI 裁判的质量校验**：RLAIF 要定期对 AI 裁判进行校准——抽取一批 AI 评分的样本让人工重新评分，计算 AI 与人工的一致率。一致率 < 75% 说明 AI 裁判有显著偏差，需要调整评分 prompt 或换更强的裁判模型。

4. **Prompt Engineering for AI Judge**：AI 裁判的评分质量高度依赖 prompt 设计。最佳实践：①给出明确的打分维度（Helpfulness、Harmlessness、Honesty）和评分标准；②用 CoT 要求 AI 先分析再打分（减少随机性）；③双向对比（先问"A vs B"，再问"B vs A"，不一致的结果需要复查）。

5. **CAI vs RLHF 的工程选择**：如果有足够预算的人工标注团队且任务安全性要求极高 → RLHF（人工反馈最可靠）；如果需要快速扩展对齐数据、任务相对标准化 → RLAIF（AI 裁判 + 少量人工校验）；如果要系统性对齐安全行为、减少对人工的依赖 → CAI（Constitution 可以系统性覆盖安全原则）。

**差距在哪**：新手把"AI 检查 AI"理解为简单的自动化，高手知道 Constitutional AI 是一套系统性的自对齐框架，RLAIF 有具体的 AI 裁判质量校验方法，以及 CAI/RLAIF/RLHF 的工程选型逻辑。

## Q：预训练数据的混合策略是什么？如何决定不同来源数据的比例？

> 来源：LLM 预训练数据工程面试

**新手答**："把各种数据混在一起训练就行了。"

**高手答**：

1. **数据来源的分类**：预训练数据通常包含多个来源：网页文本（Common Crawl，占比最大但质量最低）、书籍/论文（高质量、高密度知识）、代码（GitHub）、对话数据（对话理解）、专有数据（新闻、百科）。不同来源的质量、领域覆盖、知识密度差异巨大，直接混合会导致模型过度偏向高占比低质量来源。

2. **基于质量的上/下采样**：对高质量来源（书籍、论文、策划的网页）做上采样（重复采用），对低质量来源（Common Crawl 原始数据）做下采样或质量过滤后采样。GPT-3 的数据配比中，书籍来源被上采样约 3 倍，即使其 token 数量只占总量的 16%。

3. **Domain Mix 的评测导向**：数据配比不能拍脑袋决定，要用目标任务的 eval 集指导配比调整。做"配比消融实验"：固定总数据量，调整各来源比例，对比下游任务指标的变化，找到最优配比。Llama 3 的技术报告详细记录了这类消融实验的方法论。

4. **代码数据的特殊价值**：多个研究表明，即使是主要面向自然语言的模型，在训练数据中加入代码也能提升推理能力（Code → Chain-of-Thought 能力的正向迁移）。Llama 2 → Llama 3 的一个关键变化就是大幅增加代码数据的比例。

5. **数据比例随训练进展动态调整**：训练后期（接近收敛时）可以提高高质量数据的比例，类似于 Curriculum Learning 的思路——早期用大量数据快速学习基础语言模式，后期用高质量数据精炼模型行为。Llama 3 的"annealing"阶段就是在最后 10% 的训练中大幅提高高质量数据比例。

**差距在哪**：新手认为预训练数据是"越多越好的混合"，高手知道不同来源的质量差异决定了需要上/下采样策略，数据配比应由 eval 指导而非直觉决定，代码数据对通用推理能力有意外的正迁移价值，以及训练后期 annealing 阶段的数据精炼策略。

## Q：Curriculum Learning（课程学习）在 LLM 训练中如何应用？

> 来源：LLM 训练策略工程面试

**新手答**："从简单到难排列数据，让模型循序渐进学习。"

**高手答**：

1. **Curriculum Learning 的基本思想（Bengio et al., 2009）**：模仿人类学习从简单到复杂的规律，在训练初期用"简单"样本，随着训练进展逐步引入"困难"样本。理论上可以加快收敛、提升泛化，但在 LLM 规模下实施复杂度较高。

2. **难度度量的方法**：①基于长度（短文本 = 简单，长文本 = 复杂）；②基于困惑度（Perplexity）——用小模型预先计算每个样本的 PPL，高 PPL = 模型不熟悉的内容 = 困难；③基于任务复杂度（简单问答 < 多步推理 < 复杂代码生成）；④基于模型当前能力（在线动态评估，选当前模型刚好能学会但还有提升空间的样本）。

3. **LLM 预训练中的实践**：纯粹的从简到难排列在 Billion 级 token 的预训练中收益有限（数据顺序对自回归模型影响较小），更实用的应用是"后期精炼（Annealing）"——在训练最后阶段提高高质量/高难度数据的比例，类似逆向 Curriculum，用于精炼模型输出质量。

4. **SFT 中的 Curriculum 效果更显著**：在数据量有限的 SFT 阶段，训练顺序对结果影响更大。实践方法：①先用容易任务热身（简单格式化任务），再引入复杂任务（多步推理、长上下文）；②"难例挖掘（Hard Example Mining）"——把模型当前做错的样本加权上采样，让模型重点学习薄弱环节。

5. **Anti-Curriculum（反课程）和自适应课程**：部分研究发现"从难到易"（Anti-Curriculum）在某些任务上效果反而更好——困难样本先建立鲁棒特征，简单样本再做精炼。实际工程中不应教条地遵循"简单到难"，要在目标任务的 eval 集上做消融实验验证。

**差距在哪**：新手把 Curriculum Learning 理解为"排序数据从简到难"就结束，高手知道 LLM 预训练中纯 Curriculum 效益有限，真正价值在于 SFT 阶段的难例挖掘和预训练后期的 Annealing 策略，且需要通过消融实验而非教条应用来验证效果。
