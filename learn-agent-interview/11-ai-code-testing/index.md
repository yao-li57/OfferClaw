# AI 代码测试

## Q：LLM 应用的单元测试策略（Mock vs 真实调用）

> 来源：AI 工程师面试、测试架构讨论

**新手答**："测试 LLM 应用就直接调真实 API，看输出对不对。"

**高手答**：

**Mock 优先原则**：单元测试应使用 Mock Provider，将 LLM 调用替换为确定性返回值，确保测试速度（毫秒级 vs 秒级）、零成本、离线可运行，以及结果完全可重现。

**Mock 的分层设计**：粗粒度 Mock（返回预设文本）适合测试业务逻辑；细粒度 Mock（模拟流式事件、工具调用结构）适合测试 Agent 循环和工具编排逻辑。Mock 需要覆盖正常路径和错误路径（超时、限流、格式错误）。

**真实调用的适用场景**：集成测试和端到端测试才使用真实 API；Prompt 评测（判断模型输出质量）必须使用真实调用；新 Prompt 上线前的回归校验使用真实调用但控制调用量。

**测试隔离策略**：使用依赖注入将 Provider 作为可替换参数传入，而非在代码内部硬编码 API 调用，这是 Mock 可行的前提条件。

**成本与速度控制**：真实调用测试应标记为慢测试（`@slow` 或单独 CI pipeline），不在每次提交时运行；本地开发默认运行 Mock 测试套件，真实调用测试仅在合并前或按需触发。

**差距在哪**：新手混淆单元测试与集成测试的边界，高手清楚 Mock 设计层次，并通过依赖注入实现测试隔离，用分层 CI 控制成本。

## Q：Agent 的集成测试设计

> 来源：AI 系统可靠性面试、QA 工程师岗位

**新手答**："集成测试就是把各个模块连起来跑一遍，看有没有报错。"

**高手答**：

**Agent 集成测试的核心难点**：Agent 是多步骤、有状态、非确定性的系统，传统"输入-输出"断言不适用。需要对执行轨迹（trace）而非单点输出做断言。

**轨迹级断言**：断言 Agent 在给定输入下是否调用了预期工具序列（顺序或无序）、是否在正确步骤请求了人工确认、是否在工具失败后触发了重试，而非仅断言最终输出文本。

**场景驱动测试**：构建覆盖关键业务路径的场景集（happy path、工具调用失败、超出权限、LLM 返回格式错误），每个场景定义输入、期望轨迹和终止条件，而非依赖随机输入。

**状态隔离**：每个集成测试用例应在独立的 Session 和数据库状态下运行，避免测试间状态污染。使用测试专用数据库或事务回滚保证隔离。

**超时与资源边界**：集成测试必须设置迭代次数上限和超时，防止 Agent 无限循环导致测试挂起。应测试 Agent 在达到资源上限时是否优雅退出而非崩溃。

**差距在哪**：新手将集成测试理解为"拼接模块跑通"，高手关注轨迹断言、场景覆盖、状态隔离，并把资源边界本身作为测试对象。

## Q：基于 LLM 的测试用例生成

> 来源：测试自动化讨论、AI 辅助开发面试

**新手答**："让 LLM 帮我写测试用例，生成一些边界情况，省时间。"

**高手答**：

**LLM 生成测试用例的优势场景**：枚举边界值（字符串长度边界、空值、特殊字符）；生成自然语言输入的多样化变体（同义替换、语气变化、多语言）；生成对抗性 prompt（越权请求、注入攻击、格式破坏）。这类任务人工成本高但 LLM 效率显著。

**质量控制流程**：LLM 生成的测试用例不能直接使用，需要人工审核去除重复、低质量和错误预期值的用例；对生成的用例运行覆盖率分析，确认其覆盖了新路径而非重复已有测试。

**预期值（Oracle）问题**：LLM 生成输入容易，但生成正确的预期输出困难——LLM 可能生成与被测系统同样的错误答案。解决方案：用规则或独立参照系生成预期值，或使用"差异测试"（两个版本的输出对比而非绝对断言）。

**测试用例版本管理**：LLM 生成的测试用例应纳入版本控制，记录生成时的模型版本和 prompt，便于复现和审计。避免每次测试时动态生成（引入不稳定性）。

**生成覆盖度量**：用变异测试（Mutation Testing）评估 LLM 生成测试用例的质量——能杀死的变异体比例越高，测试用例质量越好。

**差距在哪**：新手把 LLM 生成测试用例当作"自动化省力"的工具，高手关注 Oracle 问题、质量控制流程，以及将生成用例纳入标准工程管理流程。

## Q：非确定性输出的测试方法（快照测试、语义断言）

> 来源：LLM 应用测试策略讨论、QA 架构设计面试

**新手答**："LLM 输出不确定，所以很难测，只能人工看。"

**高手答**：

**快照测试（Snapshot Testing）**：将首次运行的 LLM 输出保存为基准快照，后续运行与快照对比，检测非预期变化。适合检测 Prompt 改动或模型升级导致的输出漂移，而非验证绝对正确性。快照需要定期人工审查并更新，防止"测试腐烂"（快照与期望行为脱节）。

**语义断言（Semantic Assertion）**：用另一个 LLM（或规则）对输出做语义级验证，例如"输出是否包含代码示例"、"输出语气是否符合专业标准"、"关键信息是否被提及"。相比字符串匹配更鲁棒，允许表达方式的合理变化。

**属性测试（Property-based Testing）**：验证输出的属性而非具体内容：输出长度在合理范围内；JSON 输出符合 schema；输出不包含禁止词汇；工具调用参数格式正确。这些属性是确定性的，适合在 CI 中自动化运行。

**多次运行统计**：对于关键功能，在相同输入下运行 N 次，统计通过率。通过率低于阈值（如 90%）则标记为不稳定，触发 Prompt 优化。避免用单次结果判断功能是否可靠。

**温度控制**：测试时将 temperature 设为 0（或接近 0）提高确定性，减少测试噪音。生产环境的温度配置与测试环境分离，防止温度差异掩盖真实 bug。

**差距在哪**：新手将不确定性视为"无法测试"的障碍，高手能组合快照测试、语义断言、属性测试和统计化测试，构建分层可自动化的测试体系。

## Q：Agent 的回归测试与 Golden Set

> 来源：LLM 系统持续迭代讨论、MLOps 工程师面试

**新手答**："回归测试就是改完代码再跑一遍之前的测试用例。"

**高手答**：

**Golden Set 的定义与构建**：Golden Set 是经过人工标注的高质量测试案例集，包含输入、期望输出（或期望行为属性）和验收标准。应从真实用户交互中筛选，覆盖典型场景、边界场景和历史 bug 场景，是 Agent 回归测试的核心资产。

**Golden Set 的维护策略**：每次发现新 bug 或重要边界场景时，将其加入 Golden Set；定期清理过时的用例（业务场景已不再适用的）；用覆盖率分析识别 Golden Set 中的盲区。Golden Set 是活文档，不是一次性建设。

**回归评测流程**：模型升级、Prompt 修改或工具变更前，在 Golden Set 上运行完整评测；对比新旧得分的差异，重点分析得分下降的用例；建立"零回归"门控（关键指标不能低于基线的 X%）作为上线条件。

**自动化 Evaluation Pipeline**：使用 LLM-as-Judge 对 Golden Set 输出打分，结合人工抽查，实现高频率、低成本的回归检测。Judge 的 prompt 和评分标准需要版本化，确保历史得分可比较。

**Golden Set 的数量与质量权衡**：数量不是越多越好——100 个精心设计的用例比 1000 个随机用例价值更高。优先覆盖高风险、高频率的场景，保持 Golden Set 精简以控制评测成本。

**差距在哪**：新手将回归测试理解为"重跑旧测试"，高手理解 Golden Set 的构建、维护策略和自动化评测 pipeline 的设计，并能量化回归门控标准。

## Q：LLM 评测中的"测试不稳定性"问题

> 来源：LLM 系统工程面试、测试可靠性讨论

**新手答**："LLM 输出有随机性，测试偶尔失败是正常的，忽略就行。"

**高手答**：

**不稳定性的根源分类**：模型随机性（temperature > 0 导致同输入不同输出）；环境依赖（外部 API 状态、网络延迟）；状态污染（测试间共享状态未清理）；评测 prompt 对输出格式敏感（轻微措辞差异导致判断翻转）。不同根源需要不同的应对策略。

**区分"合理变化"与"真实失败"**：建立基准通过率（同输入 N 次运行的通过率分布），当单次失败率在基准范围内视为合理变化；当失败率超出基准范围（如从 95% 降至 70%）视为真实回归。避免单次失败直接阻断流水线。

**减少模型随机性的工程手段**：评测时 temperature=0；使用 seed 参数（支持的模型）确保可重现；对输出做格式标准化后再比较（去除无意义的空格、大小写差异）。

**隔离外部依赖**：评测中的外部 API 调用（非 LLM 本身）应使用 Mock 或 VCR（录制/回放），隔离网络不稳定对测试结果的影响。LLM 调用本身如需稳定，可使用缓存层（相同 hash 的输入返回缓存结果）。

**不稳定性监控**：在 CI 中追踪测试的历史通过率趋势，设置告警（如某测试过去 7 天通过率持续下降）。将不稳定测试标记并隔离在单独队列，定期集中处理而非放任。

**差距在哪**：新手将不稳定性归因于"LLM 随机性"并选择忽略，高手能分类根源，区分合理变化与真实失败，并通过统计化评测和工程手段系统性降低不稳定性。

## Q：AI 功能的 End-to-End 测试框架

> 来源：AI 产品质量保障讨论、测试架构面试

**新手答**："E2E 测试就是模拟用户操作，从界面到后端全跑一遍。"

**高手答**：

**AI 功能 E2E 测试的特殊挑战**：传统 E2E 测试断言确定性的 UI 状态；AI 功能的 E2E 输出不确定，需要断言输出质量而非精确内容，且执行时间长（多步骤 Agent 可能耗时数十秒）。

**测试层次设计**：E2E 测试应只覆盖最重要的用户旅程（5-10 个核心场景），不求全面覆盖。细粒度覆盖留给单元测试和集成测试。E2E 用于验证"整个系统协同工作"，而非替代低层测试。

**AI 输出的 E2E 断言策略**：结构性断言（返回了正确格式的响应）；关键信息存在性断言（输出包含必要字段或关键词）；LLM-as-Judge 打分（对输出质量的语义评估）；业务指标断言（任务完成率、工具调用成功率）。避免使用字符串精确匹配。

**测试数据管理**：E2E 测试需要稳定的测试数据集（用户账号、知识库、历史对话），数据状态在测试前重置。避免使用生产数据（隐私合规风险）和实时变化的数据（导致测试不稳定）。

**E2E 测试的运行策略**：不在每次提交时运行（耗时高、费用高），应在合并主干前、发版前触发；支持并行运行多个测试场景；失败时保存完整的 Agent 执行轨迹（trace）便于调试，而不只是保存最终输出。

**差距在哪**：新手将 AI 功能 E2E 测试等同于传统 UI 自动化，高手理解 AI E2E 测试的特殊断言策略、层次设计原则，以及测试数据管理和触发策略的工程要点。

## Q：AI 应用的 CI/CD 流水线设计

> 来源：MLOps 工程师面试、AI 平台工程讨论

**新手答**："CI/CD 就是代码改了自动跑测试，然后自动部署，AI 应用跟普通应用差不多。"

**高手答**：

**分层流水线设计**：AI 应用的 CI 需要在传统代码测试之外增加三个独立阶段：Prompt 测试阶段（验证 Prompt 修改不引发回归）、模型评测阶段（在 Golden Set 上跑评分）、成本门控阶段（检查本次变更的预估 token 消耗是否超出预算）。三个阶段触发条件不同，不应全部在每次提交时运行。

**触发策略**：每次提交触发轻量测试（Mock 单元测试 + 属性断言，耗时 < 2 分钟）；合并主干前触发中量测试（集成测试 + 部分 Golden Set，耗时 < 10 分钟）；发版前触发全量评测（完整 Golden Set + E2E，耗时可接受 30-60 分钟）。用 CI 标签（如 `[skip-eval]`）支持在文档改动时跳过评测阶段。

**Prompt 版本管理**：Prompt 应纳入 Git 版本控制，修改 Prompt 文件与修改代码享有相同的 review 流程。在 CI 中通过 diff 检测 Prompt 变更，仅在 Prompt 变更时触发完整评测，避免无关代码提交浪费评测算力。

**评测结果存储与对比**：每次 CI 运行的评测结果（得分、通过率、失败用例列表）保存到数据库或对象存储，与上一次基线对比并生成差异报告。门控条件示例：核心场景通过率 >= 95%，平均质量得分不低于基线 0.05 分。

**模型部署的金丝雀策略**：切换模型版本（如 claude-3.5-sonnet → claude-3.7-sonnet）时，先在 5% 流量上运行新模型，收集真实用户反馈指标（任务完成率、用户满意度），指标稳定后逐步扩量，而非一次性全量切换。

**差距在哪**：新手将 AI 应用 CI/CD 等同于普通应用的自动化部署，高手能设计分层触发策略，将 Prompt、评测、成本纳入流水线，并以数据驱动门控决策。

## Q：LLM 应用的负载测试与压力测试策略

> 来源：AI 基础设施面试、性能工程讨论

**新手答**："负载测试就是多发几个请求，看服务有没有崩。"

**高手答**：

**LLM 应用性能瓶颈的特殊性**：LLM 服务的瓶颈不在 CPU/内存，而在 Token 吞吐量（Tokens/s）和并发请求数。Streaming API 的延迟分为首 Token 时间（TTFT, Time to First Token）和总完成时间，两者需要分别测量，因为用户感知的"卡顿"主要由 TTFT 决定。

**压测工具选型**：使用 `k6` 或 `Locust` 构建自定义压测脚本，而非通用 HTTP 压测工具（JMeter 对 SSE/Streaming 支持差）。压测脚本需要实现 SSE 流读取逻辑，逐事件记录 TTFT 和每个 token 到达时间戳。

**Streaming 压测的关键指标**：TTFT（首个 token 到达时间，目标 < 500ms）；Token 吞吐率（tokens/s，反映模型推理速度）；并发下的 TTFT 退化率（并发 10 vs 并发 100 的 TTFT 对比）；连接超时率（长时间 streaming 连接的断开比例）。

**阶梯式负载策略**：从 5 并发起步，每 2 分钟增加 10 并发，观察 TTFT P95 和错误率的变化曲线，找到"拐点"（性能急剧退化的并发阈值）。在拐点以下留出 30% 余量作为生产容量规划依据。

**速率限制与队列测试**：故意超过 API 速率限制，验证应用的限流重试逻辑是否正确（指数退避、Jitter）；测试请求队列在高负载下的排队深度和超时行为；验证 streaming 连接中断后客户端的重连逻辑。

**差距在哪**：新手把 LLM 压测理解为普通 HTTP 压测，高手关注 TTFT 等 Streaming 特有指标，能设计阶梯式负载方案，并量化容量规划依据。

## Q：如何测试 Agent 的工具调用链

> 来源：Agent 工程师面试、LLM 工具编排测试讨论

**新手答**："测试工具调用就是传参数进去，看返回值对不对。"

**高手答**：

**三层测试策略**：工具本身的单元测试（隔离测试工具函数的正确性）、工具调用编排的集成测试（验证 Agent 在特定输入下是否调用了正确工具序列）、多步骤工具链的端到端测试（验证工具结果被正确传递和利用）。三层分别测试不同的关注点，不能用一层替代另一层。

**工具调用的 Mock 设计**：在集成测试中，Mock LLM 的输出为预设的 `tool_use` 事件序列，而非 Mock 工具本身。这样可以测试 Agent Loop 的工具解析、权限检查、Hook 触发等编排逻辑，而无需真实 LLM 调用。示例：Mock 返回 `{"type":"tool_use","name":"search","input":{"query":"test"}}` 触发工具调用路径。

**多步骤链的断言方式**：收集完整的工具调用轨迹（工具名、参数、返回值、调用顺序），而非只断言最终输出。使用轨迹快照或轨迹模式匹配（"步骤 1 调用 A，步骤 2 调用 B，B 的参数来自 A 的返回值"）进行验证。

**工具失败场景覆盖**：必须测试工具抛出异常、返回错误码、返回空结果时 Agent 的行为（是否重试、是否降级、是否向用户报告错误）。工具的异常路径测试与正常路径同等重要。

**并发工具调用测试**：对于支持并行工具调用的 Agent，测试多个工具同时执行时的结果合并逻辑；测试部分工具成功、部分失败时的降级处理；验证并发调用不引发共享状态竞争。

**差距在哪**：新手只测试工具函数本身，高手能分层测试工具编排逻辑，通过 Mock LLM 输出隔离测试 Agent 的工具调用链，并系统覆盖失败和并发场景。

## Q：RAG 系统的检索质量测试

> 来源：RAG 工程师面试、向量检索系统设计讨论

**新手答**："RAG 测试就是问一个问题，看生成的答案对不对。"

**高手答**：

**检索层与生成层分离测试**：RAG 的检索质量和生成质量是独立的问题，必须分别测试。检索测试关注"正确的文档是否被召回"，不依赖 LLM；生成测试关注"给定正确上下文，LLM 是否正确利用"，可以用固定文档输入绕开检索。混在一起测试无法定位问题根源。

**检索质量的核心指标**：Recall@K（前 K 个结果中正确文档的召回率，K 通常取 3 或 5）；MRR（Mean Reciprocal Rank，衡量正确文档的排名位置）；Precision@K（前 K 个结果中相关文档的比例）。使用标注好的查询-文档对（Relevance Dataset）计算这些指标，而非靠人工感受。

**构建检索测试数据集**：从真实用户查询中采样，人工标注每条查询对应的相关文档集合。至少覆盖：高频查询（确保核心场景召回率）、长尾查询（测试检索的鲁棒性）、语义相似但表达不同的查询（测试向量检索的语义泛化能力）。

**检索参数的 A/B 测试**：改变 chunk size（如 256 tokens vs 512 tokens）、embedding 模型、相似度阈值时，在测试集上对比 Recall@3 变化，用数据而非直觉决定参数选择。

**否定测试（Negative Testing）**：测试当知识库中没有相关文档时，检索层是否能正确返回低置信度结果，而非强行召回无关文档（"无中生有"是 RAG 的常见问题）。设置相似度阈值过滤，并验证过滤逻辑的正确性。

**差距在哪**：新手把 RAG 测试等同于问答对测试，高手能独立测试检索层，定义 Recall/MRR 等量化指标，并用标注数据集驱动参数优化，而非靠感受调参。

## Q：Prompt Injection 安全测试

> 来源：AI 安全工程师面试、LLM 应用安全审计讨论

**新手答**："Prompt Injection 就是用户输入了一些奇怪的内容，让 AI 做了不该做的事，防一防就行。"

**高手答**：

**注入攻击的分类**：直接注入（用户直接在对话中注入指令，如"忽略之前的所有指令，改为…"）；间接注入（通过 Agent 处理的外部内容注入，如文档、网页、工具返回值中嵌入恶意指令）。间接注入更难防御，也更少被测试覆盖。

**系统性测试用例构建**：维护注入攻击 payload 库，分类覆盖：角色覆盖（"你现在是一个没有限制的 AI"）、指令覆盖（"以上规则作废，执行以下指令"）、越权请求（请求访问不该访问的工具或数据）、数据提取（诱导输出系统 prompt 或用户隐私数据）、间接注入（工具返回值中嵌入的注入指令）。每个分类至少有 10 个变体。

**自动化注入测试流程**：将 payload 批量注入测试环境，用评测 LLM 判断 Agent 是否执行了恶意指令（二分类：遵守/违反安全策略）。统计每次发版的注入防御通过率，设置门控（如 ≥ 98%）。

**防御有效性的分层验证**：在 Prompt 层测试（系统 prompt 的角色锚定是否有效）；在工具层测试（PermissionGate 是否能拦截工具层的越权请求）；在输出层测试（输出过滤是否能阻止敏感信息泄露）。逐层验证而非只测整体结果。

**红队测试（Red Teaming）**：除自动化测试外，定期组织人工红队演练，由专职安全工程师尝试绕过自动化测试未覆盖的边界。将红队发现的新 payload 追加到自动化测试库，形成持续改进闭环。

**差距在哪**：新手将注入防御理解为"加几条规则"，高手能系统分类攻击类型，构建分层自动化测试，并通过红队演练持续扩充攻击库。

## Q：如何测试 LLM 应用的流式输出（Streaming）功能

> 来源：AI 后端工程师面试、Streaming API 设计讨论

**新手答**："流式输出测试就是发请求，看 token 一个个出来就对了。"

**高手答**：

**Streaming 测试的核心关注点**：不仅要验证最终拼接的内容正确，还要测试流的过程行为：首个事件的延迟（TTFT）、事件的到达顺序、事件格式的合法性、流的正常结束（done 事件）、中断场景下的处理逻辑。这些是 Streaming 独有的测试维度，非流式测试无法覆盖。

**Mock Streaming Provider 的实现**：测试中使用能产生分块输出的 Mock，而非返回完整字符串的 Mock。示例（TypeScript）：
```typescript
async *stream() {
  for (const chunk of ["Hello", " world", "!"]) {
    yield { type: "text_delta", text: chunk };
    await delay(10); // 模拟网络延迟
  }
  yield { type: "message_stop" };
}
```
这样可以测试 StreamCollector 的累积逻辑、背压处理和提前终止逻辑。

**中断与恢复测试**：模拟客户端在流传输中途断开连接，验证服务端资源是否被正确释放（连接、内存、未完成的 LLM 调用）；验证客户端重连后是否能正确处理（重新请求 vs 断点续传）；测试 LLM API 侧中断（如超时）时的错误传播路径。

**事件格式合规性测试**：对流中的每个 SSE 事件做 schema 验证，确保 `data:` 字段是合法 JSON，事件类型符合预定义枚举，不出现格式错误的部分 token 或乱码。这类测试可以作为属性断言在 CI 中自动运行。

**背压与慢消费者测试**：模拟客户端处理事件速度远低于生成速度（慢消费者），验证服务端缓冲区行为；在高并发下测试事件队列的内存增长是否有上限；验证慢消费者不影响其他并发流的正常传输。

**差距在哪**：新手只验证流式输出的最终内容，高手测试流的过程行为，能实现细粒度 Mock Streaming Provider，并系统覆盖中断、格式合规和背压场景。

## Q：AI 服务的混沌工程——如何测试 LLM 服务降级和恢复

> 来源：AI 基础设施稳定性面试、SRE 工程讨论

**新手答**："混沌工程就是随机断网，看系统会不会崩。AI 服务也一样。"

**高手答**：

**LLM 服务混沌实验的特殊场景**：LLM 服务的故障模式与普通服务不同：API 限流（429）比宕机更常见；响应延迟从 1 秒变为 30 秒（而非直接报错）；模型服务可用但质量下降（幻觉增加、格式错误频率上升）；多个 Provider 同时故障。需要针对这些场景设计专项混沌实验。

**故障注入层次**：在网络层注入（使用 Toxiproxy 模拟延迟、丢包、连接断开）；在协议层注入（返回格式错误的 SSE 事件、截断 JSON）；在应用层注入（Mock Provider 按比例返回错误，如 30% 概率返回 429）。不同层次的注入验证不同的防御逻辑。

**降级策略验证**：注入主 Provider 故障，验证系统是否自动切换到备用 Provider（Fallback）且用户无感知；验证切换延迟是否在可接受范围（如 < 2 秒）；注入所有 Provider 同时故障，验证系统是否优雅降级（返回友好错误提示而非 500 崩溃）。

**重试逻辑验证**：注入间歇性故障（50% 概率失败），验证指数退避+Jitter 重试逻辑的正确性；验证重试不会导致请求被执行多次（幂等性）；验证重试次数耗尽后的行为（日志、告警、用户通知）。

**恢复时间目标（RTO）测试**：在故障恢复后，测量系统从降级状态恢复正常的时间；验证熔断器（Circuit Breaker）的半开状态能正确检测服务恢复；记录并报告 MTTR（Mean Time to Recovery）作为稳定性指标。

**差距在哪**：新手把混沌工程理解为"断网测试"，高手针对 LLM 服务的特有故障模式（限流、延迟、质量下降）设计专项实验，并量化 Fallback 延迟和 RTO 指标。

## Q：多模型版本之间的兼容性测试

> 来源：模型升级工程讨论、AI 平台迁移面试

**新手答**："模型升级就换个版本号，输出变了就重新调一下 Prompt。"

**高手答**：

**兼容性测试的三个维度**：行为兼容性（新模型在 Golden Set 上的通过率是否达标）、格式兼容性（新模型是否严格遵守 JSON schema、工具调用格式等结构化输出约定）、成本兼容性（新模型的 token 消耗和延迟是否在预算范围内）。三个维度需要分别量化，不能只看"感觉输出变好了"。

**并行评测（Shadow Testing）**：在生产流量上同时调用新旧两个模型，将新模型输出记录但不返回给用户，事后批量对比两组输出的差异。重点分析"旧模型通过但新模型失败"的用例（回归）和"旧模型失败但新模型通过"的用例（改进）。

**格式破坏检测**：在 Golden Set 上统计新模型的 JSON 解析失败率、工具调用参数格式错误率、输出截断率。格式问题往往比内容质量问题更容易导致线上故障，需要优先排查。如果新模型格式错误率 > 0.5%，阻断升级直到找到原因。

**Prompt 迁移策略**：不同模型版本对 Prompt 的敏感度不同（claude-3.5 和 claude-3.7 的指令跟随方式存在差异）。升级前在测试集上对比 Prompt 修改前后的通过率变化，使用 A/B 测试框架选出最优 Prompt，而非凭经验猜测。

**回滚门控设计**：定义明确的回滚触发条件（如核心场景通过率下降超过 2%，或格式错误率超过 1%），在金丝雀阶段自动监控，触发条件满足时自动回滚到旧模型版本，无需人工干预。

**差距在哪**：新手将模型升级视为"换个版本号"，高手能从行为、格式、成本三个维度量化兼容性，通过 Shadow Testing 和自动化门控控制升级风险。

## Q：测试 Agent 的并发场景

> 来源：AI 后端稳定性面试、多租户 Agent 系统设计讨论

**新手答**："并发测试就是多开几个用户同时发请求，看有没有报错。"

**高手答**：

**并发场景的核心风险**：Agent 是有状态的系统，多 Session 并发时最主要的风险是状态隔离失败——一个 Session 的内存、工具调用状态、上下文数据被另一个 Session 读写。这类 bug 在单 Session 测试中完全无法复现，必须在并发场景下专项测试。

**状态隔离测试方法**：同时启动 N 个 Session（N >= 10），每个 Session 使用唯一的 Session ID 和独立的测试数据；在所有 Session 运行完成后，验证每个 Session 的最终状态、工具调用轨迹、Memory 写入结果与预期完全匹配，不存在跨 Session 的数据污染。

**竞争条件（Race Condition）测试**：专门设计会触发竞争的场景：两个 Session 同时写入同一用户的 Memory 记录（验证写操作原子性）；多个 Session 同时请求需要全局速率限制的工具（验证限流计数器的并发安全性）；Session 状态机在并发下的转换正确性（防止从 active 跳转到非法状态）。

**Sub-agent 并发池测试**：测试并发池的上限控制（超出上限的请求是排队还是报错）；测试 Sub-agent 异常退出时是否正确释放并发槽（防止并发槽泄漏导致系统逐渐挂死）；在高并发下统计 Sub-agent 的任务完成率和平均延迟。

**压力下的资源泄漏检测**：运行 100+ 个并发 Session 各 10 轮对话后，检测进程的内存和数据库连接数是否回到初始水位（排查连接泄漏和内存泄漏）。使用 Node.js 的 `--inspect` 或类似工具采集 heap 快照对比。

**差距在哪**：新手将并发测试理解为"压测吞吐量"，高手将状态隔离和竞争条件作为并发测试的核心关注点，并通过资源泄漏检测验证系统的长期稳定性。

## Q：AI 测试的成本控制策略

> 来源：AI 工程成本优化面试、CI 资源管控讨论

**新手答**："真实 LLM 调用比较贵，能少调就少调，或者用便宜的模型。"

**高手答**：

**分层测试的成本分配原则**：Mock 测试层（零成本，每次提交触发，覆盖逻辑正确性）、小模型/缓存层（低成本，合并前触发，覆盖核心功能）、完整模型评测层（高成本，发版前触发，覆盖质量基线）。80% 的覆盖由 Mock 完成，真实 LLM 调用只用于无法 Mock 的质量验证。

**请求缓存（Prompt Caching）**：在 CI 中为相同 hash 的 Prompt + 输入启用缓存，相同测试用例重复运行时直接返回缓存响应，不发起新的 LLM 调用。适用于稳定的 Golden Set 评测——Golden Set 不变时，大多数调用可以命中缓存，将成本降低 60-80%。

**小模型代理策略**：对于只需要验证格式合规性、结构正确性的测试，使用最小可用模型（如 haiku 或 gpt-4o-mini 替代 opus/gpt-4）；质量评分类测试才使用高质量模型。在 CI 配置文件中通过环境变量 `TEST_MODEL=claude-haiku-3` 切换测试用模型，不硬编码。

**批量调用优化**：将 Golden Set 的评测请求批量提交（使用 Batch API），而非逐个串行请求。Batch API 通常有 50% 的价格折扣，且不占用并发配额。非时间敏感的回归评测全部走 Batch，发版前的紧急评测才走实时 API。

**成本监控与预算告警**：在 CI pipeline 中追踪每次运行的 token 消耗，存储历史数据；当单次 CI 运行 token 消耗超过阈值（如 $5）时发告警；每月生成测试成本报告，识别高成本测试用例并优先优化。

**差距在哪**：新手靠"少调用"控制成本，高手通过分层测试、Prompt Caching、小模型代理和 Batch API 的组合，在不降低覆盖率的前提下将 CI 真实 LLM 调用成本降低 70% 以上。

## Q：工具（Tool）的测试策略

> 来源：Agent 工具开发面试、工具注册中心设计讨论

**新手答**："工具就是函数，直接写单元测试调用函数，看返回值对不对。"

**高手答**：

**工具测试的四个层次**：纯函数逻辑测试（工具核心计算逻辑的单元测试，无任何外部依赖）、工具接口契约测试（验证工具的输入 schema 校验、输出格式是否符合 Agent 期望）、工具与 Agent 的集成测试（验证 Agent 能正确解析 LLM 的工具调用指令并触发工具）、工具的权限和 Hook 测试（验证 PermissionGate 和 Pre/Post Hook 对工具调用的控制逻辑）。

**Schema 验证测试**：每个工具的 inputSchema 应有专项测试，覆盖：合法输入（各字段类型正确）、缺少必填字段、类型错误、超出范围的值、注入字符（SQL 注入、命令注入格式的字符串）。Schema 验证测试是工具安全的第一道防线，必须 100% 覆盖。

**工具的幂等性测试**：对于会产生副作用的工具（写数据库、发通知、调用外部 API），测试重复调用的安全性（是否会产生重复副作用）；Agent 在工具执行中断后重试时，重复执行的工具是否幂等。非幂等的副作用工具是 Agent 系统中的常见 bug 来源。

**外部依赖的 Mock 策略**：工具的单元测试应 Mock 所有外部依赖（数据库、HTTP API、文件系统），只测试工具自身逻辑；集成测试阶段使用真实依赖（测试数据库、沙箱 API）验证完整调用链。避免在单元测试中使用真实外部依赖，降低测试速度和稳定性。

**RiskLevel 与权限测试**：验证不同 RiskLevel 的工具在不同权限配置下的行为：`critical` 级工具是否总是请求用户确认；`high` 级工具在规则允许时是否跳过确认；速率限制是否在超出配额时正确阻断调用。权限逻辑的测试应独立于工具业务逻辑。

**差距在哪**：新手只测试工具函数的返回值，高手能分四个层次测试工具，重点关注 Schema 校验、幂等性、权限逻辑，以及工具与 Agent 编排逻辑的集成验证。

## Q：如何测试 Context 压缩逻辑

> 来源：Agent 上下文管理面试、LLM 长对话系统设计讨论

**新手答**："Context 压缩就是对话太长了截断一下，测测有没有报错就行。"

**高手答**：

**压缩逻辑测试的核心关注点**：不仅要测试"压缩后不报错"，还要测试压缩的触发时机是否正确（在 token 超出阈值时触发，未超出时不触发）、压缩策略的选择是否正确（summary vs aggressive 的切换条件）、压缩后的内容是否保留了关键信息，以及压缩是否破坏了消息格式（role/content 结构）。

**分层压缩的单元测试**：分别为 `none`、`summary`、`aggressive` 三种压缩模式编写测试用例，构造恰好落在各模式触发阈值边界的 messages 输入，验证正确的模式被选中。边界测试示例：目标 token = 60k，输入 58k（触发 none），62k（触发 summary），120k（触发 aggressive）。

**摘要质量验证**：对于 summary 模式，验证摘要结果的 token 数确实低于压缩前；摘要的格式是一条 system 或 assistant 消息（不破坏对话结构）；使用语义相似度检查摘要是否保留了关键信息（工具调用结果、用户明确设定的目标等关键信息不能在摘要中丢失）。

**Token 估算精度测试**：Context 管理依赖 `chars / 3.5` 的 token 估算，需要测试估算值与真实 tokenizer 结果的偏差率（应在 ±10% 以内）；测试中文、代码、JSON 等不同内容类型下的估算精度（这些内容的 chars/token 比例与英文自然语言差异显著）。

**压缩后的对话连贯性测试**：在 Agent 集成测试中，验证经过压缩的对话历史能被 LLM 正确理解（Agent 不因上下文截断而"忘记"用户之前设定的重要背景）。设计专门的"长对话连贯性"测试场景，在 20+ 轮对话后仍能引用早期对话中的关键信息。

**差距在哪**：新手将压缩测试理解为"不报错"，高手能测试触发时机、策略选择、内容保留质量和 token 估算精度，并通过长对话连贯性测试验证压缩对实际效果的影响。

## Q：如何测试 Memory 系统

> 来源：Agent 记忆机制面试、个性化 AI 系统设计讨论

**新手答**："Memory 测试就是写一条记忆进去，然后查出来，看数据对不对。"

**高手答**：

**Memory 系统测试的四个关键验证点**：写入正确性（Memory 内容被完整存储，不丢字、不截断）、查询相关性（query 接口返回的是与当前 Session 最相关的记忆，而非随机记录）、注入时机正确性（Memory 在正确的时机被注入到 Context 层，不早注入也不漏注入）、Session 隔离性（用户 A 的 Memory 不出现在用户 B 的 Context 中）。

**查询相关性测试**：构造多条不同主题的 Memory 记录，用不同的 query 分别查询，验证返回结果的相关性排序是否合理。示例：存储"用户偏好 Python"和"用户目标公司是字节跳动"，用 query "编程语言" 时应优先返回第一条。若 Memory 使用 FTS5 全文检索，测试关键词匹配的精确性和 BM25 排序的正确性。

**注入位置验证**：在集成测试中，捕获每轮对话发给 LLM 的完整 messages，断言 Memory 层的内容出现在系统 prompt 的正确位置（按优先级排序）；验证 Memory 注入不会超出 Context 预算（Memory 条目过多时应有截断策略）。

**隔离性测试**：创建两个不同 sessionId 的用户，分别写入不同的 Memory 记录，交叉查询（用 A 的 sessionId 查询 B 的 Memory），断言返回为空。隔离性测试是 Memory 系统安全测试的基础，隔离失败会导致用户数据泄露。

**Memory 生命周期测试**：测试 Memory 的增删改逻辑（覆盖、追加、删除特定记录）；测试 Session 结束后 Memory 的持久化（进程重启后 Memory 仍可查询）；测试 Memory 容量上限触发时的淘汰策略（LRU、时间衰减等）是否按设计执行。

**差距在哪**：新手只测试 Memory 的 CRUD 正确性，高手能测试查询相关性、注入时机、Session 隔离以及生命周期管理，并将隔离性测试视为安全测试的必要组成部分。

## Q：Contract Testing for LLM Services

> 来源：AI 平台工程面试、微服务契约测试讨论

**新手答**："契约测试就是测一下 API 接口返回的格式对不对。"

**高手答**：

**LLM API 契约测试的独特性**：LLM API 不是普通 REST API——其契约包含：请求格式（messages 结构、工具定义格式）、响应格式（流式事件类型、工具调用结构）、行为语义（支持的模型名、最大 token 限制、支持的功能集合）。这些契约在模型版本升级或提供商更新 API 时可能静默变化，导致应用在无代码改动的情况下突然故障。

**消费者驱动契约测试（CDCT）**：由应用侧（消费者）定义对 LLM API 的期望契约（使用 Pact 等工具），而非信任提供商的文档。消费者定义的契约包括：特定请求下的响应结构（必须包含哪些字段）、错误码在特定场景下的行为（如超出 token 限制时返回 400 而非 500）。当提供商 API 更新时，验证新版本是否仍满足消费者契约。

**关键契约用例设计**：工具调用响应格式（`tool_use` 事件的 schema 是否稳定）；流式结束事件（`message_stop` 是否总在流末尾出现）；错误响应格式（429 限流和 500 错误的 body 结构）；Prompt Caching 头部（`cache_read_input_tokens` 字段是否存在且类型正确）。

**自动化定期检测**：在 CI 中定期（每天一次）向 LLM 提供商发送标准化的契约探测请求，验证响应是否符合预期格式。一旦检测失败（提供商悄悄修改了 API 行为），立即触发告警，在影响生产环境前发现变更。

**多 Provider 的契约统一**：维护一份统一的 Provider 接口规范（`LLMProvider` interface），为每个 Provider（Claude、OpenAI、DeepSeek）分别编写适配器契约测试，验证各 Provider 的实现是否符合统一接口约定。切换 Provider 时运行契约测试套件，而非手动 review 差异。

**差距在哪**：新手把契约测试理解为"测接口格式"，高手能设计消费者驱动的契约，涵盖结构、语义、行为三个维度，并通过自动化定期探测实现对提供商 API 变更的持续监控。

## Q：测试覆盖率在 AI 代码中的局限性

> 来源：AI 工程质量讨论、测试策略架构面试

**新手答**："覆盖率越高测试质量越好，AI 代码也要追求高覆盖率。"

**高手答**：

**行覆盖率的根本局限**：行覆盖率衡量"每行代码是否被执行过"，对 AI 代码尤其无效。原因：LLM 的 Prompt 是字符串常量，100% 行覆盖不代表 Prompt 的语义被测试过；分支覆盖无法捕获 LLM 返回格式变化导致的解析错误；工具的 schema 定义代码即使行覆盖 100%，也无法发现 schema 与 LLM 实际期望不符的问题。

**必要的补充指标体系**：行为覆盖率（Golden Set 中有多少业务场景被测试集覆盖）；失败模式覆盖率（错误路径：工具失败、LLM 超时、格式错误——各有多少被测试用例覆盖）；Prompt 变体覆盖率（Prompt 中的条件分支——不同角色、不同场景——有多少被测试到）；注入攻击覆盖率（已知攻击类型中有多少被纳入安全测试）。

**变异测试（Mutation Testing）作为补充**：用 Stryker 等工具对业务逻辑代码（非 LLM 调用代码）做变异测试，评估测试用例检测 bug 的能力。变异体杀死率（Mutation Score）比行覆盖率更能反映测试的有效性。但变异测试成本高，应集中在核心业务逻辑模块。

**语义覆盖的测量方法**：对 Golden Set 中的场景做覆盖分析，将场景按业务维度（技能领域、用户类型、对话阶段）分类，识别覆盖空白的维度。补充 Golden Set 时优先填补维度空白，而非随机增加用例数量。

**"测试死角"的识别策略**：在 Agent 执行中记录每个工具和代码路径的调用频次，识别测试中从未触发的路径（而非只看代码行是否执行）；通过生产环境的错误日志反向识别测试未覆盖的真实故障模式，将其补充到测试集。

**差距在哪**：新手将行覆盖率视为测试质量的代理指标，高手理解 AI 代码测试质量需要行为覆盖、失败模式覆盖、语义覆盖等多维度指标衡量，行覆盖率仅作为最低门槛而非目标。
