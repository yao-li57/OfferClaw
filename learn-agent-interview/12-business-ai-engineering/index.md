# 商业 AI 工程

## Q：LLM 应用的 ROI 计算与成本控制

> 来源：AI 产品经理面试、技术负责人岗位

**新手答**："ROI 就是用 AI 节省了多少人力，成本就是 API 调用费用。"

**高手答**：

**ROI 的完整计量框架**：收益侧包括直接收益（人力替代、处理量提升、错误率降低）和间接收益（用户体验改善带来的留存提升、新业务场景开拓）。成本侧包括 LLM API 调用费、基础设施费、开发维护人力、数据标注和评测成本，以及风险成本（合规罚款、业务中断）。

**Token 成本的量化与控制**：建立 token 消耗基线（按场景统计平均 prompt + completion token 数）；识别高成本场景（长上下文、多轮对话、高频调用）；用 Prompt 压缩、上下文截断、结果缓存（相同 hash 输入复用结果）、模型降级（低复杂度任务用小模型）等手段控制成本。

**成本与质量的权衡点**：用成本-质量曲线分析不同模型/参数配置下的性价比；识别"成本边际收益递减"的拐点——超过某个成本阈值后质量提升微乎其微，该点是最优配置。

**ROI 的时间维度**：AI 项目初期成本高（数据准备、模型评测、工程开发），收益滞后；需要建立 12-24 个月的 ROI 预测模型，而非只看当期成本，避免因短期负 ROI 而过早放弃。

**成本监控体系**：建立实时成本仪表盘（按功能、按用户、按时间段细分）；设置成本告警（单日消耗超阈值通知）；定期进行成本审计（识别异常调用、低效 Prompt）。

**差距在哪**：新手将 ROI 简化为"省人力减 API 费"，高手能构建完整的收益-成本框架，量化 token 成本控制策略，并建立持续的成本监控机制。

## Q：AI 功能的用户体验设计原则

> 来源：AI 产品设计讨论、产品工程师面试

**新手答**："AI 功能体验好就是回答准确、速度快。"

**高手答**：

**透明性原则**：用户需要知道他们在与 AI 交互，以及 AI 的能力边界。不要隐瞒 AI 身份；主动告知 AI 可能出错；对不确定的输出显示置信度提示（"此建议仅供参考，请专业人士核实"）。

**可控性原则**：用户应能随时中断 AI 任务、修改 AI 的中间结果、撤销 AI 的操作。"AI 帮你做了 X，你可以修改或撤销"比"AI 已完成，结果如下"有更好的用户体验。Agent 执行长任务时需要提供暂停和干预入口。

**渐进披露（Progressive Disclosure）**：默认展示简洁结果，允许用户按需查看推理过程、参考来源和置信度。避免用冗长的 AI 输出淹没用户；提供"展开详情"而非强制显示全部内容。

**失败体验设计**：AI 失败时的体验比成功时更重要。失败时应给出具体原因（而非通用错误信息）、提供降级方案（AI 不行时如何人工完成）、以及反馈入口（让用户报告问题）。

**延迟感知管理**：LLM 响应延迟通常在 2-10 秒，超过 3 秒用户感知明显。工程手段：流式输出（streaming）让用户看到逐字生成而非等待完整结果；骨架屏或进度指示器；对长任务异步执行并通知完成，而非阻塞用户。

**差距在哪**：新手将 AI 体验等同于功能准确性，高手从透明性、可控性、失败体验和延迟感知等多维度设计，覆盖 AI 功能的完整使用链路。

## Q：LLM 系统的合规性（GDPR、数据留存）

> 来源：企业 AI 落地讨论、安全合规工程师面试

**新手答**："合规就是用户数据不能泄露，加个加密就行。"

**高手答**：

**数据最小化原则**：只收集和处理完成任务必要的数据。用户对话中可能包含个人信息（姓名、地址、健康信息），应在传入 LLM 前进行 PII（个人身份信息）检测和脱敏，避免将敏感数据发送给第三方 LLM 服务。

**数据留存策略**：GDPR 要求数据不能超期留存。对话日志应设置自动删除策略（如 90 天）；不同数据类型的留存期不同（运营日志 vs 个人数据）；用户有权要求删除其数据（右被遗忘），系统需支持按用户 ID 级联删除所有相关数据。

**第三方 LLM 服务的数据处理协议**：使用 OpenAI、Anthropic 等云服务时，需签署数据处理协议（DPA），明确数据不被用于模型训练、数据驻留地区是否符合业务要求（欧盟数据不能存储在美国）。需要更高合规要求时考虑私有化部署。

**Audit Trail（审计追踪）**：合规要求系统保留可审计的操作记录——谁在何时发送了什么查询、AI 返回了什么。审计日志需要防篡改（只追加，不可修改）、加密存储、访问受控。这与 GDPR 的数据最小化存在张力，需要权衡。

**员工 AI 使用的合规管控**：企业内部 AI 工具需要防止员工将机密业务信息发送给公共 LLM 服务。工程手段：数据分类标签系统（标记机密数据）、DLP（数据防泄露）集成、AI 使用策略培训和技术拦截。

**差距在哪**：新手将合规简化为"加密"，高手能覆盖数据最小化、留存策略、第三方 DPA、审计追踪和企业内部合规管控等完整合规工程链路。

## Q：如何向非技术利益相关方解释 AI 系统的局限性

> 来源：技术负责人面试、AI 项目管理讨论

**新手答**："直接告诉他们 AI 会犯错，准确率不是 100%，让他们有心理准备。"

**高手答**：

**用业务语言而非技术语言**：将"幻觉"解释为"AI 有时会自信地给出错误答案，就像一个博学但偶尔记错事实的顾问"；将"上下文窗口限制"解释为"AI 一次能处理的文件长度有限，超长文档需要分段处理"。避免使用模型参数、向量检索等术语。

**用数据量化而非定性描述**：不要说"AI 可能出错"，而是说"在我们的测试中，AI 在 X 类场景的准确率为 92%，在 Y 类场景为 78%，Y 类场景需要人工复核"。量化的局限性比定性描述更能帮助利益相关方做资源规划。

**展示局限性的业务影响**：将局限性翻译成业务影响："因为 AI 无法处理实时数据，库存信息会有最多 2 小时的延迟，可能影响紧急采购决策"。让利益相关方理解局限性在他们的工作流中意味着什么，而非仅技术上的限制。

**提供缓解方案而非单纯陈述问题**：每个局限性应伴随缓解措施："AI 在法律条文解读上准确率较低，但我们设计了人工复核环节，所有 AI 建议在生效前须经法务确认"。展示团队已经管控风险，而非只是披露风险。

**建立持续沟通机制**：AI 系统的局限性会随模型更新、数据变化而改变，应建立定期汇报机制（月度或季度）向利益相关方同步系统表现和局限性变化，而非仅在项目启动时一次性说明。

**差距在哪**：新手将沟通局限性理解为"设置预期"，高手用业务语言、量化数据、业务影响翻译和缓解方案构建完整的局限性沟通框架，建立利益相关方的持续信任。

## Q：AI 功能的 Feature Flag 与灰度发布

> 来源：AI 产品发布策略讨论、平台工程师面试

**新手答**："Feature Flag 就是开关，灰度就是先给一部分用户用。"

**高手答**：

**AI 功能 Feature Flag 的特殊性**：传统功能 Feature Flag 控制代码路径；AI 功能的 Feature Flag 还需要控制模型版本、Prompt 版本、工具配置等多个维度。应将这些配置纳入统一的 Feature Flag 体系，而非分散管理。

**灰度维度设计**：按用户分群（内部员工 → Beta 用户 → 5% 普通用户 → 100%）；按使用场景（仅在某类 query 上启用新模型）；按地区（数据合规要求）；按风险级别（低风险功能先全量，高风险功能长期保持灰度）。多维度灰度能更精确地控制风险暴露面。

**灰度期间的评测指标**：定义灰度成功标准（核心指标不下降超过 X%）；监控 AI 特有指标（任务完成率、用户反馈评分、幻觉率、成本变化）；与对照组（旧功能）进行 A/B 对比，统计显著性达标后再扩量。

**快速回滚机制**：Feature Flag 的价值在于出问题时能秒级回滚，而无需代码回退或重新部署。AI 功能的回滚应覆盖模型版本、Prompt 版本和工具配置的一键还原，回滚后自动触发告警通知相关团队。

**Canary 分析自动化**：建立自动化的 Canary 分析 pipeline，在灰度期间持续对比实验组和对照组的关键指标，当差异超过阈值时自动暂停扩量并告警，减少人工监控负担。

**差距在哪**：新手停在"开关"和"分流"的概念，高手理解 AI 功能 Feature Flag 的多维度配置管理、灰度期间的 AI 特有评测指标，以及自动化 Canary 分析和快速回滚的工程实现。

## Q：LLM 系统的 SLA 设计

> 来源：平台工程师面试、企业 AI 系统设计讨论

**新手答**："SLA 就是保证系统的正常运行时间，99.9% 可用性之类的。"

**高手答**：

**LLM 系统 SLA 的特殊挑战**：传统 SLA 关注可用性（能不能访问）和延迟（响应多快）；LLM 系统还需要关注质量 SLA（AI 输出是否满足质量标准）和成本 SLA（单请求成本是否在预算内）。质量和成本维度是 LLM 系统独有的 SLA 层次。

**延迟 SLA 的设计**：LLM 响应延迟波动大（P50 vs P99 差距可达 5-10 倍），不能只承诺平均延迟。应分别定义 P50、P95、P99 延迟目标；流式输出场景定义 TTFT（首 token 时间）和整体完成时间两个指标；区分同步接口（直接返回）和异步接口（轮询/webhook）的不同 SLA 标准。

**质量 SLA 的量化**：将质量指标纳入 SLA 合同：任务完成率 ≥ X%（基于 Golden Set 评测）；幻觉率 ≤ Y%（基于 Faithfulness 评测）；工具调用成功率 ≥ Z%。质量 SLA 需要持续监控而非一次性测试，建立每日自动评测 pipeline。

**供应商依赖管理**：LLM 系统的 SLA 受上游模型供应商（OpenAI/Anthropic）SLA 约束。应设计多供应商 fallback（主供应商不可用时自动切换备用供应商）；在 SLA 文件中明确"供应商不可用"的免责条款；与企业客户协商合理的依赖链透明度。

**降级策略（Graceful Degradation）**：定义系统在部分失败时的降级行为：LLM 不可用时提供规则引擎兜底；质量低于阈值时触发人工接管；高负载时优先保障核心功能，非核心 AI 功能降级或排队。降级策略本身是 SLA 的组成部分。

**差距在哪**：新手将 SLA 理解为"可用性百分比"，高手能设计 LLM 特有的延迟（分位数）、质量和成本三维 SLA，处理供应商依赖风险，并定义完整的降级策略。

## Q：AI 产品的竞争壁垒分析

> 来源：技术策略面试、AI 创业公司产品讨论

**新手答**："AI 产品的壁垒就是模型更好、数据更多。"

**高手答**：

**数据壁垒**：拥有竞争对手无法获取的专有数据（用户行为数据、行业垂直数据、标注数据集）是最持久的壁垒。数据壁垒的关键不是数据量而是数据的独特性和质量——公开数据没有壁垒，私有数据飞轮（用户使用 → 产生数据 → 模型改进 → 吸引更多用户）才是壁垒。

**工作流嵌入壁垒**：AI 功能深度嵌入用户的日常工作流后，切换成本极高（数据迁移、习惯重建、历史上下文丢失）。这比"更好的 AI"更难被竞争对手复制——即使竞争对手的模型更好，用户已有的上下文积累和工作流整合难以迁移。

**网络效应壁垒**：多用户协作场景（共享知识库、团队对话历史）产生网络效应，用户越多价值越高，单个用户无法独立迁移。这在 To B 场景尤为显著。

**模型本身不是壁垒**：基础模型能力差距正在缩小，API 切换成本低。纯粹依赖"我们用更好的模型"作为竞争策略极其脆弱——模型供应商的能力提升会快速抹平这一优势。

**工程执行速度壁垒**：在 AI 领域快速迭代的工程能力（评测体系完善、灰度发布成熟、Prompt 工程积累）使团队能比竞争对手更快将新模型能力转化为产品价值。这是软性但真实的壁垒。

**差距在哪**：新手将竞争壁垒归结为模型和数据量，高手能区分数据壁垒的独特性要求、工作流嵌入壁垒、网络效应，并清楚模型本身为何不构成持久壁垒。
