# 记忆与上下文管理

## Q：什么是 Sliding Window 上下文策略？有什么缺陷？

> 来源：Agent Loop 工程面试

**新手答**："就是只保留最近几条消息，早的删掉。"

**高手答**：

1. **实现原理**：维护一个固定大小的消息窗口，当总 token 数超过阈值（如 context limit 的 80%），从最早的非系统消息开始删除，直到 token 总量降到阈值以内。

2. **缺陷一：关键信息丢失**：第 1 轮中用户提供的重要约束（"我的工作年限是 3 年"、"我应聘的是 Staff 级别"）被删除后，后续分析会丢失关键背景，产生不一致的结论。

3. **缺陷二：无区分删除**：按时间顺序删除，没有考虑消息的重要性。一个包含关键信息的早期消息，和一个无意义的"好的"确认消息被同等对待。

4. **改进方向**：加入 Message Pinning（固定关键消息不被删除）、重要性权重（高重要性的消息排在删除队列末尾）或在删除前先做 Summarization（把多条早期消息压缩成一条摘要保留）。

5. **使用场景**：Sliding Window 适合简单对话类应用，不适合需要长程记忆的 Agent 任务（用户目标在第 1 轮说明，Agent 需要记住到第 20 轮）。

**差距在哪**：新手把 Sliding Window 当万能解决方案，高手知道它是最简单的 context 管理策略，但在需要长程一致性的 Agent 场景下缺陷明显，需要配合其他机制。

## Q：Summarization 压缩策略如何实现？质量损失如何控制？

> 来源：大规模对话 Agent 工程面试

**新手答**："让模型把前面的对话总结一下，用总结替换原始消息。"

**高手答**：

1. **触发时机**：当 context 达到目标大小的 70-80% 时触发，而不是等到 90% 才压缩（给压缩步骤本身留有 token 空间，且提前压缩减少压缩紧迫性带来的质量损失）。

2. **保留结构**：摘要不是自由文本，而是结构化的——用户目标 / 已确认信息 / 已完成步骤 / 待完成事项 / 关键约束。结构化摘要比自由文本摘要在后续对话中被更准确地引用。

3. **保留最近 N 轮**：不要对最近的 2-3 轮做摘要，保留原始消息。模型需要最近的对话上下文来保持对话连贯性，摘要只针对早期消息。

4. **摘要质量验证**：生成摘要后，可以用 LLM 对摘要做一致性校验——给模型看原始对话和摘要，问"摘要中是否遗漏了任何关键信息？"，如果有则补充。这个二次校验在高精度场景中值得投入。

5. **摘要标记**：在 messages 中把摘要标记为特殊类型（`role: 'system', type: 'summary'`），让下游处理逻辑能区分原始消息和摘要，方便调试和回放。

**差距在哪**：新手把摘要当简单的"让模型总结一下"，高手知道摘要的结构、触发时机、保留范围和质量验证，每一个设计决策都影响长对话中信息保持的准确性。

## Q：什么是 Aggressive Compression？什么情况下触发？

> 来源：Agent Context 管理深度面试

**新手答**："就是更激进地压缩上下文。"

**高手答**：

1. **触发条件**：当 token 使用量超过目标上限的 2 倍（即使做了普通压缩仍不够）时触发。这通常发生在工具调用返回了大量数据、或用户粘贴了超长内容的场景。

2. **压缩策略**：摘要化所有早期工具调用结果（只保留结论，删除原始数据）；对 Tool calls 历史只保留最近 K 次的；对 user-assistant 对话轮次做大幅度摘要（10 轮 → 1 段摘要）。

3. **信息保留优先级**：System Prompt（100% 保留）→ 用户在最初说明的目标和约束（尽力保留）→ 关键工具结果（摘要保留）→ 中间过程消息（可以删除）。

4. **质量代价的接受**：Aggressive Compression 会损失信息，这是不可避免的代价。目标是让 Agent 能够继续运行，而不是保证完美记忆。损失必须记录在 metadata 中，方便调试。

5. **预防优于治疗**：设计良好的 Agent 应该通过 Tool 结果截断（防止单次 Tool 返回大量数据）和早期 Summarization（在压力累积到需要 Aggressive 之前就开始压缩）来避免频繁触发 Aggressive 压缩。

**差距在哪**：新手不知道有"分级压缩"概念，高手知道压缩策略是渐进的——轻度到重度，有明确触发阈值，并且每级压缩都有对应的质量代价评估。

## Q：Token 估算用 chars/3.5 合理吗？什么时候需要精确 tokenizer？

> 来源：LLM 工程成本控制面试

**新手答**："差不多就行，精确计算太复杂了。"

**高手答**：

1. **chars/3.5 的适用范围**：对英文文本，平均约 4 chars/token，中文约 1.5-2 chars/token，chars/3.5 是混合语言的粗略估算。对于 80% 的场景（判断是否接近 context limit），精度足够。

2. **需要精确 tokenizer 的场景**：精确计费（计算用户账单或内部成本分摊）、接近 context limit 的边界判断（差 100 个 token 就会截断的情况）、Token 预算精确分配（每个模块有固定 token 配额）。

3. **精确 tokenizer 的代价**：每次调用 tiktoken 或模型的 tokenize API 有延迟（约 1-5ms），在高频 context 计算路径上会累积为可感知的延迟。通常用增量估算（只 tokenize 新增消息，缓存已知消息的 token 数）来优化。

4. **不同模型的差异**：GPT-4 用 cl100k_base 分词器，Claude 用自己的分词器，两者对同一文本的 token 数可能差 10-20%。使用多 Provider 时要注意这个差异。

5. **实践建议**：Context 管理中使用估算（性能优先）+ 在实际 API 调用前做一次精确计算（safety check），两者结合，而不是选其一。

**差距在哪**：新手要么坚持用估算（不关心精度），要么追求全精确（不关心性能），高手在不同场景选择不同精度，在关键路径用精确计算，在高频路径用估算 + 安全边距。

## Q：5 层 Context 的优先级设计有什么工程意义？

> 来源：Agent 架构设计面试

**新手答**："分层是为了组织清晰，方便修改。"

**高手答**：

1. **优先级决定注意力分配**：当 context 需要压缩时，低优先级层先被压缩/删除，高优先级层永远保留。system(100) > immediate(90) > knowledge(80) > memory(60) > session(40) 的设计确保了核心约束永远存在于 context 中。

2. **各层职责的清晰划分**：系统层（agent 角色定义）、即时层（当前轮次的特殊指令）、知识层（RAG 检索到的参考内容）、记忆层（用户历史偏好）、会话层（多轮对话历史）。各层独立更新，不互相污染。

3. **工具的 Token 预算分配**：知识层有 token 上限（如最多 3000 tokens 的 RAG 内容），记忆层有 token 上限（如最多 500 tokens 的记忆）。各层预算独立管理，防止某层无限扩张占用其他层的空间。

4. **调试可视化**：分层 context 让调试变得可视——可以单独查看每一层的内容，快速定位"模型为什么有这个行为"（是因为 knowledge 层注入了某个信息，还是 memory 层记录了某个偏好）。

5. **A/B 测试粒度**：可以对单个层做 A/B 测试（如测试不同的 knowledge 层格式对回答质量的影响），而不影响其他层，实验边界清晰。

**差距在哪**：新手把分层看作代码组织问题，高手知道优先级系统是 context 资源管理的核心机制，它决定了在资源紧张时"谁被牺牲"，这直接影响 Agent 在长对话中的行为一致性。

## Q：如何处理单条消息超过 context 限制的情况？

> 来源：文档处理 Agent 工程面试

**新手答**："截断到限制大小就行了。"

**高手答**：

1. **问题场景**：用户粘贴了一个 200 页的 PDF 内容，或 Tool 返回了一个 10 万行的日志文件，单条消息就超过了 context 限制。

2. **不能简单截断**：截断可能切断关键信息（如 PDF 后半段的结论，或日志最后的异常堆栈）。截断前要先判断应该从头截断还是从尾截断，基于任务类型（要找错误就看尾部，要理解概览就看头部）。

3. **分块处理策略**：把超长内容分成重叠的 chunk，逐块处理，汇总各块的结论后再做整体判断（Map-Reduce 模式）。适合需要全文理解的任务。

4. **关键段提取**：用轻量 LLM 或规则先提取最相关的段落（如搜索关键词所在的上下文段落），只把提取后的精简内容加入主 context。适合有明确查询目标的任务。

5. **用户提示**：当检测到用户输入超长时，在接受内容的同时告知用户"您的内容较长，我将分段处理"或"请明确您最关注哪部分，我可以重点分析"，管理用户期望。

**差距在哪**：新手把超长输入当错误（报错或静默截断），高手知道超长输入是高价值场景（用户有真实的深度分析需求），需要专门的分块和提取策略来优雅处理。

## Q：多轮对话中如何保持 Context 的版本管理？

> 来源：对话 Agent 系统设计面试

**新手答**："在 messages 数组里追加就行了，不需要版本管理。"

**高手答**：

1. **为什么需要版本管理**：Agent 的某次中间步骤可能基于"错误的中间状态"（如工具调用失败后的局部信息），如果能回滚到某个 checkpoint，可以重新尝试不同路径，而不是从头开始。

2. **Checkpoint 机制**：每次重要状态变化（新的用户消息、成功的工具调用结果）创建一个 checkpoint，记录当时的完整 messages 列表。checkpoint 存入 DB，不仅仅在内存中。

3. **Rewind 能力**：允许 Agent 或用户回滚到某个 checkpoint，从那个状态重新执行。在调试和 A/B 测试中极有价值——可以用相同的"历史状态"测试不同的 Prompt 策略。

4. **Branch 策略（高级）**：从同一 checkpoint 出发，创建多个"分支"（不同的执行路径），对比哪条路径效果更好。类似 git branching，但对话状态是 DAG 而非线性。

5. **存储代价**：每个 checkpoint 存储完整 messages 列表，对于长对话成本高。优化：只存储 delta（每个 checkpoint 相比上一个的变化），而不是完整快照。

**差距在哪**：新手把对话 context 当不可变的线性历史，高手知道 checkpoint + rewind 是 Agent 调试、自我纠错和 A/B 测试的关键能力，值得在架构中一等公民对待。

## Q：Working Memory（工作记忆）在 Agent 中如何实现？

> 来源：Cognitive Architecture 面试

**新手答**："工作记忆就是当前对话的 context，就是 messages 列表。"

**高手答**：

1. **Working Memory 的认知科学来源**：工作记忆是人类在执行任务时"当前活跃的信息集合"——不是所有已知信息，而是当前正在被使用的信息。在 Agent 中，这对应"当前任务循环中正在被引用和更新的信息"。

2. **与消息历史的区别**：消息历史是所有曾经发生过的对话（类似长期记忆的一部分），Working Memory 是当前正在处理的任务状态（如"当前正在分析的面试题是 X，已经完成了初步评分，待完成改进建议"）。

3. **实现方式**：在 System Prompt 的即时层（immediate layer）维护一个动态更新的"任务状态"字段，每轮循环后更新（Agent 执行了哪一步，当前结论是什么，下一步是什么）。这个字段就是 Working Memory 的工程映射。

4. **Scratchpad 模式**：让 LLM 在输出最终答案前，先在 `<scratchpad>` 标签中写下当前的中间状态和推理过程，然后基于 scratchpad 给出最终回答。scratchpad 内容在下一轮被保留在 context 中，充当 Working Memory。

5. **与 Tool 结果的关系**：Tool 执行结果是外部世界的信息，Working Memory 是 Agent 对这些信息的"加工状态"。好的 Agent 不是简单转述工具结果，而是把工具结果整合进任务状态（Working Memory 更新），再基于更新后的状态决策下一步。

**差距在哪**：新手把 Working Memory 和消息历史混为一谈，高手知道 Working Memory 是任务执行状态的显式表示，通过 scratchpad 或即时层状态字段维护，这是多步任务执行中保持"目标感"的关键机制。

## Q：对话 Session 内消息历史的存储结构如何设计？

> 来源：AI 产品后端工程师面试

**新手答**："把 messages 数组存到数据库里就行。"

**高手答**：

1. **规范化存储**：不要把整个 messages 数组序列化为一个 JSON Blob 存储，而是每条消息独立一行（id, session_id, role, content, tool_call_id, tool_calls, created_at）。这支持按消息查询、统计、分析，而不是每次加载整个 session 的全量数据。

2. **Tool Call 的关联存储**：assistant 消息的 tool_calls 和 tool 消息的 tool_call_id 要关联存储，支持"给定一个工具调用，找到对应的结果"的查询，用于 debug 和审计。

3. **内容存储的大小控制**：工具调用结果可能很大，不要原样存入 DB，对超过一定大小的 content 做截断（DB 存摘要/hash，原始内容存 object storage），防止单条消息占用过多 DB 空间。

4. **索引设计**：最常见的查询是"按 session_id 获取所有消息"，建立 `(session_id, created_at)` 联合索引。对需要分析的字段（role、tool call 类型）建索引，支持统计查询。

5. **软删除**：消息通常不物理删除（有合规和审计需求），用 `deleted_at` 字段实现软删除。用户"清除对话"操作只设置 deleted_at，不删除 DB 行。

**差距在哪**：新手把消息历史当 session state 的一部分（序列化整体存储），高手把每条消息当独立数据实体，设计了支持查询、关联、统计的规范化 schema，这为系统的可运维性和可分析性打好基础。

## Q：长期记忆应该如何存储和检索？

> 来源：对话 AI 系统架构面试

**新手答**："用向量数据库存记忆，检索相似的返回就行。"

**高手答**：

1. **存储选型的取舍**：向量 DB（适合语义相似度检索）、KV store（适合精确键值查找，如 session_id → 用户 profile）、关系型 DB（适合结构化记忆，如"用户的技术栈"字段）。通常需要混合存储，而非单一方案。

2. **记忆的结构化分类**：不是所有信息都适合做 embedding 后存向量 DB。"用户在 Google 工作"这类事实型记忆，适合结构化字段存储（entity + attribute + value 三元组）；"用户上次对话中表现出来的薄弱点"适合做语义 embedding 后存向量 DB。

3. **检索策略**：纯语义相似度检索可能漏掉重要但措辞不同的记忆。混合策略：相似度检索（找语义相关）+ 时间权重（近期记忆权重更高）+ 重要度权重（标记为重要的记忆权重更高）+ 精确关键词匹配（用户姓名/特定词汇）。

4. **记忆的分层**：用户级记忆（跨 session 保存，如用户的技术背景、目标职位）、session 级记忆（仅在当前 session 内有效，如当前对话中提到的具体问题）。两层使用不同的检索范围。

5. **检索 Limit 和 Token 预算**：记忆检索结果要设上限（如 limit=5），并在注入 context 时设 token 预算（如最多 500 tokens 的记忆）。不是越多越好，无关记忆会稀释注意力。

**差距在哪**：新手把记忆系统等同于向量 DB，高手知道记忆是多种类型的信息的集合，需要混合存储、多策略检索、分层管理，向量 DB 只是其中一个组件。

## Q：记忆的重要度评分和衰减如何实现？

> 来源：智能记忆系统设计面试

**新手答**："重要的记忆打个高分，过期的删掉就行。"

**高手答**：

1. **初始重要度评分**：记忆写入时评分，来源于多个信号：①记忆类型（用户明确表达的偏好 > 隐式推断的偏好）；②用户行为（用户主动提供的信息 > 系统推断的信息）；③对任务的相关性（和核心目标相关的信息评分高）。

2. **时间衰减**：重要度随时间衰减，实现公式参考遗忘曲线（Ebbinghaus）：`effective_score = base_score * exp(-decay_rate * time_since_last_access)`。"许久未被引用"的记忆重要度自然降低。

3. **访问增强**：每次记忆被检索并使用（在生成回答时被引用），重要度提升（access_count++ 且重置衰减计时器）。高频使用的记忆不会被衰减删除。

4. **阈值清理**：定期扫描重要度低于阈值的记忆（如 effective_score < 0.1 且超过 30 天未被访问），归档或删除，防止记忆库无限膨胀。

5. **隐私衰减**：用户可以触发特定记忆的强制衰减（类似"让我忘记 X"），或者在 GDPR 合规场景下，接受删除请求后标记对应记忆为 `deleted=true`（软删除，保留审计记录）。

**差距在哪**：新手把记忆管理等同于"存和删"，高手用遗忘曲线模型实现动态重要度管理——记忆有生命周期，使用频率和时间共同决定它是否应该继续存在于系统中。

## Q：如何防止记忆污染（错误记忆的检测与修正）？

> 来源：高可靠 AI 对话系统面试

**新手答**："让模型检查记忆对不对就行了。"

**高手答**：

1. **记忆污染的来源**：LLM 推断错误（把用户的假设性陈述当事实存储）；多次对话中信息矛盾（用户第 1 次说在 A 公司，第 3 次说在 B 公司）；过时信息未更新（存储了 6 个月前的职位，但用户已经换工作）。

2. **写入时的置信度标注**：记忆写入时记录来源和置信度——用户明确陈述的事实（高置信）vs LLM 从对话中推断的（中置信）vs 仅提及一次的细节（低置信）。低置信记忆在检索时标注为"未经确认"。

3. **矛盾检测**：新记忆写入前，检查是否和已有记忆有逻辑矛盾（如同一属性有不同值：location=Beijing vs location=Shanghai）。发现矛盾时，新值替换旧值，同时标记"记忆已更新"，方便审计。

4. **定期验证**：对高重要度的关键记忆（用户的核心目标、技术背景），在对话中自然地做"确认型追问"（"您之前提到您在 Google，目前还在那边吗？"），主动更新过时记忆，而不是永远依赖旧数据。

5. **用户可见性**：允许用户查看和编辑系统对他/她的记忆（"系统记住了我的哪些信息？"），赋予用户修正权，同时增加用户对系统的信任。

**差距在哪**：新手把记忆当只追加的数据，高手知道记忆需要"主动维护"——写入时验证置信度，更新时检测矛盾，定期主动确认关键记忆是否仍然正确，记忆质量才能随时间保持。

## Q：多用户 Agent 如何保证记忆隔离？

> 来源：多租户 AI 平台设计面试

**新手答**："每个用户用不同的数据库就行了。"

**高手答**：

1. **Row-level Security**：所有记忆记录都有 user_id 字段，所有查询都强制附带 WHERE user_id = ? 条件。不是靠应用层逻辑保证，而是靠数据库层面的行级安全策略（防止应用层 bug 导致越权）。

2. **Session 与 User 的关系**：一个 user 可以有多个 session，记忆按 user_id 范围存储，session 内的短期记忆在 session 结束后选择性提升为用户级长期记忆（不是所有 session 内容都归纳为长期记忆）。

3. **共享知识 vs 私有记忆的分层**：全局知识（如面试题库）和用户记忆（如用户的工作经历）存储在不同的表中，查询时全局知识无需 user_id 过滤，用户记忆严格 user_id 隔离。

4. **Embedding 隔离**：如果记忆向量存入向量 DB（如 Pinecone），要用 namespace 或 filter 机制确保向量检索时只检索当前用户的向量，不出现跨用户的向量相似度命中。

5. **审计和调试的隔离**：生产中 debug 某用户问题时，工程师只能访问该用户的记忆（需要显式的用户授权或按流程的访问申请），不能因为 debug 权限直接查看所有用户的私有记忆。

**差距在哪**：新手认为隔离是"查询时加个 user_id 条件"的小事，高手知道多租户隔离是安全边界，需要 DB 层、应用层、向量 DB 层的多层保证，以及严格的审计访问控制。

## Q：System Prompt 的动态组装如何实现？

> 来源：Agent 应用工程面试

**新手答**："把几个字符串拼接在一起就好了。"

**高手答**：

1. **分层组装模式**：维护 5 个独立的 context 层（system/immediate/knowledge/memory/session），每层有独立的内容来源和更新逻辑，最终按优先级 joinContext() 拼接为完整输入。

2. **各层的 Token 预算**：每层有独立的 token 上限，防止某层（如 knowledge 层注入的 RAG 结果）无限扩张侵占其他层空间。当某层内容超出预算时，在该层内部做截断，而不影响其他层。

3. **按轮次动态更新**：每轮 Agent 循环：knowledge 层被新的 RAG 检索结果替换、memory 层被最新的记忆检索结果替换、immediate 层被当前轮次的特殊指令替换，而 system 层始终固定（角色定义和核心约束）。

4. **构建顺序的重要性**：System Prompt 的最终内容要按 LLM 的注意力分布来排列，最重要的约束放在最前，最相关的参考内容（RAG 结果）放在最后（紧邻用户问题）。

5. **调试可视化**：提供 buildSystemPrompt(debug=true) 模式，输出每一层的内容和 token 数，方便在调试时检查"这次 LLM 调用的 context 到底是什么"，而不是盯着合并后的长字符串。

**差距在哪**：新手用字符串拼接实现 System Prompt，高手用分层组装模式——每层独立管理、有预算、可调试，这是 Agent 行为可控性的关键工程基础。

## Q：Context 中知识层与 Session 层发生冲突时如何处理？

> 来源：RAG 系统应用工程面试

**新手答**："模型会自己判断，不用特别处理。"

**高手答**：

1. **冲突的典型场景**：知识库中说"某 API 的返回格式是 JSON"，但用户当前对话中描述了"他们遇到了这个 API 返回 XML 的情况"。两条信息在 context 中共存，模型需要决策。

2. **优先级说明**：在 System Prompt 中明确说明冲突处理原则："当知识库信息与用户提供的当前实际信息冲突时，以用户当前信息为准"。显式指令比依赖模型的内部权衡更可靠。

3. **知识库内容的时间标注**：知识库中的内容可以附带时间戳或版本号，当 LLM 使用知识库信息时，它可以意识到"这是 2023 年的文档，用户描述的是 2024 年的实际情况"，从而做出合理判断。

4. **后处理验证**：对于高风险回答（如技术规范类），在输出后检查回答是否引用了知识库中已知的错误信息，如果检测到冲突，追加"注意：根据您提供的信息，实际情况可能与标准文档不同"的提示。

5. **知识更新机制**：当对话中发现知识库内容可能过时，可以把"用户提供的修正信息"作为新的高优先级知识记录，在后续对话中优先使用，而不是每次都出现冲突。

**差距在哪**：新手信任模型能自动解决冲突，高手知道模型的冲突处理行为不可预测，需要显式的冲突处理规则、知识时效标注和后处理验证来保证一致性。

## Q：如何设计 Context-Aware Retrieval（上下文感知检索）？

> 来源：高级 RAG 系统工程面试

**新手答**："把用户的问题直接拿去检索就行了。"

**高手答**：

1. **问题**：用户在多轮对话中的每条消息都很短（"继续"、"那么第二个问题呢"），这样的消息直接去检索 embedding 无法反映真实意图，导致检索结果无关。

2. **对话历史感知 Query**：不直接用当前用户消息做检索，而是用 LLM 生成"考虑对话历史后，此时用户实际想检索的主题是什么"作为检索 query。类似 query rewriting，但加入了对话历史上下文。

3. **Multi-query 策略**：对于模糊或多维度的问题，用 LLM 生成 3-5 个不同角度的 query，各自检索，合并去重后排序。这覆盖了单 query 可能遗漏的语义角度。

4. **Hypothetical Document Embedding（HyDE）**：让 LLM 先生成"如果知识库有一篇完美回答这个问题的文章，它会是什么"，用这篇假设文章做检索，而不是用原始问题。假设文章的语义更接近知识库中的文档语义。

5. **Metadata 结合**：用对话历史中提到的技术栈、级别、公司类型等信息作为 Metadata Filter，缩小检索范围（如：只检索 dimension=architecture 的文档），提高精准度。

**差距在哪**：新手把 RAG 检索当 keyword search（直接用用户原始输入），高手知道在多轮对话中检索需要感知上下文、重写 query、多角度探索，才能找到真正相关的内容。

## Q：记忆系统的技术选型如何决策？

> 来源：AI 基础设施工程师面试

**新手答**："用 Redis 存记忆，快。"

**高手答**：

1. **结构化事实记忆（用户 Profile）**：关系型 DB（SQLite / PostgreSQL）。支持 SQL 查询、字段索引、事务，适合"用户的技术栈"、"目标职级"等结构化属性。

2. **语义记忆（对话摘要、经验总结）**：向量 DB（Pinecone / Chroma / pgvector）。支持相似度检索，适合"找和当前任务语义相关的历史对话"。

3. **会话级短期记忆**：内存（Map/Dict） + 可选的 Redis。当前 session 的 working memory 读写频繁，内存最快；如果需要跨进程共享或持久化，用 Redis。

4. **大规模离线分析**：数据仓库（BigQuery / 本地 SQLite）。记忆的统计分析、模式挖掘不需要实时，走批处理即可，不要占用 OLTP 数据库资源。

5. **实际建议**：对于 < 1000 用户的 MVP，SQLite + 向量 BLOB（用 pgvector 或 SQLite 的 JSON 字段存 embedding）就够了，不需要引入 Redis 或 Pinecone，等规模增长再迁移。

**差距在哪**：新手按"最快"或"最流行"选存储，高手根据记忆类型的访问模式（结构化查询/语义检索/实时读写/离线分析）分别选择最合适的存储，在 MVP 阶段也知道从简开始，避免过度工程化。

## Q：如何处理 Embedding 模型替换时的记忆迁移？

> 来源：AI 平台版本迁移工程面试

**新手答**："重新生成所有 embedding 就行了。"

**高手答**：

1. **向量不兼容性**：不同 Embedding 模型（甚至同一模型的不同版本）生成的向量处于不同的向量空间，不能混用。用 model-A 检索用 model-B 生成的向量，结果无意义。

2. **全量重建方案**：最干净的方案——用新模型对所有记忆重新生成 embedding，原子切换（旧向量表备份、新向量表生成完成后切换）。代价是重建时间长和 API 成本（大量 embedding 调用）。

3. **双写过渡方案**：迁移期间，新写入的记忆用新旧两个模型都生成 embedding，存入两个向量表。检索时两个表都检索，结果合并。迁移完成后下线旧表。代价是迁移期间双倍存储和 API 成本。

4. **版本标注**：`embeddings` 表中记录 `model TEXT NOT NULL` 字段，检索时只对 model 字段匹配的向量做相似度计算。这让不同版本模型的向量可以共存于同一张表，通过版本 filter 隔离。

5. **增量迁移**：按重要度排序，优先迁移高重要度记忆，低重要度的记忆可以在首次被访问时"懒迁移"（访问时检测到使用旧 model，重新生成新 embedding 后再返回结果）。

**差距在哪**：新手认为换 Embedding 模型是"重新跑一次"的简单操作，高手知道向量迁移需要考虑不兼容性、迁移期间的一致性、成本控制，以及在大规模时如何保证零停机。

## Q：记忆写入应该实时触发还是对话结束后批处理？

> 来源：记忆系统工程设计面试

**新手答**："实时写入更安全，不会丢数据。"

**高手答**：

1. **实时写入的代价**：每条用户消息都触发 LLM 做"这条消息值得记忆吗，如果是记什么"的判断，增加延迟（额外的 LLM 调用）和成本，且大量消息不值得记忆，是浪费。

2. **对话结束后批处理的优势**：一次性处理整个 session，可以看到完整对话上下文再决策（第 1 条消息+第 10 条消息合并看，才能提炼出真正有价值的结论）。避免记录孤立片段。

3. **混合触发策略**：特殊信号触发实时记忆（用户明确说"记住这一点"、用户明确提供重要信息如"我是 Staff 工程师"）→ 即时记录；常规对话结束后批量提炼摘要 → 异步写入。

4. **幂等的异步写入**：对话结束后的记忆写入异步执行，不阻塞用户收到最终回答。写入失败要有重试机制，操作要幂等（同一 session 的记忆只写入一次，多次处理不产生重复记忆）。

5. **用户显式触发**：提供 `/remember [内容]` 命令，允许用户主动控制记忆写入，这比完全靠系统自动判断更准确（尤其是用户认为重要但 LLM 认为普通的信息）。

**差距在哪**：新手默认实时写入（最直觉的方案），高手知道记忆写入的时机设计影响成本、质量和延迟，混合触发策略（高优先级实时+常规异步批量）才是生产中的平衡点。

## Q：如何在不侵犯隐私的情况下使用跨 Session 的用户记忆？

> 来源：AI 产品合规与工程面试

**新手答**："用户用了这个系统，数据就是我的了。"

**高手答**：

1. **明确告知和用户授权**：在用户首次使用时明确告知"系统会记住你的信息用于改善后续体验"，并提供开关（允许用户关闭跨 session 记忆）。这是 GDPR 的基本要求，也是建立用户信任的基础。

2. **记忆的最小化原则**：只记录完成任务所必要的信息，不记录与服务无关的个人信息。例如：记录"用户的技术方向是后端"（服务相关）→ 合理；记录"用户今天心情不好"→ 不必要。

3. **数据留存期限**：设置记忆的默认留存期（如 90 天后自动归档），而不是永久保留。告知用户留存期，并提供提前删除的途径。

4. **用户的数据访问权**：允许用户查看"系统对我的所有记忆"，并允许编辑或删除特定记忆。`GET /user/memories` 和 `DELETE /user/memories/{id}` 这两个接口应该是基本功能，不是高级功能。

5. **记忆的脱敏处理**：对于高敏感度信息（如用户在对话中提到的薪资、医疗状况），在写入记忆前做脱敏处理，只记录抽象层面的信息（"用户有薪资期望"而不是"用户期望年薪 50 万"）。

**差距在哪**：新手把"用户数据"当系统资产无限使用，高手知道个人数据有主体权利，GDPR/CCPA 等法规要求明确授权、最小化收集、用户可访问/删除，违规不只是罚款风险，更是用户信任的崩塌。

## Q：Context 管理的单元测试如何设计？

> 来源：AI 工程质量保障面试

**新手答**："测试很难写，因为依赖 LLM，不确定性太大。"

**高手答**：

1. **ContextManager 的纯函数部分可测试**：`buildSystemPrompt()` 是确定性的（给定相同的各层内容，输出相同的 Prompt），可以直接 assert 输出是否符合预期格式、各层内容是否正确拼接、优先级排序是否正确。

2. **压缩逻辑可测试**：给定 token 估算超过阈值的 messages 列表，assert 压缩后的列表满足：token 估算 ≤ 目标大小、系统消息被保留、最近 N 条消息被保留、删除的是最早的非系统消息。

3. **Token 估算测试**：对已知 token 数的字符串（用精确 tokenizer 验证），测试估算函数的误差范围是否在可接受范围内（如 ±15%）。

4. **Mock LLM 的 context 测试**：用 MockProvider 替代真实 LLM，让 MockProvider 返回固定响应，专注测试 ContextManager 在不同 token 使用量下的行为（如是否在正确的阈值触发压缩）。

5. **快照测试（Snapshot Testing）**：记录某个确定性输入下的 buildSystemPrompt() 完整输出作为快照，下次改动后对比快照，任何预期之外的变化会被发现。

**差距在哪**：新手认为所有 AI 相关代码都因为 LLM 的不确定性而无法测试，高手知道 ContextManager 的核心逻辑（层拼接、Token 估算、压缩策略）都是确定性的，可以和 LLM 解耦做单元测试。

## Q：生产中 Context 管理的关键监控指标是什么？

> 来源：AI 平台 SRE 面试

**新手答**："看错误率就行了，出错了才需要关注。"

**高手答**：

1. **Context 利用率分布**：P50/P95/P99 的 context 利用率（实际使用 token 数 / context limit）。P95 利用率 > 90% 说明经常接近溢出，要优化；P50 利用率 < 20% 说明平均 token 很少，可能有优化空间（用更小的模型）。

2. **压缩触发率**：每小时/每天触发 Summarization 和 Aggressive Compression 的次数/比例。高压缩率说明典型会话比预期长，可能需要重新评估 context 管理策略或增加 token 预算。

3. **Tool 结果 Token 占比**：工具调用结果消耗的 token 占总 context 的比例。如果这个比例 > 50%，说明 Tool 结果没有被充分截断，是 context 膨胀的主要来源。

4. **每 Session 的 Token 总成本**：追踪每个 session 的总 token 消耗，检测异常高消耗 session（可能是工具调用失控或用户输入超长内容）。

5. **压缩质量代理指标**：压缩后 session 中 LLM 的回答质量是否下降（如答案对用户前期提供信息的遗忘率）。可以通过周期性的压缩前后一致性测试来追踪。

**差距在哪**：新手的 context 监控只关注"有没有报错"，高手关注 context 的利用效率和成本—— P95 利用率、压缩频率、Tool 占比是发现 context 管理问题的早期信号，不是等到溢出报错才意识到。

## Q：Token 成本如何在 Context 设计中作为约束？

> 来源：AI 产品成本控制面试

**新手答**："尽量减少发送的 token 就行了。"

**高手答**：

1. **Token 预算作为设计输入**：在 Agent 设计阶段，明确每轮对话的 token 预算（如每轮 ≤ 5000 tokens），这个预算决定了各层的 token 分配（system 层 500，knowledge 层 2000，session 层 2000，其他 500）。

2. **成本 vs 质量的显式权衡**：更多的 RAG 结果 → 更准确的回答，但 token 成本更高。在产品设计时要明确在哪个范围内质量提升是值得的（例如：从 1 条 RAG 结果增加到 3 条，成功率提升 15%；从 3 条增加到 10 条，成功率只提升 2%，边际收益递减）。

3. **模型分级的 Token 策略**：简单任务用 GPT-4o mini / Claude Haiku（每 1k token 便宜 10-30 倍），复杂推理任务用高端模型。Token 成本不只是"发多少 token"，还包括"用哪个模型"。

4. **Prompt Cache 的利用**：Anthropic 的 Prompt Caching 对重复的 System Prompt 部分（不变的 context）做缓存，缓存命中时 input token 成本减少 90%。维持长 System Prompt 不变（只更新末尾的动态部分）可以大幅降低成本。

5. **长期趋势监控**：追踪每日/每周的平均 token 消耗，检测 creep（逐渐增长）。往往是某次无意的 Prompt 改动把 System Prompt 增加了 500 tokens，但没有人注意到，直到成本账单超预算。

**差距在哪**：新手把 token 成本当事后统计的数字，高手把它作为设计约束前置——预算分配、质量边际收益分析、模型分级、Prompt Cache 是把成本控制在架构层面的核心手段。

## Q：如何 debug Context 相关的模型行为异常？

> 来源：AI 工程调试实践面试

**新手答**："改一改 Prompt 再试试。"

**高手答**：

1. **先检查 Context 本身**：在调试前，先完整打印出这次 LLM 调用的实际 context（所有层拼接后的完整内容）。很多"模型行为异常"其实是"context 中有错误的信息"——模型的行为是正确的，只是被喂了错误的输入。

2. **Layer-by-layer 排查**：逐层检查哪一层的内容有问题——是 knowledge 层检索到了无关内容？还是 memory 层包含了过时信息？分层结构让排查可以有序进行，而不是面对一团字符串。

3. **对比最小复现**：构造最小化的 context（只保留 system 层），测试模型行为；然后逐层加入其他内容（先加 memory，再加 knowledge，再加 session），直到行为异常出现，定位到是哪一层引入了问题。

4. **Token 占比分析**：打印每次 LLM 调用的各层 token 数，确认 context 的构成是否符合预期（如 knowledge 层占了 80%，说明 RAG 检索了太多内容，可能是 Tool 结果没有截断）。

5. **历史 Context 对比**：当行为"突然"变化时，对比本次 context 和上次正常时的 context 差异，找出哪部分内容发生了变化。这需要每次调用都有 context 快照存储。

**差距在哪**：新手调试 LLM 行为异常靠"猜"（改 Prompt 试试），高手从 context 本身出发系统排查——先打印实际 context，再逐层隔离，再最小复现，再对比历史。把不确定性转化为可分析的工程问题。

## Q：什么是 Contextual Retrieval？和普通 RAG 有什么区别？

> 来源：Anthropic 技术文档 + RAG 系统架构面试

**新手答**："就是更好的 RAG 吧？"

**高手答**：

1. **背景问题**：普通 RAG 把文档切成 chunk 后，每个 chunk 独立存储 embedding，但 chunk 离开原始文档后丢失了上下文——同一个 chunk 在不同文档中意义可能完全不同（"这个问题的解决方案是..."，解决的是什么问题？chunk 本身不知道）。

2. **Contextual Retrieval 的思路**：在存储每个 chunk 之前，用 LLM 为 chunk 生成"上下文描述"（"这段内容讨论了 X 文档中关于 Y 主题的 Z 方面"），把上下文描述和 chunk 内容合并后再生成 embedding。

3. **效果提升**：根据 Anthropic 的报告，Contextual Retrieval 将检索失败率降低约 49%（结合 BM25 混合检索后降低 67%）。上下文感知的 embedding 更准确反映 chunk 在整体文档中的语义位置。

4. **工程代价**：为每个 chunk 生成上下文描述需要额外的 LLM 调用（build-kb 阶段），对大型知识库成本可观。可以用更小的模型（如 Claude Haiku）做上下文生成，而不是用最贵的模型。

5. **与 HyDE 的区别**：HyDE 是在查询时让 LLM 生成假设文档来改善检索；Contextual Retrieval 是在索引时为每个 chunk 添加上下文来改善检索。两者是互补的，可以组合使用。

**差距在哪**：新手知道 RAG 但不了解 Contextual Retrieval，高手知道 chunk 的上下文丢失是 RAG 质量的关键瓶颈，以及 Anthropic 提出的这个解决方案的原理、工程代价和效果。
