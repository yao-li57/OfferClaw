# Agent 架构设计

## Q：ReAct、CoT、ToT、GoT 这几种 Agent 架构模式有什么区别？实际项目中你怎么选择？

> 来源：某大厂 AI 工程师面试

**新手答**："ReAct 是推理加行动，CoT 是链式思考，ToT 是树形思考，GoT 是图形思考，复杂任务用 ToT 或 GoT，简单任务用 ReAct。"

**高手答**：

**CoT（Chain-of-Thought）**：强迫模型在给出最终答案前输出中间推理步骤。核心价值在于把隐式推理转为显式，便于调试和验证。适合数学推理、逻辑推断等线性可分解问题。缺点是一旦推理链走偏，后续全错，没有回溯能力。

**ReAct（Reason + Act）**：把 CoT 的推理步骤与工具调用交织。每一步 Thought → Action → Observation 形成一个 cycle，观察结果反馈给下一轮推理。工程上最成熟，几乎所有主流 Agent 框架都以此为基础。适合需要与外部环境交互的任务（搜索、代码执行、API 调用）。主要缺陷是线性推进，无法探索多条路径，遇到局部最优容易卡死。

**ToT（Tree-of-Thoughts）**：在每个推理节点维护多个候选分支，用 BFS 或 MCTS 进行搜索。关键工程挑战是评估函数（value function）设计——用 LLM 自评估还是外部 verifier？LLM 自评估容易偏向自己生成的输出（self-consistency bias）。Token 消耗是 ReAct 的 N 倍，N 为分支数。适合有明确正确性标准的任务（定理证明、代码生成+验证）。

**GoT（Graph-of-Thoughts）**：允许推理节点之间有任意有向边，支持多条思维路径的汇聚（aggregation）和分叉（refine）。比 ToT 表达能力更强，但调度复杂度显著上升，工程实现难度高，目前主要停留在研究层面，生产环境鲜有落地。

**选型原则**：先问任务是否需要工具调用 → 是则 ReAct；再问是否存在多条解题路径且需要搜索 → 是则 ToT；再问是否对 Token 成本敏感 → 敏感则降级回 ReAct + 自我修正（self-refinement）。GoT 除非有充分的 research 预算和可以容忍的延迟，否则不推荐生产使用。

**差距在哪**：新手把这四种模式当成选单从高到低选，高手理解每种模式解决的核心问题是什么（搜索空间结构、回溯能力、工具交互），以及背后的 Token 成本和工程复杂度权衡。面试官真正想考的是候选人是否在生产环境踩过坑，能说出 ToT 的 value function 偏差、GoT 的调度开销等具体痛点。

## Q：Agent Loop 中如何设计迭代终止条件？只设置最大迭代次数够不够？

> 来源：Agentic AI 架构师面试

**新手答**："设置一个最大循环次数，比如 10 次，超过就停止，防止死循环。"

**高手答**：

**最大迭代次数是最后一道防线，不是主防线**：只靠 maxIterations 会导致两类问题：任务提前中断（迭代次数不够）或资源浪费（无效迭代被计入次数）。需要分层设计终止条件。

**正常终止检测**：当模型输出纯文本响应（无 tool_use block）时，认为 Agent 认为任务已完成。但 LLM 有时会在完成后多说一句话再调用工具，需要区分"最终答案"和"过渡性文本"。可以在系统提示中明确约定终止 token 或终止格式（如 `<FINAL_ANSWER>`）。

**停滞检测（Stagnation Detection）**：连续 N 次调用同一工具、传入相同参数，结果相同，说明 Agent 陷入局部循环。解法是对每次 tool call 的（tool_name, params_hash）做 sliding window 去重，触发后强制注入 hint message 或降级到更简单的策略。

**Token 预算终止**：当本轮已消耗 token 超过预算阈值（如 80%），主动终止并输出已有进展的部分结论，而不是等到 context 窗口撑满后报错。

**时间维度终止**：全局超时（整个任务 deadline）和单步超时（单次工具调用 timeout）分开控制。全局超时触发时需要做 graceful shutdown：完成当前工具调用，输出已有结论，持久化中间状态。

**用户意图检测**：如果 Agent 在循环中发现任务目标已无法完成（如依赖的外部资源不可用），应主动终止并上报，而不是无意义地重试耗尽迭代次数。

**差距在哪**：新手只考虑防死循环，高手考虑的是如何在任何终止条件触发时都能优雅退出并保留价值（部分结论 + 状态快照）。面试官想考察候选人是否理解"终止"本身也是一个需要精心设计的状态转换，而不是简单地 break 出循环。

## Q：Agent 的状态机应该如何设计？状态转换出错了怎么处理？

> 来源：AI 基础设施工程师面试

**新手答**："Agent 有 idle、running、done 三个状态，用一个变量记录就行了。"

**高手答**：

**状态枚举的完整性**：最小完备的状态机应包含：`idle`（等待输入）、`active`（正在处理用户请求）、`paused`（等待用户确认高风险操作）、`error`（工具调用失败，等待恢复策略决策）、`completed`（本轮任务结束）。缺少 `paused` 和 `error` 状态是最常见的设计缺陷，导致在需要人工介入时无处"停住"。

**状态转换的合法性校验**：每次状态转换都要 validate 前置状态。比如只有 `idle` 可以转到 `active`，`active` 不能直接转到 `completed` 而不经过工具调用或文本输出阶段。用状态转换矩阵或有限状态机库管理，非法转换应抛出异常而不是静默忽略。

**状态持久化与幂等恢复**：生产级 Agent 的状态不应只存在内存中。每次状态转换后写入持久化存储（数据库或消息队列），包含：当前状态、消息历史快照、已执行工具调用列表、时间戳。崩溃重启后从持久化状态恢复，重放未完成的步骤。关键：重放必须是幂等的，工具调用需要携带幂等键。

**错误状态的处理策略**：进入 `error` 状态时，需要决策：自动重试（适合临时性错误如网络超时）、降级（换备用工具或备用模型）、升级人工（critical 错误或超过重试上限）。三种策略的触发条件要明确编码，不能靠经验判断。

**并发安全**：多用户场景下，同一 session 的状态机更新必须是原子的。用数据库乐观锁或 session-level 锁防止并发写冲突。

**差距在哪**：新手把状态机当成一个枚举变量，高手把它当成系统可靠性的核心契约。面试官考察的是候选人是否理解：状态机是 Agent 行为可预测性和可恢复性的基础，每一个缺失的状态都是一个潜在的生产事故入口。

## Q：Planner 和 Executor 分离的架构是怎么设计的？什么情况下值得引入这层分离？

> 来源：LLM 系统设计面试

**新手答**："Planner 负责生成计划，Executor 负责执行，分开了更清晰。"

**高手答**：

**为什么要分离**：单体 ReAct Agent 把规划和执行混在同一个 LLM 调用中，导致两个问题：1）规划质量受执行结果干扰（Agent 看到一个工具失败就修改整体计划，而不是只处理局部失败）；2）难以并行化执行——如果计划中有多个独立步骤，单体 Agent 只能串行。

**Planner 的职责与输出格式**：Planner 接收用户 goal，输出结构化执行计划，通常是 DAG（有向无环图）形式，节点是子任务，边是依赖关系。输出格式需要机器可解析（JSON/YAML），包含：任务 ID、任务描述、依赖任务 ID 列表、所需工具、预估 token 消耗、失败处理策略（retry/skip/abort）。

**Executor 的职责**：按 DAG 拓扑序执行子任务，管理子任务状态，将结果反馈给依赖节点。Executor 不应修改计划本身，只负责执行和状态上报。当子任务失败时，Executor 通知 Planner，由 Planner 决定是否重新规划。

**Re-planning 的触发时机**：不是每次工具失败都需要 Re-planning——网络超时应由 Executor 自己重试；但如果工具返回的结果证明原始假设错误（如搜索结果表明目标文件不存在），则需要通知 Planner 重新规划。区分"执行层失败"和"规划层失败"是关键。

**引入分离的成本**：增加了两次 LLM 调用（一次 plan，一次或多次 execute），延迟上升；Planner 输出格式需要严格约束，否则 Executor 解析失败；需要额外的协调逻辑。建议只有任务明确需要并行化或任务步骤数 > 5 时才引入。

**差距在哪**：新手理解的分离是"代码分模块"，高手理解的是"关注点分离带来的并行执行能力和规划稳定性"。面试官考察候选人是否能说出 Re-planning 的触发条件，以及何时分离比单体 Agent 更差（简单任务的额外开销）。

## Q：同步 Agent 和异步 Agent 架构有什么本质区别？什么业务场景下必须用异步？

> 来源：AI 产品后端架构面试

**新手答**："同步就是等待结果返回，异步就是不等待，用 async/await 实现。"

**高手答**：

**本质区别在于任务生命周期与请求生命周期的解耦**：同步 Agent 要求客户端连接保持到任务完成，HTTP 连接/WebSocket 连接是任务状态的载体。异步 Agent 把任务状态持久化到外部存储，客户端通过轮询或回调获取结果，连接断开不影响任务执行。

**同步架构的上限**：适合任务完成时间 < 30 秒、用户可以等待的场景。超过此时限，HTTP gateway 超时、移动客户端后台杀进程、网络抖动等都会导致任务状态丢失。LLM 的 streaming 可以延长用户感知等待时间，但无法解决后端任务持续时间的限制。

**异步架构的核心组件**：任务队列（Kafka/RabbitMQ/SQS）负责任务分发；Worker 进程负责 Agent 执行；状态存储（Redis/PostgreSQL）记录任务进度；Webhook 或 SSE 推送通知客户端。每个组件都需要独立的可靠性设计。

**必须用异步的场景**：1）任务执行时间超过 30 秒（如完整简历优化流程）；2）任务需要多轮人工确认（human-in-the-loop），执行可能暂停数小时；3）任务执行成本高需要去重（相同任务不重复执行）；4）需要任务优先级队列（VIP 用户优先）；5）多步骤任务需要在不同机器上分布式执行。

**异步 Agent 的陷阱**：任务 at-least-once 语义导致工具调用被重复执行，必须保证工具幂等；任务状态与消息队列 offset 的一致性（处理了消息但状态写入失败怎么办）；用户长时间轮询的成本（建议改用 SSE 推送）。

**差距在哪**：新手用编程语言的 async/await 理解异步，高手从系统架构层面理解任务生命周期与连接生命周期的解耦，并能说出异步引入的一致性问题（at-least-once + 幂等）。面试官通常会追问"任务执行了一半 Worker 崩溃怎么办"来考察候选人对状态一致性的理解。

## Q：在 Single-agent 和 Multi-agent 之间如何做架构选型？Multi-agent 的主要工程挑战是什么？

> 来源：某互联网大厂 AI 平台面试

**新手答**："复杂任务用 Multi-agent，简单任务用 Single-agent。Multi-agent 就是多个 Agent 分工合作。"

**高手答**：

**Single-agent 的上限与瓶颈**：单个 Agent 的 context window 是硬上限，复杂长任务会撑满；单个 Agent 无法并行执行独立子任务；所有工具共享同一个系统提示，随工具数增加，提示变长、LLM 工具选择准确率下降（实测工具数 > 20 时命中率开始明显下降）。

**Multi-agent 的三种典型拓扑**：1）Orchestrator-Worker：一个协调 Agent 分解任务，多个专家 Agent 并行执行子任务，结果汇总；2）Pipeline（流水线）：Agent A 的输出是 Agent B 的输入，串行处理，适合有明确步骤顺序的工作流；3）Debate/Critique：多个 Agent 对同一问题给出不同角度的答案，最终 Agent 综合评判，提高推理质量，代价是 Token 消耗成倍增加。

**Multi-agent 的核心工程挑战**：

- **上下文传递**：Agent 间通信如何传递足够的上下文而不传递冗余信息？直接传原始消息还是传摘要？这影响信息损失和 Token 成本之间的权衡。
- **错误传播**：上游 Agent 的错误结论会被下游 Agent 当作事实继续推理，错误被放大。需要在每个 Agent 边界做结果校验。
- **循环依赖检测**：Agent A 委托 Agent B，Agent B 又委托 Agent A，形成死锁。需要在调度层维护调用图并检测环路。
- **成本爆炸**：N 个 Agent 并行 × M 次迭代 × K tokens/call，成本呈乘数级增长，需要全局 Token 预算控制。
- **调试复杂度**：分布式调用链难以追踪，必须从第一天起接入 distributed tracing（如 OpenTelemetry）。

**选型决策树**：任务能否在 128k context 内完成 → 是则 Single-agent；子任务之间是否独立可并行 → 是则 Orchestrator-Worker；是否需要多视角校验 → 是则 Debate 模式；否则考虑 Pipeline。

**差距在哪**：新手把 Multi-agent 当成银弹，高手知道它是有代价的工具。面试官考察的重点是候选人是否踩过错误传播和成本爆炸的坑，以及是否知道在什么条件下 Single-agent + 更大 context 模型反而是更好的选择。

## Q：流式输出（Streaming）对 Agent 架构有哪些深层影响？不只是"更快显示"这么简单吧？

> 来源：LLM 应用架构师面试

**新手答**："流式输出就是 token 一个个出来，用户看起来更快，用 SSE 或 WebSocket 实现就好了。"

**高手答**：

**对工具调用检测的影响**：非流式模式下，整个 LLM 响应到达后统一解析是否包含 tool_use；流式模式下，tool_use block 可能分散在多个 chunk 中，需要维护一个流式解析状态机，实时拼接 JSON 片段，在 stop_reason 为 `tool_use` 时才能确认工具调用完整。任何解析错误都可能导致工具参数截断。

**对并行工具调用的影响**：Claude 等模型支持在一次响应中返回多个 tool_use block（并行工具调用）。流式模式下需要检测多个 tool_use block 的边界，并在所有 block 接收完毕后才能开始并行执行，否则后续工具可能依赖前一个工具的结果。

**对错误处理的影响**：流式传输中途网络断开，已接收的部分 token 不完整，不能直接拼接使用。需要区分"流正常结束"（received stop_reason）和"流异常中断"（网络错误/超时），前者可以使用已接收内容，后者需要重试整个调用。

**对 Token 计数的影响**：流式模式下输入 token 数在请求时已知，但输出 token 数需要流结束后才知道（或通过 usage block 获取）。实时 Token 预算控制需要在流传输过程中估算已输出 token 数，达到阈值时主动中断（需要 provider API 支持）。

**对用户体验设计的影响**：在工具调用阶段，模型不输出文本 token，用户界面会出现"静止"。需要在工具执行期间显示进度提示（如"正在搜索…"），这些提示不是来自 LLM 输出流，而是应用层注入的状态更新，需要设计统一的事件流协议来混合 LLM token 和状态更新事件。

**差距在哪**：新手只关注流式输出的展示效果，高手理解流式模式改变了响应解析的时序，引入了状态机解析、并发控制、错误区分等工程复杂度。面试官考察候选人是否实际实现过流式 Agent，能说出 tool_use block 的流式解析细节。

## Q：如何设计 Agent 的扩展性，使得新增一个工具不需要改动核心 Agent 代码？

> 来源：AI 工程基础架构面试

**新手答**："把工具放到一个列表里，新增工具就往列表加，不改 Agent 代码。"

**高手答**：

**Tool Registry 模式**：工具以声明式方式注册到 Registry，包含：name、description（供 LLM 理解）、input schema（JSON Schema）、handler 函数、risk level、rate limit 配置。Agent Loop 从 Registry 查询可用工具列表，动态构建传给 LLM 的 tools 参数，无需硬编码工具信息。

**系统提示的解耦**：常见反模式是在系统提示里手工列举所有工具能力，每次新增工具都要同步更新提示。正确做法是系统提示描述 Agent 的通用行为准则，具体工具能力通过 tools 参数传给 LLM，提示只描述"你有一套工具可以使用"而不枚举每个工具。当然，如果工具有复杂的使用约束（先调用 A 再调用 B），可以在提示中描述这些约束，但要与工具 description 保持一致。

**工具版本管理**：同一工具可能有多个版本（v1/v2），需要支持在 Registry 中同时注册不同版本，通过路由规则决定特定 session 或特定用户使用哪个版本（灰度发布）。工具 schema 变更时需要 backward compatibility 策略，避免存量 session 中 LLM 生成的参数格式与新 schema 不匹配。

**动态工具加载**：某些场景需要运行时动态加载工具（如用户上传了自定义工具定义）。需要在 Registry 中支持动态注册和注销，同时要防范恶意工具（Tool Injection 攻击），在注册时做 schema 校验和沙箱隔离。

**工具的 capability 声明**：工具应声明自己的副作用级别（read-only / write / external-call），Agent 在构建工具列表时可以根据当前任务上下文过滤工具，避免在只需要查询的场景中向 LLM 暴露写操作工具，降低误操作风险。

**差距在哪**：新手理解的扩展性是"代码可以修改但改动少"，高手理解的是"工具和 Agent 是两个独立的关注点，通过 Registry 接口解耦"。面试官考察候选人是否设计过支持灰度发布和动态加载的工具系统，能说出工具版本管理的挑战。

## Q：长时间任务的 checkpoint 和 resume 机制如何设计？如何保证恢复后不重复执行？

> 来源：AI Agent 可靠性工程面试

**新手答**："定期保存进度，出错了从上次保存的地方继续。"

**高手答**：

**Checkpoint 的粒度设计**：过细（每步都保存）导致 I/O 开销大；过粗（整个任务结束才保存）导致失败时损失大。最优策略是在"语义边界"做 checkpoint：每次完成一个工具调用（代表一个有意义的执行单元）后保存，既保证了恢复粒度，又避免了细碎的保存开销。

**Checkpoint 数据结构**：需要包含完整恢复所需的所有信息：session_id、当前迭代次数、完整的消息历史（含所有工具调用和结果）、已执行工具调用的幂等键集合、当前 Token 消耗量、任务元数据（原始目标、超时时间）。特别注意：不能只保存"进度百分比"，必须保存能让 Agent 从相同认知状态继续的完整上下文。

**幂等性保证**：每个工具调用在执行前生成一个幂等键（通常是 tool_name + params + checkpoint_id 的哈希），执行成功后将幂等键写入 checkpoint。恢复时，检查每个待执行的工具调用是否已有幂等键记录，有则跳过执行、直接使用缓存的结果，无则正常执行。这是防止重复执行的核心机制。

**恢复策略的选择**：Full Replay（从头重放消息历史给 LLM，让它继续）vs Direct Resume（直接把保存的消息历史注入 context，告知 Agent 已完成的步骤）。Full Replay 保证了 LLM 推理的连贯性，但 Token 成本高；Direct Resume 成本低，但 LLM 可能因为缺少推理过程而产生不一致行为。实践中推荐 Direct Resume + 注入简洁的任务进度摘要。

**分布式 Checkpoint 的一致性**：在多节点 Worker 环境中，Checkpoint 写入和任务队列 ack 必须是原子操作。否则会出现：写 checkpoint 成功但 ack 失败 → 任务被重新分配给另一个 Worker → 两个 Worker 并行执行同一任务。解法是两阶段提交（2PC）或使用事务性消息队列。

**差距在哪**：新手认为"保存状态 + 从头继续"就是 checkpoint，高手知道关键在幂等键（防重复执行）和原子性（分布式一致性）。面试官会追问"如果 checkpoint 写入和工具执行之间崩溃了怎么办"来考察对 exactly-once 语义的理解。

## Q：Agent 如何与传统微服务架构集成？双方的接口契约应该怎么设计？

> 来源：企业级 AI 架构设计面试

**新手答**："Agent 就是一个服务，通过 REST API 和其他服务通信就行了。"

**高手答**：

**集成模式的选择**：Agent 作为微服务有三种角色：1）作为"智能网关"——接收用户自然语言请求，翻译成对下游微服务的调用；2）作为"编排者"——协调多个微服务完成复杂业务流程，替代传统的 BFF（Backend For Frontend）；3）作为"增强者"——在传统请求处理链路中作为某个步骤的智能处理节点。不同角色对接口契约有不同要求。

**同步调用的契约设计**：Agent 调用下游微服务时，应使用标准 REST/gRPC 接口，但需要额外约定：请求头携带 trace_id（用于 distributed tracing）、correlation_id（用于关联同一 Agent 任务内的所有下游调用）、agent_session_id（用于审计）。下游服务不需要知道调用方是 Agent 还是人类客户端。

**异步任务的回调设计**：长时间微服务操作（如批量数据处理）应使用事件驱动模式，Agent 提交任务后注册回调 endpoint，微服务完成后推送结果。Agent 需要维护"等待中的回调"状态，超时后重新查询或重试提交。

**幂等性要求**：Agent 可能因重试机制多次调用同一微服务接口，下游服务必须支持幂等操作，通常通过 idempotency-key 请求头实现。在系统设计文档中明确哪些接口支持幂等，哪些不支持（只能执行一次的操作如发送邮件需要特殊处理）。

**错误语义对齐**：微服务的 HTTP 错误码（4xx/5xx）需要映射到 Agent 可理解的语义错误。400 → 参数生成有误，Agent 可以尝试修正参数重试；404 → 目标资源不存在，Agent 应修改任务策略；503 → 服务不可用，Agent 应退避重试。这个映射逻辑应集中在 Tool 的 handler 层，不要让 Agent Loop 直接处理 HTTP 状态码。

**差距在哪**：新手把 Agent 当成普通服务，高手理解 Agent 与微服务的集成有独特挑战：重试语义、幂等要求、trace 链路的贯穿、以及错误码到语义错误的映射。面试官考察候选人是否在企业级项目中处理过这些集成细节。

## Q：LLM 调用的冷热路径应该如何设计？什么请求走缓存，什么请求必须走 LLM？

> 来源：LLM 系统性能优化面试

**新手答**："可以把相同的问题缓存起来，下次直接返回，省钱又快。"

**高手答**：

**冷热路径的定义**：热路径指可以通过缓存或预计算快速响应的请求；冷路径指必须实时调用 LLM 进行推理的请求。设计目标是尽可能多的请求走热路径，只有真正需要 LLM 智能的请求才走冷路径。

**语义缓存（Semantic Cache）的实现**：不能直接用请求文本做缓存键（自然语言的细微差异导致文本完全不同），需要对请求文本做语义向量化，用 cosine similarity 判断是否与历史请求相似（阈值通常 0.95 以上）。命中则返回历史响应，未命中则调 LLM 并将结果写入缓存。关键挑战：相似性阈值的选择（过高则缓存命中率低，过低则返回不准确的历史结果）；缓存失效策略（时间 TTL vs 内容变更触发）。

**Prompt Caching 的使用**：Anthropic Claude 等模型支持 prompt caching，当系统提示和历史消息相同时，可以复用模型的 KV cache，显著降低延迟和成本（Claude Sonnet 3.5 缓存命中可节省 90% 输入 token 费用）。工程上需要把稳定的内容（系统提示、知识库摘录）放在消息序列前面，把变化的内容（用户最新消息）放在后面，以最大化缓存命中率。

**必须走冷路径的场景**：涉及实时数据（当前时间、实时价格）；个性化推理（依赖用户最新状态）；工具调用结果的解读（工具输出是新信息，无法预缓存）；安全敏感操作（不应该用缓存的历史决策替代实时判断）。

**分层缓存架构**：L1 - 精确匹配缓存（完全相同的请求，Redis，TTL 短）；L2 - 语义缓存（向量相似度，向量数据库，TTL 长）；L3 - Prompt Caching（模型层，自动管理）；L4 - 冷路径（实时 LLM 调用）。每层命中都有对应的监控指标，用于调优缓存策略。

**差距在哪**：新手的"缓存"停留在精确文本匹配层面，高手理解语义缓存、Prompt Caching、分层缓存架构的层次关系，并能说出每种缓存的适用场景和失效风险。面试官常追问"语义缓存命中了但返回了错误结果怎么办"来考察对缓存安全性的理解。

## Q：Token 预算控制应该在架构的哪一层实现？如何在保证任务质量的前提下控制成本？

> 来源：LLM 产品成本优化架构面试

**新手答**："可以限制每次对话的 token 数，超了就告诉用户重新开始。"

**高手答**：

**Token 预算的多层控制**：Token 预算不是单点控制，而是多层协同：1）请求层：每次 LLM 调用设置 max_tokens 上限，防止单次输出过长；2）上下文层：Context Manager 在每次调用前检查消息历史总长度，超过阈值时压缩（摘要/截断）；3）会话层：单个用户会话的累计 Token 消耗上限，超出时降级（使用更便宜的模型或减少工具调用）；4）任务层：整个多步任务的 Token 总预算，根据任务复杂度动态分配给各子步骤。

**动态 Token 分配**：为不同组件分配 Token 配额：系统提示（固定，通常占 10-15%）、历史消息（滚动压缩，目标 40-50%）、工具结果（按需，可截断，20-30%）、当前输出（预留，10-20%）。当某个组件超出配额时，有明确的优先级顺序决定压缩或截断哪部分。

**模型降级策略**：当任务剩余 Token 预算不足以支撑当前模型完成任务时，自动降级到更便宜的模型（如从 Claude Opus 降到 Claude Haiku）。降级不是简单地换模型，需要同步调整提示策略（更便宜的模型需要更详细的指令）、期望精度（接受更低质量的输出），并在响应中标注"以低精度模式完成"。

**Token 成本的可观测性**：每次 LLM 调用记录：input_tokens、output_tokens、cache_read_tokens、cache_write_tokens（区分缓存命中和写入）、模型名称、cost（按价格表计算）。按 session、user、task_type 汇总，建立成本 dashboard，识别高成本路径（通常是少数几种工具组合占总成本的 80%）。

**质量与成本的权衡机制**：对于不同优先级的任务，预先配置不同的 Token 预算策略（如"快速模式"使用较少 Token 但接受较低质量，"精确模式"使用更多 Token 保证质量）。在用户界面暴露这个选项，让用户或业务方根据需求自主选择，而不是系统单方面决定。

**差距在哪**：新手的 Token 控制是"限制然后中断"，高手的 Token 控制是"多层预算 + 优先级压缩 + 动态降级 + 成本可观测性"。面试官考察候选人是否在生产环境管理过 LLM 成本，能说出成本分析方法（Pareto 分析哪些工具组合最贵）。

## Q：在 Agent 架构中如何处理工具调用的并行化？并行工具调用有哪些坑？

> 来源：高并发 AI 系统架构面试

**新手答**："并行调用就是同时执行多个工具，用 Promise.all 或者 asyncio.gather 就行了。"

**高手答**：

**并行工具调用的前提条件**：工具之间无数据依赖（A 的输出不是 B 的输入）且无资源竞争（不同时写同一文件或数据库行）。在执行前必须做依赖分析——通过 LLM 的 tool_use 输出中各工具调用的参数来推断依赖关系，或在工具 schema 中声明资源依赖。

**LLM 层面的支持**：Claude、GPT-4 等模型支持在单次响应中输出多个 tool_use block，表示这些工具可以并行执行。工程层需要检测到这种模式，并在 tool result 回复时将所有并行工具的结果打包成多条 role:tool 消息一起返回，而不是逐个返回（否则会触发额外的 LLM 调用轮次）。

**并发度控制**：无限并行会导致下游服务过载。需要维护全局并发池（semaphore），限制同时执行的工具调用数。不同风险级别的工具可以有不同的并发上限：只读工具（并发 10）、写操作工具（并发 2）、外部 API 调用（并发 5，受对方 rate limit 限制）。

**部分失败的处理**：并行执行中某个工具失败时，有三种策略：fail-fast（任一失败则取消其他，适合强依赖场景）、best-effort（继续等待其他工具完成，用部分结果）、conditional（根据失败工具的重要性决定是否继续）。这个策略应该在工具 schema 中声明，而不是 hardcode 在 Agent Loop 里。

**结果顺序问题**：并行执行的工具完成顺序不确定，但 LLM 消息历史要求顺序确定。需要在收集所有工具结果后，按原始 tool_use 的顺序排列 tool_result 消息，再发给 LLM，确保 LLM 能将结果与工具调用正确对应。

**差距在哪**：新手认为并行是性能优化，高手知道并行引入了依赖分析、并发控制、部分失败处理、结果顺序等复杂性。面试官常追问"两个工具都写数据库同一行怎么办"来考察对资源竞争的理解。

## Q：如何设计 Agent 对新能力（非工具，是推理能力）的扩展？比如需要让 Agent 具备多步数学推理能力？

> 来源：AI 研究工程师面试

**新手答**："在系统提示里告诉 Agent 你会数学就好了，或者加一个计算器工具。"

**高手答**：

**能力注入的层次**：新能力可以通过多个层次注入：提示层（告诉模型如何思考）、工具层（提供外部计算能力）、模型层（使用针对该能力微调的模型）、混合层（提示 + 工具协同）。对于数学推理，这四层的效果和成本差异显著。

**提示层的上限**：通过 CoT 提示可以显著提升基础数学推理，但对于复杂多步计算（精确数值、符号代数、大数运算），LLM 的计算精度天然受限，提示无法突破这个上限。提示层适合逻辑结构正确但对数值精度要求不高的推理。

**工具层的设计**：数学工具不应只是"计算器"，而应是结构化的数学推理工具：接受数学表达式（LaTeX 或 Python 语法），返回结构化结果（包含中间步骤、最终结果、使用的定理）。工具本身可以调用 SymPy（符号数学）、NumPy（数值计算）、或专用数学求解器。工具的输入 schema 要足够严格，防止 LLM 生成语法错误的表达式。

**模型路由策略**：对于需要复杂推理的请求，在 Query Engine 层将其路由到具有更强推理能力的模型（如 Claude Opus 或 o1 系列），而简单请求走轻量模型。路由依据可以是任务类型标签（在用户请求分类阶段打标）或 token 复杂度估算。这是"模型能力"级别的扩展，区别于工具扩展。

**混合方案的协同**：推荐的模式是：LLM 负责问题分解和推理结构（"这个问题分三步：1. 求导 2. 令导数为零 3. 验证二阶条件"），工具负责每步的精确计算，LLM 负责最终结果的语义解释。这样既保证推理的正确逻辑结构，又保证数值精度，是 LLM 与工具各自发挥优势的最佳模式。

**差距在哪**：新手混淆了"告诉模型它能做什么"和"真正给模型这个能力"，高手清楚能力注入的四个层次和每层的边界。面试官考察候选人能否区分提示工程能解决的问题和需要工具介入的问题，以及何时需要在模型路由层做决策。

## Q：Agent 系统中如何实现请求的优先级调度？VIP 用户的请求怎么保证不被普通用户请求阻塞？

> 来源：AI 平台工程面试

**新手答**："用优先级队列，VIP 用户的请求放到队列前面。"

**高手答**：

**多级队列设计**：将请求分为多个优先级队列（Critical / High / Normal / Low），不同队列的 Worker 资源分配不同。Critical 队列始终有专用 Worker 保留，不与其他队列共享；High 队列可以抢占 Normal 队列的 Worker 资源（在 Worker 空闲时）。这种设计保证了高优先级请求的响应时间上界（SLA），而不仅仅是"排在前面"。

**优先级提升（Priority Aging）**：低优先级请求在队列中等待超过阈值时间后，自动提升优先级，防止饿死（starvation）。这对实时性要求不高但不能无限等待的后台任务很重要（如定时触发的简历分析任务）。

**LLM 并发槽位的抢占**：在 LLM Provider 有并发限制时（如同时最多 N 个请求），高优先级请求可以抢占低优先级请求正在等待的槽位。实现方式：低优先级请求在进入等待状态时注册一个可取消的 promise，高优先级请求到达时取消等待中的低优先级请求，将其放回队列重排。

**Token 配额的优先级感知**：高优先级用户不仅请求先处理，还应有更高的 Token 配额。低优先级用户在系统负载高时会被压缩配额（减少最大输出 token 数或使用更便宜的模型）。这是优先级调度在资源层面的体现，而不只是队列层面。

**可观测性**：每个优先级队列需要独立监控：队列深度、平均等待时间、P99 等待时间、饿死事件数。当高优先级队列积压时，触发自动扩容；当低优先级队列等待时间超过阈值时，触发告警（可能需要增加 Worker 资源）。

**差距在哪**：新手的优先级调度只解决"谁先被取出队列"的问题，高手理解优先级调度需要涵盖队列调度、资源抢占、Token 配额、防饿死机制的完整体系。面试官常追问"如果高优先级请求持续涌入，低优先级请求会一直等吗"来考察 starvation 的处理。

## Q：Agent 的系统提示（System Prompt）如何做分层管理？为什么不能把所有指令塞在一个大提示里？

> 来源：LLM 应用架构面试

**新手答**："系统提示就是在最开始告诉 AI 它是什么角色，一段文字写清楚就好了。"

**高手答**：

**分层管理的必要性**：单一大提示有三个工程问题：1）维护困难——不同团队（安全团队、产品团队、领域专家）需要修改不同部分，共享一个字符串导致冲突；2）动态注入困难——某些内容（用户记忆、实时知识）需要每次请求动态插入，不能写死；3）Token 浪费——某些内容（调试信息、开发指令）只在特定场景需要，全量携带浪费 Token。

**分层架构设计**：按优先级划分层次，优先级决定内容冲突时谁覆盖谁，也决定放置顺序（高优先级在前，模型注意力更集中）：System 层（100）：核心行为规范、安全边界、禁止行为，由安全团队维护，不可被其他层覆盖；Immediate 层（90）：当前任务上下文、特定场景的专项指令；Knowledge 层（80）：从知识库检索的相关背景知识；Memory 层（60）：从用户记忆中召回的个性化信息；Session 层（40）：当前对话的临时状态和进度信息。

**动态组装的实现**：每次 LLM 调用前，Context Manager 按优先级顺序拼接各层内容，并在总 Token 超过阈值时按逆优先级顺序截断（先截低优先级内容）。各层的内容更新独立进行：Memory 层在每个用户回复后更新，Knowledge 层在工具调用后可能更新，System 层很少变更。

**层间隔离与冲突解决**：如果两个层都包含对同一行为的指令（如一个说"回答要简洁"，另一个说"回答要详细"），需要有明确的优先级规则。可以在每层开头添加优先级声明，也可以在组装时检测内容冲突并自动解决（优先级高的指令覆盖低的）。

**提示测试与版本管理**：分层提示的每一层都应该有独立的版本号，变更时通过 A/B 测试验证效果，用 Git 管理变更历史。层之间的组合也需要集成测试，防止某两层组合后产生相互矛盾的指令。

**差距在哪**：新手把系统提示当成静态文本，高手把它当成有层次的动态内容管理系统。面试官考察候选人是否在生产环境管理过复杂的提示体系，能说出分层的设计原则和动态组装的实现细节。

## Q：如何设计 Agent 的可观测性系统？仅有日志够不够？

> 来源：AI 系统 SRE 面试

**新手答**："记录日志，出了问题看日志排查就行了。"

**高手答**：

**可观测性三大支柱在 Agent 系统的特殊性**：Logs（日志）、Metrics（指标）、Traces（追踪）在传统系统中已经成熟，但在 Agent 系统中有额外挑战：LLM 调用的不确定性（相同输入可能产生不同输出）、工具调用链的动态深度（事先不知道会调用多少个工具）、Token 消耗的成本维度（必须跟踪成本，不只是性能）。

**Traces 的关键性**：Agent 的一次用户请求可能触发 3-10 次 LLM 调用和多次工具调用，形成复杂调用树。必须使用 distributed tracing 将所有调用关联到同一个 trace_id，并记录每次 LLM 调用的：输入消息列表（hash 而非全文，保护隐私）、使用的模型、输入/输出 token 数、延迟、工具调用列表及各工具的执行时间。没有 trace，多步 Agent 的 bug 几乎无法定位。

**LLM 特有的指标**：除常规指标（QPS、P99 延迟、错误率），还需要：Token 消耗率（input/output/cache 分别统计）、工具调用成功率（按工具类型分别统计）、迭代次数分布（了解任务复杂度分布）、任务完成率（是否达到用户目标，需要业务层定义）、幻觉检测率（如果有自动检测的话）。

**LLM 响应的结构化日志**：不能只记录"调用了 LLM"，要记录：请求的结构化摘要（参数、模型、温度）、响应类型（text / tool_use）、工具调用的名称和参数哈希、stop_reason、是否命中缓存。原始 LLM 响应全文通常不应全量存储（成本 + 隐私），但需要在需要时可以重放（存储消息历史 ID）。

**告警策略**：关键告警：工具调用错误率 > 5%（可能是工具服务故障）、LLM 平均迭代次数 > 8（可能有无限循环风险）、P99 任务完成时间 > SLA（用户体验下降）、单次会话 Token 消耗 > 阈值（可能被恶意利用）。次级告警：缓存命中率下降（可能是提示变更导致）、特定工具的调用频率异常（可能有 Bug 导致重复调用）。

**差距在哪**：新手认为日志等于可观测性，高手理解可观测性是 Logs + Metrics + Traces 三支柱，且 Agent 系统有成本维度（Token 消耗）这个传统系统没有的关键指标。面试官常追问"如果 Agent 在生产环境突然变慢，你怎么排查"来考察 trace 设计的实用性。

## Q：Agent 如何在不重启服务的情况下热更新系统提示或工具定义？

> 来源：AI 产品运营工程师面试

**新手答**："重新部署就好了，或者把提示存在数据库里，读取时动态加载。"

**高手答**：

**热更新的必要场景**：紧急安全修复（发现提示有漏洞需要立即修复，不能等下次发布）；A/B 测试（对不同用户群动态切换不同提示版本）；节假日/活动期间的临时调整（特殊场景的提示增强）；工具 schema 的非破坏性更新（新增可选参数）。

**提示的热更新设计**：提示存储在外部配置系统（如 AWS Parameter Store、Consul、数据库），Agent 在每次请求时（或按配置的刷新间隔）拉取最新版本。为避免频繁 I/O，使用本地缓存 + TTL（如 60 秒），TTL 到期后后台异步拉取最新版本，避免每次请求都阻塞等待配置加载。紧急更新可以通过推送通知（如 Redis PubSub）让所有 Agent 实例立即失效本地缓存。

**版本化与回滚**：每次提示更新要有版本号，每个 Agent 调用日志中记录当时使用的提示版本。发现新版提示导致质量下降时，可以通过配置系统立即回滚到上一版本，无需代码回滚。版本历史至少保留 30 天，支持任意版本的快速切换。

**工具定义热更新的约束**：工具 schema 的热更新比提示更危险——如果正在进行的 Agent 会话使用了旧 schema 生成了工具调用参数，而 handler 已经切换到新 schema，会导致参数解析失败。因此工具 schema 更新必须是向前兼容的（旧参数在新 schema 中仍然有效），或者使用 session-level 的 schema 版本锁定（会话开始时锁定工具版本，不受后续更新影响）。

**灰度发布机制**：新提示或工具定义不应立即全量推送，应通过用户分组（如 user_id % 100 < 10 则使用新版本）或 session flag 进行灰度，监控关键指标（任务完成率、错误率、用户满意度）稳定后再全量。热更新系统需要内置灰度能力，不能每次都靠手工修改代码实现。

**差距在哪**：新手把热更新等同于"从数据库读配置"，高手理解热更新涉及缓存一致性（推/拉模式）、会话一致性（版本锁定）、安全更新（工具 schema 向前兼容）、灰度发布等多个工程细节。面试官考察候选人是否处理过生产环境的紧急提示修复场景。

## Q：当用户请求需要跨多个 Agent 版本完成时（如任务开始在 v1，完成时已升级到 v2），如何处理兼容性？

> 来源：AI 平台版本管理面试

**新手答**："版本升级后所有任务都用新版本就好了，旧任务重新开始。"

**高手答**：

**版本兼容性的核心挑战**：长时间异步任务（如数小时的文档分析）在执行中途可能遇到 Agent 服务版本升级。如果直接切换到新版 Agent，可能出现：新版 Agent 不理解旧版工具调用产生的中间状态格式；旧版的 checkpoint 数据结构在新版中无法反序列化；新版系统提示改变了 Agent 的行为模式，导致对历史上下文的解读不一致。

**滚动兼容策略**：在部署新版本时，保持旧版本 Worker 继续运行直到所有进行中的任务完成（drain 模式）。新请求路由到新版 Worker，旧请求继续在旧版 Worker 上完成。这要求同时运行两个版本的能力，以及根据 session 版本标签路由请求的基础设施。

**Checkpoint 的 Schema 版本化**：每个 checkpoint 记录写入时的 Agent 版本号（或 schema 版本号）。恢复时根据 checkpoint 版本选择对应的反序列化器和恢复策略。对于无法兼容的版本跨度，提供迁移脚本（checkpoint migration），将旧格式转换为新格式，或者直接触发任务重新执行（如果允许的话）。

**向前兼容的消息格式**：Agent 的工具调用参数、工具结果、中间推理步骤，都应使用可扩展格式（如 JSON with optional fields），新版本新增的字段在旧版本中被忽略，旧版本产生的消息在新版本中仍然有效。避免使用位置依赖的格式（如 CSV、固定长度字段）。

**版本化的 API 契约**：如果 Agent 以 API 形式暴露（外部系统调用），API 版本需要独立于 Agent 内部版本演进。外部调用者使用 /v1/agent/ 接口，即使内部 Agent 升级到 v3，v1 接口的请求语义保持不变，通过版本适配层转换。

**差距在哪**：新手认为版本升级就是替换，高手理解长任务的版本兼容性涉及滚动部署、checkpoint 迁移、消息格式向前兼容、API 版本隔离等多个维度。面试官考察候选人是否遇到过版本升级导致进行中任务失败的事故，以及如何系统性解决。

## Q：如何设计 Agent 的 A/B 测试框架？测试指标应该如何定义？

> 来源：AI 产品增长工程师面试

**新手答**："随机把用户分两组，一组用 A 版本，一组用 B 版本，比较哪组效果好。"

**高手答**：

**Agent A/B 测试的特殊挑战**：传统 A/B 测试假设每次用户操作是独立的，但 Agent 系统中同一会话的多个 turn 之间有强依赖，不能在会话中途切换版本（会导致上下文不一致）。因此 Agent 的 A/B 测试必须以 session 为实验单元（不是单次请求），一个会话从开始到结束使用同一个版本。

**实验单元的稳定性保证**：用户第一次与 Agent 交互时分配实验组，该用户的所有后续会话在实验结束前都属于同一组（用 user_id 做 hash 分组）。同一用户在不同设备上应该进入同一实验组（避免"交叉污染"——同一用户的不同行为分到不同组影响统计独立性）。

**Agent 特有的评估指标设计**：
- **任务完成率**：Agent 最终是否达成了用户的初始目标。需要定义"完成"的标准（可以是 LLM 自动评估、规则判断、用户反馈）。
- **任务效率**：达成同等目标所需的工具调用次数、LLM 轮次数、总 Token 消耗。
- **错误率**：Agent 主动报告无法完成、工具调用失败率、用户在会话中途放弃的比例。
- **用户体验指标**：会话时长（过长可能表示 Agent 在兜圈子）、用户满意度（显式评分或隐式行为如重开会话）。

**离线评估与在线实验的结合**：在 A/B 实验上线前，先用历史会话数据离线评估（用 LLM Judge 评估新版 Agent 在历史会话上的质量），过滤掉明显更差的版本。只有离线评估达到阈值的版本才进入在线实验，避免把质量差的版本暴露给真实用户。

**实验的统计有效性**：Agent 任务通常有较低的实验参与率（用户每天只用几次），需要较长的实验周期（通常 1-2 周）才能获得足够样本。用 bootstrap 方法估算统计置信区间，而不是简单的 t-test（Agent 指标通常非正态分布）。在实验中加入 holdout 组（不参与任何实验的用户）监控系统整体健康度。

**差距在哪**：新手把 Agent A/B 测试当成普通功能实验，高手理解 session 粒度的实验设计、多维度评估指标（质量+效率+体验）、以及离线评估过滤的必要性。面试官常追问"如何定义 Agent 任务完成"来考察候选人对评估指标设计的思考深度。

## Q：如何在 Agent 系统中实现 Human-in-the-Loop？在哪些节点需要人工介入？

> 来源：企业 AI 助理产品架构面试

**新手答**："在危险操作前弹出确认框，用户点确认才执行。"

**高手答**：

**人工介入的触发层次**：不是所有需要人工的场景都一样。按紧迫性和影响分级：1）同步确认（实时等待）——高风险不可逆操作（删除数据、发送通讯、支付）；2）异步审批（任务暂停等待）——中风险操作（批量修改、外部 API 调用），Agent 保存状态暂停，人工审批后恢复；3）事后通知（不阻塞执行）——低风险操作，执行后通知人工，有时间窗口可撤销。错误地把所有操作都变成同步确认会严重损害用户体验。

**暂停状态的设计**：Human-in-the-Loop 要求 Agent 支持 `paused` 状态（状态机设计的重要性）。Agent 在需要人工介入时：1）持久化当前状态（完整消息历史 + 待确认操作的详细描述）；2）向用户推送审批请求（含操作描述、风险说明、影响范围）；3）进入 `paused` 状态并释放计算资源（不占用 Worker）。收到人工响应（approve/reject/modify）后恢复执行。

**审批界面的信息设计**：人工审批请求必须包含足够信息让审批者做出决策：将要执行的操作（具体参数，不是模糊描述）、操作的背景上下文（Agent 为什么要执行这个操作）、操作的影响范围（将改变什么数据/发送给谁）、可选的替代方案（如果 Agent 有多个选择的话）。不能只显示"是否允许执行此操作？"这种无信息量的确认框。

**超时与默认行为**：人工审批有超时限制（如 24 小时），超时后触发默认行为：对于可逆操作，可以有"超时自动批准"或"超时自动拒绝"两种策略，根据操作风险级别选择；同时向原任务的提交者发送通知。超时策略需要在工具定义中声明，不能由 Agent 运行时随机决定。

**审批记录的合规性**：所有人工介入事件需要完整记录：谁审批的（审批者身份）、何时审批、审批决定（批准/拒绝/修改）、操作的最终执行结果。这对于合规审计（特别是金融、医疗行业）是必须的。记录应该防篡改（写入后不可修改，用 append-only 存储）。

**差距在哪**：新手的 Human-in-the-Loop 是"弹确认框"，高手的 Human-in-the-Loop 是一个完整系统：暂停状态管理、审批信息设计、超时策略、合规记录。面试官考察候选人是否在生产 Agent 系统中设计过真正的人工介入流程，而不只是前端弹窗。

## Q：如何衡量 Agent 的整体系统质量？有哪些关键指标？

> 来源：AI 系统质量保障面试

**新手答**："看准确率和用户满意度就行了。"

**高手答**：

**质量指标的多维度体系**：单一指标无法全面反映 Agent 质量，需要从以下维度综合评估：

**任务维度**：Goal Completion Rate（任务完成率）——Agent 是否最终达成用户意图；Step Efficiency（步骤效率）——完成相同任务所需的工具调用次数（越少越好，但过少可能表示 Agent 走了捷径而非真正完成）；Recovery Rate（从错误中恢复的比例）——当工具失败时，Agent 能自我纠正的比例。

**质量维度**：Factual Accuracy（事实准确率）——输出中可验证事实的正确比例（需要 LLM Judge 或人工评估）；Instruction Following Rate（指令遵循率）——用户明确指定格式/约束时，Agent 满足的比例；Hallucination Rate（幻觉率）——Agent 编造工具调用结果或不存在事实的比例。

**效率维度**：P50/P99 Task Completion Time（任务完成时间分位数）；Token Efficiency（每完成一个用户意图消耗的 Token 数，成本的核心指标）；Tool Call Success Rate（工具调用成功率，分工具类型统计）。

**用户维度**：Session Abandonment Rate（会话中途放弃率，用户不等 Agent 完成就离开）；Repeat Request Rate（用户对同一意图重复发请求，可能是因为上次没完成好）；Explicit Feedback Score（用户显式评分，如点赞/点踩）。

**评估的自动化设计**：人工评估成本高，需要建立自动评估流水线：用 LLM Judge（通常比任务 Agent 更强的模型）评估每次会话质量，Judge 使用结构化评分 rubric（避免评分不一致）；定期抽样人工复核 LLM Judge 的评分，确保 Judge 的准确性；将评估结果与系统变更关联，实现指标驱动的迭代。

**差距在哪**：新手用单一指标衡量 Agent，高手建立多维度评估体系并自动化评估流水线。面试官考察候选人是否在生产 Agent 上建立过系统的质量评估机制，能说出 LLM Judge 的设计和使用局限性（Judge 本身也可能出错）。

## Q：如何处理 Agent 中的上下文窗口限制？长对话必然导致 context 溢出吗？

> 来源：LLM 上下文管理架构面试

**新手答**："超过 context 限制就把旧的对话删掉，留最新的部分。"

**高手答**：

**溢出预防优于溢出处理**：不应该等到 context 溢出时才处理，而是在每次 LLM 调用前主动管理 context 大小。设定目标 context 大小（如 60% of max_tokens），在超过目标时触发压缩，保留 20% 的安全余量用于当前输出。这样既防止溢出，又避免接近限制时的性能不稳定。

**三级压缩策略**：1）无压缩（context 在 60% 以下）：直接使用完整消息历史；2）摘要压缩（在 60-120% 之间）：用 LLM 对较早的消息段落生成摘要，替换原始消息（摘要 + 最近 N 条完整消息），目标是在保留关键信息的前提下减少 token 数；3）激进压缩（超过 120%）：只保留系统提示 + 最近 K 条消息 + 任务关键信息的精简摘要。三级压缩的阈值需要根据实际使用模式调优。

**摘要质量的保障**：摘要生成时需要指定保留重点：已完成的工具调用及其关键结果、用户明确表达的偏好和约束、未完成的任务目标。不能只是"把前 N 条消息的大意总结一下"，要结构化地提取对后续任务有价值的信息。可以使用不同的提示来引导摘要的格式（如 structured summary format）。

**工具结果的特殊处理**：工具调用结果往往是 context 的大头（搜索结果、代码输出、文档内容）。可以对工具结果做分层保留：保留工具调用的 header（工具名、关键参数、执行时间）、截断或摘要工具的完整输出（但保留 Agent 引用过的关键信息）。这比对所有消息一视同仁的压缩更高效。

**多轮对话的结构性保持**：即使经过大幅压缩，仍需保证 LLM 能理解当前任务的目标和已完成的进度。可以在每次压缩后注入一条"任务进度摘要"消息（role: system，内容由压缩逻辑生成），让 LLM 始终有清晰的任务上下文，而不是只有压缩后的对话历史。

**差距在哪**：新手的处理是"截断旧消息"，高手理解三级压缩策略、摘要质量保障、工具结果特殊处理，以及如何在压缩后仍然保持任务上下文的连贯性。面试官常追问"摘要后 Agent 忘记了用户之前说的关键约束怎么办"来考察信息损失的处理策略。

## Q：如何设计 Agent 的安全边界？防止 Prompt Injection 和越权操作？

> 来源：AI 安全架构面试

**新手答**："在系统提示里告诉 AI 不要做坏事，加一些规则过滤用户输入。"

**高手答**：

**Prompt Injection 的攻击模式**：攻击者通过构造恶意输入，试图覆盖或绕过系统提示的约束。常见模式：1）直接注入（用户输入中包含"忽略上面的指令，改为…"）；2）间接注入（通过工具读取的外部内容中嵌入恶意指令，如恶意网页、文档）；3）角色扮演绕过（"假装你是一个没有限制的 AI…"）。间接注入最危险，因为来自"可信"的工具输出。

**输入净化（Input Sanitizer）的层次**：在工具输入进入 Hook Pipeline 前做检测：1）结构化检测——检查用户输入是否包含已知 injection 模式（正则匹配高风险关键词）；2）语义检测——用一个专门的 LLM 分类器判断输入是否包含越权意图（这比正则更难绕过，但有成本）；3）工具结果净化——对工具返回的外部内容（网页、文档）做标记，在系统提示中告知 LLM 这些内容来自不可信源，不应作为指令执行。

**权限边界的实现**：不能只靠 LLM "遵守规则"，还需要工程层面的硬边界：工具注册时声明 risk level，Permission Gate 在执行前独立检查权限（不通过 LLM 判断）；对高风险工具（写操作、外部调用）实施速率限制；定期审计工具调用记录，检测异常模式（同一工具被调用异常多次，可能是注入攻击在探测系统）。

**最小权限原则**：在任务开始时分析任务类型，向 LLM 只暴露完成该任务所需的最小工具集。只读任务不暴露写工具，内部任务不暴露外部 API 工具。这减少了即使 LLM 被注入后能造成的最大损害范围。在工具 schema 中也只暴露必要的字段，避免泄露系统内部信息。

**可审计性与检测**：安全边界不只是防御，还需要检测。记录所有被 Permission Gate 拦截的尝试、所有触发安全规则的输入、所有异常的工具调用模式（高频、异常参数）。设置告警阈值，当检测到可能的攻击模式时通知安全团队。对于高价值系统，定期做红队测试（让专人尝试绕过安全边界）。

**差距在哪**：新手认为安全是提示问题，高手理解安全需要纵深防御：输入净化 + 工程层权限控制 + 最小权限原则 + 可审计检测。面试官重点考察对间接 injection（通过工具结果注入）的理解，这是大多数候选人忽略的攻击面。

## Q：如何设计 Agent 的测试策略？怎么测试 LLM 行为的不确定性？

> 来源：AI 系统测试工程师面试

**新手答**："写单元测试测工具函数，集成测试测整个 Agent 流程。"

**高手答**：

**测试层次的完整设计**：Agent 测试需要覆盖四个层次：单元测试（工具函数、Hook、状态机转换等确定性逻辑）、集成测试（工具与 Agent Loop 的集成，用 Mock Provider 替代 LLM）、行为测试（测试特定场景下 Agent 的决策行为，使用真实 LLM 但固定 seed）、端到端测试（完整用户场景，允许 LLM 自由输出，以任务完成为通过标准）。

**Mock Provider 的设计**：最关键的测试基础设施是 Mock LLM Provider，它能返回预设的响应序列（工具调用 → 特定工具结果 → 最终文本）。Mock Provider 让确定性的工具、状态机、权限控制逻辑可以在不消耗 API 费用的情况下充分测试。Mock 要支持：返回 tool_use 响应、模拟错误响应（rate limit、context overflow）、模拟流式响应的分块返回。

**LLM 行为的测试策略**：LLM 输出是随机的，不能直接做精确断言。替代方案：1）语义断言——使用 LLM Judge 判断输出是否满足预期语义（如"输出是否包含了计划的三个步骤"）而不是精确字符串匹配；2）结构断言——测试输出格式是否符合要求（JSON schema 校验、关键字段存在性）；3）行为断言——测试 Agent 在给定场景下是否调用了预期工具（路径断言），不断言工具的具体参数内容。

**回归测试套件的建立**：从生产会话中挑选代表性场景（包含成功案例、边界案例、之前出过 bug 的案例）建立 golden test set。每次代码或提示变更后运行这套测试，用 LLM Judge 自动评估新版本的通过率，设定质量门槛（如不能低于当前版本通过率 2% 以上才能发布）。

**随机性的控制**：对于需要确定性的测试，使用支持 seed 参数的 LLM API，固定 seed 后输出确定（注意：不同模型版本即使相同 seed 也可能输出不同，所以模型版本也要固定）。对于测试需要覆盖多种 LLM 输出变体的情况，使用 property-based testing 思路，生成多个变体场景并验证 Agent 在所有变体上都能正确处理。

**差距在哪**：新手的测试只覆盖确定性逻辑，高手设计了覆盖 LLM 随机性的完整测试策略（Mock Provider + 语义断言 + 回归测试套件）。面试官考察候选人是否在生产 Agent 上建立过有效的测试防护网，以及如何处理"提示变更后如何验证没有引入 regression"这个难题。

## Q：如何在 Agent 架构层支持多模态输入（图片、语音、文档）？

> 来源：多模态 AI 系统设计面试

**新手答**："把图片转成 base64 传给支持多模态的模型就好了。"

**高手答**：

**多模态内容的统一表示**：在消息数据结构层，需要支持 content 字段包含多种类型的块（content blocks）：text block、image block（base64 或 URL 引用）、document block（PDF 等）、tool_result block。这要求消息格式从简单字符串升级为结构化的内容数组，所有处理消息的代码（上下文压缩、摘要生成、日志记录）都需要能处理这种结构。

**大文件的处理策略**：直接在消息中内嵌大文件（如高分辨率图片、长 PDF）会快速耗尽 context window，且多次传输成本高。应该使用文件引用策略：文件上传到对象存储（S3/OSS），在消息中传递 URL 引用，LLM API 支持从 URL 拉取内容（如 Claude 的 Document Block 支持 base64 或 URL）。对于需要多次引用的文件，使用 Anthropic Files API 等持久化 token 的方案，避免重复 encode。

**语音的特殊处理**：语音不能直接传给 LLM（大多数 LLM 不支持原始音频输入）。需要先经过 ASR（自动语音识别）转为文本，再处理。ASR 的延迟和准确率会影响整体体验。对于实时语音场景，需要流式 ASR（边说边识别），与 Agent 的流式 token 输出结合，形成实时交互体验。这是一个完整的实时 pipeline，架构复杂度显著高于文本输入。

**多模态内容的压缩挑战**：图片和文档的 context 压缩不能像文本一样做摘要（无法用更少的 token 表示同一图片）。压缩策略：1）降低分辨率（减少 base64 大小）；2）提取关键信息（在早期轮次中用 LLM 生成图片的文本描述，后续轮次用文本描述替代原图）；3）淘汰机制（只保留最近 N 张图片，旧图片用描述替代）。

**工具的多模态输出**：工具可以返回图片（如代码执行结果的图表、截图工具的截图）。需要在工具 result 的数据结构中支持图片类型，并确保这些图片能被后续 LLM 调用正确处理（作为图像输入传递）。同时要控制工具返回的图片数量和大小，防止 context 被工具图片输出撑满。

**差距在哪**：新手只考虑"传 base64 给模型"，高手考虑统一内容表示、大文件引用策略、语音 pipeline、多模态压缩挑战、工具多模态输出的完整架构。面试官考察候选人是否真正设计过多模态 Agent，能说出在长对话中管理图片 token 消耗的具体策略。

## Q：Sub-agent（子 Agent）架构下如何管理并发和资源隔离？

> 来源：大规模 AI 系统架构面试

**新手答**："多个子 Agent 同时运行，用线程或协程管理并发。"

**高手答**：

**并发控制的层次**：Sub-agent 并发管理需要两个层次：1）资源层——并发池（semaphore）限制同时运行的 Agent 数量，防止因并发过高耗尽 LLM API 配额或内存；2）调度层——优先级队列决定哪个 Sub-agent 先被启动，支持高优先级子任务抢先低优先级子任务。并发池大小需要根据 LLM Provider 的并发限制和可用内存动态调整。

**资源隔离的必要性**：没有隔离时，一个失控的 Sub-agent（如陷入循环的子任务）会消耗大量 Token 和 CPU，影响其他 Sub-agent 的执行。隔离手段：1）Token 预算隔离——每个 Sub-agent 有独立的 Token 配额，超出后强制终止，不影响其他 agent；2）时间隔离——每个 Sub-agent 有独立的超时时间，不能因为某个子任务慢而阻塞整体进度；3）状态隔离——每个 Sub-agent 有独立的消息历史和内存，不共享（防止一个 agent 的幻觉污染另一个 agent 的上下文）。

**父子 Agent 的通信协议**：父 Agent 向子 Agent 传递任务时，需要传入：任务描述（goal）、可用工具子集（最小权限）、上下文信息（执行本任务需要的背景，但不是全部父 Agent 的上下文）、Token 预算、超时时间。子 Agent 向父 Agent 返回：结构化结果（success/failure）、输出内容、消耗资源统计（Token 数、耗时）、错误信息（如果失败）。通信格式要机器可解析，不依赖 LLM 的自然语言理解。

**Sub-agent 失败的级联影响控制**：某个 Sub-agent 失败不应自动导致整个任务失败。父 Agent 需要判断该子任务是否对整体任务是 critical 的，并决定：重试子任务、用备用策略替代、继续其他子任务并标注此子任务为失败、或上报给更高层 Agent 请求决策。这种容错能力是 Multi-agent 系统可靠性的关键。

**调试与 Trace 的关联**：Sub-agent 的所有调用都必须携带父 Agent 的 trace_id，并生成自己的 span，形成完整的调用树。在分布式 tracing 系统中，可以看到整个任务的 Agent 调用树结构，快速定位是哪个 Sub-agent 出了问题、花费了多少 Token、耗时多久。没有这个能力，Multi-agent 系统几乎无法调试。

**差距在哪**：新手把 Sub-agent 并发看成普通的并发编程问题，高手理解需要 Token 预算隔离、时间隔离、状态隔离、以及失败级联控制。面试官考察候选人是否在 Multi-agent 系统中处理过资源隔离问题，能说出具体的隔离实现和调试方案。

## Q：当 Agent 依赖的 LLM 模型被弃用或升级时，如何设计平滑迁移策略？

> 来源：AI 平台运维工程师面试

**新手答**："改一下代码里的模型名，重新部署就好了。"

**高手答**：

**模型变更的影响面评估**：模型升级不只是换名字，可能影响：输出格式（新模型可能用不同的 JSON 格式返回工具调用）、推理行为（相同提示下新模型可能做出不同的工具选择决策）、token 用量（新模型的 tokenizer 不同，相同文本的 token 数不同，影响 context 窗口管理）、API 参数（某些参数在新模型中被弃用或语义改变，如 max_tokens 改为 max_completion_tokens）。每一项都需要评估和测试。

**迁移测试的体系设计**：在切换到新模型前，用离线评估框架对比旧模型和新模型在代表性任务集上的表现差异。关注：工具调用格式的兼容性（新模型是否能生成与旧 schema 兼容的参数）；任务完成率的变化（新模型是否在某些场景下表现下降）；Token 消耗的变化（新模型是否明显更贵或更便宜）；延迟的变化。

**渐进式迁移策略**：不要在某一天同时切换所有流量，应该用 Shadow Mode 先验证：新模型在生产流量的 Shadow 副本上执行（不影响用户），记录结果；对比新旧模型的输出差异和质量指标；Shadow Mode 稳定后，用灰度发布（5% → 20% → 50% → 100%）逐步切换真实流量。每个阶段设定回滚触发条件（如任务完成率下降 > 2%）。

**提示的模型适配**：不同模型对提示的响应方式不同，从一个模型迁移到另一个时，可能需要调整提示（新模型可能对某些指令模式更敏感或更迟钝）。将模型版本与提示版本绑定管理，针对不同模型维护不同版本的提示，通过模型路由层自动选择配套提示。

**并行运行期的成本控制**：Shadow Mode 期间需要同时运行两个模型，成本翻倍。应该只在一个有代表性的流量样本（如 10%）上运行 Shadow，不需要对全量流量做并行处理。Shadow Mode 的持续时间要足够长（至少覆盖完整的业务周期，如一周），以捕获不同时间段的使用模式差异。

**差距在哪**：新手认为模型迁移是配置更改，高手知道需要影响面评估、离线测试、渐进式迁移、提示适配和成本控制的完整体系。面试官常追问"如果新模型在某类任务上明显变差，但整体指标还好，你会迁移吗"来考察候选人对局部性能下降的处理策略。

## Q：如何设计 Agent 的配置管理体系，使得不同环境（开发/测试/生产）的 Agent 行为一致可控？

> 来源：AI 系统 DevOps 面试

**新手答**："用环境变量区分不同环境的配置，开发用便宜模型，生产用好模型。"

**高手答**：

**配置的分类管理**：Agent 配置不是一个整体，需要按性质分类：1）基础设施配置（API 密钥、服务端点、数据库连接）——按环境不同，用 Secret Manager 管理，不能写死在代码；2）模型配置（使用哪个模型、temperature、max_tokens）——按环境和成本策略不同，用配置文件管理；3）行为配置（系统提示、工具白名单、权限规则）——需要版本管理和 A/B 能力，用配置数据库管理；4）业务配置（速率限制、Token 预算、优先级规则）——按业务需求不同，支持热更新。

**环境一致性的保障**：最大的风险是生产和测试行为不一致导致测试通过但生产出问题。关键措施：1）测试环境使用与生产相同的提示版本（不是"差不多的"），通过配置 ID 确保一致性；2）测试环境使用相同的模型（而非更便宜的替代品），除非有明确的成本 vs 一致性权衡决策；3）所有配置变更都先在测试环境验证，再通过流水线自动推送到生产。

**配置变更的审计追踪**：所有配置变更需要记录：谁改的、改了什么（diff）、何时改的、为什么改（关联 ticket）、变更后的效果（关联指标变化）。这对生产故障定位极为重要——"三天前配置改了什么"是排查 Agent 行为突变的第一个问题。

**Feature Flags 与配置的结合**：某些 Agent 能力的开关（是否启用某个工具、是否开启某个安全规则）应该通过 Feature Flags 管理，而不是代码发布。Feature Flags 支持按用户/按环境/按百分比动态开关，实现了能力的渐进式发布和快速回滚。关键能力（如新的高风险工具）必须通过 Feature Flag 控制，不能"一次性全量发布"。

**配置校验与类型安全**：配置加载时必须做严格校验（类型检查 + 值域检查 + 业务规则检查），而不是在运行时才发现配置错误。使用 Zod/Pydantic 等 schema 校验库，在服务启动时 fail fast，而不是在处理第一个用户请求时才崩溃。关键配置的校验错误必须触发告警并阻止部署，不能静默使用默认值。

**差距在哪**：新手的配置管理是环境变量 + 条件判断，高手的配置管理是分类存储 + 版本控制 + 审计追踪 + Feature Flags + 启动校验的完整体系。面试官考察候选人是否经历过"生产行为与测试不一致"的故障，以及如何从体系层面防止类似问题再次发生。

## Q：在 Agent 系统中如何实现"可解释性"，让用户和运营人员能理解 Agent 为什么做了某个决策？

> 来源：企业级 AI 系统透明度设计面试

**新手答**："把 Agent 的思考过程显示出来，让用户看到推理步骤。"

**高手答**：

**可解释性的受众与需求**：不同受众对可解释性的需求不同：终端用户需要"Agent 为什么会给我这个建议"的通俗解释；运营人员需要"这次会话中 Agent 调用了哪些工具、每步输出了什么"的操作级详情；开发者需要"LLM 的完整输入输出、token 用量、工具参数"的调试级详情；管理层需要"Agent 的整体决策模式和质量指标"的聚合统计。一套可解释性方案无法满足所有需求，需要分层设计。

**用户面的可解释性设计**：在用户界面展示 Agent 的"工作过程"：当前正在执行什么步骤（如"正在搜索您的历史记录"）、已完成哪些步骤及每步的简要结果（如"找到 3 份相关简历"）、最终结论的依据摘要（如"基于您在 XX 公司的 3 年经验，建议…"）。关键：展示的应是用户可理解的业务语义，而不是技术细节（不要在用户界面显示工具名和 JSON 参数）。

**运营面的可解释性工具**：提供 Session Replay 功能：运营人员可以回放任意一次会话，查看每个 turn 的完整信息（用户输入、Agent 内部推理步骤、工具调用详情、工具返回结果、最终输出）。Session Replay 需要保存足够的信息（但要注意隐私脱敏），使运营人员无需联系开发者就能自主排查用户投诉。

**决策链的结构化记录**：将 Agent 的每次决策（为什么选择工具 A 而不是工具 B、为什么停止迭代）结构化记录，而不只是保存原始 LLM 输出。可以通过在系统提示中要求 Agent 在每次决策前先输出结构化的推理标签（如 `<decision reason="搜索结果已足够，无需继续" action="finalize">`），使决策链可程序化分析。

**因果解释与相关解释的区别**：大多数可解释性工具提供的是相关解释（"Agent 看到了 X 然后做了 Y"），而非真正的因果解释（"因为 X，所以做 Y"）。需要诚实地告知用户和运营人员这一局限性，避免对解释过度信任。对于高风险决策，建议人工校验解释的合理性，而不是只依赖 Agent 的自我解释。

**差距在哪**：新手把可解释性等同于"显示推理步骤"，高手理解可解释性是面向不同受众的分层设计，包括用户面的业务语义展示、运营面的 Session Replay、结构化决策链记录，以及对解释可靠性局限性的诚实声明。面试官考察候选人是否在企业 AI 系统中处理过合规或用户投诉场景，能说出可解释性工具的实际工程设计。

## Q：如何为 Agent 系统设计合理的 Token 预算分配策略，在多个用户并发时保证 SLA？

> 来源：AI 平台容量规划面试

**新手答**："给每个用户分配固定的 token 配额，超了就拒绝请求。"

**高手答**：

**静态配额的缺陷**：固定配额在负载不均匀时效率低（低峰期配额浪费，高峰期配额不够）。更好的模型是动态配额：设定全局 Token 吞吐量上限（基于 LLM Provider 的 rate limit），在用户间按优先级动态分配。

**令牌桶（Token Bucket）算法的应用**：为每个用户维护一个 Token 桶，以固定速率补充 token（如每分钟 10000 tokens），桶的容量上限防止突发使用（如一次性消耗 100000 tokens）。用户请求时从桶中扣除预估的 token 消耗，请求完成后补偿实际与预估的差额（退还多扣的或补扣少扣的）。

**优先级感知的容量控制**：在系统负载高时（全局 Token 消耗接近 Provider 上限），降低低优先级用户的桶补充速率（降级服务），保证高优先级用户不受影响。这比简单的"高负载时拒绝所有新请求"更优雅，可以给低优先级用户降速而非完全拒绝。

**预测性容量规划**：分析历史 Token 消耗模式（按小时、按工作日/周末），预测未来负载。在预计高峰期前预先申请更高的 Provider 配额（很多 LLM Provider 允许提前预约更高 rate limit）；在预计低谷期降低配额节省成本。将容量规划与业务事件日历关联（如重大招聘季前增加配额）。

**SLA 的分层保证**：不是所有请求都有相同的 SLA 要求。分级承诺：Critical（金融、医疗等实时决策）→ 99.9% 的请求在 5s 内开始响应；High（VIP 用户）→ 99% 在 10s 内开始响应；Normal（普通用户）→ 95% 在 30s 内开始响应。对应分配不同的 Token 桶容量和补充速率，并监控每个 SLA 级别的达标率。

**差距在哪**：新手用简单配额控制，高手用令牌桶算法 + 优先级感知降级 + 预测性容量规划 + 分级 SLA 的完整体系。面试官常追问"如果 LLM Provider 突然降低 rate limit 怎么办"来考察候选人对突发容量变化的应对能力。

## Q：如何设计 Agent 的国际化与多语言支持？不只是 UI 翻译，还涉及推理语言的选择？

> 来源：全球化 AI 产品架构面试

**新手答**："系统提示用英文，用户输入什么语言就用什么语言回答。"

**高手答**：

**推理语言 vs 输出语言的分离**：LLM 在不同语言下的推理能力不同——大多数主流 LLM 在英文训练数据上最充分，英文推理能力最强。但用户期望用母语交互。最优策略是"英文推理 + 目标语言输出"：在系统提示中要求 LLM 用英文进行内部推理（tool selection、reasoning）但用用户语言输出最终结果。这通常比直接用用户语言推理质量更高，特别是对推理能力要求高的任务。

**工具调用的语言中立性**：工具的参数传递应该语言中立（结构化数据而非自然语言描述），确保工具行为不受输出语言影响。工具返回的数据（如搜索结果）如果是特定语言，可能需要 Agent 翻译后呈现给用户，这是额外的 Token 消耗，需要在 Token 预算中预留。

**多语言 Prompt 的测试**：同一任务用不同语言提问，Agent 的行为应该一致（工具选择、推理路径相同，只是输出语言不同）。需要建立多语言评估集，定期测试 Agent 在各支持语言下的表现，防止某种语言下的退化（通常会发现低资源语言如中文小语种的表现明显差于英文）。

**文化适配 vs 语言翻译**：国际化不只是语言，还包括文化适配：日期格式、货币符号、礼貌措辞风格（日本文化要求更正式的敬语）、内容敏感度（不同地区对某些话题的合规要求不同）。Agent 的系统提示需要注入用户地区的文化规范，而不只是语言参数。这些规范要由当地团队审核，不能靠 AI 自己推断。

**语言检测与路由**：用户输入语言需要自动检测（不能要求用户手工设置），根据检测结果选择对应的文化规范和系统提示变体。对于多语言混合输入（如中英文夹杂），需要有明确的策略（以哪种语言为主？），而不是让 LLM 随机决定。

**差距在哪**：新手认为多语言是翻译问题，高手理解推理语言与输出语言的分离、工具语言中立性、多语言评估、文化适配等维度。面试官考察候选人是否在实际全球化产品中处理过语言和文化的工程挑战，能说出"英文推理质量高于中文推理"这个实践观察。
