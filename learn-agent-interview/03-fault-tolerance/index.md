# 容错与可靠性

## Q：LLM 调用的重试策略应该如何设计？

> 来源：AI 后端工程师面试

**新手答**："失败了再试一次就行了。"

**高手答**：

1. **指数退避（Exponential Backoff）**：第一次重试等 1 秒，第二次 2 秒，第三次 4 秒……避免在服务故障期间持续打满请求，加重服务端压力。上限通常设为 60 秒。

2. **Jitter（抖动）**：在退避时间上加随机值（如 ±30%），防止大量并发请求在同一时刻重试形成"惊群效应"（thundering herd）。AWS 的经验表明 jitter 是 at-scale 系统中必须的。

3. **错误分类决定是否重试**：**可重试错误**（网络超时、429 Rate Limit、502/503 服务器错误）→ 重试；**不可重试错误**（400 Bad Request / 401 Unauthorized / context length exceeded）→ 直接报错，重试没有意义。把 network error 和 API error 混为一谈是常见 bug。

4. **最大重试次数**：通常 3 次，极限场景 5 次。无限重试会占用线程/协程资源，且 LLM 服务通常会对 IP/API Key 做速率限制，太多重试反而触发封禁。

5. **超时设置**：每次重试需要独立计时，而不是复用上次请求的剩余时间。同时设置"总体 deadline"——如果用户请求的整体超时是 30 秒，那么重试链的总时长不能超过这个值。

**差距在哪**：新手重试逻辑是"失败了就再试"，高手的重试逻辑是一套精细的状态机——区分错误类型、控制退避速率、加 jitter、设总 deadline，防止重试本身成为系统故障的加速器。

## Q：什么是 graceful degradation？在 LLM 系统中如何实现？

> 来源：AI 系统可靠性工程师面试

**新手答**："就是出错了显示错误信息，让用户知道失败了。"

**高手答**：

1. **核心思想**：当系统组件不可用时，提供降级但仍然有用的功能，而不是完全不可用。用户看到的是"功能受限"而非"完全失败"。

2. **降级阶梯**：设计清晰的降级路径——第一级：使用缓存的上次结果；第二级：调用更小/更便宜的模型；第三级：使用规则引擎的预设答案；第四级：返回"服务暂时不可用，已记录您的请求"。每级降级要有明确触发条件。

3. **功能降级 vs 质量降级**：功能降级是停止提供某些非关键功能（如实时推荐，不影响核心流程）；质量降级是用低质量替代（用 GPT-3.5 替代 GPT-4）。优先选择质量降级保留功能，其次才是功能降级。

4. **降级标记**：降级响应要在 response metadata 中标记降级原因和等级，方便监控告警和用户透明提示（如"当前使用简版回答，完整分析稍后补全"）。

5. **自动恢复**：降级是临时状态，要定期探测主服务是否恢复，恢复后自动切回，而不是持续降级直到人工干预。

**差距在哪**：新手把降级等同于"显示错误"，高手的降级是一套设计好的应急响应矩阵，核心目标是在任何情况下都给用户提供某种程度的价值，而不是让用户面对黑屏。

## Q：Circuit Breaker（熔断器）模式如何应用于 LLM 调用层？

> 来源：大规模 AI 应用架构面试

**新手答**："没用过，好像是微服务里的概念。"

**高手答**：

1. **三态状态机**：Closed（正常，请求通过）→ Open（熔断，请求直接失败，不调 LLM）→ Half-Open（探测，允许少量请求通过测试服务是否恢复）。

2. **触发条件设置**：当 N 秒内的错误率超过 X%（如 10 秒内 50% 失败）时打开熔断器。阈值要基于业务容忍度和历史数据设置，不要用默认值。

3. **LLM 场景的价值**：LLM API 出现大规模故障时，没有熔断器会导致大量请求堆积在重试队列中，消耗线程资源，用户等待时间增加，最终可能拖垮整个应用服务。熔断器让应用"快速失败"，立即降级而非等待超时。

4. **Per-Provider 熔断**：不同 Provider（OpenAI、Anthropic）应该独立管理熔断状态。OpenAI 故障时熔断 OpenAI 路径，自动切换到 Anthropic，而不是两者都熔断。

5. **与 Retry 的配合**：熔断器和重试是互补的——Open 状态下不触发重试（避免在已知故障时浪费资源），Half-Open 状态下允许有限重试进行探测。

**差距在哪**：新手把 LLM 调用当普通 API 调用，遇到故障就重试到超时。高手知道在规模化系统中，下游故障会通过重试风暴放大到上游，熔断器是阻断这种放大的关键基础设施。

## Q：Agent 的 Timeout 应该如何设计？全局超时还是每工具超时？

> 来源：Agent 工程稳定性面试

**新手答**："设一个总超时时间就够了，超了就返回错误。"

**高手答**：

1. **两层超时是必须的**：每工具超时（Tool-level timeout）+ 全局 Agent 超时（Session-level timeout）。只有全局超时会导致"单个慢工具拖垮整个 Agent"；只有工具超时而没有全局超时会导致工具不断重试占用全局时间。

2. **工具级超时按 Tool 类型分级**：低延迟工具（本地 DB 查询）超时 5 秒；中延迟工具（外部 API）超时 15-30 秒；高延迟工具（执行代码、生成报告）超时 60-120 秒。不要给所有工具同一个超时值。

3. **全局超时要考虑用户体验**：Web 场景通常 30-60 秒；CLI 场景可以更长（3-5 分钟）；异步任务（用户提交后通过通知获取结果）可以设 15-30 分钟。超时后要返回"已完成的部分结果"而非空响应。

4. **超时传递（Deadline Propagation）**：父 Agent 的剩余时间要传递给子 Tool 调用。如果父 Agent 还有 10 秒，每个 Tool 的 timeout 应该 <= 10 秒，而不是独立设置 30 秒（会等到 10 秒截止再超时，产生不必要的等待）。

5. **超时后的状态清理**：超时不仅仅是返回错误，还要清理已启动但未完成的工具调用（关闭数据库连接、取消 HTTP 请求、释放锁），防止资源泄漏。

**差距在哪**：新手把超时当简单的"等多久"，高手把超时当资源预算分配——层级化、类型化、传递化，并且超时后要做完整的资源清理而非简单抛异常。

## Q：Token 耗尽时应该如何降级？

> 来源：LLM 应用成本控制面试

**新手答**："Token 用完了就报错告诉用户。"

**高手答**：

1. **预算分层**：在 Agent Loop 层面设置 token 预算（如每 session 最多 50k tokens），而不是等到 API 拒绝才知道超限。预算快耗尽时提前触发降级，而不是在边界处突然失败。

2. **提前压缩 Context**：当 Token 使用量达到预算的 80% 时，触发 Context 压缩（Summarization），而不是等到 100% 再压缩。压缩后可以继续对话，用户几乎感知不到。

3. **切换更小的模型**：当 Token 成本接近预算上限时，自动降级到更便宜的模型（GPT-4 → GPT-3.5 → 本地模型）。在 metadata 中标记降级，但不要中断用户对话。

4. **功能性截断**：长任务可以分阶段执行——先完成最关键的子任务，剩余任务标记为"待完成"，提示用户继续请求（新 session）。比直接返回错误的体验好得多。

5. **每 session 成本监控**：对高 token 消耗的 session 发出告警，检查是否有异常 pattern（如模型被诱导产生极长的输出，或者工具结果没有被截断直接全量塞进 context）。

**差距在哪**：新手把 Token 耗尽当异常处理（try-catch），高手把它当资源管理问题——预算感知、提前预警、多级降级，让 Token 耗尽是一个可控的状态迁移而不是突发的崩溃。

## Q：如何检测和处理模型幻觉？

> 来源：高可靠 LLM 应用工程师面试

**新手答**："幻觉没法完全解决，只能靠用户自己判断。"

**高手答**：

1. **幻觉的工程分类**：事实性幻觉（编造不存在的事实）、格式幻觉（声称输出了 JSON 但实际没有）、工具幻觉（声称调用了工具但没有），三类处理方式不同。

2. **格式验证（最易处理）**：对结构化输出做 Schema 验证，不符合格式的输出直接认为是"格式幻觉"并要求模型重试，同时在重试 Prompt 中明确告知哪里出了问题。

3. **RAG 引用检测**：要求模型在输出中标注信息来源（"根据文档 X 第 Y 段"），然后在后处理中验证这些引用是否真实存在于提供的 context 中。无法验证的声明标记为"待核实"。

4. **置信度采样**：对关键信息用不同 temperature 多次采样（3-5 次），如果多次结果一致性高则置信度高；如果结果差异大则标记为低置信度，需要人工审核或降级处理。

5. **对高风险域设置硬约束**：在医疗、法律、金融等高风险域，不依赖模型的自我约束，而是在应用层维护事实数据库，对模型输出与数据库做交叉验证，不一致时拒绝输出。

**差距在哪**：新手把幻觉当"AI 的固有缺陷"无可奈何，高手把不同类型的幻觉当工程问题逐一解决——能验证的验证，不能验证的标记，高风险的拒绝。

## Q：Agent 陷入工具调用死循环如何检测和处理？

> 来源：Agent Loop 工程面试

**新手答**："设置 max_iterations 就行了，跑够次数就停。"

**高手答**：

1. **max_iterations 是必要但不充分的兜底**：必须有，但只依赖它会浪费大量 token 才停止。要有更快的检测机制。

2. **重复 Tool 调用检测**：记录最近 N 次 Tool 调用的（tool_name, input_hash）对，如果发现完全相同的调用出现 2 次以上，判定为循环，立即中止并记录"模型陷入重复调用"。

3. **无进展检测**：评估每次工具调用后 context 的"信息增益"（如果两次调用结果几乎相同，说明没有新信息进入 context），连续 3 次无进展则提前终止。

4. **Tool 调用频率限制**：每个 Tool 在同一 session 内设置调用次数上限（如 search_knowledge 最多调用 5 次），超出后该 Tool 对当前 session 不可用，强制模型换其他工具或直接给用户答案。

5. **循环模式日志**：把检测到的循环模式结构化记录（哪个 Tool、哪种输入触发了循环），定期分析这些记录，找出触发循环的 Prompt 或工具设计问题，在源头修复。

**差距在哪**：新手把循环防护等同于 max_iterations，高手知道等到 iterations 耗尽是资源浪费，要用重复检测和无进展检测在几轮内就发现循环，快速失败，而不是等 10 轮结束。

## Q：生产环境下如何设计 Fallback Provider 策略？

> 来源：多 Provider AI 平台架构面试

**新手答**："主要用 OpenAI，备用用 Anthropic，主的挂了切过去。"

**高手答**：

1. **失败触发 vs 主动路由**：Fallback 有两种模式——①被动故障切换（主 Provider 失败时切换，保证可用性）；②主动负载均衡（根据延迟、成本、任务类型动态分配，提升性能和成本效率）。生产中两者都需要。

2. **Provider 健康检测**：每隔 30-60 秒向各 Provider 发送探测请求（lightweight probe），维护每个 Provider 的健康状态。不能等到真实请求失败才知道 Provider 挂了。

3. **切换透明性**：Fallback 切换对上层代码完全透明——通过抽象的 LLMProvider interface，上层不知道底层用的是哪个 Provider。不要在业务代码中写 `if provider == 'openai': ... else: ...`。

4. **模型能力等价性**：切换 Provider 时要确保 Fallback 模型有足够能力完成任务，不能用 GPT-3.5 做 Fallback 来替代 GPT-4 执行高复杂度任务。要维护一个"能力等级"映射表。

5. **切换成本评估**：Fallback 可能带来额外成本（价格更贵的备用 Provider）或质量损失（能力较弱的备用模型）。要在可用性和成本之间做明确的 SLA 决策，而不是无脑切换。

**差距在哪**：新手把 Fallback 当"备用服务器"，高手把它当多维度路由系统——主动健康检测、能力等价验证、成本感知、对上层透明，而不是简单的"主挂切备"。

## Q：如何保证多步任务的原子性？

> 来源：Agent 事务设计面试

**新手答**："步骤一步一步执行，哪步出错了重试就行。"

**高手答**：

1. **多步任务的原子性问题**：Agent 执行"发送邮件 → 更新数据库 → 通知 Webhook"时，如果邮件发了但数据库写入失败，系统进入不一致状态。这是典型的分布式事务问题。

2. **副作用分类处理**：**可逆操作**（数据库写入、文件创建）：使用事务，失败后回滚；**不可逆操作**（发送邮件、发 Slack 消息、调 Webhook）：放在流程最后执行，所有可逆操作成功后再触发不可逆操作，确保最终状态一致。

3. **幂等性设计**：每步操作都要有幂等性保证，支持重试不产生副作用。发送邮件时附带 idempotency_key，防止重试导致邮件重复发送。

4. **Saga 模式**：对于跨服务的长事务，使用 Saga 模式——每步操作都有对应的补偿操作（Compensating Transaction）。步骤 N 失败时，倒序执行步骤 1~N-1 的补偿操作来回滚。

5. **Checkpoint 和 Resume**：对于长时间任务，记录每步完成的 checkpoint，失败时从 checkpoint 恢复而不是从头重跑，减少重复执行和副作用风险。

**差距在哪**：新手把多步任务当串行脚本（一步一步跑，失败了重跑），高手把它当分布式事务——区分可逆/不可逆操作，用 Saga 模式管理跨服务一致性，用幂等性保证重试安全。

## Q：Agent 进程崩溃后如何恢复状态？

> 来源：AI 系统高可用架构面试

**新手答**："进程重启，让用户重新输入就行。"

**高手答**：

1. **Session 持久化**：Agent Loop 的每一轮（每次 LLM 调用 + Tool 调用结果）都要持久化到数据库，包括完整的 messages 列表和当前状态。进程崩溃后能从数据库恢复到最后一个一致状态。

2. **状态机快照**：每轮循环结束后，将 Agent 的完整状态（session_id、messages、iteration_count、tool_results、当前 context 层）序列化保存。状态机快照是 checkpoint 恢复的基础。

3. **重启后的续跑策略**：进程重启后，检查是否有 active 状态但中断的 session，对每个中断 session 评估是否可以续跑（如果中断发生在 LLM 调用前，可以安全重试；如果发生在 Tool 执行中途，需要验证 Tool 的幂等性后再重试）。

4. **用户通知机制**：对于因进程崩溃中断的 session，向用户发送"我们检测到服务短暂中断，您的任务已恢复执行"的通知，而不是让用户等待超时后才发现失败。

5. **分布式部署下的锁**：在多实例部署中，session 续跑要先获取分布式锁（防止两个实例同时恢复同一 session），锁的持有时间不超过 session timeout，防止 deadlock。

**差距在哪**：新手依赖 stateless 设计回避问题（重启就是全新），高手在有状态的长任务场景下设计完整的持久化和恢复机制，把进程崩溃从"灾难"变成"短暂中断"。

## Q：流式输出中断后如何重连和续传？

> 来源：LLM 前后端工程师面试

**新手答**："中断了重新请求就好了，让用户刷新页面。"

**高手答**：

1. **SSE/WebSocket 的中断处理**：网络抖动时 SSE 连接会断开，浏览器 EventSource 会自动重连，但重连后 Server 不知道客户端已收到多少内容，会重发全部内容，导致重复显示。

2. **服务端 Chunk 记录**：Server 发送每个 SSE chunk 时记录 event_id（序列号），客户端在 Last-Event-ID header 中发送已收到的最后 ID，Server 从该 ID 之后继续发送，实现真正的续传。

3. **前端去重显示**：即使实现了续传，前端也要根据 event_id 去重，防止网络抖动导致短暂收到重复的 chunk。

4. **LLM 调用不支持续传**：LLM API 本身通常不支持中途续传（断了就要重新生成）。如果网络在 LLM 生成 50% 时断开，Server 要缓存已生成内容，重连后先发送缓存内容，再决定是否继续请求 LLM 补全剩余部分（或直接返回已生成的完整结果如果 LLM 已完成）。

5. **超时区分**：要区分"LLM 慢导致的长时间无数据（正常）"和"连接真正断开（需要重连）"，通常用 heartbeat（每 5 秒发一个空事件）来保持连接并允许客户端检测真正的断连。

**差距在哪**：新手把流式中断当网络问题（刷新就好），高手知道流式续传需要 Server 端缓存、event_id 记录、客户端去重三层配合，且要和 LLM 侧的生成状态做好同步。

## Q：如何处理 LLM 返回空响应或格式完全错误？

> 来源：LLM 应用稳定性工程面试

**新手答**："捕获异常，告诉用户出错了重试。"

**高手答**：

1. **区分空响应的原因**：安全过滤被触发（模型认为 Prompt 违规，返回空）、context 过长被截断（token 超限导致输出不完整）、API 故障（服务端问题）、模型 bug（极少见）。不同原因处理方式不同。

2. **安全过滤处理**：不要直接把"被过滤"的信息暴露给用户。检测到安全拒绝后，用预定义的友好信息回应（"这个问题超出了我的服务范围"），并在日志中记录触发原因，供人工审核。

3. **格式错误的重试策略**：第一次格式错误：在重试 Prompt 中携带原始输出和具体错误（"你的上次输出不是有效 JSON，错误是 SyntaxError at position X，请重新输出"）。第二次仍然格式错误：降级到纯文本处理，或返回"生成失败"而非无限重试。

4. **输出截断检测**：检查输出是否在 JSON 对象中途截断（末尾没有 `}`）、句子中途截断（没有句号结尾）。检测到截断后，用 continuation Prompt 请求模型补全，而不是重试全部生成。

5. **兜底静默答案**：对于每种关键功能，预定义一个"安全降级响应"（如"当前无法生成诊断，请稍后再试"），确保在所有异常情况下用户都能收到有意义的响应，而不是裸 500 错误。

**差距在哪**：新手统一用"出错了，请重试"处理所有 LLM 异常，高手根据错误类型做精细化处理：安全拒绝有固定回应、格式错误带上下文重试、截断走续写、真正的服务故障才走降级。

## Q：如何防范 Rate Limit 触发的 Retry Storm？

> 来源：大规模 LLM 平台工程面试

**新手答**："被限速了就等一下再重试。"

**高手答**：

1. **Retry Storm 的成因**：系统中有 100 个并发请求，同时触发 429 Rate Limit，如果所有请求都在 1 秒后重试，100 个请求再次同时打过来，继续触发 429，形成雪崩。

2. **Jitter + 指数退避**：Rate Limit 的重试间隔要用带 jitter 的指数退避，每次重试的等待时间 = `base_delay * 2^attempt + random(0, base_delay)`，把重试请求打散到时间轴上。

3. **全局限速器（Client-side Rate Limiter）**：在客户端维护一个令牌桶（Token Bucket），按 Provider 的 RPM 限制控制发送速率。新请求先消耗令牌，令牌不足时排队等待，而不是发出后等 429。

4. **请求优先级队列**：Rate Limit 触发时，不是先到先得重试，而是按请求优先级排队。用户交互型请求（实时对话）优先级高于后台批处理任务，确保关键请求能优先获得可用容量。

5. **多 Provider 分流**：当主 Provider 触发 Rate Limit 时，自动分流部分请求到备用 Provider，而不是全部请求排队等主 Provider。前提是备用 Provider 能力满足需求。

**差距在哪**：新手的重试是"出了问题再说"，高手在触发 Rate Limit 之前就用客户端限速器主动控制速率，触发后用 jitter 分散重试，用优先级队列保护关键请求——从根本上防止 Retry Storm 的形成。

## Q：错误应该如何分类？什么错误可以重试、什么不行？

> 来源：AI 系统工程师面试

**新手答**："网络错误重试，其他错误报错就行。"

**高手答**：

1. **四分类体系**：
   - **可重试 + 系统错误**：超时（408/504）、服务不可用（503）、速率限制（429）、网络中断 → 指数退避重试
   - **不可重试 + 系统错误**：服务端 500 但带有"Invalid request"含义的 → 不重试，记录告警
   - **不可重试 + 用户/配置错误**：400 Bad Request、401 Unauthorized、403 Forbidden、422 Invalid Parameters → 立即失败，报告给调用方修复
   - **不可重试 + 资源限制**：context length exceeded（413）、账单欠费（402）→ 立即失败，走降级路径

2. **HTTP 状态码不够用**：LLM API 的 400 可能是"你的 Prompt 里有违禁词"（不可重试），也可能是"你的 JSON 格式有问题"（开发 bug，不应重试）。需要解析 response body 中的 error code 做更细粒度判断。

3. **超时的区分**：连接超时（服务器没响应，可重试）vs 读取超时（服务器响应中途断开，谨慎重试，因为操作可能已经在服务端执行了）。

4. **上报而非静默**：所有不可重试错误都要上报到监控系统，而不是静默吞掉。"不可重试"意味着这是需要人工干预的问题，不报告就不会被修复。

5. **重试上报**：每次重试本身也要记录日志（第几次重试、距上次多少 ms、触发原因），便于分析哪类错误的重试频率高，是否有系统性问题。

**差距在哪**：新手用 try-catch 一刀切，高手建立错误分类体系——知道哪些错误是系统暂时故障（重试有意义）、哪些是配置/开发问题（重试没有意义且会掩盖 bug）、哪些需要立即人工介入。

## Q：如何处理上下文窗口溢出？

> 来源：LLM 应用开发工程师面试

**新手答**："出错了就截断前面的消息，或者让用户开新对话。"

**高手答**：

1. **预防优于补救**：在每次 LLM 调用前预估 token 数（chars/3.5 快速估算），当预估超过 80% context limit 时主动触发压缩，而不是等到 API 返回 context_length_exceeded 错误。

2. **三级压缩策略**：
   - **轻度（60-80% 利用率）**：仅保留最近 K 轮消息，截断早期对话
   - **中度（80-95% 利用率）**：对早期消息做 Summarization，压缩为摘要后保留
   - **重度（>95% 利用率）**：激进摘要 + 截断 Tool 调用历史，只保留系统消息、摘要和最近 2-3 轮

3. **Important Message Pinning**：有些消息绝对不能被截断（用户在第 1 轮提供的关键约束、密钥信息），需要 Pinning 机制确保这些消息无论压缩到何种程度都不会丢失。

4. **Tool 结果的截断**：Tool 调用结果通常是上下文膨胀的主要来源（检索到的大量文档、API 返回的长 JSON）。对 Tool 结果设置最大 token 数限制（如 2000 tokens），超出部分截断并标注"[内容已截断]"。

5. **透明度**：Context 被压缩时，在 metadata 中记录压缩比例和截断部分，方便 Debug。压缩后出现"模型好像忘记了前面说的话"这类问题，可以快速定位到压缩策略有问题。

**差距在哪**：新手等到溢出才处理（截断 or 报错），高手用三级预防+压缩策略在溢出前介入，既保证对话连续性，又保证关键信息不丢失。

## Q：如何在不中断用户的情况下降级到更小的模型？

> 来源：AI 成本优化 + 用户体验面试

**新手答**："直接换小模型就好了，用户可能感觉不到。"

**高手答**：

1. **透明性决策**：对用户是否透明取决于降级幅度。从 GPT-4o 降到 GPT-4 mini（小幅质量损失）可以静默降级；从 GPT-4 降到 GPT-3.5（显著质量差异）要告知用户"当前服务繁忙，使用简化版分析"。

2. **Prompt 适配**：不同模型对 Prompt 的响应方式不同，不能直接用 GPT-4 的 Prompt 跑 GPT-3.5。降级时要切换到为目标模型优化的 Prompt 版本（针对小模型可能需要更明确的格式指引）。

3. **任务类型判断**：不是所有任务都适合降级。"给我推荐一本书"可以用小模型；"分析这段复杂代码的 bug 原因"则不应降级。在 Agent 层面维护任务难度评估，低难度任务降级，高难度任务保留原模型。

4. **降级标记在 Response 中**：返回结果时携带 `model_used` 和 `degraded: true` 字段，让监控系统能统计降级频率，让产品团队能评估用户体验影响。

5. **自动恢复**：降级应该是临时的。当主模型恢复可用（健康检查通过）后，新请求自动切回主模型，而不是一直在降级状态。同时对降级期间的高价值用户请求标记为"待补处理"，可以异步用主模型重新生成结果。

**差距在哪**：新手把模型降级当简单的变量替换（换个 model name），高手知道模型降级涉及 Prompt 适配、任务筛选、用户透明度、自动恢复的完整工程问题。

## Q：SLA 保障下如何分配 P95/P99 延迟预算？

> 来源：AI 平台性能工程师面试

**新手答**："LLM 就是慢，没法保证延迟。"

**高手答**：

1. **LLM 系统延迟的构成**：总延迟 = 排队等待 + 网络传输 + LLM 生成时间 + Tool 调用时间 + Context 处理时间。每个组件的延迟需要分别建模，而不是笼统地说"LLM 慢"。

2. **预算分配原则**：如果端到端 P99 SLA 是 30 秒，典型分配：LLM 调用 20 秒、Tool 调用总计 7 秒、其他处理 3 秒。超出预算的组件触发降级（减少 Tool 调用次数、用流式提前返回）。

3. **P95 vs P99 的不同应对**：P95 的优化靠工程效率（减少串行等待、并行 Tool 调用）；P99 的优化靠超时截断（超时后返回已有结果）+ 降级（用更快的模型）。试图用同一个策略优化 P95 和 P99 通常两个都达不到。

4. **流式响应的不同 SLA**：对于流式输出，用户关心的 SLA 是 Time-to-First-Token（TTFT，通常 < 2 秒）而不是总生成时间。TTFT 的优化方向不同于总延迟——减少 pre-processing、并行预热 context、用小模型先返回草稿。

5. **降级决策的实时性**：P99 超时通常需要在运行时做决策（这个请求是否会超时）。可以根据历史数据训练一个延迟预测模型，在请求开始时预测完成时间，对预测超时的请求提前激活降级路径。

**差距在哪**：新手把 LLM 慢当不可控因素放弃保障，高手把延迟拆分成可量化的组件，给每个组件分配预算，用流式 TTFT 改善感知延迟，用预测降级保障 P99。

## Q：如何设计工具执行副作用失败时的补偿机制？

> 来源：AI 事务可靠性面试

**新手答**："出错了就报告失败，让用户重试。"

**高手答**：

1. **副作用的不可逆性分类**：写数据库（可回滚）、发邮件（不可回滚）、发 Slack 消息（不可回滚）、调用 Webhook（取决于对方）、扣款（极难回滚）。设计补偿机制前要先分类。

2. **可回滚操作用数据库事务**：把可回滚的多步操作包在事务里，任何一步失败整体回滚，保证一致性。

3. **不可回滚操作的前置验证**：发邮件、发消息这类不可回滚的操作，在执行前做完整性检查——收件人是否有效、内容是否符合规范、是否有重复发送风险（幂等性检查）。减少需要"补偿"的场景。

4. **Saga 补偿事务**：对于无法避免部分失败的多步流程，为每一步定义补偿操作（如"发邮件"的补偿是"发一封撤回/修正邮件"，"创建订单"的补偿是"取消订单"）。失败时按倒序执行补偿链。

5. **人工介入 Queue**：对于既不能自动重试、也没有自动补偿的高风险失败（如扣款失败但服务已提供），推入人工处理队列，配套完整的失败上下文（操作内容、失败原因、时间戳），由人工处理后标记完成。

**差距在哪**：新手把失败当终态（报告 error 让用户重试），高手知道在有副作用的系统中，失败的恢复策略比失败的检测更难设计，Saga 补偿模式是解决这类问题的成熟工程答案。

## Q：多 Provider 环境下如何实现服务健康检查？

> 来源：AI 基础设施工程师面试

**新手答**："等到调用失败了才知道 Provider 挂了。"

**高手答**：

1. **主动探测而非被动检测**：每隔 30 秒向每个 Provider 发送轻量健康探测请求（短 Prompt、低 token 的测试调用），实时维护健康状态，而不是等真实请求失败才知道故障。

2. **探测请求的开销控制**：探测请求要极轻量——1-5 tokens 的输入/输出，不消耗大量 API 额度，同时能反映服务的实际状态（避免探测和真实请求走不同代码路径）。

3. **延迟趋势监控**：不只监控成功/失败，还要监控每次探测的延迟。当 P95 延迟从正常的 1 秒升到 5 秒时，虽然没有失败，也应该降低该 Provider 的路由权重，提前规避即将到来的故障。

4. **多维度健康状态**：可用性（能不能调通）、延迟（快不快）、质量（返回内容是否正常）三维度独立评估。某些故障只影响质量（模型返回垃圾但 HTTP 200），纯成功/失败检测发现不了。

5. **熔断器与健康检查的联动**：健康检查结果直接驱动熔断器状态——连续 3 次探测失败 → Open 熔断；探测恢复 → Half-Open；连续 3 次探测成功 → Closed。健康检查是熔断器的"眼睛"。

**差距在哪**：新手用被动检测（请求失败才知道故障），高手用主动探测建立实时健康视图，结合延迟趋势实现预测性切换，在故障影响用户之前就完成 Provider 迁移。

## Q：Agent Loop 的超时控制应该如何分层设计？

> 来源：Agent 工程稳定性深度面试

**新手答**："设一个总超时，Loop 超过时间就停掉。"

**高手答**：

1. **三层超时模型**：单次 LLM 调用超时（30-120 秒，防止单次生成卡死）+ 单个 Tool 调用超时（按 Tool 类型分级，5-120 秒）+ 整个 Agent Loop 的会话超时（覆盖所有迭代的总时间上限，通常 3-15 分钟）。三层各司其职，不能只有一层。

2. **超时的传递（Deadline Propagation）**：会话 deadline 要向下传递给每一次 LLM 调用和 Tool 调用。如果会话还剩 20 秒，Tool 的独立超时是 60 秒，实际应取 min(60, 20)=20 秒，避免 Tool 超时时间超过父级 deadline 导致无谓等待。

3. **LLM 调用超时的特殊性**：LLM 生成是流式的，超时策略要区分"首 token 超时"（TTFT，通常 < 5 秒，超过说明服务有问题）和"生成总超时"（整个 completion 的时间上限）。只设总超时会让用户等 60 秒才知道服务卡了。

4. **超时后的局部结果保留**：LLM 调用超时时，如果已经通过流式收到部分响应，可以截断后作为不完整结果返回，而非直接丢弃——"已生成的部分分析如下，后续可继续请求"比空响应体验好得多。

5. **超时与重试的协调**：超时不等于需要重试。会话总超时触发时不应再重试（会继续占用超时的时间槽），单次 LLM 调用超时可以重试（如果会话仍有剩余时间）。错误类型决定是否重试，而非统一处理。

**差距在哪**：新手把超时当"等多久停止"，高手把超时当三层预算的分配和传递系统——每层超时独立保护、父子层超时传递防止等待浪费、LLM 流式超时分首 token 和总时间两个维度控制。

## Q：分布式 Agent 系统如何用分布式锁避免多实例操作同一资源？

> 来源：分布式 Agent 架构面试

**新手答**："部署多个 Agent 实例，让负载均衡分配请求，一般不会冲突。"

**高手答**：

1. **冲突场景**：多实例 Agent 在以下场景会产生竞争——同一 session 在多实例上并发续跑（崩溃恢复时触发）；多个 Agent 同时修改同一份用户数据；批处理任务的多实例争抢同一批任务。

2. **Redis 分布式锁实现**：用 `SET key value NX PX ttl` 命令原子性获取锁（NX=不存在才设置，PX=毫秒级过期）。value 存锁持有者的 instance_id，释放时先检查 value 确认是自己的锁再删除（用 Lua 脚本保证原子性）。

3. **锁 TTL 的设置**：TTL 要大于 Critical Section 的最大执行时间，但不能无限大（防止持有者崩溃后锁永远不释放）。对于长任务，在执行中定期续期（Watchdog 模式：每隔 TTL/3 时间自动续期），任务完成后主动释放。

4. **锁竞争的退避策略**：获取锁失败时不是立即重试（会造成惊群），而是等待随机的退避时间后重试（Jitter Backoff），超过最大等待时间后报告"资源被占用，请稍后再试"。

5. **锁粒度的最小化**：锁的粒度要尽可能小——不锁整个 Agent，而是锁具体的资源（`lock:session:{id}`、`lock:resume:{user_id}`），让不相关的操作可以并发执行，减少竞争面积。

**差距在哪**：新手认为"一般不会冲突"，高手知道分布式环境下崩溃恢复和并发请求必然产生竞争，Redis NX+TTL 是实现分布式锁的工程标准，Watchdog 模式解决长任务的锁续期问题。

## Q：消息队列在 Agent 容错中扮演什么角色？DLQ 如何设计？

> 来源：高可靠 Agent 平台架构面试

**新手答**："用消息队列是为了解耦，容错主要靠重试。"

**高手答**：

1. **消息队列的容错价值**：生产者（API 接收用户请求）和消费者（Agent 处理器）解耦，消费者崩溃时消息不丢失（队列持久化），重启后自动从队列继续消费。这把"用户请求在处理中服务崩溃"从数据丢失变为短暂延迟。

2. **重试策略与队列结合**：消息处理失败后不直接丢弃，而是放回队列（带延迟重试），第一次重试延 30 秒，第二次延 5 分钟，第三次延 30 分钟（指数退避），最多重试 N 次。重试次数记录在消息 metadata 中。

3. **死信队列（DLQ）设计**：超过最大重试次数的消息自动转移到 DLQ（Dead Letter Queue）。DLQ 中的消息不自动重试，需要人工分析失败原因——是代码 bug、数据异常还是下游故障。修复后手动重投 DLQ 消息。

4. **DLQ 的监控告警**：DLQ 有消息进入时立即触发告警（Slack、PagerDuty）。DLQ 堆积说明系统有系统性失败，是需要立刻响应的信号。DLQ 消息量是衡量系统健康状况的核心指标之一。

5. **幂等消费保证**：消息队列的 at-least-once 语义意味着消息可能被消费多次（ACK 丢失后重新投递）。消费者必须实现幂等处理（基于 message_id 做去重），防止同一 Agent 任务被执行两次产生重复副作用。

**差距在哪**：新手用消息队列只是为了"解耦"，高手知道消息队列的核心容错价值在于持久化 + 重试 + DLQ 三位一体，让任何处理失败都有明确的恢复路径，而不是静默丢失。

## Q：Agent 的 State Checkpointing 如何设计？崩溃后如何恢复？

> 来源：有状态 Agent 高可用面试

**新手答**："定期保存进度，崩溃了从上次保存的地方继续。"

**高手答**：

1. **Checkpoint 内容的完备性**：一个完整的 Checkpoint 包含：当前 messages 数组（完整对话历史）、iteration_count（已执行轮次）、session_state（状态机当前状态）、tool_results_cache（已完成的 Tool 调用结果）、context_layers（当前 context 各层内容）。缺任何一项，续跑结果可能不确定。

2. **Checkpoint 的时机**：在每轮迭代的"安全点"保存——LLM 调用完成且结果已记录后、Tool 调用完成且结果已注入 context 后。不在 Tool 执行中途保存（半完成状态续跑会产生重复副作用）。

3. **事务性写入**：Checkpoint 写入要原子——用数据库事务或文件的原子替换（写入临时文件，完成后 rename 替换旧文件）。防止写入中途崩溃产生损坏的 Checkpoint。

4. **崩溃恢复的判断逻辑**：重启后扫描处于 `active` 状态但超过一定时间未更新的 session（心跳超时）。对这些 session 评估：Checkpoint 是否完整（校验 hash）→ 最后的 Tool 调用是否幂等可重试 → 决定是续跑还是标记失败通知用户。

5. **Checkpoint 的存储策略**：活跃 session 的 Checkpoint 存 Redis（快速读写），同时异步备份到持久存储（PostgreSQL/S3）。Redis 重启导致 Checkpoint 丢失时，从持久存储恢复。不要只存一处。

**差距在哪**：新手的 Checkpoint 是"定期备份对话历史"，高手的 Checkpoint 是完整可恢复的执行状态——包含上下文层、工具缓存、状态机状态，配合幂等性保证让"崩溃恢复"从概念变为可靠的工程实现。

## Q：超时如何在工具调用链中正确传递（Deadline Propagation）？

> 来源：分布式系统工程面试

**新手答**："每个地方分别设超时，各自管好就行了。"

**高手答**：

1. **Deadline vs Timeout 的区别**：Timeout 是"从现在起等多久"，Deadline 是"绝对截止时刻"。在调用链中传递 Deadline（绝对时间戳）而不是 Timeout（相对时间），可以避免逐层传递时时间被累加错误。

2. **Context 携带 Deadline**：每个 Agent 请求创建时记录 `deadline = now() + maxAllowedTime`，通过请求上下文（Go Context / AsyncLocalStorage）在整个调用链中自动传递，不需要每一层手动传参。

3. **每层取 min(本层超时, 剩余时间)**：Tool 调用时，实际超时 = min(tool.maxTimeout, deadline - now())。这确保无论哪一层的超时设置多宽，都不会超过顶层的 deadline。

4. **剩余时间检查（Early Exit）**：在每个 Tool 调用开始前，先检查 `deadline - now()`，如果剩余时间已经不足以完成该工具的最短执行时间（如 < 500ms），直接跳过并返回"时间不足，已跳过"，而不是发起注定超时的请求。

5. **跨进程/服务的 Deadline 传递**：当 Agent 调用外部微服务时，通过 HTTP header（如 `X-Request-Deadline: <timestamp>`）传递 deadline，服务方根据剩余时间决定是否执行请求，而不是盲目执行导致超时后工作白费。

**差距在哪**：新手在每一层独立设超时，导致调用链总超时 = 各层超时之和，远超用户可接受的等待时间。高手传递绝对 Deadline，每层取 min 确保任何子调用都不超过顶层约束，配合 Early Exit 避免注定失败的工作。

## Q：大 Context 下如何防止内存 OOM？

> 来源：LLM 应用内存管理工程面试

**新手答**："context 太大就压缩一下，一般没有 OOM 问题。"

**高手答**：

1. **Token 估算 vs 实际内存**：`chars/3.5` 的 token 估算只反映 LLM 侧的 context 窗口，不反映应用进程的内存。大量并发 session 每个持有完整 messages 数组时，Node.js/Python 进程的堆内存会线性增长，最终 OOM。

2. **Session 的内存上限**：为每个 Session 的 messages 数组设置内存上限（如每 session 最多占用 10MB）。达到上限时触发内存侧压缩（清理 tool_use 中间状态、压缩重复内容），而不是继续增长到 OOM。

3. **按需加载 vs 全量加载**：长 session 的历史消息不全部加载到内存，而是按需分页——最近 N 轮消息保持在内存中，更早的消息写入数据库，需要时按 offset 查询。这把单 session 的内存占用从 O(历史长度) 降到 O(N)。

4. **工具结果的及时释放**：Tool 调用的原始结果（可能几 MB）在注入 context 后（已截断为 2000 tokens）应立即释放原始对象引用，让 GC 回收。不要在 session 对象上保留所有 Tool 调用的原始输出。

5. **内存监控与主动保护**：在服务启动时注册内存使用监控（如 Node.js 的 `process.memoryUsage()`），当堆内存超过阈值（如 80% of max heap）时，拒绝接受新 session，现有 session 加速压缩，主动保护服务稳定性而不是等 OOM 崩溃。

**差距在哪**：新手把 OOM 防护等同于"context 压缩"（token 层面），高手知道应用进程的内存 OOM 是独立的工程问题——需要内存上限、按需加载、及时释放、主动监控四个层次协同保护，token 压缩只是其中一环。

## Q：Bulkhead 隔离模式如何应用于 Agent 系统？

> 来源：AI 系统弹性架构面试

**新手答**："好像是微服务里的概念，把服务分开部署。"

**高手答**：

1. **Bulkhead 的核心思想**：来自造船业的水密舱壁——某个舱进水不会蔓延到其他舱。在 Agent 系统中，不同类型的请求使用独立的线程池/协程池/连接池，一类请求的过载不影响其他类型请求的处理能力。

2. **按请求类型隔离**：实时交互请求（用户对话，低延迟要求）和批处理请求（批量分析简历，高吞吐但可延迟）使用独立的线程池。批处理任务堆积时，实时请求的线程池不受影响，用户体验不退化。

3. **Provider 级别的连接池隔离**：OpenAI 和 Anthropic 使用独立的 HTTP 连接池（各自最多 20 个连接）。OpenAI 被限速导致连接被占满时，Anthropic 的连接池不受影响，Fallback 切换立刻生效。

4. **并发限制的参数**：每个 Bulkhead 配置：最大并发数（线程池大小）、等待队列长度（超过则拒绝，而非无限排队）、等待队列超时（等太久直接拒绝）。三个参数共同定义了每个隔离舱的容量上限。

5. **Bulkhead 与 Circuit Breaker 的组合**：Bulkhead 防止过载蔓延；Circuit Breaker 防止故障蔓延。二者组合——Bulkhead 限制某类请求的并发上限，Circuit Breaker 在该类请求失败率过高时切断。缺一不可。

**差距在哪**：新手知道"服务要分开部署"，高手知道 Bulkhead 是进程内的资源隔离——独立线程池、独立连接池，让不同类型请求的资源消耗互不侵占，单类故障不会拖垮整个服务。

## Q：LLM 输出 JSON 解析失败时如何重试和降级？

> 来源：LLM 应用鲁棒性工程面试

**新手答**："解析失败了让模型再生成一次。"

**高手答**：

1. **失败原因分类再处理**：①截断（输出被 max_tokens 截断导致 JSON 不完整）→ 用 continuation prompt 要求补全；②格式偏差（用了单引号、多了注释）→ 用 lenient 解析（JSON5 / js-yaml）或正则修复后再解析；③结构错误（缺少必要字段）→ 带错误详情重试；④安全拒绝（模型拒绝输出 JSON）→ 降级为自由文本。

2. **带错误上下文的重试 Prompt**：重试时不能只说"请重新输出 JSON"，要附带具体错误——"你上次的输出解析失败：`SyntaxError: Unexpected token at position 234`，原始输出为：`{...}`，请修复并重新以合法 JSON 格式输出"。精确的错误信息让模型能定向修复。

3. **重试次数上限与升级**：最多 2-3 次格式重试，超过后进入降级路径。无限重试不仅浪费 token，还可能让模型陷入"修这里破那里"的循环。第 3 次失败后记录告警，人工分析是否需要改进 Prompt 或 Schema。

4. **降级为自由文本处理**：当 JSON 解析持续失败时，切换为从自由文本中提取关键信息的模式（用正则或 NLP 提取，或再次调用 LLM 将自由文本转 JSON）。这增加了一次 LLM 调用成本，但保证了功能可用。

5. **根本性修复：结构化输出**：使用 OpenAI 的 Structured Output（JSON Schema 约束）或 Anthropic 的 tool_use 模式强制 LLM 输出符合 Schema 的 JSON，从根本上消除格式错误问题。这是比重试策略更根本的解决方案。

**差距在哪**：新手把 JSON 解析失败当"重试就好"的小问题，高手知道不同的失败原因要差异化处理——截断走续写、格式偏差走修复解析、结构错误走带上下文重试，最终用结构化输出从根本上消除问题。

## Q：Agent 收到 SIGTERM 时如何实现 Graceful Shutdown？

> 来源：Agent 运维工程面试

**新手答**："收到信号就停止，进行中的请求让用户重试就行。"

**高手答**：

1. **捕获 SIGTERM 并设置停止标志**：注册 SIGTERM 处理函数，设置 `isShuttingDown = true`，停止接受新的 session 请求（负载均衡层收到健康检查失败后会将实例从轮询列表摘除），但不立刻终止进程。

2. **等待进行中的 Session 完成**：对所有 `active` 状态的 session，允许当前迭代轮次完成（LLM 调用结束、Tool 调用结束），然后优雅地将 session 状态持久化。等待的最大时间通常是 30-60 秒（Kubernetes terminationGracePeriodSeconds）。

3. **超时后的强制清理**：如果等待时间超过 graceful period 仍有 session 未完成，执行强制清理：将剩余 active session 的状态保存为 `interrupted`（可恢复）而非 `failed`（不可恢复），附带"服务维护中，您的任务将在重启后自动恢复"的通知。

4. **资源的有序释放**：停止接受新请求 → 等待进行中的 session → 关闭数据库连接池 → 关闭 HTTP 服务器 → 退出进程。顺序很重要：数据库连接要在 session 全部结束后才关闭，防止 session 处理中途失去数据库连接。

5. **与 Kubernetes 生命周期的配合**：Pod 收到 SIGTERM 后，Kubernetes 会等待 terminationGracePeriodSeconds 再发 SIGKILL。要保证 graceful shutdown 在这个窗口内完成。对长任务（如代码分析），要在收到 SIGTERM 时主动降低任务优先级，优先处理最快能完成的 session。

**差距在哪**：新手把 SIGTERM 等同于"立刻停止"，高手把 Graceful Shutdown 设计为有序的四阶段流程——停止接受新请求、等待进行中任务、超时后保存可恢复状态、有序释放资源，让重新部署对用户透明无感知。
