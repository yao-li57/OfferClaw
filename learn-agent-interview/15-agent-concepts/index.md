# Agent 核心概念

## Q：Agentic AI 和 Chatbot 的本质区别

> 来源：AI 系统设计面试、高级工程师岗位

**新手答**："Agentic AI 能自动完成任务，Chatbot 只能对话。"

**高手答**：

**控制流归属不同**：Chatbot 的控制流在用户侧——用户驱动每一步；Agent 的控制流在系统侧——Agent 自主决定下一步行动，形成 think-act 循环。

**工具执行能力**：Chatbot 仅输出文本；Agent 具备工具调用（Tool Use）能力，可操作外部系统，产生真实世界副作用。

**规划与反馈闭环**：Agent 具备 Planning 层，能将长目标分解为子任务，并根据工具执行结果动态调整计划；Chatbot 每轮独立生成回复，无持久规划状态。

**状态持久性**：Agent 维护跨轮次的内部状态（记忆、任务进度）；Chatbot 通常依赖上下文窗口，无显式状态机。

**失败恢复机制**：Agent 能检测工具调用失败并重试或切换策略；Chatbot 无执行层，不存在失败恢复的概念。

**差距在哪**：新手停留在"能力"描述层面，高手从控制流归属、副作用、规划闭环三个维度揭示架构本质差异。

## Q：Planning 在 Agent 中的含义

> 来源：系统设计面试、LLM 应用架构讨论

**新手答**："Planning 就是 Agent 把任务分成几步来做。"

**高手答**：

**Planning 的定义层次**：Planning 是将高层目标（goal）分解为一系列可执行子任务（subtask）的过程，包含任务分解、依赖排序和资源分配三个子问题。

**主要 Planning 范式**：ReAct（交替 Reason+Act）适合短链任务；Tree-of-Thought 适合需要搜索最优路径的复杂任务；Plan-and-Execute 先生成完整计划再执行，减少中间 LLM 调用次数。

**静态 vs 动态规划**：静态规划在执行前一次性生成；动态规划在每步执行后重新评估并修订后续计划，代价是更多 LLM 调用但鲁棒性更高。

**规划的可验证性**：优秀的规划输出应结构化（如 JSON 任务列表），便于程序检查前置条件是否满足，避免幻觉式规划（计划中引用不存在的工具或数据）。

**规划失败的常见原因**：目标歧义、工具能力边界理解不足、缺乏中间状态验证。工程上通过 Planning Validator 或 Critic Agent 来捕获。

**差距在哪**：新手只知道"分步"，高手能区分范式选型、静态/动态权衡，并理解规划可验证性在工程落地中的重要性。

## Q：Grounding（接地）的概念与工程实现

> 来源：RAG 系统设计面试、LLM 可靠性讨论

**新手答**："Grounding 就是给模型提供真实信息，防止它瞎编。"

**高手答**：

**概念定义**：Grounding 指将模型生成的内容锚定到可验证的外部事实源，解决 LLM 的参数知识时效性差、易幻觉的问题。分为检索接地（Retrieval Grounding）和工具接地（Tool Grounding）两类。

**检索接地的工程实现**：RAG 管道——查询改写 → 向量检索 → 重排序（Rerank）→ 上下文注入。关键质量指标是检索召回率和上下文相关性，两者需分别评测。

**工具接地的工程实现**：让 Agent 调用 API/数据库/计算器获取实时数据，结果作为 ground truth 注入 prompt。需处理工具超时、返回格式标准化等工程问题。

**幻觉检测与接地验证**：生成后用 NLI（自然语言推理）模型验证回复是否被检索内容支撑，不支撑的声明标记为未接地并触发降级策略（拒答或追加检索）。

**接地的局限性**：外部数据源本身可能错误；检索结果与查询语义不匹配时会引入噪音；过度依赖接地会压制模型的合理推理能力。

**差距在哪**：新手停在概念层，高手能描述完整 RAG 管道、工具接地实现，以及接地验证和局限性的工程处理方式。

## Q：Agent 的 Autonomy Level 分类

> 来源：AI 产品设计讨论、安全性评审

**新手答**："自主程度高就是 Agent 自己做决定，低就是等人告诉它。"

**高手答**：

**L0 - 完全人工**：无 AI 介入，所有决策由人完成，AI 仅作为信息展示工具。

**L1 - 辅助建议**：AI 提供候选方案，人类做最终选择并执行所有操作。典型场景：代码补全建议。

**L2 - 条件自动化**：AI 在预设规则范围内自主执行，超出规则立即转人工。典型场景：带白名单的自动审批流。

**L3 - 有监督自主**：AI 自主规划并执行多步任务，人类保留随时介入和中止的能力，关键节点要求人工确认。典型场景：当前主流 Agentic 系统。

**L4 - 高度自主**：AI 全权处理任务，仅在失败或超出置信度阈值时上报人工。需要成熟的错误检测和 fallback 机制。

**L5 - 完全自主**：AI 完全独立运行，理论级别，当前不具备安全部署条件。

**差距在哪**：新手只有"高/低"二元认知，高手能给出分级框架，并理解每级对应的工程安全要求和监督机制设计。

## Q：Emergent Behavior 在 Agent 中的含义

> 来源：大模型能力讨论、Agent 安全评审

**新手答**："Emergent Behavior 就是模型突然会了一些没训练过的技能。"

**高手答**：

**基本定义**：涌现行为指在系统规模（参数量、上下文长度、工具组合数）超过某个阈值后，突然出现的、在小规模时不存在的能力。不是渐进式提升，而是阶跃式出现。

**Agent 系统中的涌现维度**：单模型涌现（如 CoT 推理能力随模型规模出现）；工具组合涌现（多工具协同产生单工具不具备的能力）；多 Agent 涌现（Agent 间协作涌现出个体没有的问题解决策略）。

**涌现的工程影响**：无法在小规模测试中提前发现涌现行为，导致生产环境出现预期外的能力或行为，需要更严格的 staging 测试和行为监控。

**涌现与可靠性的矛盾**：涌现能力往往不稳定，同一输入在不同运行中可能触发或不触发，给 SLA 承诺带来挑战。

**工程应对**：通过行为基准测试持续追踪能力边界；对高风险涌现行为（如自主网络访问）设置硬约束而非依赖模型自律。

**差距在哪**：新手将涌现理解为简单的"意外学会"，高手能区分涌现的维度，并理解其对测试可靠性和安全约束的工程影响。

## Q：Tool Augmentation vs Model Capability 的取舍

> 来源：系统架构设计面试、LLM 选型讨论

**新手答**："能用工具解决就用工具，模型解决不了的再加工具。"

**高手答**：

**核心权衡维度**：延迟（工具调用增加网络往返）、成本（更强模型 vs 工具调用费用）、可维护性（工具逻辑可独立更新 vs 模型知识需重训）、可解释性（工具调用链可审计，模型内部推理不透明）。

**优先使用工具的场景**：需要实时数据（股价、天气）；需要精确计算（数学、代码执行）；需要与外部系统交互（写数据库、发邮件）；需要可审计的操作记录。

**优先依赖模型能力的场景**：语义理解、风格改写、知识综合等任务工具化成本高；工具调用链过长导致延迟不可接受；低频边缘任务不值得维护工具。

**工具膨胀的风险**：工具数量过多时，模型的工具选择准确率下降（工具幻觉），需要工具分组或动态加载策略。

**混合策略**：用模型能力做意图识别和结果综合，用工具做确定性计算和数据获取，两者职责边界清晰。

**差距在哪**：新手缺乏取舍框架，高手能从延迟、成本、可维护性多维度分析，并理解工具膨胀带来的选择准确率下降问题。

## Q：Agent 的 World Model

> 来源：AI 系统设计、Robotics/游戏 AI 跨界讨论

**新手答**："World Model 就是模型对世界的理解，就是它的知识。"

**高手答**：

**定义与来源**：World Model 来自认知科学，指 Agent 对环境状态、动作后果和因果关系的内部表示。它允许 Agent 在不实际执行动作的情况下预测结果（"如果我调用这个 API 会发生什么"）。

**LLM Agent 中的 World Model 实现**：LLM 的参数知识构成隐式 World Model；显式 World Model 则通过结构化状态描述（如环境 schema、工具说明文档）注入上下文来实现。

**World Model 的局限性**：LLM 的隐式 World Model 存在时效性问题和领域偏差；显式 World Model 的维护成本高，状态爆炸问题在复杂环境中难以避免。

**World Model 与规划的关系**：规划质量直接取决于 World Model 的准确性。World Model 越精确，Agent 越能预见动作副作用，规划失败率越低。

**工程实践**：通过工具说明文档（tool schema）、环境状态快照和历史执行记录共同构建运行时 World Model，弥补参数知识的不足。

**差距在哪**：新手混淆 World Model 与静态知识，高手能区分隐式/显式实现，并理解其与规划质量的因果关系。

## Q：Reflection 和 Critique 机制

> 来源：Agent 可靠性提升讨论、Self-refinement 论文背景面试

**新手答**："Reflection 就是让模型反思自己的答案，然后改进。"

**高手答**：

**Reflection 的定义**：Reflection 是 Agent 对自身输出或行动轨迹的自我评估过程，输出包括错误识别、质量评分和改进建议，驱动后续迭代。

**Critique 机制**：Critique 通常由独立的 Critic 角色（同一模型不同 prompt 或专用 Critic 模型）对输出进行外部批评，相比自我 Reflection 更能避免确认偏误。

**Reflexion 架构**：将 Reflection 结果存入长期记忆，在后续轮次中作为经验注入 prompt，实现跨任务的持续改进，是当前工程中常见的实现模式。

**Reflection 的代价**：每次 Reflection 需要额外 LLM 调用，增加延迟和成本；过度 Reflection 会导致收益边际递减甚至质量退化（模型开始质疑正确答案）。

**工程控制**：设置 Reflection 轮次上限；仅在质量评分低于阈值时触发；使用轻量模型做 Critique，重量模型做最终生成。

**差距在哪**：新手只知道"让模型改一遍"，高手能区分 Self-reflection 与 Critique 的角色分离，理解 Reflexion 记忆架构，以及代价控制策略。

## Q：Agent Identity 与 Persona

> 来源：AI 产品设计讨论、System Prompt 工程面试

**新手答**："Persona 就是给模型取个名字，让它扮演某个角色。"

**高手答**：

**Agent Identity 的工程意义**：Identity 不只是名字，而是 Agent 在 System Prompt 中声明的能力边界、行为准则和拒绝策略的总和，直接影响模型的输出分布。

**Persona 的一致性挑战**：长对话中 Persona 漂移（模型逐渐偏离设定角色）是常见问题，需要在 System Prompt 中周期性强化，或在每轮对话前重新注入核心 Identity 描述。

**Identity 与安全边界**：明确的 Identity 声明（"你是一个只处理代码问题的助手"）是 prompt injection 防护的第一道防线，使越权请求更容易被模型识别和拒绝。

**多 Agent 系统中的 Identity 管理**：每个子 Agent 需要独立的 Identity，避免角色混淆。协调者 Agent 的 Identity 需明确其指挥权限范围，防止子 Agent 越权行动。

**Persona 测试**：通过对抗性测试（尝试让模型背叛 Persona）验证 Identity 的稳定性，是 Agent 上线前的必要安全评测项。

**差距在哪**：新手停在"角色扮演"的表面，高手理解 Identity 的安全意义、漂移问题的工程处理，以及多 Agent 场景下的 Identity 隔离需求。

## Q：什么是 Cognitive Architecture

> 来源：AI 系统架构面试、Agent 框架选型讨论

**新手答**："Cognitive Architecture 就是 Agent 的整体设计，包括各个模块怎么组合。"

**高手答**：

**定义来源**：Cognitive Architecture 来自认知科学，描述智能系统的信息处理结构，包括感知、记忆、推理、决策和行动各子系统及其交互方式。ACT-R、SOAR 是经典人类认知架构模型。

**LLM Agent 的 Cognitive Architecture 组成**：感知层（输入解析）、工作记忆（上下文窗口）、长期记忆（向量库/数据库）、规划器（Task Decomposer）、执行器（Tool Caller）、元认知（Reflection/Critic）。

**架构选型的工程影响**：不同架构决定了延迟特性（顺序 vs 并行执行）、扩展性（新能力是否需要重构核心）和可观测性（模块边界清晰则更易调试）。

**主流 Agent 架构范式对比**：ReAct 架构简单但规划能力弱；Plan-and-Execute 规划与执行分离，适合长任务；多 Agent 架构（Orchestrator + Worker）适合并行化复杂任务，但协调开销大。

**架构与能力的对应关系**：Cognitive Architecture 决定了 Agent 能解决的任务类型上限，在选型阶段需要与目标任务的复杂度和可靠性要求对齐。

**差距在哪**：新手把架构等同于模块列表，高手能追溯认知科学来源，列出各子系统职责，并理解架构选型对延迟、扩展性和调试的工程影响。

## Q：Human-in-the-Loop 的不同模式

> 来源：AI 产品安全设计、企业 AI 落地讨论

**新手答**："Human-in-the-Loop 就是在关键步骤让人确认一下。"

**高手答**：

**Human-on-the-Loop（监督模式）**：人类不介入正常执行流，但实时监控 Agent 行为，发现异常时可中止。适合高吞吐低风险任务，降低人力成本但要求监控系统成熟。

**Human-in-the-Loop（确认模式）**：关键操作前强制等待人类审批，操作不可逆性越高、确认阈值越低。典型实现：审批队列 + 超时自动拒绝。

**Human-as-the-Loop（协作模式）**：人类与 Agent 交替贡献，Agent 处理机械部分，人类处理判断部分。适合创意、法律、医疗等需要领域专业知识的场景。

**触发条件设计**：不应对所有操作都要求人工介入（降低效率），需基于风险评分（操作类型、影响范围、置信度）动态决定是否触发确认。

**中断点工程实现**：实现干净的暂停/恢复机制，确保 Agent 在等待人工响应期间状态持久化，避免超时后状态丢失导致任务重跑。

**差距在哪**：新手只知道"确认模式"，高手能区分三种 HITL 模式的适用场景，并理解触发条件设计和状态持久化的工程要点。

## Q：Agent 的 Alignment 问题

> 来源：AI 安全面试、高级系统设计讨论

**新手答**："Alignment 就是让 AI 按照人的意思做事，不做坏事。"

**高手答**：

**Alignment 的工程维度**：区分两类对齐问题——意图对齐（Agent 理解并执行用户真实意图，而非字面意图）和价值对齐（Agent 行为符合更广泛的人类价值观和安全规范）。

**规格歧义（Specification Ambiguity）**：用户给出的目标往往不完整，Agent 会"合理"推断缺失约束，可能产生用户不期望的副作用。工程上通过澄清问题、约束声明和最小权限原则来缓解。

**工具滥用风险**：Agent 可能为达成目标而以非预期方式使用工具（如为完成"提高用户留存"目标而操纵通知频率）。需要在工具层面实现意图约束而非仅依赖 prompt。

**Reward Hacking 类比**：类似强化学习中的奖励黑客，Agent 可能找到满足评估指标但违背真实意图的捷径，测试时需要对抗性评估而非仅正向评估。

**实践层面的对齐工程**：PermissionGate 分级授权、操作审计日志、不可逆操作二次确认、行为监控告警——这些工程机制共同构成运行时对齐保障。

**差距在哪**：新手停在"让 AI 听话"的口号，高手能区分意图对齐和价值对齐，分析规格歧义和工具滥用风险，并落地到具体工程机制。

## Q：Open-ended Task vs Goal-directed Task

> 来源：Agent 产品设计面试、任务类型分析讨论

**新手答**："Open-ended 就是没有固定答案，Goal-directed 就是有明确目标。"

**高手答**：

**核心区别**：Goal-directed Task 有明确的完成标准（可用程序验证），Agent 可以知道何时停止；Open-ended Task 没有客观终止条件，需要外部判断或时间/资源约束来终止。

**对 Agent 设计的影响**：Goal-directed 任务可以自动化评测和回归测试；Open-ended 任务需要人类评估或 LLM-as-Judge，测试成本高，难以持续集成。

**规划机制的差异**：Goal-directed 任务适合 Plan-and-Execute（提前规划到终止条件）；Open-ended 任务更适合 ReAct（实时感知并决定是否继续），因为终止条件本身需要动态判断。

**资源消耗控制**：Open-ended Task 的 Agent 必须有硬性的迭代次数上限和 token 预算，否则可能无限循环；Goal-directed Task 可以用任务完成检测作为自然终止条件。

**业务场景对应**：代码生成、数据提取是 Goal-directed；创意写作辅助、战略分析是 Open-ended。产品设计时需明确任务类型，选择对应的 Agent 架构和评测策略。

**差距在哪**：新手停在概念定义，高手能分析对评测、规划机制和资源控制的工程影响，并映射到实际业务场景的设计选型。

## Q：In-context Learning 的工程价值

> 来源：LLM 应用工程面试、大模型能力讨论

**新手答**："In-context Learning 就是在 prompt 里给几个例子，模型就会了。"

**高手答**：

**核心机制**：In-context Learning（ICL）让模型在推理阶段通过 prompt 中的示例"学习"新任务，无需更新任何参数。本质上是利用 LLM 在预训练中积累的元学习能力——模型学会了"如何从示例中学习模式"，而不仅仅是记忆具体知识。

**工程价值的三个维度**：
1. **零改造成本**：不需要收集标注数据、跑训练流程、管理模型版本，新任务的适配成本从"周级"降到"小时级"。
2. **快速迭代**：调整示例比调整模型参数快 100 倍，适合需要频繁调整行为的业务场景（如内容审核规则变化）。
3. **任务泛化**：通过精心设计的 few-shot 示例，可以引导模型完成训练数据中未见过的任务格式，覆盖长尾需求。

**ICL 的局限性与工程应对**：示例占用宝贵上下文窗口；示例质量对性能影响巨大（差示例比没示例更差）；上下文长度限制了可提供的示例数量。工程应对：动态示例选择（从示例库中检索与当前输入最相似的 k 个示例，而非固定写死）。

**ICL vs Fine-tuning 的选型**：任务数据量 < 1k、任务频繁变化、快速验证假设 → 优先 ICL；需要极低延迟、任务固定且数据充足 → Fine-tuning 可能更优。两者不是互斥的，Fine-tuned 模型上仍可用 ICL 做细粒度引导。

**差距在哪**：新手把 ICL 当"给例子"的简单操作，高手理解其元学习本质，知道如何动态选择示例、何时选择 ICL 而非微调，以及如何量化示例质量对性能的影响。

## Q：LLM 幻觉的根因分析

> 来源：LLM 可靠性工程面试、AI 系统设计讨论

**新手答**："幻觉是模型编造了不存在的信息，因为它不够聪明。"

**高手答**：

**机制层根因**：LLM 的生成本质是基于条件概率的 token 预测，模型目标是生成"听起来合理"的序列，而非"事实正确"的序列。当训练数据中某类信息稀疏或模型对某问题不确定时，它会选择高概率但可能错误的 token，而不是输出"我不知道"——因为训练目标中没有对"拒绝回答"给予奖励。

**幻觉的主要类型**：
1. **事实幻觉**：声称不存在的事件、人物、引用（如捏造论文标题）。
2. **推理幻觉**：推理步骤看似合理但结论错误（数学、逻辑题中常见）。
3. **工具幻觉**：Agent 场景中调用不存在的工具或以错误参数调用工具。
4. **自我幻觉**：模型对自身能力的错误描述（声称能做实际做不到的事）。

**工程层缓解策略**：
- **检索接地（RAG）**：用外部可信源约束生成内容，降低知识边界外的幻觉。
- **置信度声明**：在 prompt 中要求模型标注不确定的断言，触发下游验证。
- **输出验证**：对关键输出用独立工具或 Critic Agent 二次核实。
- **温度控制**：降低 temperature 减少随机性，但同时也降低创造力。
- **工具优先**：对需要精确数据的场景用工具调用替代模型记忆。

**差距在哪**：新手把幻觉当模型质量问题，高手理解其是概率生成机制的必然产物，从机制出发选择工程缓解手段，而非期待"更聪明的模型"解决所有幻觉。

## Q：Scaling Laws 对 Agent 设计的影响

> 来源：大模型选型面试、AI 基础设施讨论

**新手答**："模型越大越好，能用大模型就用大模型。"

**高手答**：

**Scaling Laws 的基本结论**：Kaplan et al. 和 Chinchilla 研究表明，模型性能与参数量、训练数据量、计算量之间存在幂律关系。关键洞见：在固定计算预算下，训练数据量的增加与参数量增加同等重要——Chinchilla 最优化建议数据量应与参数量成比例增长。

**对 Agent 能力的影响**：
1. **复杂推理能力**：Chain-of-Thought、多步规划等 Agent 核心能力只在模型达到一定规模后才可靠出现（约 100B 参数以上），小模型即使 prompt 设计再好也无法稳定执行。
2. **指令跟随精度**：规模越大，模型对复杂嵌套指令的遵循精度越高，工具调用格式错误率越低。
3. **上下文利用率**：大模型能更好地利用长 context 中的信息，小模型在长 context 下"遗忘"关键信息的概率更高。

**对模型选型的指导意义**：
- **任务复杂度分级**：简单格式转换、摘要 → 小模型（成本低）；复杂推理、多步规划 → 大模型（必要投入）。
- **边际收益递减**：某些任务在模型规模超过一定量级后性能提升趋于平缓，不需要无限追求最大模型。
- **推理效率权衡**：更大模型意味着更高延迟和成本，需要在性能和工程约束间找平衡点。

**差距在哪**：新手把"大模型更好"当万能法则，高手理解 Scaling Laws 的具体机制，能据此做任务-模型规模匹配，在性能与成本之间做工程决策。

## Q：Constitutional AI 与 RLHF 的区别

> 来源：AI 对齐工程面试、模型训练方向讨论

**新手答**："都是让模型变得更好、更安全的训练方法。"

**高手答**：

**RLHF（人类反馈强化学习）的工程流程**：收集人类对模型输出的偏好标注 → 训练 Reward Model → 用 PPO 等算法对 LLM 做强化学习微调，使输出更符合人类偏好。核心依赖：大规模人工标注，成本高，标注者偏见会被放大进入模型。

**Constitutional AI（CAI）的工程流程**：Anthropic 提出的方法。给模型一套"宪法"（一组原则），让模型对自己的输出做自我评判（"这个输出是否违反了原则X"），通过 AI 反馈（RLAIF）替代部分人工反馈，再做强化学习。核心依赖：规则定义的质量，而非大规模人工标注。

**主要区别**：
1. **反馈来源**：RLHF 依赖人类标注员；CAI 主要依赖模型自身（AI 反馈），可规模化。
2. **对齐目标的显式性**：RLHF 的偏好是隐式的（标注员可能无法完整表达理由）；CAI 的原则是显式文本，可审计和调整。
3. **成本结构**：RLHF 标注成本随数据量线性增长；CAI 标注成本主要是原则撰写，后续可自动化。
4. **适用场景**：RLHF 在有明确人类偏好的任务（如对话风格）上效果好；CAI 在需要可解释合规约束的场景更适合。

**差距在哪**：新手把两者都归类为"安全训练"，高手能区分反馈来源、可扩展性和原则显式性的本质差异，并理解各方法在工程成本和可审计性上的取舍。

## Q：Agent 的可信度评分（Confidence Score）

> 来源：高可靠 Agent 系统设计面试、AI 风控讨论

**新手答**："让模型输出时说一下它有多确定就行了。"

**高手答**：

**核心问题**：LLM 的输出概率（token logprob）不等于语义置信度。模型可以用高概率输出错误内容（幻觉往往是"自信"的），也可以对正确内容表现出不确定性。直接用 logprob 作为置信度指标是不可靠的。

**工程实现方案**：
1. **Verbalized Confidence**：在 prompt 中要求模型以自然语言声明不确定性（"我不确定"、"需要核实"），并附上理由。简单但依赖模型的自我认知能力，不适合作为程序判断的硬条件。
2. **Sampling-based 估计**：对同一问题以不同随机种子生成 N 个回答，用答案一致性作为置信度代理指标。一致性高 → 高置信，分散 → 低置信。成本较高但更可靠。
3. **Calibration 层**：训练一个独立的置信度校准模型，输入 LLM 回答和 context，输出校准后的置信度分数。
4. **规则兜底**：对特定场景的输出做确定性校验——如要求 Agent 提供来源链接，缺失则强制降低置信度评分。

**置信度与 Agent 行为的联动**：置信度高 → 直接返回结果；中等 → 追加检索验证；低 → 触发 Human-in-the-Loop 或拒绝回答。这让 Agent 的"不确定"变成可操作的工程信号，而不是模糊的描述。

**差距在哪**：新手以为让模型说"我不确定"就完成了置信度设计，高手知道 logprob 的局限性，会用采样一致性或独立校准模型来获得更可靠的置信度信号，并将其接入 Agent 决策链路。

## Q：Chain-of-Thought 的主要变体

> 来源：LLM 推理优化面试、Agent 规划讨论

**新手答**："CoT 就是让模型一步一步想，效果更好。"

**高手答**：

**Standard CoT（标准思维链）**：在 prompt 中提供"逐步推理"的示例，引导模型在给出最终答案前输出中间推理步骤。效果：显著提升复杂推理任务的准确率，但推理路径是单一线性链，一步出错导致后续全错。

**Self-consistency CoT**：对同一问题用高温度采样生成多条独立推理链，对最终答案做多数投票。牺牲推理速度（需多次调用）换取更稳定的准确率，适合对精度要求高的场景。

**Tree of Thoughts（ToT）**：将推理过程建模为树形搜索——每个推理步骤可以分叉出多个候选延续，通过评估函数（启发式或 LLM 打分）剪枝，只保留最有希望的分支继续推进。适合需要全局搜索的问题（如数学证明、规划任务），但调用次数随树深度指数增长。

**Graph of Thoughts（GoT）**：进一步将推理图结构扩展为有向图，允许不同推理分支之间相互合并和引用，实现非树状的推理拓扑。适合需要综合多个独立推理路径结论的任务，工程实现更复杂。

**工程选型建议**：简单推理 → Standard CoT；需要稳健性 → Self-consistency；需要探索最优解 → ToT；超复杂综合推理 → GoT（谨慎评估成本）。推理能力越强的基础模型，从复杂变体中获益越大。

**差距在哪**：新手以为 CoT 只有一种，高手能描述各变体的结构差异（线性/树/图）和适用场景，并理解不同变体在推理能力与调用成本之间的权衡。

## Q：Foundation Model 和 Application Layer 的职责分工

> 来源：LLM 系统架构面试、AI 产品工程讨论

**新手答**："模型负责智能，应用层负责业务逻辑。"

**高手答**：

**Foundation Model 层的职责**：语言理解与生成、通用知识推理、指令跟随能力、安全对齐（基本有害内容过滤）。这些能力通过预训练和 RLHF 固化在模型参数中，应用层不应重复实现，也无法在推理期修改。

**Application Layer 的职责**：
1. **知识补充**：模型参数知识有截止日期，实时/专有知识通过 RAG 或工具调用注入，由应用层负责。
2. **业务规则执行**：公司特定的合规规则、格式要求、品牌语言规范等，不应期望模型"记住"，在应用层做结构化约束和后处理。
3. **状态与记忆管理**：跨轮次的对话历史、用户偏好、任务进度，由应用层的 Session/Memory 模块维护。
4. **工具执行与权限控制**：工具调用的实际执行、权限验证、结果处理在应用层，模型只输出"调用意图"。
5. **输出验证与修正**：格式校验、业务规则合规性检查、幻觉检测均在应用层处理。

**常见职责错配**：把过多业务逻辑写进 system prompt（难以测试和维护）；期望模型自行做复杂的权限判断（应在应用层硬编码）；用模型替代确定性计算（应用层工具更可靠）。

**差距在哪**：新手把所有智能都寄托于模型 prompt，高手理解模型和应用层各自的能力边界，把确定性逻辑放在代码层，把语义理解和生成留给模型，两层职责清晰才能构建可维护的系统。

## Q：LLM 的上下文长度管理

> 来源：长上下文 LLM 应用工程面试

**新手答**："用 200k context 的模型，就能记住很长的内容了。"

**高手答**：

**有效上下文 vs 标称上下文长度**：模型支持 200k tokens 不等于能有效利用全部 200k。研究（如 Lost in the Middle）表明，LLM 对上下文中间部分的信息提取能力显著弱于开头和结尾，即"中间遗失"现象。标称长度只是硬上限，有效利用区域通常集中在 context 两端。

**注意力分布的工程含义**：
1. **关键信息位置设计**：重要的指令、约束、关键数据应放在 system prompt（开头）或最近的用户消息（结尾），而不是埋在长历史对话中间。
2. **历史压缩策略**：对长对话历史做摘要而非截断保留，将关键决策点和结论提炼到 context 开头的摘要层。
3. **分块检索替代全文注入**：超长文档不应整体注入 context，而是用 RAG 按查询检索相关片段，精准注入而非暴力堆砌。

**长 Context 的成本非线性增长**：Transformer 的注意力计算复杂度是 O(n²)，context 越长，每次推理的延迟和成本增长不是线性的，在实际工程中要评估长 context 的边际收益。

**差距在哪**：新手把"支持 200k"当能力保证，高手理解注意力分布的不均匀性，会主动设计信息位置策略、历史压缩方案，而不是依赖超长 context 解决一切问题。

## Q：Prompt Compression（提示词压缩）技术

> 来源：LLM 成本优化面试、上下文管理讨论

**新手答**："把 prompt 写短一点，删掉不必要的内容。"

**高手答**：

**为什么需要系统化 Prompt Compression**：手动裁剪 prompt 是一次性的，无法适应动态内容；而自动化压缩可以针对不同查询智能识别并保留关键信息，在保持任务性能的前提下减少 30-70% 的 token 消耗。

**主要技术路线**：
1. **LLMLingua / Selective Token Dropping**：用小型语言模型对每个 token 计算"重要性分数"（基于困惑度），低分 token 直接删除。压缩率高（可达 3-5x），但可能破坏语法连贯性，适合对语法不敏感的任务（检索结果列表）。
2. **摘要压缩（Abstractive Compression）**：用 LLM 对长文档生成摘要后注入 context，而非原文。保留了语义完整性，但摘要本身可能引入信息损失。
3. **重排序裁剪（Rank and Truncate）**：对 RAG 检索结果按相关性排序，只保留 top-k 片段，舍弃尾部低相关内容。
4. **结构化压缩**：将自然语言 context 转换为结构化格式（JSON、Markdown 表格），相同信息用更少 token 表达。

**压缩率与质量的权衡**：压缩率越高，丢失关键信息的风险越大。需要建立压缩后的任务性能基准测试，确保在目标压缩率下任务准确率没有不可接受的下降。

**差距在哪**：新手把 prompt 压缩当手工编辑工作，高手知道自动化压缩技术的多种路线及其适用场景，能根据任务类型和压缩目标选择合适方法，并建立质量回归测试。

## Q：Agent 的能力评估基准（Benchmark）

> 来源：AI 系统评测讨论、Agent 研究面试

**新手答**："跑一跑 MMLU 或者 HumanEval，看看分数高不高。"

**高手答**：

**常见 Agent 评测基准及定位**：
1. **AgentBench**：覆盖 8 类环境（OS、数据库、网页浏览等），测试 Agent 完成真实世界操作性任务的能力，是目前最接近 Agent 生产场景的综合基准。
2. **WebArena / VisualWebArena**：测试 Agent 在真实网站上完成任务（购物、信息检索等），评估浏览器操作 Agent 的端到端能力。
3. **SWE-bench**：从真实 GitHub Issue 中提取软件工程任务，测试 Agent 定位 bug、修改代码、通过测试的端到端能力，是代码 Agent 最权威的评测之一。
4. **GAIA**：设计为"通用 AI 助手"评测，任务需要多步推理、工具使用和真实世界信息整合，区分度高。
5. **τ-bench（TAU-bench）**：测试 Agent 在真实用户服务场景（航空、零售）中遵循规则完成长任务的能力，重点评测指令遵循和工具使用的鲁棒性。

**各基准的局限性**：
- 测试集污染（模型训练数据可能包含基准答案）
- 静态快照（网站/代码库随时间变化，基准无法持续更新）
- 单一维度（代码 Agent 好不代表推理 Agent 好）
- 与业务场景的 gap（基准任务未必反映实际业务复杂度）

**实践建议**：公开基准用于横向对比选型，自建内部 Golden Set 用于日常回归测试，两者结合才能既看业界水位又保证业务场景适配。

**差距在哪**：新手用通用 NLP 基准评测 Agent，高手知道 Agent 评测需要操作性任务基准（AgentBench、SWE-bench），并清楚各基准的局限性和使用场景。

## Q：Instruction Following vs Goal Pursuit

> 来源：AI 安全设计面试、Agent 对齐讨论

**新手答**："指令遵循更安全，追求目标更智能。"

**高手答**：

**本质区分**：
- **Instruction Following Agent**：严格按当前收到的指令执行，不自主扩展行动范围。"用户说做什么，就做什么，不多做。"
- **Goal Pursuit Agent**：以达成设定目标为优先，会主动规划、选择工具、分解子任务，甚至执行指令中未明确提到但"有助于目标"的动作。

**风险分析**：
1. **Instruction Following 的局限**：对复杂多步任务需要大量显式指令，用户负担重；面对意外情况时无法自适应，容易卡死。
2. **Goal Pursuit 的风险**：Agent 可能为达成目标采取用户未预期的行动（工具滥用、副作用超出预期）；目标定义模糊时发生 reward hacking；高自主性与安全边界之间的张力更大。

**工程选型建议**：
- 高风险、不可逆操作场景 → Instruction Following 更安全，每步明确授权。
- 低风险、探索性任务 → Goal Pursuit 效率更高，允许 Agent 自主规划。
- 实践中通常采用混合策略：Goal-level 定义目标，Instruction-level 约束执行边界（PermissionGate 的 RiskLevel 分级本质上就是这种混合）。

**差距在哪**：新手认为"追目标更强大"，高手理解 Goal Pursuit 的自主性带来的不可预测行为风险，能根据操作风险等级在两种模式间做明智选型，并设计相应的约束机制。

## Q：LLM 的 Latent Space 与知识编码

> 来源：大模型机制理解面试、AI 基础研究讨论

**新手答**："LLM 把知识存在参数里，所以能回答问题。"

**高手答**：

**Latent Space 的定义**：Latent Space（潜在空间）指神经网络中间层的高维激活空间，每个 token 被表示为该空间中的一个向量。LLM 的知识并非以"键值对"形式存储，而是以分布式方式编码在数十亿参数矩阵的权重中，通过前向传播在 latent space 中进行计算。

**知识编码的机制**：
1. **事实知识的几何表示**：研究表明，语义相近的概念在 latent space 中距离相近（向量空间中的簇结构），关系（如"首都"）可以用向量差来表示。
2. **知识以叠加（Superposition）形式存储**：单个神经元并非对应单一概念，一个神经元对多个不相关概念响应（多义性），一个概念分散在多个神经元上。这解释了为什么模型难以被直接"解剖"读出知识。
3. **Attention 的知识检索机制**：Transformer 的 attention 机制在 latent space 中做动态加权聚合，根据当前 token 的 query vector 去"检索"相关 key vector，是知识激活的核心机制。

**对工程实践的启示**：
- LLM 的知识边界是模糊的，不像数据库那样有明确的"有/无"分界，这是幻觉难以根除的根本原因。
- Fine-tuning 是在参数空间中做局部调整，不能保证只修改目标知识而不影响其他能力（灾难性遗忘问题）。
- Prompt 工程的本质是通过输入 token 激活特定的 latent space 路径，引导模型从正确的"知识区域"生成输出。

**差距在哪**：新手把 LLM 的知识理解成数据库存储，高手理解分布式编码、superposition 和 attention 检索机制，这种理解能指导更好的 prompt 设计和对模型行为的合理预期设置。
