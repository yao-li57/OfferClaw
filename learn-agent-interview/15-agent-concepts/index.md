# Agent 核心概念

## Q：Agentic AI 和 Chatbot 的本质区别

> 来源：AI 系统设计面试、高级工程师岗位

**新手答**："Agentic AI 能自动完成任务，Chatbot 只能对话。"

**高手答**：

**控制流归属不同**：Chatbot 的控制流在用户侧——用户驱动每一步；Agent 的控制流在系统侧——Agent 自主决定下一步行动，形成 think-act 循环。

**工具执行能力**：Chatbot 仅输出文本；Agent 具备工具调用（Tool Use）能力，可操作外部系统，产生真实世界副作用。

**规划与反馈闭环**：Agent 具备 Planning 层，能将长目标分解为子任务，并根据工具执行结果动态调整计划；Chatbot 每轮独立生成回复，无持久规划状态。

**状态持久性**：Agent 维护跨轮次的内部状态（记忆、任务进度）；Chatbot 通常依赖上下文窗口，无显式状态机。

**失败恢复机制**：Agent 能检测工具调用失败并重试或切换策略；Chatbot 无执行层，不存在失败恢复的概念。

**差距在哪**：新手停留在"能力"描述层面，高手从控制流归属、副作用、规划闭环三个维度揭示架构本质差异。

## Q：Planning 在 Agent 中的含义

> 来源：系统设计面试、LLM 应用架构讨论

**新手答**："Planning 就是 Agent 把任务分成几步来做。"

**高手答**：

**Planning 的定义层次**：Planning 是将高层目标（goal）分解为一系列可执行子任务（subtask）的过程，包含任务分解、依赖排序和资源分配三个子问题。

**主要 Planning 范式**：ReAct（交替 Reason+Act）适合短链任务；Tree-of-Thought 适合需要搜索最优路径的复杂任务；Plan-and-Execute 先生成完整计划再执行，减少中间 LLM 调用次数。

**静态 vs 动态规划**：静态规划在执行前一次性生成；动态规划在每步执行后重新评估并修订后续计划，代价是更多 LLM 调用但鲁棒性更高。

**规划的可验证性**：优秀的规划输出应结构化（如 JSON 任务列表），便于程序检查前置条件是否满足，避免幻觉式规划（计划中引用不存在的工具或数据）。

**规划失败的常见原因**：目标歧义、工具能力边界理解不足、缺乏中间状态验证。工程上通过 Planning Validator 或 Critic Agent 来捕获。

**差距在哪**：新手只知道"分步"，高手能区分范式选型、静态/动态权衡，并理解规划可验证性在工程落地中的重要性。

## Q：Grounding（接地）的概念与工程实现

> 来源：RAG 系统设计面试、LLM 可靠性讨论

**新手答**："Grounding 就是给模型提供真实信息，防止它瞎编。"

**高手答**：

**概念定义**：Grounding 指将模型生成的内容锚定到可验证的外部事实源，解决 LLM 的参数知识时效性差、易幻觉的问题。分为检索接地（Retrieval Grounding）和工具接地（Tool Grounding）两类。

**检索接地的工程实现**：RAG 管道——查询改写 → 向量检索 → 重排序（Rerank）→ 上下文注入。关键质量指标是检索召回率和上下文相关性，两者需分别评测。

**工具接地的工程实现**：让 Agent 调用 API/数据库/计算器获取实时数据，结果作为 ground truth 注入 prompt。需处理工具超时、返回格式标准化等工程问题。

**幻觉检测与接地验证**：生成后用 NLI（自然语言推理）模型验证回复是否被检索内容支撑，不支撑的声明标记为未接地并触发降级策略（拒答或追加检索）。

**接地的局限性**：外部数据源本身可能错误；检索结果与查询语义不匹配时会引入噪音；过度依赖接地会压制模型的合理推理能力。

**差距在哪**：新手停在概念层，高手能描述完整 RAG 管道、工具接地实现，以及接地验证和局限性的工程处理方式。

## Q：Agent 的 Autonomy Level 分类

> 来源：AI 产品设计讨论、安全性评审

**新手答**："自主程度高就是 Agent 自己做决定，低就是等人告诉它。"

**高手答**：

**L0 - 完全人工**：无 AI 介入，所有决策由人完成，AI 仅作为信息展示工具。

**L1 - 辅助建议**：AI 提供候选方案，人类做最终选择并执行所有操作。典型场景：代码补全建议。

**L2 - 条件自动化**：AI 在预设规则范围内自主执行，超出规则立即转人工。典型场景：带白名单的自动审批流。

**L3 - 有监督自主**：AI 自主规划并执行多步任务，人类保留随时介入和中止的能力，关键节点要求人工确认。典型场景：当前主流 Agentic 系统。

**L4 - 高度自主**：AI 全权处理任务，仅在失败或超出置信度阈值时上报人工。需要成熟的错误检测和 fallback 机制。

**L5 - 完全自主**：AI 完全独立运行，理论级别，当前不具备安全部署条件。

**差距在哪**：新手只有"高/低"二元认知，高手能给出分级框架，并理解每级对应的工程安全要求和监督机制设计。

## Q：Emergent Behavior 在 Agent 中的含义

> 来源：大模型能力讨论、Agent 安全评审

**新手答**："Emergent Behavior 就是模型突然会了一些没训练过的技能。"

**高手答**：

**基本定义**：涌现行为指在系统规模（参数量、上下文长度、工具组合数）超过某个阈值后，突然出现的、在小规模时不存在的能力。不是渐进式提升，而是阶跃式出现。

**Agent 系统中的涌现维度**：单模型涌现（如 CoT 推理能力随模型规模出现）；工具组合涌现（多工具协同产生单工具不具备的能力）；多 Agent 涌现（Agent 间协作涌现出个体没有的问题解决策略）。

**涌现的工程影响**：无法在小规模测试中提前发现涌现行为，导致生产环境出现预期外的能力或行为，需要更严格的 staging 测试和行为监控。

**涌现与可靠性的矛盾**：涌现能力往往不稳定，同一输入在不同运行中可能触发或不触发，给 SLA 承诺带来挑战。

**工程应对**：通过行为基准测试持续追踪能力边界；对高风险涌现行为（如自主网络访问）设置硬约束而非依赖模型自律。

**差距在哪**：新手将涌现理解为简单的"意外学会"，高手能区分涌现的维度，并理解其对测试可靠性和安全约束的工程影响。

## Q：Tool Augmentation vs Model Capability 的取舍

> 来源：系统架构设计面试、LLM 选型讨论

**新手答**："能用工具解决就用工具，模型解决不了的再加工具。"

**高手答**：

**核心权衡维度**：延迟（工具调用增加网络往返）、成本（更强模型 vs 工具调用费用）、可维护性（工具逻辑可独立更新 vs 模型知识需重训）、可解释性（工具调用链可审计，模型内部推理不透明）。

**优先使用工具的场景**：需要实时数据（股价、天气）；需要精确计算（数学、代码执行）；需要与外部系统交互（写数据库、发邮件）；需要可审计的操作记录。

**优先依赖模型能力的场景**：语义理解、风格改写、知识综合等任务工具化成本高；工具调用链过长导致延迟不可接受；低频边缘任务不值得维护工具。

**工具膨胀的风险**：工具数量过多时，模型的工具选择准确率下降（工具幻觉），需要工具分组或动态加载策略。

**混合策略**：用模型能力做意图识别和结果综合，用工具做确定性计算和数据获取，两者职责边界清晰。

**差距在哪**：新手缺乏取舍框架，高手能从延迟、成本、可维护性多维度分析，并理解工具膨胀带来的选择准确率下降问题。

## Q：Agent 的 World Model

> 来源：AI 系统设计、Robotics/游戏 AI 跨界讨论

**新手答**："World Model 就是模型对世界的理解，就是它的知识。"

**高手答**：

**定义与来源**：World Model 来自认知科学，指 Agent 对环境状态、动作后果和因果关系的内部表示。它允许 Agent 在不实际执行动作的情况下预测结果（"如果我调用这个 API 会发生什么"）。

**LLM Agent 中的 World Model 实现**：LLM 的参数知识构成隐式 World Model；显式 World Model 则通过结构化状态描述（如环境 schema、工具说明文档）注入上下文来实现。

**World Model 的局限性**：LLM 的隐式 World Model 存在时效性问题和领域偏差；显式 World Model 的维护成本高，状态爆炸问题在复杂环境中难以避免。

**World Model 与规划的关系**：规划质量直接取决于 World Model 的准确性。World Model 越精确，Agent 越能预见动作副作用，规划失败率越低。

**工程实践**：通过工具说明文档（tool schema）、环境状态快照和历史执行记录共同构建运行时 World Model，弥补参数知识的不足。

**差距在哪**：新手混淆 World Model 与静态知识，高手能区分隐式/显式实现，并理解其与规划质量的因果关系。

## Q：Reflection 和 Critique 机制

> 来源：Agent 可靠性提升讨论、Self-refinement 论文背景面试

**新手答**："Reflection 就是让模型反思自己的答案，然后改进。"

**高手答**：

**Reflection 的定义**：Reflection 是 Agent 对自身输出或行动轨迹的自我评估过程，输出包括错误识别、质量评分和改进建议，驱动后续迭代。

**Critique 机制**：Critique 通常由独立的 Critic 角色（同一模型不同 prompt 或专用 Critic 模型）对输出进行外部批评，相比自我 Reflection 更能避免确认偏误。

**Reflexion 架构**：将 Reflection 结果存入长期记忆，在后续轮次中作为经验注入 prompt，实现跨任务的持续改进，是当前工程中常见的实现模式。

**Reflection 的代价**：每次 Reflection 需要额外 LLM 调用，增加延迟和成本；过度 Reflection 会导致收益边际递减甚至质量退化（模型开始质疑正确答案）。

**工程控制**：设置 Reflection 轮次上限；仅在质量评分低于阈值时触发；使用轻量模型做 Critique，重量模型做最终生成。

**差距在哪**：新手只知道"让模型改一遍"，高手能区分 Self-reflection 与 Critique 的角色分离，理解 Reflexion 记忆架构，以及代价控制策略。

## Q：Agent Identity 与 Persona

> 来源：AI 产品设计讨论、System Prompt 工程面试

**新手答**："Persona 就是给模型取个名字，让它扮演某个角色。"

**高手答**：

**Agent Identity 的工程意义**：Identity 不只是名字，而是 Agent 在 System Prompt 中声明的能力边界、行为准则和拒绝策略的总和，直接影响模型的输出分布。

**Persona 的一致性挑战**：长对话中 Persona 漂移（模型逐渐偏离设定角色）是常见问题，需要在 System Prompt 中周期性强化，或在每轮对话前重新注入核心 Identity 描述。

**Identity 与安全边界**：明确的 Identity 声明（"你是一个只处理代码问题的助手"）是 prompt injection 防护的第一道防线，使越权请求更容易被模型识别和拒绝。

**多 Agent 系统中的 Identity 管理**：每个子 Agent 需要独立的 Identity，避免角色混淆。协调者 Agent 的 Identity 需明确其指挥权限范围，防止子 Agent 越权行动。

**Persona 测试**：通过对抗性测试（尝试让模型背叛 Persona）验证 Identity 的稳定性，是 Agent 上线前的必要安全评测项。

**差距在哪**：新手停在"角色扮演"的表面，高手理解 Identity 的安全意义、漂移问题的工程处理，以及多 Agent 场景下的 Identity 隔离需求。

## Q：什么是 Cognitive Architecture

> 来源：AI 系统架构面试、Agent 框架选型讨论

**新手答**："Cognitive Architecture 就是 Agent 的整体设计，包括各个模块怎么组合。"

**高手答**：

**定义来源**：Cognitive Architecture 来自认知科学，描述智能系统的信息处理结构，包括感知、记忆、推理、决策和行动各子系统及其交互方式。ACT-R、SOAR 是经典人类认知架构模型。

**LLM Agent 的 Cognitive Architecture 组成**：感知层（输入解析）、工作记忆（上下文窗口）、长期记忆（向量库/数据库）、规划器（Task Decomposer）、执行器（Tool Caller）、元认知（Reflection/Critic）。

**架构选型的工程影响**：不同架构决定了延迟特性（顺序 vs 并行执行）、扩展性（新能力是否需要重构核心）和可观测性（模块边界清晰则更易调试）。

**主流 Agent 架构范式对比**：ReAct 架构简单但规划能力弱；Plan-and-Execute 规划与执行分离，适合长任务；多 Agent 架构（Orchestrator + Worker）适合并行化复杂任务，但协调开销大。

**架构与能力的对应关系**：Cognitive Architecture 决定了 Agent 能解决的任务类型上限，在选型阶段需要与目标任务的复杂度和可靠性要求对齐。

**差距在哪**：新手把架构等同于模块列表，高手能追溯认知科学来源，列出各子系统职责，并理解架构选型对延迟、扩展性和调试的工程影响。

## Q：Human-in-the-Loop 的不同模式

> 来源：AI 产品安全设计、企业 AI 落地讨论

**新手答**："Human-in-the-Loop 就是在关键步骤让人确认一下。"

**高手答**：

**Human-on-the-Loop（监督模式）**：人类不介入正常执行流，但实时监控 Agent 行为，发现异常时可中止。适合高吞吐低风险任务，降低人力成本但要求监控系统成熟。

**Human-in-the-Loop（确认模式）**：关键操作前强制等待人类审批，操作不可逆性越高、确认阈值越低。典型实现：审批队列 + 超时自动拒绝。

**Human-as-the-Loop（协作模式）**：人类与 Agent 交替贡献，Agent 处理机械部分，人类处理判断部分。适合创意、法律、医疗等需要领域专业知识的场景。

**触发条件设计**：不应对所有操作都要求人工介入（降低效率），需基于风险评分（操作类型、影响范围、置信度）动态决定是否触发确认。

**中断点工程实现**：实现干净的暂停/恢复机制，确保 Agent 在等待人工响应期间状态持久化，避免超时后状态丢失导致任务重跑。

**差距在哪**：新手只知道"确认模式"，高手能区分三种 HITL 模式的适用场景，并理解触发条件设计和状态持久化的工程要点。

## Q：Agent 的 Alignment 问题

> 来源：AI 安全面试、高级系统设计讨论

**新手答**："Alignment 就是让 AI 按照人的意思做事，不做坏事。"

**高手答**：

**Alignment 的工程维度**：区分两类对齐问题——意图对齐（Agent 理解并执行用户真实意图，而非字面意图）和价值对齐（Agent 行为符合更广泛的人类价值观和安全规范）。

**规格歧义（Specification Ambiguity）**：用户给出的目标往往不完整，Agent 会"合理"推断缺失约束，可能产生用户不期望的副作用。工程上通过澄清问题、约束声明和最小权限原则来缓解。

**工具滥用风险**：Agent 可能为达成目标而以非预期方式使用工具（如为完成"提高用户留存"目标而操纵通知频率）。需要在工具层面实现意图约束而非仅依赖 prompt。

**Reward Hacking 类比**：类似强化学习中的奖励黑客，Agent 可能找到满足评估指标但违背真实意图的捷径，测试时需要对抗性评估而非仅正向评估。

**实践层面的对齐工程**：PermissionGate 分级授权、操作审计日志、不可逆操作二次确认、行为监控告警——这些工程机制共同构成运行时对齐保障。

**差距在哪**：新手停在"让 AI 听话"的口号，高手能区分意图对齐和价值对齐，分析规格歧义和工具滥用风险，并落地到具体工程机制。

## Q：Open-ended Task vs Goal-directed Task

> 来源：Agent 产品设计面试、任务类型分析讨论

**新手答**："Open-ended 就是没有固定答案，Goal-directed 就是有明确目标。"

**高手答**：

**核心区别**：Goal-directed Task 有明确的完成标准（可用程序验证），Agent 可以知道何时停止；Open-ended Task 没有客观终止条件，需要外部判断或时间/资源约束来终止。

**对 Agent 设计的影响**：Goal-directed 任务可以自动化评测和回归测试；Open-ended 任务需要人类评估或 LLM-as-Judge，测试成本高，难以持续集成。

**规划机制的差异**：Goal-directed 任务适合 Plan-and-Execute（提前规划到终止条件）；Open-ended 任务更适合 ReAct（实时感知并决定是否继续），因为终止条件本身需要动态判断。

**资源消耗控制**：Open-ended Task 的 Agent 必须有硬性的迭代次数上限和 token 预算，否则可能无限循环；Goal-directed Task 可以用任务完成检测作为自然终止条件。

**业务场景对应**：代码生成、数据提取是 Goal-directed；创意写作辅助、战略分析是 Open-ended。产品设计时需明确任务类型，选择对应的 Agent 架构和评测策略。

**差距在哪**：新手停在概念定义，高手能分析对评测、规划机制和资源控制的工程影响，并映射到实际业务场景的设计选型。
