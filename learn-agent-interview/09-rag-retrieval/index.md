# RAG 与检索增强

## Q：Naive RAG 的基本流程是什么？有哪些核心缺陷？

> 来源：RAG 系统工程师面试

**新手答**："切 chunk，存向量，搜索，生成，就是 RAG 了。"

**高手答**：

1. **基本流程**：文档分块（Chunking）→ Embedding 向量化 → 存入向量 DB → 用户查询时 Embed query → 向量相似度检索 top-K chunks → 把 chunks 注入 Prompt 生成回答。

2. **缺陷一：检索质量不稳定**：简单的向量相似度检索无法处理表述不同但语义相同的查询，且无法处理需要多文档联合推理的问题。

3. **缺陷二：chunk 上下文丢失**：chunk 离开原文档后丧失上下文，独立的 chunk 可能无法表达完整语义（"这个问题的解决方案是..."，哪个问题？chunk 不知道）。

4. **缺陷三：生成不忠实**：LLM 可能不完全使用检索到的内容，混入训练数据中的旧知识，导致输出和检索内容不一致（Faithfulness 低）。

5. **缺陷四：没有迭代机制**：一次检索决定最终答案，没有反馈机制。检索结果不佳时，LLM 只能用有限信息生成答案，而不能主动补充检索。

**差距在哪**：新手认为 Naive RAG 的流程就是 RAG 的全部，高手知道这只是起点，Real-world RAG 需要解决上下文丢失、检索质量、生成忠实度三个核心问题，进阶到 Advanced RAG。

## Q：RAG、Fine-tuning、Long Context 三者如何选型？

> 来源：AI 技术选型架构面试

**新手答**："一般用 RAG，精度要求高用 Fine-tuning。"

**高手答**：

1. **RAG 适合**：知识动态变化（需要实时更新）、知识量大（超过 context 上限）、需要可溯源（能引用来源）、不需要改变模型行为风格。RAG 是"给模型加外部知识"。

2. **Fine-tuning 适合**：需要改变模型的输出风格/格式/语气（模型本身的"说话方式"）、高频使用的领域知识（让模型内化，减少每次 RAG 检索的成本）、需要特定的推理模式（如特定的代码风格）。Fine-tuning 是"改变模型本身"。

3. **Long Context 适合**：文档量少（< 几十万 tokens）、需要跨文档推理、可以接受高 token 成本。Long Context 是"把所有信息直接塞进去"。

4. **组合策略**：三者不是互斥的——可以用 Fine-tuning 让模型学会如何更好地利用 RAG 检索内容；或者 Long Context 处理少量关键文档 + RAG 处理大量历史文档。

5. **决策树**：知识会变化或超大量 → RAG；需要改变模型行为/格式 → Fine-tuning；小量文档 + 需要全局理解 → Long Context；都不确定 → 先用 RAG（成本最低、迭代最快）。

**差距在哪**：新手用"什么更好"来选型，高手用"这个任务的核心需求是什么"来选型——知识性问题 vs 行为性问题 vs 全局理解问题，三类问题对应不同的解决方案。

## Q：文档分块策略有哪些？如何选择？

> 来源：RAG 工程实践面试

**新手答**："按 1000 个字符切一下，加点 overlap 就行了。"

**高手答**：

1. **Fixed-size Chunking**：按固定字符数（500-1000）切分，可选 overlap（100-200 字符）。最简单、最快，适合结构均匀的文档（如 Wiki 文章）。缺点：不考虑语义完整性，可能切断句子和段落。

2. **Recursive Character Splitting**：按层级分隔符（`\n\n` → `\n` → `. ` → ` `）优先切分，尽量在自然边界切割。是最常用的生产方案，LangChain 的默认实现。

3. **Semantic Chunking**：计算相邻句子的 embedding，当相似度跌破阈值时切分，保证每个 chunk 内容语义连贯。质量最高，但计算代价大（每个句子都要 embed），适合高价值文档。

4. **Token-based Chunking**：直接按 token 数切分（而不是字符数），精确控制 chunk 的 token 消耗。适合对 token 预算有精确要求的场景。

5. **特殊文档类型**：代码 → 按函数/类切分；Markdown → 按标题层级切分；PDF → 按页面或段落切分；表格 → 保持行结构完整（不在行中间切）。通用切分策略对结构化文档效果差。

**差距在哪**：新手用固定切分一招走天下，高手根据文档类型和质量要求选策略，知道 Semantic Chunking 质量最好但成本最高，Recursive 是生产中平衡点，特殊类型需要特殊处理。

## Q：如何选择 Embedding 模型？

> 来源：RAG 系统设计面试

**新手答**："用 OpenAI 的 text-embedding-3-small，效果好。"

**高手答**：

1. **评测维度**：MTEB 排行榜（多语言 embedding 基准）是最重要的参考，但要看与你的任务相关的子任务（Retrieval、STS 等），而不是整体排名。

2. **成本与性能**：text-embedding-3-small（1536 dims，便宜）vs text-embedding-3-large（3072 dims，贵 5x，质量提升有限）vs 本地模型（BGE-M3、E5-large，免费，延迟低，隐私保护好）。

3. **多语言场景**：处理中文或多语言文档，优先选多语言 Embedding（如 BGE-M3、multilingual-e5）而不是英文模型，跨语言检索质量差异巨大。

4. **领域专用**：通用 Embedding 模型在专业领域（医疗、法律、代码）的表现可能不如领域微调模型。如果知识库非常领域专用，评估 Fine-tuning embedding 模型是否值得。

5. **实际测量**：在你自己的数据集上测试（构建 100-200 条 query-ground_truth 对），比 MTEB 排名更可靠。模型在通用 Benchmark 好不代表在你的具体数据集上好。

**差距在哪**：新手用"大厂出的就好"逻辑选模型，高手知道 Embedding 模型选择需要在你的实际数据上评测，多语言、领域专用、成本都是必须考虑的维度。

## Q：向量数据库如何选型？

> 来源：AI 基础设施工程师面试

**新手答**："用 Pinecone，全托管方便。"

**高手答**：

1. **pgvector（PostgreSQL 扩展）**：在已有 PostgreSQL 的项目中零额外基础设施，支持 IVFFLAT 和 HNSW 索引，SQL 查询支持复杂 Filter，适合中小规模（< 1000 万向量）且已用 PostgreSQL 的团队。

2. **Chroma**：轻量 Python 优先，本地开发友好，零配置，适合原型和研究。生产规模有限，持久化和分布式能力弱。

3. **Pinecone**：全托管，零运维，适合想快速上线、不想管理基础设施的团队。缺点是数据在第三方、成本随规模增长快、数据迁移难。

4. **Weaviate / Qdrant**：自托管，功能完整（Metadata Filter、多向量、BM25混合），适合需要完全控制权、大规模（千万向量以上）的场景。运维成本相对高。

5. **选型决策**：< 100 万向量 + 已有 PostgreSQL → pgvector；快速原型 → Chroma；不想运维 → Pinecone；大规模 + 自托管 → Weaviate/Qdrant；MVP 阶段 → SQLite + JSON 存向量（用余弦相似度手算），不要过早引入专业向量 DB。

**差距在哪**：新手选向量 DB 看知名度，高手根据规模、运维能力、是否需要混合检索、成本限制来选，在 MVP 阶段往往 pgvector 甚至 SQLite 就够了。

## Q：HNSW 和 IVF 索引算法有什么区别？如何选择？

> 来源：向量数据库工程师面试

**新手答**："不太了解，用默认设置就行了。"

**高手答**：

1. **HNSW（Hierarchical Navigable Small World）**：图索引，查询时间 O(log n)，查询速度快，精度高（ANN 精度接近暴力检索），内存占用大（每个向量需要额外存储图的连接）。适合查询性能优先、内存充足的场景。

2. **IVF（Inverted File Index）**：聚类索引，先量化后聚类，内存占用小，支持大规模向量（数十亿）。缺点是查询时需要设置 `nprobe`（搜索多少个 cluster），精度和速度的权衡需要调参。

3. **构建时间对比**：HNSW 构建时间较长（建图），但查询时无需额外参数；IVF 构建快，但查询时 `nprobe` 直接影响精度，生产中需要谨慎调参。

4. **混合方案**：IVFPQ（IVF + Product Quantization）把向量压缩后存储，极大减少内存，代价是精度损失。Faiss 常用 IVFPQ 处理数十亿向量。

5. **实践建议**：数百万向量以下 → HNSW（Qdrant/pgvector 默认）；数十亿向量 → IVF 或 IVFPQ；不知道选哪个 → 用 HNSW，等规模增长后再评估切换。

**差距在哪**：新手用默认配置不关心索引算法，高手知道索引算法直接影响查询延迟、精度和内存，在不同规模下有明确的选型建议。

## Q：稠密检索（Dense Retrieval）和稀疏检索（BM25）有什么互补性？

> 来源：混合检索系统设计面试

**新手答**："稠密检索是向量，稀疏检索是关键词，向量更先进。"

**高手答**：

1. **各自的优劣**：稠密检索（Dense）擅长语义理解，处理同义词替换和跨语言查询（"automobile" 可以找到 "car" 的文档）；稀疏检索（BM25）擅长精确关键词匹配，对专有名词（人名、产品型号、代码变量名）召回准确（dense embedding 无法有效区分相似的专有名词）。

2. **实际互补场景**：用户搜索"GPT-4o"，dense embedding 可能把 GPT-3.5 的文档也检索出来（向量空间中接近）；BM25 只会返回精确包含"GPT-4o"的文档。专有名词多的领域，BM25 是必要的补充。

3. **Hybrid Search（混合检索）实现**：并行执行 Dense 和 BM25 检索，各取 top-K，用 RRF（Reciprocal Rank Fusion）或加权求和融合两路结果，去重后返回最终 top-K。

4. **Reciprocal Rank Fusion（RRF）**：`score(d) = Σ 1/(k + rank(d))`，k 通常为 60。不需要校准两路分数的量纲（Dense 和 BM25 的分数范围完全不同，直接加权求和需要归一化），是最简单且效果好的融合方法。

5. **性能代价**：并行执行两路检索，BM25 通常基于倒排索引，延迟低（< 10ms），不会显著增加总查询延迟。

**差距在哪**：新手认为向量检索"更先进"所以不需要 BM25，高手知道两者解决的是不同问题，在含有专有名词和精确匹配需求的场景中，混合检索是标配而非可选项。

## Q：Reranker 的工作原理是什么？何时值得用？

> 来源：RAG 质量优化工程面试

**新手答**："Reranker 就是对检索结果再排一次序。"

**高手答**：

1. **Bi-encoder vs Cross-encoder**：初步检索用 Bi-encoder（query 和 document 分别编码成向量，dot product 相似度，速度快但精度有限）；Reranker 是 Cross-encoder（query 和 document 一起输入模型，模型能看到两者交互，精度高但速度慢，不能预计算）。

2. **为什么 Cross-encoder 更准**：Bi-encoder 的相似度是"两个向量的距离"，缺乏细粒度的语义交互；Cross-encoder 在 Attention 层让 query 和 document 的 token 互相交互，能捕捉到更细微的相关性差异。

3. **两阶段 Retrieve-Rerank**：向量检索（Bi-encoder）召回 top-50 候选（保证高 recall），Reranker（Cross-encoder）精排 top-3（保证高 precision）。两阶段结合，兼顾速度和精度。

4. **何时值得用**：结果精度要求高（如法律文档搜索、医疗问答）、初步检索召回了很多噪音（precision 低）、可以接受额外 200-500ms 的延迟。如果初步检索已经足够精准，或者延迟 SLA 严格，Reranker 的价值有限。

5. **常用 Reranker**：ms-marco-MiniLM 系列（轻量快速）；BGE Reranker（中文友好）；Cohere Rerank API（托管服务，无需本地部署）。

**差距在哪**：新手把 Reranker 当 RAG 必备组件无脑加，高手知道 Reranker 有延迟代价，要在初步检索的精度、质量要求、延迟 SLA 之间权衡是否值得引入。

## Q：RAG 的端到端评测指标体系是什么？

> 来源：RAG 工程质量保障面试

**新手答**："看答案准不准确就行了。"

**高手答**：

1. **检索层指标**：Recall@K（相关文档是否在前 K 个结果中）、Precision@K（前 K 个结果中有多少是相关的）、MRR（Mean Reciprocal Rank，相关文档排名越靠前，MRR 越高）。这些指标评估检索本身，与 LLM 无关。

2. **生成层指标（RAGAS 框架）**：Faithfulness（生成答案中的每个声明是否能在检索内容中找到依据）、Answer Relevance（答案是否真正回答了问题）、Context Precision（检索到的内容中有多少是相关的）、Context Recall（用于回答问题的相关信息是否都被检索到）。

3. **端到端指标**：Answer Correctness（生成答案与 ground truth 的一致性，需要有 ground truth），可用 BERTScore 或 LLM-as-judge 评分。

4. **为什么要分层**：Recall@K 好但 Faithfulness 低 → 检索正确但 LLM 没用检索内容（Context Utilization 问题）；Recall@K 差但 Faithfulness 高 → LLM 用了它找到的内容（但找的不对），需要优化检索。分层指标帮助精准定位问题。

5. **RAGAS 的使用**：RAGAS 框架提供自动计算这些指标的工具，内部使用 LLM-as-judge 评估，可以在无 ground truth 的情况下评估 Faithfulness 和 Context Relevance，适合快速迭代阶段。

**差距在哪**：新手用"答案对不对"这个单一指标，高手用分层指标体系（检索层+生成层+端到端），能精准定位是检索问题、生成问题还是两者都有。

## Q：HyDE（假设性文档嵌入）的原理是什么？

> 来源：Advanced RAG 工程面试

**新手答**："让 LLM 写一个假设答案，然后拿去检索。"

**高手答**：

1. **问题背景**：用户的查询（通常是一个简短的问题）和知识库中的文档（通常是一段描述性文本）在向量空间中可能距离较远，即使语义相关，因为"问题形式"和"文档形式"的向量分布不同。

2. **HyDE 的思路**：让 LLM 根据查询生成一段"如果有一篇完美回答这个问题的文章，它会是什么样的"，然后用这段假设文章（而不是原始查询）做 Embedding 和检索。假设文章的形式和知识库文档更接近，检索相关性更高。

3. **核心优势**：把"查询 → 相关文档"的检索问题，转化为"文档 → 相关文档"的检索，后者的精度更高（同类文本的 embedding 距离更近）。

4. **风险**：LLM 生成的假设文章可能包含幻觉（即使是"假设性"的，LLM 可能编造内容），如果假设文章质量差，检索结果也会偏差。需要控制假设文章的生成质量（明确说明只基于已知信息生成）。

5. **工程代价**：每次检索前多一次 LLM 调用（生成假设文章），增加延迟和成本。通常在高质量要求且 recall 是关键的场景才值得用。

**差距在哪**：新手认为"用原始查询检索"是唯一方式，高手知道查询和文档的 embedding 分布差异是 RAG 检索的深层问题，HyDE 是解决"查询-文档语义 gap"的工程方案。

## Q：Multi-hop Retrieval（多跳检索）是什么？如何实现？

> 来源：复杂 RAG 系统工程面试

**新手答**："检索多次就是多跳检索？"

**高手答**：

1. **什么是多跳问题**："Google 的 CEO 出生在哪个城市？"需要先找到"Google CEO 是谁"，再找"这个人出生在哪里"。答案需要通过多个文档"跳跃"得出，单次检索无法完成。

2. **Iterative Retrieval**：第一次检索后，用 LLM 分析"还需要哪些信息"，再发起第二次检索（以第一次检索结果为基础构造新查询），循环直到 LLM 判断信息充足。简单但可能需要多次 LLM 调用。

3. **Query Decomposition**：把复杂问题分解为若干子问题，并行检索每个子问题，再用 LLM 综合所有子问题的检索结果得出最终答案。适合已知问题可以分解的场景。

4. **Knowledge Graph 辅助**：把文档中的实体和关系提取为知识图谱，多跳检索在图上进行（跟随关系边），比纯 embedding 检索更适合需要推理链的多跳问题。

5. **性能代价**：每一跳都是一次检索 + 可能的 LLM 调用，N 跳检索的延迟是单跳的 N 倍以上。生产中通常限制最大跳数（如最多 3 跳），超过则返回当前最佳答案。

**差距在哪**：新手认为 RAG 就是"一次检索一次生成"，高手知道复杂问题需要多跳推理，并知道 Iterative Retrieval vs Query Decomposition vs KG 三种实现方案的工程取舍。

## Q：Query Expansion 和 Query Decomposition 有什么区别？

> 来源：Advanced RAG 工程面试

**新手答**："都是改写查询，差不多吧？"

**高手答**：

1. **Query Expansion（查询扩展）**：在原始查询的基础上增加更多相关词汇或表述，扩大检索范围，提高 Recall。例如把"机器学习优化"扩展为"机器学习优化 学习率 梯度下降 Adam SGD"，多个词汇并行检索后合并。

2. **Query Decomposition（查询分解）**：把一个复杂问题拆分成多个独立的子问题，每个子问题单独检索，最后综合所有结果回答原始问题。解决的是"一个问题涉及多个方面"的问题，而不是"词汇不够丰富"的问题。

3. **适用场景**：Expansion → 用户查询太简短、词汇单一、需要更高 Recall；Decomposition → 问题本身涉及多个子主题、需要多文档综合回答。

4. **Multi-query（多查询变体）**：用 LLM 从不同角度重写同一查询（生成 3-5 个不同表述），各自检索，合并去重后排序。是 Expansion 的 LLM 版本，比简单关键词扩展更语义准确。

5. **工程成本对比**：Expansion 主要增加检索成本（并行检索多个 query）；Decomposition 还增加 LLM 调用成本（分解和综合都需要 LLM）。复杂问题才值得用 Decomposition。

**差距在哪**：新手把两者混为一谈（都是"改写查询"），高手知道它们解决不同问题——Expansion 扩大检索范围（解决 Recall 问题），Decomposition 处理多方面复杂问题（解决覆盖完整性问题）。

## Q：Contextual Retrieval 和普通 RAG 的核心区别是什么？

> 来源：Anthropic 技术创新面试

**新手答**："就是更好的 RAG，加了一些上下文。"

**高手答**：

1. **核心问题**：普通 RAG 的 chunk 离开原文档后丧失上下文。"第二个解决方案是使用缓存" — 缓存什么问题的解决方案？chunk 本身不知道。这导致 embedding 无法准确表达 chunk 的真实语义。

2. **Contextual Retrieval 的做法**：在存储每个 chunk 之前，用 LLM 为 chunk 生成简短的上下文描述（"本段是来自 '高并发设计' 文档的讨论缓存策略的内容，主要解决...的问题"），把上下文描述 + chunk 内容合并后再生成 embedding。

3. **效果提升**：Anthropic 的测试显示 Contextual Retrieval 将检索失败率降低约 49%，结合 BM25 混合检索后降低 67%。这是 RAG 领域近年来最重要的工程优化之一。

4. **工程实现**：在 build-kb 阶段，对每个 chunk 调用一次轻量 LLM（如 Claude Haiku）生成 150 tokens 以内的上下文描述，附加到 chunk 前面，再整体 embedding。增加了 build 阶段的成本，但检索时无额外代价。

5. **与 HyDE 的定位差异**：Contextual Retrieval 在索引时改善 chunk 的表示（离线）；HyDE 在查询时改善 query 的表示（在线）。两者解决问题的阶段不同，可以组合使用。

**差距在哪**：新手知道 RAG 但不了解 chunk 上下文丢失是核心缺陷，高手知道 Contextual Retrieval 是解决这个根本问题的方案，并知道它的实现方式、效果提升数据和工程代价。

## Q：如何为 RAG 系统构建 Golden Set 评测集？

> 来源：RAG 质量保障工程面试

**新手答**："把一些问答对收集起来就行了。"

**高手答**：

1. **Golden Set 的三元组**：（query, relevant_documents, expected_answer）。不只是问答对，还要标注"哪些文档是回答这个问题的依据"，这样才能评估 Recall（文档是否被检索到）而不只是答案质量。

2. **覆盖维度**：高频简单查询（主体功能测试）+ 边界 case（长问题、模糊问题、需要多跳推理的问题）+ 已知故障 case（历史中发现的检索失败 case）。覆盖完整才能有效防止回归。

3. **生成策略**：人工构建（质量最高，成本最高）vs 从生产数据挖掘（效率高，但需要人工审核质量）vs LLM 生成（利用 LLM 从文档中自动生成 QA 对，再人工审核）。三种方式可以混合使用。

4. **规模建议**：100-500 条是 MVP 阶段的合理范围。质量比数量重要——50 条精心设计的 case 比 500 条随意收集的 case 更有价值。

5. **动态维护**：每次在生产中发现新的检索失败模式，加入 Golden Set；每 3 个月审查并淘汰过时 case；用 CI/CD 自动运行 Golden Set 测试，任何改动都要通过测试才能合并。

**差距在哪**：新手构建 Golden Set 只关注问答质量（答案对不对），高手知道 RAG 的 Golden Set 必须包含文档层面的 ground truth，才能分别评估检索和生成两个维度的质量。

## Q：RAG 系统的缓存策略有哪些？

> 来源：RAG 性能优化工程面试

**新手答**："把常见查询的结果缓存起来就行了。"

**高手答**：

1. **Query-level 缓存**：对相同（或语义相近）的查询，缓存其检索结果和生成答案。Key = query embedding 的 hash 或精确 query 字符串，Value = (检索到的 chunk IDs, 最终答案)。命中缓存直接返回，无需调用 LLM 和向量 DB。

2. **Embedding 缓存**：对文档 chunk 的 embedding 结果缓存（这是 build 阶段的产出，存入 DB），避免重复调用 Embedding API。对用户 query 的 embedding 在 session 内缓存（同一 session 内相同查询不重新 embed）。

3. **语义相似度缓存**：不只是精确匹配，还缓存语义相似的查询——当新 query 的 embedding 与已缓存 query 的 embedding 距离 < 阈值时，直接返回缓存结果。需要一个"缓存向量索引"来做相似度搜索。

4. **Prompt Cache（LLM 侧）**：利用 Anthropic 的 Prompt Caching 功能，对 RAG 检索结果中固定的文档内容（在 context 中位置固定的部分）做 prefix caching，后续相同前缀的请求 input token 成本减少 90%。

5. **缓存失效策略**：知识库更新后，相关 chunk 的缓存要失效；缓存设置 TTL（如 24 小时），防止过期数据持续返回；对于时效敏感的查询（如"今天的..."），跳过缓存直接查询。

**差距在哪**：新手只想到"缓存答案"，高手在 Embedding 缓存、Query 缓存、语义缓存、LLM Prompt Cache 四个层次做缓存，每层解决不同的性能瓶颈。

## Q：Self-RAG 的工作机制是什么？

> 来源：前沿 RAG 研究工程师面试

**新手答**："自己检索自己？不太了解。"

**高手答**：

1. **传统 RAG 的问题**：每次都检索，即使问题不需要外部知识（如简单问候）也浪费检索资源；检索到内容后 LLM 不一定使用，造成 Context Faithfulness 低。

2. **Self-RAG 的思路**：通过特殊的 Token（`<Retrieve>`, `<IsRel>`, `<IsSup>`, `<IsUse>`）让 LLM 自己决定：①是否需要检索（Retrieve = Yes/No）；②检索到的文档是否相关（IsRel = relevant/irrelevant）；③生成内容是否由检索内容支撑（IsSup = supported/partially/not）；④生成内容是否有用（IsUse = 1-5）。

3. **核心优势**：按需检索（减少不必要的检索）；对检索结果做质量评估；生成时持续检验自己的输出是否 grounded，比 Naive RAG 的 Faithfulness 显著更高。

4. **工程实现代价**：Self-RAG 需要一个专门 fine-tuned 的模型（原始 Llama 微调的）来输出这些特殊 Token，通用模型（GPT-4）无法直接使用 Self-RAG 机制。如果不想 fine-tuning，可以用 Prompt 模拟类似逻辑，但效果有折扣。

5. **实用价值**：Self-RAG 的思想（让 LLM 主动评估检索结果的相关性）可以用 Prompt 工程实现部分功能——在生成答案时，要求 LLM 标注使用了哪些检索内容，不使用的检索内容可以丢弃，提高 Faithfulness。

**差距在哪**：新手不了解 Self-RAG，高手知道它解决的核心问题（按需检索 + 生成忠实度），了解其 fine-tuning 实现代价，并能把它的思想用更实用的方式（Prompt 工程）部分实现。

## Q：大规模知识库（百万文档）的检索性能优化有哪些方向？

> 来源：AI 平台性能工程师面试

**新手答**："用更好的硬件就行了。"

**高手答**：

1. **ANN 索引（近似最近邻）**：百万向量的精确 KNN 搜索是 O(n)，用 HNSW 或 IVF 索引将查询时间降到 O(log n) 或 O(√n)，代价是近似搜索（recall = 95-99%，通常可接受）。

2. **量化压缩**：用 PQ（Product Quantization）或 SQ（Scalar Quantization）把 float32 向量（1536 dims = 6KB）压缩为更小的表示（如 INT8 = 1.5KB），减少内存占用 4x，同时加速向量比较。代价是精度损失约 1-3%。

3. **分片（Sharding）**：把向量 DB 按 document_category 或 time_range 分片，查询时只检索相关分片，减少搜索空间。类似数据库分区，要求查询时有 filter 维度。

4. **二阶段检索**：先用轻量 BM25 或 TF-IDF 快速过滤（从 100 万 → 10000 候选），再在候选集上做 dense 向量检索（10000 → 50），大幅减少精确 KNN 搜索的范围。

5. **预热 + 缓存**：热门查询的检索结果缓存（语义缓存），高频查询直接命中缓存，不访问向量 DB。同时对向量索引做预热（启动时加载进内存），避免冷启动的磁盘 I/O 延迟。

**差距在哪**：新手把性能问题当资源问题（买更好的硬件），高手知道百万向量场景需要 ANN 索引、量化压缩、分片策略的组合，从算法层面而不仅仅是资源层面解决性能问题。

## Q：RAG 如何与 Agent Tool 结合使用？

> 来源：Agentic RAG 工程设计面试

**新手答**："把 RAG 检索作为 Agent 的一个工具就行了。"

**高手答**：

1. **基础模式（RAG as Tool）**：把向量检索封装为 `search_knowledge` Tool，Agent 决定何时调用检索，检索结果作为 Tool 结果注入 context。Agent 比 Naive RAG 灵活，可以多次检索、按需检索。

2. **Iterative RAG（迭代式检索）**：Agent 在多轮循环中根据前一步的检索结果决定是否需要再次检索、以什么查询检索。实现了"用检索结果驱动下一次检索"的迭代精化，适合复杂多跳问题。

3. **Parallel Retrieval（并行检索）**：对于分解后的多个子问题，Agent 并行发起多个检索 Tool 调用，减少总延迟（串行 N 次 → 并行 1 轮）。

4. **检索+执行的混合**：Agent 可以先检索相关知识，再用代码执行 Tool 验证知识是否正确（如检索到"函数 X 的用法"，再用 code_execute Tool 实际运行验证）。RAG 提供知识，Tool 提供验证。

5. **检索策略的动态选择**：不同问题适合不同检索策略（简单事实查询用 keyword，复杂推理用 semantic，最新信息用 web search），Agent 根据问题类型动态选择调用哪个检索 Tool。

**差距在哪**：新手把 RAG as Tool 当最终形态，高手知道这只是 Agentic RAG 的起点——Iterative、Parallel、检索+验证、动态策略，每种模式解决 Naive RAG 的不同局限性。

## Q：向量数据库如何处理数据更新？

> 来源：RAG 系统运维工程面试

**新手答**："修改了文档就重新生成 embedding 覆盖进去。"

**高手答**：

1. **删除-插入（最简单）**：删除旧 chunk 的向量，插入新 chunk 的向量。缺点是文档更新期间有短暂的"不可用窗口"（旧删了新还没插），对于大文档更新影响明显。

2. **基于 ID 的原子更新**：用固定的 chunk_id（基于文档 ID + chunk 序号生成），更新时直接 upsert（INSERT OR REPLACE），保证任何时刻向量 DB 中只有一个版本的 chunk，避免不可用窗口。

3. **增量更新 vs 全量重建**：文档小范围修改（< 20% 内容变化）→ 增量更新受影响的 chunks；大规模知识库内容重构 → 全量重建（新建索引 → 导入完成 → 原子切换到新索引，Zero-downtime migration）。

4. **版本化知识库**：在 `embeddings` 表中加入 `version` 字段，更新时写入新 version 而不是覆盖旧 version，旧 version 保留作为回滚依据。上线新 version 后确认无问题再清理旧 version。

5. **更新触发器**：知识库更新不应该是人工定期操作，而是通过 webhook 或监控（文件系统变更、API 数据源更新）自动触发。更新流程：检测变更 → 重新分块 → 重新 embedding → upsert 向量 DB → 更新 FTS 索引。

**差距在哪**：新手把知识库更新当"重新跑一遍 build-kb"的手动操作，高手设计了增量更新、原子切换、版本化、自动触发的更新流程，让知识库的时效性和稳定性都有保障。

## Q：RAG Pipeline 的关键监控指标是什么？

> 来源：AI 平台可观测性面试

**新手答**："看系统有没有报错就行了。"

**高手答**：

1. **检索层监控**：平均检索延迟（P50/P95/P99）、检索命中率（query 是否返回了结果，空结果比例）、检索相关性得分分布（低分结果比例高说明 embedding 质量或知识库质量有问题）。

2. **生成层监控**：Faithfulness 的周期性 LLM-as-judge 抽样评估（检测 LLM 是否在使用检索内容）、平均生成长度（突然变短或变长可能是模型行为异常）、拒绝回答率（知识库中找不到相关内容时的处理方式）。

3. **知识库质量监控**：每周跑一次 Golden Set 测试，追踪 Recall@3 / Precision@3 趋势；监控知识库的新增/更新频率（长时间未更新是时效性问题的信号）。

4. **成本监控**：每次 RAG Pipeline 调用的 Embedding API 成本和 LLM 生成成本，追踪每日/每周总成本，检测异常增长。

5. **用户反馈信号**：追踪"用户在 AI 回答后连续追问同一问题"的比例（可能表明回答未解决问题）、用户明确反馈"信息不准确"的比例，这些行为信号是 RAG 质量的最真实体现。

**差距在哪**：新手的监控只关注系统可用性（有没有报错），高手的监控覆盖检索质量（延迟、命中率、相关性）、生成质量（Faithfulness、周期评测）、成本（避免超支）、用户体验（行为信号），形成多维度的质量感知体系。

## Q：Chunk 大小对 RAG 性能有什么影响？如何调优？

> 来源：RAG 优化工程面试

**新手答**："Chunk 大一点信息更完整，小一点检索更精准。"

**高手答**：

1. **Chunk 大小影响 Embedding 精度**：Chunk 太大（>1000 tokens）→ embedding 是多个话题的平均，检索时对任何单一话题的匹配度都不高（Precision 低）；Chunk 太小（<50 tokens）→ 缺乏足够的语义上下文，embedding 不稳定（Recall 低）。

2. **经验范围**：通用文本 256-512 tokens 是经验起点，但不同场景差异大——技术文档可能 128-256 tokens 效果更好（结构化、密度高），长篇叙述文本可能 512-1024 tokens 更好（需要上下文连贯）。

3. **调优方法**：构建评测集 → 测试不同 chunk size（128/256/512/1024）→ 在评测集上对比 Recall@3 和 Precision@3 → 选择 Pareto 最优的 chunk size。**用数据说话，而不是经验拍板**。

4. **Parent-Child 双层策略**：不必在大 chunk（高 recall）和小 chunk（高 precision）之间强选一个。Small child chunk（128 tokens）用于精准 embedding 检索，检索命中后自动扩展到 Parent chunk（512 tokens）供 LLM 使用，兼得两者优点。

5. **与 Overlap 的关系**：Overlap（相邻 chunk 的重叠内容）增加可以弥补 Fixed-size 切分的边界问题，通常设为 chunk_size 的 10-20%。Overlap 太大会引入冗余，增加检索噪音。

**差距在哪**：新手凭直觉选 chunk size 并一次性确定，高手知道 chunk size 是需要用评测数据调优的超参数，并且 Parent-Child 策略可以从根本上解决大小 chunk 的权衡困境。

## Q：如何防止 RAG 系统产生幻觉？

> 来源：RAG 可靠性工程面试

**新手答**："RAG 有了参考文档就不会幻觉了。"

**高手答**：

1. **RAG 不能消除幻觉，只能降低**：LLM 在使用检索内容时，仍然可能混入训练数据中的相关内容，或者对检索内容做"创造性解读"，产生的答案虽然看起来合理但实际上超出了检索内容。

2. **强制引用约束**：在 Prompt 中要求"只使用提供的参考文档回答，不使用任何外部知识，如果文档中没有答案则明确说明"，减少 LLM 混入训练数据的概率。

3. **来源引用（Citation）**：要求 LLM 在回答中标注信息来源（[文档 1]、[文档 2]），后处理中验证每个引用是否真实存在于检索内容中，无法验证的声明标记为"未经文档支撑"。

4. **Faithfulness 后置验证**：用 LLM-as-judge 对生成答案做 Faithfulness 检验（把答案和检索内容一起输入，问"答案中的每个声明是否可以在文档中找到依据"），低 Faithfulness 的回答不直接返回，触发重新生成或降级处理。

5. **No-answer 处理**：当检索内容无法回答用户问题时，诚实地说"根据现有文档无法找到相关信息"比强行生成一个不靠谱的答案更好。需要在 Prompt 中明确授权 LLM 说"我不知道"。

**差距在哪**：新手认为有 RAG 就解决了幻觉，高手知道 RAG 降低幻觉但不消除，需要强制引用约束、来源验证、Faithfulness 检验三层保护，并且在知识不足时允许说"不知道"。

## Q：Metadata 过滤在 RAG 中如何实现？有什么价值？

> 来源：RAG 系统工程面试

**新手答**："向量检索已经能找到相关内容了，不需要再过滤。"

**高手答**：

1. **Metadata 过滤的价值**：向量检索是语义层面的相似度，无法处理结构化约束（"只看 2024 年的文档"、"只看技术分类下的内容"、"只看某个作者的文章"）。Metadata Filter 把这些结构化约束加入检索，精准缩小范围。

2. **典型 Metadata 字段**：document_type（文章/文档/FAQ）、category（技术/产品/法务）、date（发布日期）、language（语言）、version（版本号）、source（来源系统）。根据业务需求设计。

3. **Pre-filter 和 Post-filter**：Pre-filter 在向量检索之前用 Metadata 过滤候选集（缩小搜索空间，速度快但可能过度过滤）；Post-filter 在向量检索后对结果做过滤（可能导致最终结果数量不足）。通常用 Pre-filter 效率更高。

4. **混合查询（Hybrid Filter）**：`WHERE category = 'architecture' AND date > '2024-01-01' AND vector_similarity > 0.7`，把 Metadata Filter 和向量相似度阈值组合。pgvector 和 Qdrant 都支持这种查询。

5. **动态 Metadata（Query-time Extraction）**：在用户查询中自动提取 Metadata 信息（用 NER 或 LLM 提取"用户想看哪个时间段、哪个类别的内容"），动态构建 Filter，而不是要求用户手动指定 Filter 条件。

**差距在哪**：新手把 RAG 当纯语义问题，高手知道真实业务场景中结构化约束（时间、类别、来源）往往和语义相关性同等重要，Metadata Filter 是把 RAG 从"通用搜索"变成"精准业务搜索"的关键。

## Q：RAG 系统的端到端延迟如何优化？

> 来源：RAG 性能工程面试

**新手答**："用更快的向量数据库就好了。"

**高手答**：

1. **延迟拆分分析**：Query Embedding（~50ms）→ 向量检索（20-100ms）→ Reranker（100-500ms，如有）→ LLM 生成（1-10s）。总延迟由最慢的组件决定，先分析各组件的延迟占比，针对性优化。

2. **并行化**：Query Embedding 和 BM25 检索可以并行（不相互依赖）；多路 Query 检索可以并行。并行化把串行的多步骤变为并行，最大化利用等待时间。

3. **流式返回（Streaming）**：LLM 生成通常占总延迟的 70% 以上，用 SSE 流式返回让用户在 LLM 开始生成后立即看到部分输出（TTFT < 1s），即使总生成时间较长，用户体验显著改善。

4. **缓存查询 Embedding**：同一 session 内的相似查询不重新 embed（Session 级别 embedding 缓存）。高频查询的 embedding 结果长期缓存（Redis TTL 24h）。

5. **减少 Reranker 候选数**：如果 Reranker 是延迟瓶颈，把初步检索的候选从 top-50 减少到 top-20，在精度略微损失的代价下将 Reranker 延迟减少 60%。

**差距在哪**：新手认为"更快的组件 = 更快的系统"，高手用延迟拆分找到真正的瓶颈，然后用并行化、流式返回、缓存在不换组件的情况下大幅降低用户感知延迟。

## Q：RAG 中"Lost in the Middle"问题的工程解决方案

> 来源：RAG 生成质量优化面试

**新手答**："多检索几个文档，覆盖更全面。"

**高手答**：

1. **问题证据**：斯坦福研究表明，LLM 对放在 context 中间位置的相关信息的利用率显著低于头部和尾部。把最相关的 chunk 放中间，反而是最差的位置策略。

2. **Position 策略**：把最相关的 chunk 放在 context 的末尾（紧邻用户问题），次相关的放在最前面，相关性最低的放中间。这利用了 LLM 对头部（primary position）和尾部（recent position）的注意力偏向。

3. **减少 chunk 数量**：与其注入 10 个 chunk（大量内容在中间被忽略），不如精选最相关的 3-5 个（用 Reranker 严格筛选），减少中间区域的内容量。

4. **强制 LLM 引用**：要求 LLM 在回答中标注使用了哪些文档（[Doc 1], [Doc 2]），这强迫 LLM 主动处理每个 chunk 而不是自然地忽略中间的。没有被引用的 chunk 很可能没有被真正使用。

5. **Chunk 摘要化**：对 context 中间的 chunk 先做摘要（用 LLM 提炼为 50-100 tokens 的关键信息），减少中间区域的内容量，提高信息密度，让 LLM 更容易处理。

**差距在哪**：新手的直觉是"更多信息 = 更好答案"，高手知道 LLM 的注意力分布不均，需要在 chunk 数量控制、位置策略、强制引用上做工程优化，把最重要的信息放在 LLM 最容易"看到"的位置。
