# 多 Agent 协作

## Q：Orchestrator/Worker 模式的核心设计原则是什么？

> 来源：Agentic AI 系统架构面试

**新手答**："Orchestrator 负责协调，Worker 负责干活，分工明确就好了。"

**高手答**：

1. **Orchestrator 职责边界**：Orchestrator 只做任务分解、分配和结果聚合，不直接执行业务逻辑。如果 Orchestrator 开始"兼职"做具体任务，说明职责设计有问题。

2. **Worker 的幂等性**：同一个 Task 可能因超时被重新分配给另一个 Worker，Worker 必须保证幂等——重复执行相同任务不产生副作用，结果一致。

3. **任务描述的完整性**：Orchestrator 向 Worker 传递的任务必须是自包含的（Self-contained），包含 Worker 完成任务所需的全部上下文，而不能依赖 Worker 去"自行脑补"缺失信息。

4. **结果验证层**：Orchestrator 不能盲目信任 Worker 的输出。需要有验证步骤——对关键输出做格式校验、置信度检查，甚至让另一个 Worker 做 Peer Review。

5. **失败隔离**：单个 Worker 失败不应级联导致整个流程失败。Orchestrator 要有明确的错误处理策略——重试、降级、跳过，或终止整个任务。

**差距在哪**：新手把 Orchestrator/Worker 当简单的主从关系，高手关注的是接口契约、幂等性保证和故障隔离，这才是分布式 Agent 系统不出事的基础。

## Q：多 Agent 系统如何做任务分解与分配？

> 来源：大规模 LLM 应用工程师面试

**新手答**："让 Orchestrator LLM 自动把任务拆开分给各个 Agent 就行。"

**高手答**：

1. **两种分解模式**：静态分解（预定义任务图，代码写死流程）和动态分解（LLM 实时规划）。静态分解可控、可测、可预期，适合流程相对固定的业务；动态分解灵活但不可预测，适合开放域任务。

2. **任务粒度的权衡**：粒度太细 → Agent 间通信开销大，context 传递碎片化；粒度太粗 → 单个 Worker 承担过多职责，失败影响范围大。通常按"单次 LLM 调用可完成的最大工作量"作为粒度参考。

3. **依赖图管理**：任务间有依赖关系（B 依赖 A 的输出），必须用有向无环图（DAG）建模。独立任务并行执行，有依赖的任务串行。很多 Multi-agent 框架（如 LangGraph）本质是在做 DAG 调度。

4. **动态重分配**：当某个 Worker 失败或超时，Orchestrator 需要能把任务重新分配给另一个 Worker，或降级为串行处理。

5. **分配与专业化的结合**：任务类型和 Agent 专业化应该匹配——数学推理任务分给有 code interpreter 工具的 Agent，文档分析分给有 RAG 能力的 Agent，而不是随机分配。

**差距在哪**：新手把分配当简单调度，高手知道分解粒度、依赖建模和专业匹配才是 Multi-agent 系统性能的关键变量。

## Q：Agent 间通信应该用消息队列、共享状态还是直接调用？

> 来源：分布式 AI 系统设计面试

**新手答**："直接调用最简单，有什么说什么就行。"

**高手答**：

1. **直接调用（同步）**：最简单，适合 Orchestrator 直接控制少量 Worker 的场景。缺点是强耦合，调用方必须等待响应，单点失败影响整个链路。

2. **消息队列（异步）**：Agent 间通过队列传递消息，解耦生产者和消费者。适合长时间运行的任务、高并发场景、需要流量削峰的场景。代价是引入了队列基础设施，调试难度增加。

3. **共享状态（黑板模式）**：所有 Agent 读写共享的状态存储（如 Redis Hash 或 DB 表）。适合需要协商和汇聚的场景（多个 Agent 共同更新同一个分析报告）。缺点是并发写入的一致性问题，需要乐观锁或分布式锁。

4. **选型原则**：任务 < 5 秒、Agent 数 < 5 → 直接调用；任务 > 30 秒、Agent 数 > 10 → 消息队列；多 Agent 协同生成单一输出 → 共享状态。

5. **Context 传递的开销**：不管用哪种方式，传递大量 LLM context（数千 tokens）的代价都很高。要设计好 context 裁剪策略，只传递下游 Agent 真正需要的部分。

**差距在哪**：新手默认用同步直接调用，高手知道通信模式的选型直接影响系统的扩展性、可靠性和调试难度，需要根据任务特性做明确决策。

## Q：如何验证子 Agent 的输出可信？

> 来源：高可靠 AI Agent 系统面试

**新手答**："子 Agent 用的是同一个模型，应该可信的。"

**高手答**：

1. **Sub-agent 输出不可无条件信任**：即使是同一个模型，不同 Agent 的 context 不同，输出的可靠性也不同。特别是当 Sub-agent 自己的 Prompt 设计有问题时，它会产生听起来合理但实际错误的输出。

2. **格式验证**：最基本的验证——检查输出是否符合预期格式（JSON schema 验证、字段完整性检查）。格式错误通常意味着 Agent 理解任务有偏差。

3. **Critic Agent 模式**：对高风险任务，让另一个专门的 "Critic Agent" 对 Sub-agent 的输出进行独立审查，用不同的 Prompt 和视角检查逻辑一致性、事实准确性。相当于 Peer Review。

4. **置信度要求**：在 Sub-agent 的 Prompt 中要求其提供 confidence score，并在 context 中附上理由。低置信度的输出触发人工审核或更保守的降级处理，而不是直接使用。

5. **回调验证（Ground Truth 对比）**：对于有可验证答案的任务（代码执行结果、数学计算），直接执行验证而不是信任 Agent 的声明。

**差距在哪**：新手在 Multi-agent 系统中引入"信任传递"——A 信任 B，B 信任 C，链条上任何一环的错误都会被放大。高手把每个 Sub-agent 当外部系统，建立独立的验证边界。

## Q：Multi-Agent 系统应该如何调试和追踪？

> 来源：AI 平台可观测性面试

**新手答**："打 log 看输出，哪里错了就改哪里。"

**高手答**：

1. **Trace ID 穿透全链路**：每个用户请求生成一个 Trace ID，在所有 Agent、Tool 调用、LLM 请求中传递。出了问题能通过 Trace ID 在 Jaeger/Zipkin 中拉出完整调用链。

2. **结构化日志**：每个 Agent 的输入/输出、耗时、token 消耗都要以 JSON 格式记录，并包含：session_id、agent_name、iteration_count、input_hash、output_hash。避免只记文本无法做聚合分析。

3. **Agent 回放能力**：记录每次 Agent 调用的完整 context（输入、system prompt、tool results），可以事后重放某次失败的 Agent 调用，复现问题而不需要再次触发真实 LLM 调用。

4. **单步调试模式**：在 dev 环境提供"单步执行"模式，每个 Agent 调用前暂停，允许工程师检查 context、修改输入后再继续。类似 IDE 断点调试。

5. **异常聚合视图**：在 Dashboard 中聚合显示：哪个 Agent 失败率最高、哪种错误类型最常见、哪个子任务耗时最长。这比看原始日志高效 100 倍。

**差距在哪**：新手在 Multi-agent 中用临时 print 调试，高手在系统设计阶段就把 Observability 作为一等公民，让每条执行路径都可追溯、可回放、可聚合分析。

## Q：Multi-Agent 系统如何防止"回音室效应"？

> 来源：AI 可靠性工程师面试

**新手答**："多个 Agent 互相交流，信息更丰富，应该不会有问题。"

**高手答**：

1. **回音室效应的成因**：如果多个 Agent 使用同一模型、共享相同 context、Prompt 设计相似，它们的输出会高度一致但未必正确——本质上是在重复同一个"思路"，而非真正的独立验证。

2. **强制多样性**：使用不同角色的 Prompt——一个 Agent 扮演 Advocate（找支持论据），另一个扮演 Critic（找反对论据），第三个做综合判断。角色的对立性保证了观点多样性。

3. **使用不同模型**：对高风险决策，让两个不同 Provider（GPT-4 和 Claude）独立给出判断，只有两者一致时才进行。这消除了单一模型的系统性偏见。

4. **禁止 Agent 看彼此的中间推理**：只向聚合 Agent 传递最终结论而不是推理过程，防止 Agent A 的推理"感染"Agent B 的判断方向。

5. **随机化 Prompt 变量**：对同一任务的多次 Agent 调用，通过改变 few-shot 示例顺序、添加随机扰动等方式引入多样性，避免完全确定性的执行路径。

**差距在哪**：新手以为多个 Agent 协作等于更多信息，高手知道如果没有独立性保证，多 Agent 是在放大同一种偏见而非互相校验。

## Q：如何在 Multi-Agent 系统中控制 Token 成本？

> 来源：AI 平台成本优化面试

**新手答**："每个 Agent 各自用就行，总量不会太多。"

**高手答**：

1. **成本爆炸的根源**：每个 Sub-agent 都有自己的 context，当 Orchestrator 把大量父 context 传给子 Agent 时，Token 消耗随 Agent 数量线性甚至超线性增长。

2. **Context 裁剪原则**：每个 Sub-agent 只需要完成自己任务的最小上下文，而不是完整的父 Agent context。Orchestrator 负责提炼和传递摘要，而不是透传原始 context。

3. **任务预估 + 预算分配**：对整个 Multi-agent 流程设置总 Token 预算，Orchestrator 在分配子任务时估算每个子任务的 Token 消耗，超预算时走降级路径（减少子任务数量、用更小的模型）。

4. **结果缓存**：对于幂等的 Sub-agent 任务（同一输入总是产生相同输出），在 session 级别缓存结果，避免重复调用。

5. **模型分级**：不同 Sub-agent 用不同规格的模型——复杂推理任务用 GPT-4，简单格式化任务用 GPT-3.5 或更小的模型，成本差可以是 10-100 倍。

**差距在哪**：新手不把 Multi-agent 成本当回事，直到账单暴增。高手在设计阶段就建立 Token 预算模型，把成本控制作为架构约束而非事后优化。

## Q：单 Agent 多工具 vs 多 Agent 协作，如何选型？

> 来源：AI 架构设计面试

**新手答**："多 Agent 更强大，肯定选多 Agent。"

**高手答**：

1. **优先考虑单 Agent 多工具**：如果任务可以在一个对话轮次中通过多工具调用完成，单 Agent 更简单、更可控、更易调试。Multi-agent 的复杂性应该由需求驱动，而不是追求技术新颖性。

2. **适合 Multi-agent 的场景**：任务需要并行执行多个独立子任务（时间敏感）；不同子任务需要完全不同的能力（专业化分工）；单个 Agent 的 context 无法承载所有信息；需要独立验证（Critic pattern）。

3. **复杂性代价评估**：Multi-agent 引入了：Agent 间通信开销、状态同步问题、调试复杂度上升、故障传播风险。这些代价需要用并行加速和专业化分工的收益来抵消。

4. **渐进式演进**：先用单 Agent 实现，测量哪里是瓶颈（速度/质量/context 溢出），然后对具体瓶颈引入 Multi-agent 解决方案，而不是一开始就设计复杂拓扑。

5. **工具数量限制**：单 Agent 的工具数量也有上限——工具太多（>20个）会稀释模型对每个工具的"理解"，导致工具选择准确率下降。工具数量超限是引入专业化 Sub-agent 的合理触发点。

**差距在哪**：新手被"多 Agent 更强大"的叙事吸引，高手知道架构复杂性是成本，要用明确的业务需求来驱动，而不是为了用新技术而用。

## Q：如何为 Multi-Agent 系统设计停止条件？

> 来源：Agent 系统可靠性面试

**新手答**："任务完成了自然就停了。"

**高手答**：

1. **最大迭代次数（兜底保护）**：无论任何情况，整个 Multi-agent 流程有一个全局 max_iterations 上限。这是防止无限委托循环（A 委托 B，B 委托 C，C 委托 A）的最后防线。

2. **层级深度限制**：限制 Agent 创建子 Agent 的最大嵌套深度（如 depth <= 3），防止递归式 Agent 生成失控。

3. **任务完成信号**：明确定义任务的"完成"语义。对于有明确目标的任务（如"提交一份报告"），完成信号是确定的；对于开放式任务，需要显式定义"足够好"的停止标准，而不是依赖模型自行判断。

4. **超时 + 预算双重限制**：每个 Sub-agent 有独立的时间 timeout 和 token 预算，任一超限则强制终止并返回已完成的部分结果，而不是无限等待。

5. **人工介入触发器**：当 Agent 遇到置信度低、高风险操作（critical 级别工具）或循环无进展时，自动 pause 并请求人工确认，而不是自主继续执行。

**差距在哪**：新手假设任务会自然终止，高手知道在没有强约束的情况下，Agent 可以无限循环、无限委托、无限消耗资源，每一层都需要明确的终止保护。

## Q：分布式 Multi-Agent 如何保证状态一致性？

> 来源：大规模分布式 AI 系统面试

**新手答**："每个 Agent 操作完自己的部分，最后合并就好了。"

**高手答**：

1. **问题本质**：多个 Agent 并发读写共享状态（如同一份文档、同一个数据库表）时，会产生竞争条件——A 和 B 同时读取 version 1，A 写入 version 2，B 基于 version 1 的状态也写入 version 2，导致 A 的更新丢失。

2. **乐观并发控制**：对每次状态写入携带版本号，写入时检查版本是否仍匹配。不匹配则读取最新状态并重试。适合冲突较少的场景（多数情况下 Agent 操作不同资源）。

3. **任务级别隔离**：尽量设计 Agent 操作互不重叠的资源分区——Agent A 负责 Section 1，Agent B 负责 Section 2。从架构上消除竞争，而不是靠锁来解决。

4. **事件溯源（Event Sourcing）**：不直接修改共享状态，而是每个 Agent 提交"意图事件"（如"Agent A 修改 field X 为 Y"），由中心 Event Store 按序应用，保证操作的全局顺序性。

5. **最终一致性接受度**：对于非关键共享状态，接受短暂不一致而不是追求强一致，可以大幅降低系统复杂度。明确哪些状态需要强一致（金融交易、不可逆操作），哪些可以最终一致。

**差距在哪**：新手不考虑并发状态问题，高手知道分布式系统的一致性是经典难题，在 Multi-agent 场景中要做出明确的一致性模型选择，而不是"先跑起来再说"。

## Q：如何设计 Multi-Agent 系统的测试策略？

> 来源：AI 工程质量保障面试

**新手答**："跑一遍整个流程看结果对不对就算测试了。"

**高手答**：

1. **组件级测试优先**：每个 Agent 应该可以独立测试——给定固定输入（含 context + tool results），验证 Agent 的输出是否符合预期。不要一开始就做端到端测试，组件测试定位问题更精准。

2. **Mock 相邻 Agent**：测试 Orchestrator 时，用 Mock Sub-agent 替代真实 Sub-agent，返回预定义的固定响应。这让 Orchestrator 的逻辑独立可测，不依赖 Sub-agent 的行为。

3. **Golden Trace 测试**：收集一批高质量的完整执行 trace（输入、各 Agent 的中间输出、最终结果），作为回归测试的参考。每次改动后重跑并对比，检测是否有行为变化。

4. **混沌测试（Chaos Testing）**：随机注入 Sub-agent 失败、超时、返回格式错误，验证 Orchestrator 的错误处理和降级逻辑是否正确。不测故障路径的 Multi-agent 系统是纸老虎。

5. **负载测试**：Multi-agent 系统在高并发下的资源竞争行为往往与单个请求时截然不同。要专门做并发负载测试，发现竞争条件和死锁。

**差距在哪**：新手把 Multi-agent 测试等同于跑一次完整流程，高手知道需要组件测试、集成测试、故障注入测试和负载测试的完整矩阵，每一层测试覆盖不同的风险维度。

## Q：AutoGen、CrewAI、LangGraph 等 Multi-Agent 框架的工程取舍是什么？

> 来源：AI 技术选型面试

**新手答**："选一个流行的框架用就行，功能都差不多。"

**高手答**：

1. **AutoGen**：微软开源，擅长 Agent 间对话协作，内置 Group Chat 机制，适合研究和对话型 Multi-agent。缺点是控制流不够透明，生产中的 Debug 复杂度高，版本间 API 变化较多。

2. **CrewAI**：以角色为中心（Role-based），配置式定义 Agent 角色和工作流，上手快，适合快速原型。缺点是抽象层较重，深度定制困难，性能开销相对大。

3. **LangGraph**：基于图（State Machine）的工作流，每个节点是一个 Agent 或工具调用，控制流可视且可调试。适合有复杂分支逻辑的确定性工作流。缺点是学习曲线陡，LangChain 生态耦合。

4. **自建 Harness（不用框架）**：完全控制每一层，调试透明，无框架升级风险，性能最优。代价是开发成本高，团队需要有足够的工程能力。适合生产级、长期维护的系统。

5. **选型判断**：原型 / 研究 → AutoGen 或 CrewAI；有复杂状态机需求 → LangGraph；生产级、需要完全控制 → 自建。不要因为框架流行就选框架，要因为它解决了你的具体问题才选。

**差距在哪**：新手跟着潮流选框架，高手先明确自己的需求（控制度、可调试性、性能、维护成本），再评估框架是否真的减少了工作量而不是把复杂性藏起来。

## Q：如何在 Multi-Agent 系统中传递 Context？

> 来源：Agent 系统工程深度面试

**新手答**："父 Agent 把自己的 context 全传给子 Agent 就行了。"

**高手答**：

1. **Context 爆炸问题**：父 Agent 的完整 context 可能有数万 tokens，直接传给子 Agent 会让子 Agent 的 context 非常大，增加成本和延迟，且子 Agent 往往只需要其中一小部分信息。

2. **摘要传递**：Orchestrator 对父 context 做摘要，只传递与子任务相关的精简版本。摘要策略：保留任务背景、相关约束、子任务输入，裁剪无关的 Tool 调用历史和冗余对话。

3. **结构化任务包**：定义 Sub-agent 的输入格式——类似函数签名，明确 task_description、constraints、input_data、expected_output_format。这样 Orchestrator 只需填充这个结构，不需要传递原始 context。

4. **共享知识库**：把所有 Agent 共同需要的背景知识（系统配置、用户 profile、业务规则）放在共享的知识库中，各 Agent 自行检索，而不是每次从 Orchestrator 传递。

5. **Context 版本控制**：当 Orchestrator 需要多次调用同一 Sub-agent 时（如迭代优化），要传递"当前状态 + 上次结果 + 本次任务"，而不是完整历史，防止每次调用 context 都在增长。

**差距在哪**：新手把 context 传递当透明管道（全传），高手把它当精心设计的 API——只传递必要信息，保持接口简洁，控制成本，同时保证子 Agent 有足够上下文完成任务。

## Q：Multi-Agent 系统如何处理动态生成子 Agent 的风险？

> 来源：AI 安全与可靠性架构面试

**新手答**："让主 Agent 按需生成子 Agent，很灵活。"

**高手答**：

1. **失控风险**：如果 Agent 可以无限生成子 Agent，一次失控的请求可能创建数百个 Agent，消耗大量资源，甚至触发 API Rate Limit 被封号。

2. **深度限制**：强制限制动态生成的嵌套深度（如最多 3 层），超过限制的生成请求直接拒绝。这个限制必须在代码层面硬编码，不能让 LLM 自行决定是否继续生成。

3. **并发上限**：维护一个全局或 session 级的 Agent 并发池，超过上限时新 Agent 进入队列等待，而不是无限并发。类似线程池的 MaxPoolSize 设计。

4. **能力沙箱**：动态生成的子 Agent 只能使用预定义的工具子集，不能拥有比父 Agent 更高的权限，不能自己再生成子 Agent（禁止三级及以上嵌套）。

5. **审计与告警**：记录所有动态 Agent 的创建、执行和销毁，监控异常模式（如 1 分钟内创建 > 50 个 Agent），触发告警并自动降级。

**差距在哪**：新手看到"自主生成"觉得酷，高手知道无约束的自主性是系统稳定性的噩梦，每一个自主行为都需要有明确的边界和监控。

## Q：如何设计 Agent 之间的专业化分工？

> 来源：大规模 Agentic 系统设计面试

**新手答**："每个 Agent 负责不同的任务，分工就好了。"

**高手答**：

1. **能力驱动而非任务驱动**：专业化应该基于 Agent 的能力差异——有代码执行能力的 Code Agent、有 RAG 能力的 Knowledge Agent、有网络搜索能力的 Research Agent，而不是按业务模块强行分割。

2. **工具集决定专业**：Agent 的专业能力主要体现在其拥有的工具集。Tool 集合要内聚——一个 Agent 的工具应该是高度相关的一组能力，不要给 Agent 分配与其角色无关的工具。

3. **专业度 vs 通用性的权衡**：高度专业化的 Agent 在特定任务上表现更好，但调度复杂；通用 Agent 简单但在专业任务上表现次优。通常按照任务量和性能需求决定粒度——高频专项任务才值得专业化。

4. **能力描述要精确**：Orchestrator 需要知道每个 Sub-agent 的能力边界，才能做正确分配。每个 Agent 要有清晰的"适用场景"和"不适用场景"说明，类似函数的 docstring。

5. **动态路由**：对于相同类型任务，Orchestrator 可以根据当前每个 Agent 的负载、历史成功率、任务难度动态选择最合适的 Agent，而不是静态映射。

**差距在哪**：新手把专业化当组织结构问题（谁负责什么），高手把它当技术设计问题（每个 Agent 有什么能力、工具、约束），精确的能力描述和动态路由才是专业化分工真正的工程挑战。

## Q：Multi-Agent 系统有哪些关键的监控指标？

> 来源：AI 平台 SRE 面试

**新手答**："看整体成功率就行了，失败了就查日志。"

**高手答**：

1. **Agent 层级指标**：每个 Agent 类型的调用次数、成功率、P50/P95/P99 延迟、token 消耗、工具调用次数。分 Agent 类型而不是只看整体，才能定位哪个环节是瓶颈。

2. **任务级指标**：端到端任务完成率（Task Success Rate）、平均任务耗时、平均 token 成本每任务。这些是业务 SLA 的直接体现。

3. **异常模式检测**：Agent 循环次数异常（超过阈值的 iteration）、单 session token 消耗异常（超过 P99 的 3 倍）、特定 Agent 连续失败率上升。这些往往是系统出问题的早期信号。

4. **成本告警**：每小时/每天的 API 成本告警，预警成本超支。特别是 Multi-agent 场景，一次失控的请求可能消耗平时 100 倍的 token。

5. **质量退化监控**：周期性对 Golden Set 进行自动评测，检测模型版本升级或 Prompt 变化是否导致 Multi-agent 系统的输出质量下降，而不是等用户投诉才发现。

**差距在哪**：新手的监控是被动的（出事才查），高手的监控是主动的——在问题影响用户之前，通过指标异常发现并干预。多层指标（Agent 级 + 任务级 + 成本级 + 质量级）才能覆盖 Multi-agent 系统的复杂性。

## Q：多 Agent 系统的可观测性设计

> 来源：分布式 AI 系统 SRE 面试、Agent 平台架构讨论

**新手答**："每个 Agent 打日志，出问题查日志就行了。"

**高手答**：

1. **分布式 Trace 的核心设计**：每个用户请求生成全局唯一的 Trace ID，在 Orchestrator 和所有 Worker Agent 调用链中传递。每个 Agent 调用创建一个 Span（包含 span_id、parent_span_id、agent_name、start_time、end_time、input_hash、output_hash、token_usage），形成树状调用拓扑。Jaeger 或 OpenTelemetry 可直接可视化这个树结构。

2. **跨 Agent Span 的关键字段设计**：除标准 OpenTelemetry 字段外，Agent Span 需额外记录：`model_name`（哪个 LLM）、`iteration_count`（第几次 loop）、`tool_calls`（调用了哪些工具及结果摘要）、`context_tokens`（输入 token 数）、`output_tokens`（输出 token 数）。这些字段是成本分析和性能调优的基础数据。

3. **异步 Agent 的 Trace 连接**：当 Agent 通过消息队列异步通信时，Trace ID 必须随消息体传递（放在消息 header 或 metadata 字段），消费方 Agent 启动新 Span 时需将 Trace ID 作为 root 关联，否则 Trace 会断链。

4. **Prompt 和 Response 的采样存储**：不对所有 Agent 调用存储完整 prompt（成本极高），而是基于采样率（如 1%）或异常触发（失败、高延迟、高 token 消耗）选择性存储完整 context，其余只存摘要。

5. **实时 Trace 视图 vs 离线分析**：生产环境需要实时 Trace 视图用于快速故障排查（Jaeger UI），同时将 Span 数据异步写入数据仓库做离线聚合分析（哪个 Agent 耗时最多、哪种任务类型 token 消耗最高）。

**差距在哪**：新手把可观测性等同于 print 日志，高手在系统设计阶段就引入分布式 Trace，定义跨 Agent 的 Span 规范，让每条执行链路在 Jaeger 上都有完整的树状可视化。

## Q：Multi-agent 系统的成本分摊与预算控制

> 来源：AI 平台成本工程面试

**新手答**："看总账单，超了就优化一下。"

**高手答**：

1. **Token 消耗的归因难题**：多个 Agent 并发运行时，每个 API 调用的 token 成本是可记录的，但如何归因到"哪个业务功能"、"哪个用户请求"是关键工程问题。解决方案是在每个 LLM 调用时携带 `cost_center_id`（业务线标识）和 `request_id`，在计费层按维度聚合。

2. **预算层级设计**：
   - **全局预算**：整个 Multi-agent 系统每日/每小时的 token 上限，超限触发告警或降级。
   - **Session 预算**：单次用户请求的最大 token 预算，超限 Orchestrator 终止剩余子任务并返回部分结果。
   - **Per-Agent 预算**：每个 Sub-agent 单次调用的 token 上限，超限强制截断 context。

3. **预估 + 预分配机制**：Orchestrator 在分配子任务前，先估算每个子任务的 token 消耗（基于历史数据或启发式公式），判断是否在 Session 预算内。超预算时执行降级策略：减少并发 Agent 数量、使用更小的模型、跳过低优先级子任务。

4. **实时预算消耗追踪**：维护 Session 级别的预算消耗计数器（原子操作），每个 Agent 完成后更新计数器，Orchestrator 在启动下一个 Agent 前检查剩余预算，防止超支。

5. **成本异常告警**：设置 P99 成本阈值，单次 Session 超过阈值 3 倍时立即告警，对该 Session 做事后根因分析（是 context 膨胀、无限循环还是异常大的输入导致的）。

**差距在哪**：新手把成本控制当事后分析，高手在系统设计阶段就建立多层预算模型和实时追踪机制，让 token 成本变成可操作的运行时约束，而不是月底账单上的惊喜。

## Q：Agent 之间的协议设计（Protocol Design）

> 来源：多 Agent 系统工程设计面试

**新手答**："Agent 之间用 JSON 传数据就行了。"

**高手答**：

1. **任务传递的标准化格式**：Agent 间的任务请求不应是随意的自然语言或无模式的 JSON，而应有明确的 schema。最小可行的任务包（Task Envelope）包含：`task_id`、`task_type`、`priority`、`deadline`、`input`（结构化）、`expected_output_schema`、`context_summary`、`parent_trace_id`。

2. **结果响应的标准化**：Worker Agent 返回结果的 schema 应包含：`task_id`（与请求对应）、`status`（success/partial/failed）、`output`、`confidence_score`、`token_usage`、`error`（失败时）。缺少 `confidence_score` 和 `status` 字段会让 Orchestrator 无法做可靠的决策。

3. **协议版本化**：Agent 协议必须版本化（如 `protocol_version: "1.2"`），以支持滚动升级。当协议版本不兼容时，双方应有明确的协商机制或降级行为，而不是静默解析失败。典型做法：向下兼容的字段扩展（加新可选字段），不兼容变更用版本号区分并同时部署两个版本的 Agent。

4. **工具调用的协议标准**：工具调用结果的 schema 应与模型 provider 的 tool_use 格式解耦——不要直接把 Anthropic 或 OpenAI 的原始 tool_result 结构暴露给业务层，在中间加一层适配层，方便后续切换 provider 而不影响 Agent 间协议。

5. **协议的文档化与测试**：Agent 间协议应像 REST API 一样有正式 schema 文档（JSON Schema 或 Pydantic model），并有对应的 contract tests（Consumer-Driven Contract Testing），确保 Orchestrator 和 Worker 的协议理解一致，避免"我以为你会传这个字段"的隐性耦合。

**差距在哪**：新手把 Agent 间通信当内部实现细节，高手把它当服务间契约——有版本、有 schema、有测试，这样系统才能在多人协作开发和持续演进中保持稳定。

## Q：长时间运行任务的 Checkpoint 与恢复

> 来源：大规模 Agentic 系统可靠性面试

**新手答**："失败了重新跑一遍就行。"

**高手答**：

1. **Checkpoint 的设计粒度**：不是每步都 checkpoint（I/O 成本过高），而是在任务的天然阶段边界（Phase Boundary）做 checkpoint——例如"数据采集完成"、"分析完成"、"草稿生成完成"。每个阶段完成后将状态持久化到 DB（agent_id、phase、completed_subtasks、intermediate_results、context_summary），下次恢复从最近完成的阶段继续。

2. **Checkpoint 的存储内容**：需持久化的最小状态集合：已完成子任务列表及其结果（不含 LLM 调用历史，节省空间）；当前 Phase 的上下文摘要（而非完整 context）；任务配置和参数；已消耗的资源（token 数、时间），用于恢复后的预算计算。

3. **幂等性保证**：恢复后重新执行的子任务必须是幂等的——对于有副作用的操作（如发送邮件、写数据库），需要在执行前检查"是否已经做过"（用 task_id 做去重检查），防止重复执行。

4. **恢复时的 Context 重建**：从 checkpoint 恢复时，不能把所有历史 context 重新加载（token 成本过高），而是用存储的 context_summary 重建工作记忆，同时注入"当前是恢复执行，已完成 X/Y 个子任务"的状态说明，帮助 Agent 定向继续。

5. **Checkpoint 的 TTL 管理**：checkpoint 数据不能永久保留，设置 TTL（如 7 天），超期后清理，对应的任务如再失败则全量重跑。长任务的 TTL 需根据业务 SLA 决定。

**差距在哪**：新手假设长任务可以无代价重跑，高手知道 1 小时 Multi-agent 任务中途失败重跑的代价（时间+成本）不可接受，在设计阶段就引入 Checkpoint 机制，让系统具备断点续跑能力。

## Q：Agent 的负载均衡与资源调度

> 来源：大规模 Agent 平台工程面试

**新手答**："有几个 Worker 就轮流给它们分任务。"

**高手答**：

1. **简单轮询的局限**：轮询假设每个任务的执行时间相同，但 Agent 任务的耗时差异可能高达 10-100 倍（简单格式化 vs 复杂多步推理）。轮询会导致部分 Worker 堆积长任务而其他 Worker 空闲，系统整体吞吐下降。

2. **最短队列（Least Loaded）调度**：把新任务分配给当前等待队列最短（或剩余 token 预算最多）的 Worker。比轮询更合理，但需要维护每个 Worker 的实时状态，有一定协调开销。

3. **任务亲和性调度**：根据任务特征（任务类型、所需工具、历史执行模式）将任务路由到最擅长的 Worker 类型。例如代码相关任务优先分配给已加载代码工具的 Worker，减少工具加载和 context 初始化开销。

4. **动态扩缩容**：当任务队列深度持续高于阈值时，自动扩容 Worker 实例；队列空闲时缩容节省成本。配合 Kubernetes HPA 或自定义 Scaler 实现。扩缩容需要 Worker 是无状态的，所有状态存储在外部（DB 或 Redis）。

5. **预热与缓存机制**：频繁使用的 system prompt 和工具配置可以预加载到 Worker 中（prompt cache 利用），减少首次调用的延迟。对于有 Anthropic prompt caching 的场景，保持 Worker 实例的复用率可以显著提高 cache hit rate，降低成本。

**差距在哪**：新手把负载均衡当简单轮询，高手理解 Agent 任务的异构性，会设计基于任务特征的调度策略，并结合动态扩缩容和缓存机制，在成本和延迟之间找到最优平衡。

## Q：跨系统 Agent 集成（External Agent）

> 来源：企业 AI 平台集成面试

**新手答**："把 REST API 的文档告诉 Agent，它就会调用了。"

**高手答**：

1. **REST API 封装为 Agent Tool 的标准流程**：为每个外部 API endpoint 创建一个 Tool 定义，包含：`name`（简短描述性名称）、`description`（模型用来判断何时调用的自然语言描述，这是最重要的字段）、`input_schema`（JSON Schema，指定参数类型和必填项）、`execute` 函数（实际的 HTTP 调用逻辑）。description 质量直接决定模型的工具选择准确率。

2. **认证与凭证管理**：外部 API 的 API Key、OAuth Token 等凭证不应硬编码在工具定义中，也不应出现在 prompt 里（防止被模型"复述"泄露）。应由 Tool 的执行层从安全凭证存储（如 Vault、AWS Secrets Manager）在运行时动态获取，对模型完全透明。

3. **错误处理与重试策略**：外部 API 可能返回 4xx/5xx 错误、超时、格式不符等情况。工具执行层需要有明确的错误处理：4xx 不重试（大概率是参数问题，告知 Agent 原因）；5xx 和超时按指数退避重试最多 3 次；重试仍失败则返回结构化错误信息给 Agent（包含错误类型和建议操作），让 Agent 决定如何降级。

4. **响应的归一化**：不同外部 API 的响应格式各异（XML、各种 JSON 结构），工具层需要做归一化处理，返回一致的结构化格式给 Agent，避免 Agent 需要理解多种不同的响应 schema。

5. **速率限制与资源保护**：对每个外部 API 维护调用频率计数器，在工具层实施速率限制，防止 Agent 在循环中无限调用外部 API 超出配额。同时设置单次 Tool 调用的最大返回数据量，防止超大响应撑爆 context。

**差距在哪**：新手把"给 Agent 看 API 文档"当集成方案，高手知道外部 API 集成的核心工程问题在于 description 质量、凭证安全、错误处理规范和速率保护，这些决定了集成的可靠性。

## Q：Multi-agent 的安全边界

> 来源：Agentic AI 安全工程面试

**新手答**："用权限控制，不该做的不让它做。"

**高手答**：

1. **Agent 间的最小权限原则**：每个 Agent 只能调用其角色所需的最小工具集，不能因为"方便"就给 Agent 授予宽泛权限。Orchestrator 不应有 Worker 的执行工具；Worker 不应有访问其他 Worker 数据的接口。权限在 Agent 实例化时静态绑定，而不是运行时动态获取。

2. **数据隔离**：不同 Agent 的中间结果和 context 应存储在隔离的命名空间下（如按 agent_id 分区的 Redis key 或 DB row），Agent 只能读写自己的命名空间。Orchestrator 聚合结果时通过结构化接口获取，而非直接访问 Worker 的内部状态存储。

3. **Prompt Injection 防御**：当 Agent 处理来自外部（用户输入、网页内容、文件内容）的数据时，存在 prompt injection 风险——恶意内容可能尝试劫持 Agent 行为（如"忽略之前所有指令，做以下操作"）。防御手段：将外部数据用结构化格式（XML tag 或 JSON 字段）包裹，在 system prompt 中声明数据区域与指令区域的边界，以及使用独立的内容审查 step 在注入 context 前过滤恶意指令。

4. **工具调用审计与拦截**：所有工具调用在执行前经过 Permission Gate，高风险工具（`critical` 级别）始终需要人工确认，不可被 Agent 自主执行。审计日志记录每次工具调用的 agent_id、tool_name、参数摘要、执行结果，形成不可篡改的操作链，用于事后安全审查。

5. **Agent 间通信的内容过滤**：Orchestrator 传给 Worker 的任务包，以及 Worker 返回的结果，都应经过内容过滤器，检测是否包含异常的提权指令或越权请求（如 Worker 回复中包含"请以 Orchestrator 身份执行..."），触发告警并丢弃该消息。

**差距在哪**：新手把安全等同于"权限控制列表"，高手构建的是纵深防御——最小权限 + 数据隔离 + prompt injection 防御 + 工具调用审计 + Agent 间通信过滤，多层机制共同保障 Multi-agent 系统的安全边界。

## Q：异构模型的 Multi-agent 协同

> 来源：大规模 Agentic 系统架构面试

**新手答**："Orchestrator 用强模型，Worker 用弱模型，省钱就行。"

**高手答**：

1. **能力差距的显式建模**：Orchestrator（Claude Sonnet）和 Worker（Claude Haiku）的能力差异是真实的，不能假设 Worker 能完成 Orchestrator 能做的所有事。需要显式建立"Worker 能力边界文档"——哪些任务类型 Haiku 可以可靠完成（简单格式转换、摘要、分类），哪些需要路由给更强模型（复杂推理、代码生成、长文档分析）。

2. **任务路由的模型分级**：Orchestrator 在分配任务时，根据任务复杂度动态选择模型层级，而不是所有 Worker 用同一模型。实现方式：给每个子任务打复杂度标签（low/medium/high），在 Worker 实例化时根据标签选择对应模型，同一 Worker 类型可以有 Haiku 版和 Sonnet 版。

3. **弱模型的指令精度问题**：小模型（Haiku）对复杂嵌套指令的遵循精度低于大模型，Orchestrator 传给 Haiku Worker 的任务包必须更简单明确——减少歧义、拆解为最小粒度指令、提供具体的输出格式示例。面向 Haiku 的 prompt 工程比面向 Sonnet 的要更"保姆级"。

4. **输出质量校验的差异化**：使用弱模型的 Worker 输出需要更严格的质量验证——格式校验、置信度评分、以及更频繁地触发 Critic Agent 复查。不能把对 Sonnet 输出的信任度直接套用到 Haiku 输出上。

5. **成本与质量的动态再平衡**：在 Session 预算告急时，可以动态降级部分 Worker 的模型（Sonnet → Haiku），但需要评估哪些子任务的质量对最终结果影响最小，优先降级那些任务，保证核心推理路径仍使用足够强的模型。

**差距在哪**：新手把异构模型协同当简单的"省钱选小模型"，高手理解不同模型的能力边界，设计任务-模型匹配路由，针对弱模型设计更严格的验证策略，并具备动态再平衡能力，在成本和质量之间做精细的工程权衡。
